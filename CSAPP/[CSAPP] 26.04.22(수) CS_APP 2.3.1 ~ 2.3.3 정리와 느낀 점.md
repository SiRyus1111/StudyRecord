## 1. Unsigned Addition - 부호없는 정수의 덧셈

### 정리

1. $w$비트의 부호없는 정수 두 개($x, y$)를 더하게 된다면 
$x$, $y$의 범위는 
$0 \leq x,\;y < 2^w$,
$0 \leq x,\;y \leq UMax$이다.
그리고 두 값을 더한 값($s$)의 범위는 
$0 \leq s \leq 2^{w+1} - 2$,
그러니까
$0 \leq s \leq 2UMax$가 된다.
그래서 결국 범위를 넘어서는 부분(오버플로우가 발생하는 부분)은 
$s \bmod 2^w$으로  절삭이 되는 식으로
부호없는 정수의 덧셈이 이루어진다.
- 예시 ) $w = 4$
$12 +_w^u 9 = ([1100] + [1001]) \bmod 2^w = [10101] \bmod 2^w = 21 \bmod 16 = 5$
$[1100] + [1001] = [10101]$이지만, 4비트의 정수이므로 최상위 비트는 절삭되어서 $[0101]$이 된다.
- $0 \leq x + y \leq UMax$라면, $w+1$비트로 나타냈을 때,
최상위 비트는 무조건 0이 된다. 
즉, 원래값 그대로 $w$비트 안에 표현 가능하다.
- 그리고, $UMax < x + y < 2^{w+1}$이라면, $w+1$비트로 나타냈을 때,
최상위 비트는 무조건 1이 된다.
이 상태에서 $w$비트만 남기고 최상위 비트를 절삭한다면, 
결과적으로 $2^{w}$를 뺀 것과 동일하다.
- 결국, $+_w^u$는 이렇게 정의할 수 있다.
$0 \leq x, y \leq UMax$
$x +_w^u y = 
\begin{cases} 
x + y & (x + y \leq UMax)(일반적인\; 경우) \\
x + y - 2^w & (UMax < x + y < 2^{w+1})(오버플로우가\; 발생한\; 경우) 
\end{cases}$
- 즉, 오버플로우가 발생하지 않았다면 일반적인 덧셈과 결과가 같고,
오버플로우가 발생했다면 $2^w$를 뺀 값이 결과가 된다.

2. 부호없는 정수의 덧셈의 오버플로우(overflow)의 판별
- 두 수를 연산한 결과가 
비트가 자료형의 크기, 값이 $UMax$보다 큰 경우, 
앞에서처럼 절삭이 발생한다.
이걸 오버플로우라고 한다.
- 두 수를 더한 결과가 오버플로우가 발생했는지 값을 보고 알 수 있는 방법은,
$s = x +_w^u y$라고 할 때,
$s < x$ 
$s < y$ 
라면 오버플로우가 발생한 것이다.
물론 오버플로우가 발생했다면 두 경우 전부 참이 된다.
앞에서의 예시에서도 
$9 +_w^u 12 = 5$, 
$5 < 9$, 
$5 < 12$
이므로 오버플로우가 발생했다.
- 오버플로우가 발생하지 않았다면 값($s$)은 
일반적인 정수 덧셈과 동일하므로
$x + y \geq x$
즉 $s \geq x$
의 조건을 만족하게 된다.
- 만약 오버플로우가 발생했다면, 값($s$)은 $s = x + y - 2^w$가 된다.
- 그런데 $y < 2^w$ (즉 $y \leq UMax$)이므로
그렇다면 $y - 2^w < 0$이다.
- 따라서 
$x + (y - 2^w) < x$
$s < x$
이라는 것과 다름 없다.
- 결과값은 원래의 피연산자보다 더 작아질 수 있다는 것이다.
- 즉, $s < x$이라면 오버플로우가 발생한 것이다.
물론, 대칭적으로 $s < y$인 경우에도 적용 된다.

3. 부호없는 정수의 부정(negation)
- 부호없는 정수의 $+_w^u$에 대한 역(inverse)이라고 생각하면 된다.
- 즉, 어떤 비트열 $x$가 있을 때 더해서 0이 되는 비트열이다.
- 값의 관점에서는,
$-_w^u x = 
\begin{cases} 
x & (x = 0) \\
2^w - x & (x > 0)
\end{cases}$
이렇게 정의될 수 있다.
- 그리고 비트열의 관점에서는 비트열 $x$를 $\sim x + 1$해서 만들어진다.
- 이 값은 $x$와 $\bmod 2^{w}$로 절삭이 되며 더해졌을 때,
0이 되기 때문에 부호없는 정수의 역이 된다.
$2^{w}$는 어차피 절삭이 되는 수이므로,
$(x + 2^{w} - x) \bmod 2^{w} = 0$
- 물론 이 $-_w^ux$의 범위는 $0 < 2^{w} - x < 2^{w}$까지이다. 

### 느낀 점 & 배운 점

결국, 이번 파트는 부호없는 정수의 연산이 아니라,
**부호없는 정수의 연산에서의 오버플로우**가 메인이었던 것 같아.

그냥 `일반적인 정수의 연산` + `오버플로우를 고려해서 최상위 비트를 절삭`
같은 느낌이었어.

근데 솔직히 이것도 전에 배웠던 정수 비트 표현의 절삭과 상당히 유사한 것 같아.
단지 관점의 차이일 뿐.
- 정수 비트 표현의 절삭은 그저 더 작은 워드 크기의 자료형에 기존 비트들 담았을 때의,
정수의 표현으로써의 관점,
$x \bmod 2^k$이고,
- 부호없는 정수의 연산은 현재 워드 크기의 자료형에서 넘치는 비트를 제외했을 때의,
산술 연산으로써의 관점,
$s \bmod 2^w$인 것 같아.

결국 수식은 사실상 같은 느낌인데, 
특정 비트들의 절삭이라는 작용도 같은데, 
목적이 다른 느낌이야. 

> 더 작은 자료형에 맞춰 넘치는 비트 잘라내기
vs 기존 자료형에서 산술 연산의 결과에 맞춰 넘치는 비트 잘라내기

> 단순 정수 표현 vs 정수의 산술 연산

인 것 같아.

그리고 결국 오버플로우의 문제인,
내가 기대한 값과 결과값이 다를 수 있다는 문제를 확실히 마주친 것 같아.

기존의 덧셈에 대한 상식으로는
$0 \leq x, y < 2^w$일 때
$x + y < x$
인게 말이 안되니까.

컴퓨터의 unsigned 덧셈은 결국,
우리의 일반적인 자연수 덧셈이 아니라
항상 $2^w$ 이상이면 안되는,
$s \bmod 2^w$를 고려하는
모듈러 연산($\bmod$)처럼 동작하는 것 같아.

2.27번 연습문제도 풀어봤는데, 
```c
#include <stdio.h>

// 만약 오버플로우가 발생하지 않는다면 1을 반환하는 함수
int uadd_ok(unsigned int x, unsigned int y) {

    unsigned int s = x + y;
    return s >= x; // s < x 라면 오버플로우 발생한 것. s >= x라면 오버플로우 발생하지 않은 것. true = 0 / false = 1이므로..
    
}

int main() {
    unsigned int x, y;
    printf("두 수를 공백으로 구분해 입력하세요 : ");
    scanf("%u %u", &x, &y);
    if (uadd_ok(x, y) == 0) printf("오버플로우입니다.");
    else printf("오버플로우가 아닙니다.");
}
```
```
두 수를 공백으로 구분해 입력하세요 : 2100000000 2200000000  // 21억 + 22억 > UMax(약 42 억)
오버플로우입니다.
```

이렇게 두 수를 더했을 때 오버플로우가 발생하지 않는다면 1, 발생한다면 0을 반환하는 함수를
구현하는 문제였어.

그런데 이거 의외로 실제 개발에 쓸만할 것 같더라.

두 수를 산술연산할 때 
오버플로우가 발생한다면 내가 의도한, 기대한 값이 나오지 않을 수 있으니 
연산하기 전에 두 값을 이 함수에 넣어보고
1을 반환해서 두 수를 더했을 때 오버플로우가 발생하지 않는다면 더하고,
0을 반환해서 두 수를 더했을 때 오버플로우가 발생하지 않는다면 더하지 않고 예외를 던지는 식으로..

그리고 내가 지금까지 
```cpp
0U - 1 < 0
```
같은 조건식 볼 때 어떻게 부호없는 정수 0에서 1을 빼는지 항상 궁금했고
그저 비트의 덧셈 느낌으로 `0x0000 + 0xffff = 0xffff(UMax)`처럼 계산했었는데,
뭔가 기존에 야매로 때려맞추던 과정이 
그게 실제로 부정이라는 개념으로 가능했다는게 참 신기했어.

여기서 -1은 결국 1과 더했을 때 0이 나오는 값.
unsigned char 기준
[0000 0001]과 더했을 때 0이 나와야 하니까,
[1111 1111]이라면 0이 나오겠지.

어차피 넘치는 최상위 비트($x_{w}$)는 절삭되고,
그래서 값은 $x mod 2^{w}$니까.

최상위 비트를 초과해서 올라간 1은 아예 고려 안하고..

이런게 확실히 수식으로 이해되니까 내 생각이 편해졌어. 
실제로 야매로 때려맞추는 일도 할 필요가 없어졌고.

## 2. Two's-Complement Addition - 2의 보수의 덧셈

### 정리

1. 부호없는 정수의 덧셈과 다르게,
$-2^{w-1} \leq x,\;y \leq 2^{w-1} - 1$
$TMin \leq x,\;y \leq TMax$
범위의 2의 보수 $x$와 $y$의 덧셈은 
두 값을 더한 값($s$)의 범위가
$-2^{w} \leq s \leq 2^{w} - 2$
$2TMin \leq s \leq 2TMax$이기 때문에,
2의 보수의 덧셈은 
표현하기에 너무 큰지($TMax < s$) 뿐만 아니라
너무 작지 않은지($TMin > s$)까지 고려해야한다.
그래서 너무 값이 커서 일어나는 오버플로우(positive overflow)와
너무 값이 작아서 일어나는 오버플로우(negative overflow)가 구별된다.
- positive overflow는 $s$가 $TMax$를 초과했을 때 발생한다.
그렇기 때문에 결과가 절삭되어, $x + y - 2^w$의 값이 된다.
- negative overflow는 $s$가 $TMin$ 미만일 때 발생한다.
그렇기 때문에 결과는 절삭되어, $x + y + 2^w$의 값이 된다.
- 그래서 $+_w^t$는 이렇게 정의할 수 있다.
$x +_w^t y = 
\begin{cases} 
x + y - 2^w & (TMax < x + y \leq 2^{w} - 2)(양수\; 오버플로우가\; 발생한\; 경우) \\
x + y & (TMin \leq x + y \leq TMax)(일반적인\; 경우) \\
x + y + 2^w & (-2^{w} \leq x + y < TMin)(음수\; 오버플로우가\; 발생한\; 경우)
\end{cases}$

2. 2의 보수의 덧셈은 어떻게 진행될까?
- 컴퓨터는 signed든 unsigned든 같은 방법으로 더한 후,
해석만 2의 보수 / 부호없는 정수로 한다.
- 그저 최상위 비트가 음의 가중치인지, 양의 가중치인지로 다르게 해석할 뿐이다.
- 그래서 2의 보수의 덧셈도 부호 없는 정수처럼 $s \bmod 2^w$와 동일한 과정으로 진행된다.
- $x +_w^t y = U2T_w(T2U_w (x) +_w^u T2U_w(y))$
$= U2T_w[(x_{w-1} 2^{w} + x) + (y_{w-1} 2^{w} + y) \bmod 2^w]$(unsigned처럼 최상위 비트($x_{w-1}$)를 양의 가중치로 취급해서 기존의 값에 더해주기)
$= U2T_w[(x + y) \bmod 2^w]$ 
$= U2T_w[s \bmod 2^w]$
이렇게, 같은 방법으로 더한 후 그저 2의 보수로 해석할 뿐이다.

3. 오버플로우의 공식들은 어떻게 도출된걸까?
- 여기서부터가 중요한데,
- 만약 $-2^w \leq s < TMin$, negative overflow라면, 
음수에 대한 나머지 연산($\bmod$)의 특성 때문에 $s \bmod 2^{w} = s + 2^w$이 되고,
결국 $s$의 값은 $0 \leq s < 2^{w-1}$이 되어서, 양수로 전환된다.
그리고 $s + 2^w = x + y + 2^w$이니 
위에서 설명한 negative overflow의 공식이 도출된다.
- 만약 $TMin \leq s \leq TMax$이라면,
여기는 정상 범위이므로 그대로 2의 보수로 정상적으로 표현할 수 있기 때문에 
$x + y$가 값이 되므로
위에서 설명한 일반적인 경우의 공식이 도출된다.
- 만약 $TMax < s < 2^{w} - 2$, positive overflow라면,
어차피 $s \bmod 2^w$해봤자 같은 값($s$)이 나온다.
하지만 그 비트열의 최상위 비트는 1이 되어서 
2의 보수 정수로 해석하게 된다면 음수 값이 된다.
그래서 기존의 값과 $-2^{w}$차이가 나게 되고,
$s - 2^{w} = x + y - 2^{w}$이니
위에서 설명한 positive overflow의 공식이 도출된다.

4. 2의 보수의 덧셈의 오버플로우 판별
- 음수 + 음수 = 0 / 양수면 negative overflow
- 양수 + 양수 = 0 / 음수면 positive overflow
- 같은 부호의 두 수를 더했는데 결과값의 부호가 바뀌면 overflow
- 정도로 이해해도 된다.

### 느낀 점 & 배운 점

정말 마의 구간이었어.. 
이전의 2.2.4 - Conversion between Signed and Unsigned 이후의
고비라고 해야하나..

결국, 2의 보수의 덧셈이란 이 문장 하나로 정리가 될 것 같아.
> 그저 unsigned의 덧셈처럼 **비트열끼리 더한 후**, 결과값을 2의 보수(Signed)로 해석한다.

이 정도.

그리고 내가 고생한 부분은 정리에서는 3번 파트인 오버플로우의 공식 도출이었는데,
이거 실제 책에서는 수식이 정말 빽빽해서 이해가 너무 안됐어..ㅋㅋ

그리고 그러고도 이해가 안되서 직접 오버플로우를 발생시켜서
더한 두 값($x,\;y$) / 더한 결과값($s$)의 10진수 표현과 비트 표현을 출력하는 프로그램을 만들어서
직접 체감을 해봤어.
```cpp
#include <iostream>
#include <bitset>

void neg_overflow() {
    std::cout << "negative overflow가 발생하는 두 signed char 형 정수 입력 : " << std::flush;
    int x, y;
    std::cin >> x >> y;

    signed char cx = static_cast<signed char>(x);
    signed char cy = static_cast<signed char>(y);

    std::cout << "\nx : " << static_cast<int>(cx)
              << " 0b" << std::bitset<8>(static_cast<unsigned char>(cx))
              << " / y : " << static_cast<int>(cy)
              << " 0b" << std::bitset<8>(static_cast<unsigned char>(cy)) << '\n';

    int sum_int = static_cast<int>(cx) + static_cast<int>(cy);
    signed char sum = static_cast<signed char>(sum_int);

    std::cout << "두 비트열의 덧셈 실행 : "
              << static_cast<int>(sum)
              << " 0b" << std::bitset<8>(static_cast<unsigned char>(sum)) << '\n';
}

void pos_overflow() {
    std::cout << "positive overflow가 발생하는 두 signed char 형 정수 입력 : " << std::flush;
    int x, y;
    std::cin >> x >> y;

    signed char cx = static_cast<signed char>(x);
    signed char cy = static_cast<signed char>(y);

    std::cout << "\nx : " << static_cast<int>(cx)
              << " 0b" << std::bitset<8>(static_cast<unsigned char>(cx))
              << " / y : " << static_cast<int>(cy)
              << " 0b" << std::bitset<8>(static_cast<unsigned char>(cy)) << '\n';

    int sum_int = static_cast<int>(cx) + static_cast<int>(cy);
    signed char sum = static_cast<signed char>(sum_int);

    std::cout << "두 비트열의 덧셈 실행 : "
              << static_cast<int>(sum)
              << " 0b" << std::bitset<8>(static_cast<unsigned char>(sum)) << '\n';
}

int main() {
    std::ios::sync_with_stdio(false);
    std::cin.tie(nullptr);

    neg_overflow();
    pos_overflow();

    return 0;
}
```
sum_int는 일반적인 정수 덧셈 결과고, 
sum은 그 결과를 8비트로 절삭한 뒤 다시 signed char로 해석한 값임.
```
negative overflow가 발생하는 두 signed char 형 정수 입력 : -100 -50

x : -100 0b10011100 / y : -50 0b11001110
두 비트열의 덧셈 실행 : 106 0b01101010
positive overflow가 발생하는 두 signed char 형 정수 입력 : 100 50

x : 100 0b01100100 / y : 50 0b00110010
두 비트열의 덧셈 실행 : -106 0b10010110

```
이렇게 결과가 나왔어.
-100 = 1001 1100
-50 = 1100 1110
1001 1100 + 1100 1110 = 0110 1010 = 106, 
negative overflow 발생해서 -100 + (-50) + 256 = 106
정확히 나오는거 보고 정말 신기하더라.

그리고 비트열을 출력해보니까 최상위 비트가 0이 되면서 
자릿수가 올라가버린 비트는 결국 절삭된다는게 체감이 정말 잘 되더라.

그리고 
100 = 0110 0100

positive overflow 발생해서 100 + 50 -256 = -106
정확히 이론대로 나오더라고..
정말 신기했어.

이것도 비트열을 출력해보니까 최상위 비트로 1이 올라가면서

## 3. Two's-Complement Negation - 2의 보수의 부정

### 정리

1. 2의 보수의 부정($+_w^t$의 역, $-_w^t$)은 
값의 관점에서는 단순히 -를 곱해서 만들 수 있지만,
$TMin$은 대응하는 양수가 없기 때문에,
그대로 $TMin$을 부정으로 쓴다.
- $-_w^t x =
\begin{cases} 
TMin & (x = TMin) \\
-x & (x > TMin)
\end{cases}$
- $(TMin + TMin) \bmod 2^w = (-2^{w-1} + (-2^{w-1})) \bmod 2^{w} = -2^{w} \bmod 2^{w} = 0$
- $x + (-x) \bmod 2^w = 0$
- 물론 비트열의 관점에서는 비트열 $x$를 \sim x + 1$해서 만들 수 있다.
- 여기서도 $[1000]$($TMin$)은 
$\sim [1000] + [0001] = [0111] + [0001] = [1000]$으로
그대로 $TMin$이 부정이다.


### 느낀 점 & 배운 점

솔직히 이 파트는 위의 2의 보수의 덧셈 파트에 붙여도 될 정도로 정말 짧았는데,
왜 별개의 파트로 분리했는지는 모르겠어.

하지만 결국, 부정이란 기존의 값과 더했을 때 0이 되는 값,
그래서 뭔가 이 개념을 사용해서 
양수의 비트열로 음수의 비트열을 쉽게 찾을 수 있을 것 같아.

그리고, 이 부정(negation)이라는 개념으로
뺄셈(subtraction)을 구현할 수 있을 것 같아.
예를 들어,
$x - y$라고 할 때,
$x - y = x + (\sim y + 1)$
이렇게.

ALU는 그저 더하기만 하는데,
실제로는 이 부정을 이용해서 뺄셈이 가능해지는 것 같아.

그러니까 결국 SUB(뺄셈) 명령어같은건
밑바닥에서는
```
NEG y
ADD x + y
```
이런 느낌으로 돌아가지 않을까 싶어.

결국, 부정(negation)은 단순히 음수를 만드는 연산이 아니라,
덧셈의 역을 만드는 연산이라는 생각이 들어.

이래서 컴퓨터가 2의 보수를 쓰는 것 같아.
이런 부정을 쉽게 만들 수 있어서.

1의 보수 시스템은 그저 $\sim x$만 하면
부정이 되지만,
결국 -0과 +0으로 0이 두 개가 존재해서
뺄셈(subtraction) 구현이 상당히 복잡해질 것 같아.

하지만 2의 보수는 단순히 $\sim x + 1$로,
가산기 하나로 덧셈 / 뺄셈 / 음수 표현 다 가능하게 되니까..
