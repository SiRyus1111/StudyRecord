## 1. unsigned와 signed는 그저 다른 방식으로 비트열을 해석하는 것일 뿐이다.

말 그대로야.
$U2T_w$든, $T2U_w$든 결국 같은 비트열의 최상위 비트를
양의 가중치로 해석할지(unsigned)
음의 가중치로 해석할지(signed)
만 다를 뿐이라는걸 체감했어.

이 점 때문에, CPU의 같은 덧셈 규칙으로도
signed와 unsigned 둘 다를 더할 수 있다는거.

실제로 지금 공부하고 있는 2.3.2챕터에서도
signed 정수의 덧셈은 결국 
unsigned 정수의 덧셈처럼 두 비트열을 더한 후 해석을 signed로 하는 것 뿐이더라.

결국 컴퓨터는, CPU는 unsigned든 signed든 일단 더하고,
컴파일러같은 프로그램이 해당 데이터에 대해 
unsigned / signed같은 해석을 덧붙이는 것이라고 느꼈어.

해당 비트열에 실행 시점에서 문자 / 숫자 / 코드같은 의미가 실리는건
CPU에서가 아니라 결국 컴파일러 / 어셈블러 같은 부분인 것 같아.

하드웨어가 아니라 소프트웨어.
CPU는 일단은 그저 더할 뿐인 느낌..

## 2. 비트열의 확장과 절삭

난 지금까지 비트열의 확장과 절삭에 대해
하나도 몰랐어.

그저 그냥 자료형이 확장되면 값이 보장되니~
자료형이 절삭되면 값이 보장되지 않느니~

하는 정도였어.

하지만, 이번에 확실히 체감이 된 것 같아.

확장은 기존 값을 안 건드리는 방식으로 확장이 일어나.
- unsigned는 기존의 값에서 변하지 않게 그저 확장된 부분을 0으로 채울 뿐이고,
- signed는 기존의 값에서 변하지 않게 최상위 비트와 확장된 부분의 가중치를 같게 하고..
그러니까 기존의 최상위 비트의 음의 가중치와 확장된 부분 + 기존의 최상위 비트의 총합 가중치가
같을 수 있게 확장을 하는..

그리고 절삭은 정말 신기했어.
- unsigned든 signed든 일단 $x \bmod 2^k$ 방식으로 k비트만 남기고 전부 날려버린 후,
- unsigned는 최상위 비트의 가중치를 양수로,
- signed는 최상위 비트의 가중치를 음수로 해석한다는게

너무 신기했어.

여기서는 절삭과 함께 충분히 의미가 망가질 수 있겠다는 사실도 확실히 체감했어.

## 3. C언어에서는 signed와 unsigned를 함께 연산을 할 때 전부 unsigned로 바꾼다.

이 점 때문에 생기는 문제가 정말 놀라웠어.

```cpp
0U > -1
```
이런 조건문이 
```cpp
0x00000000 > 0xffffffff
0 > UMax
```
이렇게 -1이 $UMax$로 해석되어서 결과가 false가 나오지를 않나ㅋㅋ

```cpp
5U - 6 < 0
```
이런 조건문이
```cpp
5 + (-6) < 0
[0101] + [1010] < 0
[1111] < 0
UMax < 0
```
이렇게 5 - 6의 결과가 무지막지하게 커지는 식으로 해석되어서
결과가 false가 나오지를 않나ㅋㅋ

이거 진짜 조심해야겠다 싶었어..

책에서 말한 것처럼 unsigned를 안 쓰는게 가장 확실하고 좋고 신경 덜써도 되는 해결법인 것 같아.

만약 unsigned와 signed를 어쩔 수 없이 같이 사용해야하는 경우에는
이런 문제가 발생할 수 있으니 조심조심 코드를 짜야할 것 같고.
