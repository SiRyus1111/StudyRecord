## 2. Data Sizes - 데이터(자료형)의 크기


### 요약

1. 해당 프로그램이 가질 수 있는 가상 주소값의 범위는 해당 프로그램(?)의 워드 크기에 달렸다.
- 워드가 32비트(4바이트)라면, 4바이트로 표현될 수 있는 주소값의 갯수는 약 40억개이므로,
물리적 메모리의 총 크기와 관련없이 가상 메모리를 최대 4GB밖에 사용하지 못한다.
- 워드가 64비트(8바이트)라면, 8바이트로 표현될 수 있는 주소값의 갯수는 약 $$1.84 
\times 10^{19}$$개이므로,
가상 메모리를 최대 16EB(ExaByte, 엑사바이트)까지도 사용할 수 있다.

2. (워드 크기)32bit 기기에서 컴파일된 프로그램은 (워드 크기)64bit 기기에서 실행될 수 있는 호환성이 있다.
- 하지만 그 반대는 불가능하다.

3. (C언어 기준) 자료형은 크게 정수(integer, 2 / 4 / 8 바이트)와 
부동 소수점(floating-point, 4 / 8 바이트)로 구분되고,
양수 / 음수 둘다 표현가능한 signed의 자료형과 
음수는 표현 불가능한(0은 가능) unsigned의 자료형으로 나뉜다.

4. (C언어 기준) 32bit 환경과 64bit 환경에 따라 long과 포인터가 사용하는 범위는 다르다.
- 32bit 환경인 경우 - long = 4바이트 / 포인터(void*) = 4바이트
- 64bit 환경인 경우 - long = 4 / 8바이트(환경에 따라 다름) / 포인터(void*) = 8바이트
- 물론 이 차이도 워드 크기가 32bit냐, 64bit냐의 차이 때문에 발생한다.
- 포인터의 크기는 워드 크기를 따라간다.

5. 대부분의 자료형은 별도로 unsigned로 명시를 하거나 하지 않는 이상 
signed의 자료형으로 인코딩한다.
- 여기서 char은 예외이다.
- char은 C언어 기준, 명시를 안하면 signed 형이라고 보장을 해주지 않는다.
- 그래도 char 형이 signed든 unsigned든 프로그램의 행동에는 영향을 거의 주지 않는다고 한다.
- 하지만 분명히 영향을 주는건 맞다.

### 느낀 점 & 배운 점

예전에 메이플스토리의 클라이언트가 32bit에서 64bit로 바뀌었을 때 렉이 그렇게 줄었는지 이해됐어.

하드디스크가 아닌 이전보다 큰 용량의 메모리에 캐싱해놓은 스킬 데이터를 가져오니
렉이 확연히 줄었던 것 같아.

이게 확실하진 않지만,
내 생각에는
그래도 워드의 크기가 꽤 영향을 미쳤다고 생각해.

이렇게 실행되는 환경에 따라 사용가능한 가상 메모리 주소의 범위가
매우 크게 달라지니,
이런 것들도 확실히 고려가 필요할 것 같다는 생각도 들었어.

그리고 운영체제 / 프로그램이 
32bit / 64bit로 분류되는거 보고
왜 그렇게 작은 단위로 분류되는지 항상 의문이 들었는데,
결국 워드의 크기 차이구나.
이렇게 이해됐어.

그리고 또 신기했던게,
포인터와 long 형에서도 알 수 있듯,
자료형의 크기 자체가 실행 환경에 따라 바뀔 수 있다는 것도 인상깊었어.
> 내가 64bit 환경에서 짠 소스 코드가

이런 생각도 들었어.

## 3. Addressing and Byte Ordering - 주소 지정과 바이트 정렬

### 요약

1. 컴퓨터는 두 가지 형식으로 데이터를 메모리에 저장한다.
- Little endian(리틀 엔디언) - 작은 자릿수부터 앞쪽 주소에 저장
- Big endian(빅 엔디언) - 큰 자릿수부터 앞쪽 주소에 저장

2. CPU 아키텍처가 엔디언 방식을 결정하고, 운영체제는 그 위에서 그 방식을 따른다
- x86 계열은 보통 리틀 엔디언의 바이트 정렬을 사용하고, 
SPARC 등의 CPU는 빅 엔디언의 바이트 정렬을 사용하고..
- 하지만 최근에는 Bi-endian이라는 처음 설정할 때에 두 가지 모두 호환되는 CPU가 많이 나온다고 한다.
물론 중간에 바이트 정렬 방식을 바꿀 수는 없다.
- 그리고 바이트 정렬은 한번 정하면 유지해야한다.

3. 평소에는 애플리케이션 프로그래머가 이런 바이트 정렬을 체감할 일은 많지 않다.
하지만 대표적으로 이 세 경우에서 바이트 정렬을 고려하게 된다.
- 네트워크 프로그래밍을 할 때 - 다른 장치(device)들과 네트워크로 통신할 때 데이터 수신 대상인 장치가
송신하는 장치와 다른 바이트 정렬을 사용하는 장치라면? 
데이터를 읽는 방향이 다르기 때문에 서로의 데이터를 제대로 이해할 수 없다.
그래서 따로 네트워크 바이트 정렬의 표준(빅 엔디언)으로 바꾸어서 송신하고,
네트워크 바이트 정렬(빅 엔디언)의 데이터를 수신하는 장치에 맞는 호스트 바이트 정렬로 바꿔서 수신한다.
- 기계어 수준의 코드를 분석할 때 - 바이트 정렬에 따라, 16진수의 메모리 주소 / 정수 값 등을 볼 때
바이트 단위로 뒤집혀서 보일 수 있다.
- 일반적인 자료형(타입) 시스템을 벗어나도록 작성된 프로그램을 작성할 때 - (C언어) 캐스팅 / 유니온 등
애플리케이션 프로그램들은 이런 경우가 잘 없고, 권장되지 않지만,
시스템 수준의 프로그래밍을 할 때는 꼭 필요하다.

### 느낀 점 & 배운 점

실제로 책에 있던 예제 코드를 한번 직접 실행해봤는데,
바이트 정렬이 확 와닫더라.
책을 보고 내가 짜본 예제 코드는 대략 이래.
```cpp
// CS:APP 2.1.3 - Figure 2.4 / 2.5 / 2.6
// 윈도우 환경에서는 리틀 엔디언을 쓰므로,
// 작은 단위의 바이트부터 작은 주소에 배치된다.

// 예 : 0x01234567 = 67 45 23 01

#include <stdio.h>

typedef unsigned char *byte_pointer;

void show_bytes(byte_pointer start, size_t len){
    for (int i = 0; i < len; i++){
        printf(" %.2x", start[i]);
    }
    printf("\n");
}

void show_int(int x){
    show_bytes((byte_pointer) &x, sizeof(int));
}

void show_float(float x){
    show_bytes((byte_pointer) &x, sizeof(float));
}

void show_pointer(void *x){
    show_bytes((byte_pointer) &x, sizeof(void *));
}

void test_show_bytes(int value){
    int ivalue = value;
    float fvalue = (float) ivalue;
    int *pvalue = &ivalue;
    
    show_int(ivalue);
    show_float(fvalue);
    show_pointer(pvalue);
}

int main(){

    while (1){
        int value;
        scanf("%d", &value);
        test_show_bytes(value);
        if (value == 0){
            break;
        }
    }

    return 0;
}
```
정수 하나를 입력받고, 
정수 / 실수 / 포인터의 바이트를
메모리 주소가 앞쪽인 바이트부터 출력하는 프로그램이야.
```cpp
12345
 39 30 00 00
 00 e4 40 46
 2c fe 5f 00 00 00 00 00
```
이렇게 입력 / 출력이 되었어.

그리고 12345는,
```
12345 = 0x00003039
```
이렇게 돼.

그리고 내가 지금 사용하는 OS는 윈도우 11 - 64bit라서,
리틀 엔디언을 사용해.
작은 자릿수가 메모리 주소 앞쪽에 저장돼.

그래서,
정수(int(4바이트))형으로 인식하고,
메모리 주소가 앞쪽인 바이트부터 출력하면,
```
39 30 00 00
```
이렇게 작은 자릿수가 먼저 출력돼.

그리고 포인터도, 64bit 환경이라서
8바이트 크기인 것을 알 수 있어.

hton*() / ntoh*() 을 쓰지 않고
이렇게 직접 바이트 정렬을 체감해보니, 신기하더라.

근데 float형 변수는 아예 다른 이상한 값이 나오더라,
그리고 책에서 설명하기론,
floating-point(부동 소수점) 자료형과 
integer(정수) 자료형의 비트열(bit-sequence)을 봤을 때,
```
0x00003039(integer) = 0000 0000 0000 0000 0011 0000 0011 1001
0x4640e400(floating-point) = 0100 0110 0100 0000 1110 0100 0000 0000

두 개를 잘 보면, 1000000111001이 겹친다.
```
이렇게 겹치는 비트가 있는데,
이게 우연이 아니라고 하네.
이건 나중에 부동 소수점 파트에서 추가로 알아볼거라고 해.
