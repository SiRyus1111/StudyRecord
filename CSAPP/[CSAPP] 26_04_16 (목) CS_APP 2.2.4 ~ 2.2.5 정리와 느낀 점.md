## 1. Conversion between Signed and Unsigned - 부호있는 정수와 부호없는 정수의 변환

### 정리

1. 일반적으로는, C언어에서 같은 비트 수(같은 워드 크기)의 
signed 자료형과 unsigned 자료형을 변환할 때는 비트열($\vec{x}$)은 그대로 둔 채,
값만 바뀐다.
- $U2T_w \doteq B2T_w (U2B_w(\vec{x}))$
- $T2U_w \doteq B2U_w (T2B_w(\vec{x}))$
- 이렇게 비트열은 변하지 않고, 단순히 읽는 방식(Unsigned냐, 2의 보수냐)만 바뀐다.
- 그래서 Unsigned의 자료형의 값이 $TMax$
를 초과할 때,
signed의 자료형의 값이 0보다 작을 때에
기대한 값과 다르게 보일 수 있다.

2. 2의 보수에서 부호없는 정수로 어떻게 값을 바꿀 수 있을까?
- 어차피 $B2U_w(T2B_w (x))$와 $B2T_w(T2B_w (x))$는 2진수($B$)를 
2의 보수($T$)로 바꾸나($B2T_w$), Unsigned($U$)로 바꾸나($B2U_w$)의 차이밖에 없다.
- 그리고, $B2T_w(T2B_w (x))$는 원래의 $T$와 똑같은 값이 나온다.
- 그렇다면, $B2U_w$와 $B2T_w$의 차이를 $B2T_w(T2B_w(x))$에 고려해준다면 $T2U_w$일 것이다.
$B2U_w (\vec{x}) = \sum_{i=0}^{w-1} x_i 2^i$
$B2T_w (\vec{x}) = -x_{w-1} 2^{w-1} + \sum_{i=0}^{w-2} x_i 2^i$
$$B2U_w - B2T_w = x_{w-1}(2^{w-1} - -2^{w-1})
= x_{w-1} 2^{w}$$
$$
B2U_w (\vec{x}) = B2T_w (\vec{x}) + x_{w-1} 2^w
$$
- 즉, $B2U_w (T2B_w (x)) = T2U_w (x) = B2T_w(T2B_w(\vec{x})) + x_{w-1}2^{w}$
$= T + x_{w-1}2^w$
- 그리고 $x_{w-1}$은 $T$에서는 음의 가중치를 가지는 비트이다.
1이라면 음수, 0이라면 양수.
- 지금까지 설명한걸 정말 쉽게 설명해보자면,
결국 $U$와 $T$는 최상위 비트를 읽는 방법만 다를 뿐인데,
그 다름 때문에 같은 비트열일 때의 어느 정도의 격차가 발생할지를 설명한 것이다.
결국, 음수인 경우에는 $2^w$의 격차가 발생한다는 것.
- 이 모든걸 종합하면,
$
T2U_w(t) = 
\begin{cases} 
t + 2^w & (t < 0) \\ 
t & (t \geq 0) 
\end{cases}
$
이게 $T2U_w$, $w$비트의 2의 보수$T$를 부호없는 정수$U$로 바꾸는 방법이다.
- 그저 최상위 비트를 $-2^{w-1}$에서 $2^{w-1}$로 바꿔서 해석하는 것일 뿐이다.

3. $T2U_w$ 요약
- 결국 2의 보수를 부호없는 정수로 바꾼다는건 
2의 보수의 음수 부분이 부호없는 정수의 $TMax$ 이상의 부분이 된다는것이다.
그저 최상위 비트가 음의 가중치에서 양의 가중치로 읽히는 것이다.
- 2의 보수(signed)의 음수 값들은 같은 비트열을 unsigned로 해석했을 때
$2^{w-1}$ 이상 $2^w - 1$구간의, $TMax$초과 $UMax$이하 구간의 값으로 보이게 된다.

4. 부호없는 정수에서 2의 보수로 어떻게 바꿀 수 있을까?
- 이건 사실상$T2U$의 반대로 보면 편하다.
- 어차피 $B2T_w(U2B_w (x))$와 $B2U_w(U2B_w (x))$는 원래 $U$였던 2진수($B$)를 
2의 보수($T$)로 바꾸나($B2T_w$), Unsigned($U$)로 바꾸나($B2U_w$)의 차이밖에 없다.
- 그리고, $B2U_w(U2B_w (x))$는 원래의 $U$와 똑같은 값이 나온다.
- 그렇다면, $B2T_w$와 $B2U_w$의 차이를 $B2U_w(U2B_w(x))$에 고려해준다면 $U2T_w$일 것이다.
$$B2T_w - B2U_w = x_{w-1}(-2^{w-1} - 2^{w-1})
= -x_{w-1} 2^{w}$$
$$
B2T_w (\vec{x}) = B2U_w (\vec{x}) - x_{w-1} 2^w
$$
- 즉, $B2T_w(U2B_w(x)) = U2T_w = B2U(U2B(\vec{x})) - x_{w-1} 2^w$
$= U - x_{w-1}2^w$
- 그리고 $x_{w-1}$은 $U$에서는 $TMax$ 초과를 표현할 때 쓰이는 비트이다.
1이면 $TMax$ 초과, 0이면 $TMax$ 이하.
- 이 모든걸 종합해보면,
$
U2T_w(u) = 
\begin{cases} 
u  & (u \leq TMax) \\ 
u - 2^w & (u > TMax) 
\end{cases}
$
이게 $U2T_w$, $w$비트의 부호없는 정수 $U$를 2의 보수 $T$로 바꾸는 방법이다.
- 그저 최상위 비트를 $2^{w-1}$에서 $-2^{w-1}$로 바꿔서 해석하는 것일 뿐이다.

5. $U2T_w$ 요약
- 결국 부호없는 정수를 2의 보수로 바꾼다는건
부호없는 정수의 $TMax$ 초과 부분이 음수가 된다는것이다.
그저 최상위 비트가 양의 가중치에서 음의 가중치로 읽히는 것이다.
- unsigned의 $TMax$를 초과하는 값들은 같은 비트열을 signed로 해석했을 때
$-2^{w-1}$ 이상 $-1$구간의, $0$미만 $TMin$이상 구간의 값으로 보이게 된다.

6. 추가 추론
- $0 \leq x \leq TMax$사이의 값들은 signed(2의 보수)로 해석하든 unsigned로 해석하든 
같은 워드 크기(비트 수)라면 바뀌지 않는다.
$T2U_w (x) = x$, $U2T_w (y) = y$

### 느낀 점 & 배운 점

이 챕터는 내 정신력과 이해력과 인내심의 한계를 시험하는 파트였어..

## 2. Signed versus Unsigned in C - C언어에서의 Signed와 Unsigned

### 정리

1. C언어 표준상으로는 아무것도 안 적은 정수는 signed가 기본이다.
- Unsigned를 명시하려면 뒤에 `U`나 `u`를 붙이면 된다.

2. C언어 표준에서는 딱 히 특별한 변환 규칙은 없지만, 
대부분의 시스템이 따르는 규칙으로 signed와 unsigned를 변환하는데,
앞서 설명한 $T2U_w$와 $U2T_w$를 사용한다.

3. C언어에서의 캐스팅(형 변환)은 직접 형을 명시해주는 명시적 형 변환과,
명시하지 않아도 암시적으로 형이 변환되는 암시적 형 변환이 있다.

4. 실제로도, 이렇게 C언어가 해당 정수들을 표현하는걸 보면,
```c
int x = -1; // 0xffffffff
unsigned int y = 2147483648; // 2^31(TMax + 1) - 0b100000 ... 000

printf("x = %u = %d\n", x, x);
printf("u = %u = %d\n", u ,u);
```
- 이런 코드를 실행했을 때
```c
x = 4294967295 = -1 // x = unsigned = signed
u = 2147483648 = -2147483648 // u = unsigned = signed
```
- 이렇게 signed / unsigned 별로 읽는 방식이 $T2U$ / $U2T$에 따라 다르게 나온다.
- $T2U_{32}(-1) = UMax_32 = 2^{32} - 1$
- $U2T_{32}(2^31) = 2^{31} - 2^{32} = -2^{31} = TMin$

5. 이렇게 직관적이지 않은 변환이 있을 수 있는 이유는,
앞에서 설명했듯이 C언어는 signed와 unsigned를 변환할 때 
그저 같은 비트 수(워드 크기)라면 비트열($\vec{x}$)은 그대로 둔 채,
값만 바꾸기 때문이다.
- C언어는 signed를 unsigned로 변환할 때, 그저 모든 비트는 양수라고 가정한다.


6. 앞에서처럼 산술 연산을 할 때 직관적이지 않은, 이해가 안되는 오차가 발생할 수 있다. 
대표적으로 이걸로 인해 `<`, `>`가 영향을 받는다.
- 대표적으로 `-1 < 0U`가 있다.
  - `0U`가 unsigned기 때문에, 앞의 `-1 = 0xffffffff`이 unsigned로 강제로 형 변환(캐스팅)된다.
  - 그런데, unsigned로 `-1 = 0xffffffff`는 $UMax$이므로, 
  - 실제로 `-1 < 0`의 결과는 True(1)지만, 출력값으로는 False(0)가 나온다.
- 이렇게 두 수를 비교할 때 한 수가 unsigned라면 강제로 unsigned로 캐스팅해버리는 문제가 발생할 수 있다.

### 느낀 점 & 배운 점

이번 파트는 이전에 배웠던 signed와 unsigned의 변환 이론을
C언어에서 어떻게 보이는지 직접 체감한 파트인 것 같아.

중간에 코드로 분명 같은 수를 %u(unsigned)와 %d(signed)로 해석했는데,
최상위 비트에 따라 아예 다른 값이 나오는게 정말 신기했어.

결국 이렇게 실제 코드에서도 signed와 unsigned에 따라 서로 형을 변환할 때에
최상위 비트에 따라 해석이 달라져서 문제가 생길 수 있다는걸 확실히 체감했어..

그리고, `<`와 `>`처럼 두 수를 비교할 때에,
한 수가 unsigned면 다른 수도 unsigned로 자동 형 변환(캐스팅)되어서
우리가 예상한 결과가 나오지 않을지도 모른다는게 정말 충격이었어ㅋㅋㅋ

결국, signed와 unsigned를 같이 사용할 때에는,
비트 단위로 생각을 해야할 것 같아.

unsigned의 값이 $TMax$를 넘기면 다른 signed 변수와 연산을 했을 때,
signed의 값이 음수라면 다른 unsigned 변수와 연산을 했을 때
우리가 직관적으로 기대한 값이 나올 수 없을테고,
변수의 값이 고정이 아니라 바뀔 수도 있으니
> 그 변수는 항상 $TMax$를 넘기지 않는가,

> 그 변수는 항상 양수가 되는가,

더 정확히는
> 그 변수는 항상 0 <= x <= TMax의 값을 가지는가

를 생각을 해보면서 코드를 짜야할 것 같아.

백준같은 온라인저지 사이트에서도 
자료형을 잘못 선택해서 오버플로우가 발생하는 문제를 많이 겪어봤었는데,

그것도 결국에는 값을 저장할 비트가 모자라서 생기는 문제였고,
이것도 결국에는 비트열을 읽는 방식에 따라 발생하는 문제인 점에서
비트 단위의 문제라는 공통점이 있는 것 같아.

그래서 최종 결론으로는
> 결국 우리가 정수(signed, unsigned)를 다룰 때는 
비트 레벨까지 내려가야 한다.

> 적어도 signed와 unsigned를 섞어 쓰거나,
형 변환과 오버플로우가 걸리는 부분에서는
비트 레벨까지 내려가서 생각을 해야한다.

> 그래서 개발자는 이 변수가 signed로 읽힐지, unsigned로 읽힐지도 생각하면서
코드를 작성해야한다.

라고 할 수 있을 것 같아.
