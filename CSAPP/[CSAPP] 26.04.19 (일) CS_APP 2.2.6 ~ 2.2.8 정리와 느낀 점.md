## 1. Expanding the Bit Representation of a Number - 정수 비트 표현의 확장

### 정리

1. 단순히 unsigned 자료형을 더 큰 워드 크기의 unsigned 자료형으로 확장하는건 
그저 빈 공간을 0으로 채우면 된다.
$\vec{u} = [u_{w-1}, u_{w-2}, ..., u_{0}]$
$\vec{u'} = [0,0, ... , u_{w-1}, u_{w-2}, ..., u_{0}]$
- 이 연산(Operation)을 zero extension이라고 한다.

2. 2의 보수(signed) 자료형을 더 큰 워드 크기의 2의 보수(signed) 자료형으로 확장하려면
빈 공간을 최상위 비트와 동일하게 채우면 된다.
$\vec{x} = [x_{w-1}, x_{w-2}, ..., x_{0}]$
$\vec{x'} = [x_{w-1}, ..., x_{w-1}, x_{w-1}, x_{w-2}, ..., x_{0}]$
- 이 연산을 sign extension이라고 한다.

3. 2의 보수 자료형을 sign extension하면 왜 값이 그대로라는게 보장될까?
$$
B2T_{w + k} ([x_{w-1}, ..., x_{w-1}, x_{w-2}, ..., x_{0}])
$$
$$
= B2T_{w} ([x_{w-1}, x_{w-2}, ..., x_{0}])
$$
- 이 수식을 증명하면 된다.
$B2T_{w + 1} ([x_{w-1}, x_{w-1}, x_{w-2}, ..., x_{0}])$
$= B2T_{w} ([x_{w-1}, x_{w-2}, ..., x_{0}])$
를 살펴보자.
$B2T_{w + 1} ([x_{w-1}, x_{w-1}, x_{w-2}, ..., x_{0}])$
$= -x_{w-1}2^w + \sum_{i = 0}^{w-1} x_i2^i$
$= -x_{w-1}2^w + x_{w-1}2^{w-1} + \sum_{i=0}^{w-2}x_i2^i$
$= -x_{w-1}(2^{w} - 2^{w-1}) + \sum_{i=0}^{w-2}x_i2^i$
$= -x_{w-1}2^{w-1} + \sum_{i=0}^{w-2}x_i2^i$
$= B2T_{w} ([x_{w-1}, x_{w-2}, ..., x_{0}])$
$\doteq B2T_w$
이렇게 증명될 수 있다.
- 간단히 요약해보자면,
sign extension을 하면 새로 추가된 최상위 비트는 $−2^w$의 가중치를 갖는다.
하지만 기존의 최상위 비트는 이제 일반 비트가 되어 $+2^{w−1}$의 가중치로 더해진다.
따라서 두 항은 합쳐져 다시 $−2^{w−1}$ 효과를 만들고, 
결과적으로 값이 보존된다.
그리고 +1 이상의 sign extension도 이 과정의 반복이나 다름없다.
그래서 2의 보수(signed) 자료형을 sign extension하면 값이 그대로인게 보장된다.

4. 워드 크기 확장과 signed / unsigned 간의 교체가 동시에 일어난다면 어떻게 될까?
- 워드 크기 확장과 signed / unsigned 간의 교체 중 뭐가 먼저 일어나는지에 따라 
값이, 의미가 달라지기 때문에 이 항목이 중요하다.
- C언어에서는 값 보존이 가능한 방향으로 
먼저 비트 수 확장(sign extension 또는 zero extension)이 일어난다고 이해하면 편하고,
그 다음 대상 타입의 signed / unsigned 해석이 적용된다고 보면 된다.

### 느낀 점 & 배운 점

이번에는 뭔가 기존에는 당연한 상식처럼 여기고 있었던 더 큰 자료형으로 캐스팅할 때의
규칙을 좀 더 수학적으로 이해한 것 같아.

unsigned는 그냥 확장된 자리를 0으로 채워넣고,
signed는 확장된 자리를 부호비트와 같은 비트로 채워넣는다는 규칙 자체는 이미 알만큼 알고있었지만,

signed(2의 보수) 자료형에서 확장된 자리를 부호비트와 같은 비트로 채워넣으면
값의 의미가 왜 바뀌지 않는지에 대해서 확실히 수학적으로 알게된 것 같아.

결국 확장된 부분, 그러니까 새로 추가된 상위 비트들의 가중치의 총합과 
기존($x_{w-1}$)의 음의 가중치가
동일하니까 결국 값은 변하지 않는다..

이렇게 직접 원리를 알고 나니까, 정말 신기하더라..

sign extension에 대한 예시를 한번 들어봤어. 
책에 있는 연습 문제를 그대로 풀어봤는데,

(2의 보수, signed)
- $[1100] = -8 + 4 + 0 + 0 = -4$
- $[11100] = -16 + 8 + 4 + 0 + 0 = -4$
- $[111100] = -32 + 16 + 8 + 4 + 0 + 0 = -4$

이렇게 새로 확장된 부분(기존의 음의 가중치)
- $-8$
- $-16 + 8$
- $-32 + 16 + 8$

이 전부 같기 때문에, 결국 의미는 안 변한다는게 체감이 됐어.

## 2. Truncating Numbers - 정수 비트 표현의 절삭

### 정리

1. unsigned 자료형을 더 작은 워드 크기의 자료형으로 절삭하는건,
그저 최상위 비트($u_{w-1}$)부터 더 작은 워드 크기에 해당하는 비트($u_k$)까지 싸그리 날려버리면 된다.
$\vec{u} = [u_{w-1}, u_{w-2}, ..., u_{0}]$
$\vec{u'} = [u_{k-1}, u_{k-2}, ..., u_{0}]$
- 그리고 절삭된 값은, 
$u \bmod 2^k$의 값을 가진다.

2. 왜 절삭된 값이 $u \bmod 2^k$의 값을 가질까?
- 여기서 $\bmod$는 나머지 연산자이다.
- $w$비트에서 $k$비트로 절삭한다는 것은, 
단순히 값을 해석하는 관점에서는 $u_{w-1}$부터 $u_{k}$까지의 비트들을
전부 0으로 바꾼다는 것과 크게 다르지 않다.
- $u \bmod 2^k$라는건 $u$를 $2^k$로 나누었을 때 남는 나머지라는 의미인데,
- $u_{k}$ 이상의 비트들은 전부 이렇게 $2^k$의 배수의 가중치를 가진다.
$u = (u_{w-1}2^{w-1} + \dots + u_{k+2} \underbrace{2^{k+2}}_{2^k \times 2^2} + u_{k+1} \underbrace{2^{k+1}}_{2^k \times 2^1} + u_k 2^k + \dots + u_0 2^0)$
- 그렇다면, 결국 절삭되는, 0이 되는 모든 비트의 가 중치는 $2^k$의 배수가 된다.
- 그리고, 남은 비트, 절삭되지 않은 비트들의 가중치를 다 합쳐도 $UMax_{k}$, 즉 $2^{k} - 1$이다.
- 그러므로, 원래 unsigned 자료형에서 $k$비트로 절삭한 값은,
$u_{w-1}$부터 $u_{k}$까지의 비트들을 전부 0으로 바꾼 값은
원래 값을 $2^k$로 나눈 나머지인 $u \bmod 2^k$가 된다.

3. signed 자료형을 더 작은 워드 크기의 자료형으로 절삭하는건,
딱 수식 하나로 요약할 수 있다.
- $U2T_k(x \bmod 2^k)$ 
- 일단 $x \bmod 2^k$는 $0 \sim 2^k - 1$ 사이의 정수들은
방금 전과 같이 $w$비트의 정수를 $k$비트의 정수로 절삭해낼 수 있다.
- 그리고 $U2T_k$(Unsigned To Two's-complement) 함수로 절삭해낸 정수를 
2의 보수가 아닌 Unsigned로 놓고 signed(2의 보수)로 바꾼다.
- 그리고 여기서, 최상위 비트($x_{k-1}$)은 부호 비트이므로,
음의 가중치를 가진 것($x_{k-1} \times -2^{k-1}$)으로 해석한다.
- 예시를 들어보자면,
$x = 53191$, int(4바이트, 32비트)를 short(2바이트, 16비트)로 절삭한다고 했을 때,
$53191 \bmod 2^{16} = 53191 \bmod 65536 = 53191$
$53191$(`1100 1111 1100 0111`)은 16비트일 때 최상위 비트가 1이기 때문에 $TMax$를 초과하므로
$U2T_k(53191) = 53191 - 2^{16} = 53191 - 65536 = -12345$
이렇게 절삭이 된다.
- unsigned와 비슷하게 k + 1번째 비트까지 절삭 후, 그저 해석 방법을 달리 한다.

4. 총 요약
- unsigned 절삭하는 법 : $u \bmod 2^k$
$B2U_k([u_{k-1}, u_{k-2}, ..., u_0]) = B2U_w([u_{w-1}, u_{w-2}, ..., u_0]) \bmod 2^k$
- 2의 보수(signed) 절삭하는 법 : $U2T_k(x \bmod 2^k)$
$B2T_k([x_{k-1}, x_{k-2}, ..., x_0]) = U2T_k(B2U_w([x_{w-1}, x_{w-2}, ..., x_0]) \bmod 2^k)$
로 요약할 수 있다.

### 느낀 점 & 배운 점

이번 파트 진짜 어렵더라..
특히 $x \bmod 2^k$의 의미를 파악하는게 정말로 어려웠어..
하지만 그래도 $k$번째 비트와 그 이후 비트($x_{w-1} \sim x_{k}$)의 가중치는 $2^k$의 배수이고,
$k$번째 비트의 이전 비트($x_{k-1} \sim x_0$)의 가중치를 전부 더해도 $2^k - 1$이 나온다는 것을 확실히 알았을 때
왜 굳이 $x \bmod 2^k$인지 알겠더라..

그리고 기존에 나는 그저 
> 더 크기가 큰 변수를 더 작은 변수로 캐스팅(형 변환)하는 것은 위험하다.
그래서 명시적인 형 변환을 해줘야한다.

정도로만 정수 비트 표현의 절삭을 알고있었어.

하지만 이번에 이걸 배우면서,
확실히 어떻게 절삭이 되는지를 배운 것 같아.

결국 그냥 값을 해석하는 관점에서는 최상위 비트부터 $k+1$번 비트까지는
싹다 0으로 만드는 것이라는걸 알게되었어.

그리고, signed 자료형은 절삭을 어떻게 하나 궁금했었는데,
이 것도 그저 unsigned와 동일하게 절삭한 후에
해당 비트열을 signed로, 최상위 비트를 음의 가중치로 해석한다는걸 알게되었어.

솔직히 나는 이게 좀 충격이었어.

그동안 나는 signed 자료형을 절삭할 때 기존의 최상위 비트($x_{w-1}$, 부호 비트)가 영향을 주는 줄 알았는데,
아닌게 참 신기하고, 충격이었어.

## 3. Advice on Signed versus Unsigned

### 정리

1. signed 자료형과 unsigned 자료형간의 형 변환(캐스팅)에서 생길 수 있는 문제들
- 형 변환은 프로그래머가 알아채기 어려운 곳에서도 일어나기 때문에,
프로그래머가 이러한 문제들을 간과하는 경우가 많다.
```cpp
// int형 배열의 모든 원소를 더해서 반환하는 함수
int sum(int arr[], unsigned int length){

    int result = 0;
    
    for (int i = 0; i <= length - 1; i++) {
        result += arr[i];
    }
    
    return result;
}
```
- 이 코드가 겉으로 보기에는 문제가 없어보이지만, 만약 `length`가 0으로 입력이 들어왔다면,
`length - 1`은 부호 없는 정수형인데도 `0 - 1`이 수행되고, 
`(unsigned) 0000 ... 0000`  + `(signed) 1111 ... 1111`와 다름이 없어진다.
하지만 unsigned와 signed 자료형끼리 아무런 명시 없이 연산을 하게 된다면
C언어 기준으로는 signed 자료형이 unsigned로 해석되므로
사실상 `(unsigned) 0000 ... 0000` + `(unsigned) 1111 ... 1111`이므로
결국 `0 - 1`의 결과는`1111 ... 1111`이 되어서
실제 값으로는 32비트짜리 unsigned int 형의 $UMax$가 된다.
- 그렇기 때문에 arr 배열의 존재하지 않는 인덱스에 접근하게 되어서 메모리 오류가 발생한다.
```cpp
// 문자열 s가 문자열 t보다 긴지 판단해서 s가 t보다 길면 true, s가 t보다 길지 않으면 false를 반환하는 함수
bool strlonger(char *s, char *t) {
    return (strlen(s) - strlen(t)) > 0;
}
```
- 이 코드도 겉으로 보기에는 문제가 없어보이지만, `strlen()` 함수는 반환값으로 size_t 형,
실제로는 32비트 환경에서는 unsigned int, 64비트 환경에서는 unsigned long long 형이다.
어쨌든 unsigned 자료형이다.
- 하지만, 여기서 s보다 t가 더 긴 경우,
strlen(s) = 1
strlen(t) = 2
라고 할 때,
실제로는 `1U - 2U` 를 연산하게 된다.
- 프로그래머의 직관적인 계산으로는 1 - 2를 하니 -1이 나오고 
그러면 -1 > 0은 False니 False(0)를 반환할 듯 싶지만,
실제로는 -1을 2의 보수로 표현하면 `1111 ... 1111`이 되고,
실제 해석은 unsigned 자료형으로 되니,
실제로는 UMax > 0의 논리값이 반환되므로 True를 반환한다..
- 이렇게 signed 자료형이 암시적으로 unsigned로 변환되는 문제 때문에
다양한 오류나 취약점이 초래될 수 있다.

2. 이런 문제를 해결하기 위한 방법
- 그냥 이렇게 숫자로 해석하는 값들은 unsigned 자료형을 사용하지 않는 것이 좋다.
- unsigned 자료형을 사용하는 경우는 숫자로 해석하지 않고 그저 비트들의 집합으로 해석할 때이다.
실제로 이럴 때는 상당히 유용하다.
- 대표적으로 개별의 비트를 하나의 bool 변수로 취급하는 비트 플래그 같은 상황이 있다.

### 느낀 점 & 배운 점

이 챕터는 unsigned 자료형의 위험성을 잘 보여준 것 같아.
C언어에서는 unsigned와 signed 자료형을 함께 연산할 때는
signed 자료형이 unsigned 자료형으로 캐스팅되어 연산되는데,
결국 $TMin \leq x < 0$의 값들은 unsigned로 해석될 때 
$TMax < u \leq UMax$의 값으로 해석되기 때문에 결국 의도하지 않은 값이 나오는..

그리고 해결책도 참 명쾌한 것 같아ㅋㅋ

애초에 unsigned를 사용하지 않으면 된다는..ㅋㅋ

이건 뭔가 공감되기도 해. 
어차피 $TMax < x \leq UMax$의 수들을 사용하려면
이런 문제가 있는 unsigned 자료형보다는 
더 워드 크기가 큰 signed 자료형을 사용하는게 훨씬 편하기 때문인 것 같아.
이런거에 일일히 신경쓸 필요가 없어지니까..

이 챕터를 읽으면서, 내가 만들었던 에코 서버 코드를 다시 읽어봤는데,
내 에코 서버 코드에서도 좀 문제가 있는 것 같았어.

헤더의 length 필드는 uint32_t(unsigned int 32-bit)자료형을 사용했었는데,
- 나중에 GB 단위로 송 / 수신할 때
- singed와 섞어서 연산할 때
- 범위 검사를 허술하게 할 때

는 확실히 문제가 되지 않을까 싶네.
