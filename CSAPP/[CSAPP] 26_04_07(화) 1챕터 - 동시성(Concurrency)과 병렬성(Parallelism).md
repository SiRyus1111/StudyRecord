## 1. 동시성(Concurrency)와 병렬성(Parallelism)의 차이?

솔직히 처음 단어만 봤을 때는 차이가 거의 안 느껴져.
둘 다
> 어쨌든 여러 스레드가 같이 실행된다는거 아닌가?

같은 정도의 느낌만 받을 수 있어.
적어도 나는 그랬음ㅋㅋ

하지만 달라.

일단 비유를 한번 들어볼게..

일단 동시성이란!

한 명의 요리사에게 어떤 일들을 시킬지야.
요리사 한 명이 국도 끓이고, 생선도 굽지.
국을 젓다가(스레드 A), 
생선 뒤집을 시간이 되면 국을 끓이는 것을 잠시 멈추고(Context Switching) 
생선을 뒤집어.

결국, 요리사는 한 번에 한 과정만 실행하고 있지만,
요리사라는 '시스템'은 국 끓이기과 생선 굽기라는 두 작업을 동시에 진행하고 있지.

이게 동시성(Concurrency).

이제 여기서 요리사가 두 명으로 늘어났다고 해보자.

이렇게 되면, 그냥 한 명은 국을 끓이고, 다른 한 명은 생선을 굽는 식으로 배치해주면,
실제로 같은 시점에 동시에 실행되겠지.

결국, 요리사가 늘어난 만큼, 시스템 전체의 요리 완성 시간은 확실히 줄어들겠지.

이게 병렬성(Parallelism).
```
실행 흐름이 겹쳐 보이게 만드는 것(동시성, Concurrency) vs 물리적으로 동시에 실행하는 것(병렬성, Parallelism)
```
이렇게 설명할 수도 있어.
<br>

좀 더 전문적인 느낌으로는,

- 동시성 (Concurrency): "논리적인 설계의 영역"

  - 어떻게 하면 작업을 잘게 쪼개서(Task Decomposition), 동시에 진행되는 것처럼 '관리'할 것인가?

  - 핵심 키워드: 구조, 스케줄링, 컨텍스트 스위칭, 인터리빙(Interleaving)
  
- 병렬성 (Parallelism): "물리적인 실행의 영역"

  - 어떻게 하면 쪼개놓은 작업들을 실제 여러 개의 코어(Hardware)에 태워서 '동시에 물리적으로 실행'할 것인가?

  - 핵심 키워드: 하드웨어 자원, 멀티코어, 처리량(Throughput), 가속화
  
  이렇게 요약할 수 있겠네.
  
즉, 
동시성은 
> 프로세스의 작업들을 어떻게 나눠서 실행시킬 것인가. 

이고 

병렬성은 
> 실제 그렇게 나눠서 실행시킬 실행 흐름들을 
실제 물리적인 하드웨어에서 어떻게 실행시킬 것인가. 

라고 하면 돼.

## 2. 실제로 동시성과 병렬성은 어떻게 구현될까?

지금까지 동시성과 병렬성의 개념에 대해 봤다면,
이런 의문이 들 수 있어.

> 그래서 이것들 실제로 어디에, 어떻게 씀?

솔직히 운영체제 / 멀티스레드 안 배우고 
바로 동시성과 병렬성 배우면 이게 잘 이해가 안될 것 같기도 해.

일단 나는 깊게 들어가진 않아.
딱 책에서 나온거 요약 정리정도로만 해볼게.

### 2-1. 스레드 수준 동시성

일단 스레드 수준의 동시성이 뭐냐면,
> 추상화를 바탕으로 여러 스레드가 동시에 실행되는 것처럼 보이게 하는 것

이라고 할 수 있어.

이런 스레드 수준의 동시성은
싱글 코어(Core, 하나의 CPU 집합 / 임시) 프로세서일 때도,
하드웨어적 실행 흐름(스레드)이 하나처럼 보일 때도,
실제 소프트웨어적 실행 흐름(스레드)은 
빠르게 여러 개의 소프트웨어적 실행 흐름(스레드)을 
교체해가면서 실행(컨텍스트 스위칭)하는 식으로
스레드 수준 동시성을 구현할 수 있어.

여기까지는 일단 스레드 수준 동시성의 정의고,
여기서부턴 어떻게 여기서 성능을 더 올릴 수 있는지,
그리고 어떤 방식으로 성능을 올릴 수 있는지를 설명해볼게.

일단 요즘의 컴퓨터들은 기본적으로 
```
ALU - Control Unit(제어 장치) - Registers(레지스터 집합)
```
이렇게 기존에 CPU라고 배웠던,
코어(Core)라고 하는 집합이 여러개 달려있어.

그걸 **'멀티코어 프로세서(Multi-Core Processor)'**라고 하는데,
각각의 코어에 
데이터를 저장하는 L1 캐시(d-cache)와 
인출(fetch)한 명령어를 저장하는 L1 캐시(i-cache)가 딸려있지.
그리고 대개 L2 캐시까지도 각각의 코어에 딸려있어.

그리고, 하나의 코어도 하나 이상의 일을 한번에 할 수 있어.

이건 **'하이퍼스레딩(Hyperthreading)'**이라고 하는데,
단일 코어가 여러 제어 실행 흐름을 동시에 실행할 수 있게 해주는 기술이야.

프로그램 카운터(PC), 레지스터 파일같은 실행 상태를 저장하는 CPU 하드웨어는 
한 코어에 여러개를 집어넣고,
하나만 있어도 충분한 CPU 하드웨어(ALU 등)들은
한 코어에 하나만 집어넣도록 설계되어있어.

왜 그렇냐면, 
딱 상태를 저장하는 레지스터 집합만 여러개를 둬서 비용 대비 효율을 높일 수 있기 때문이야.
ALU는 상대적으로 비싸니까.
물론 다른 이유도 있긴 해.

하이퍼스레딩이 적용되어있지 않은 프로세서는 
OS가 개입하는 스레드 사이의 전환(컨텍스트 스위칭)을 하는데 약 20000클럭이 필요한 반면에,
하이퍼스레딩이 적용된 프로세서는
하드웨어 수준에서 클럭 단위로 실행할 스레드를 결정해서 
CPU의 자원을 더욱 효율적이게 굴릴 수 있어.

보통 이렇게 하이퍼스레딩으로 만들어진 작업 흐름을
- 하드웨어적 스레드
- 논리 프로세서

라고도 해.

물론 우리가 코드에서 만드는 `std::thread`같은건
'소프트웨어적 스레드'라고 하는 하드웨어적 스레드와는 상당히 다른 개념이야.
이거 중요해서.

스레드 수준 동시성의,

첫 번째는, 한 컴퓨터에서 더 많은 작업을 동시적으로(Concurrently) 처리할 수 있게 된다는 점이야.

두 번째는, 한 애플리케이션 프로세스도 더 빠르게 동작할 수 있게 해준다는 점이야.
만약 병렬적으로 실행될 수 있도록 프로그래밍되어있다면.
이것도 이유는 첫 번째와 비슷해.

### 2-2. 명령어 수준 병렬성

명령어 수준 병렬성이란,
> 한 스레드 안에서 여러 명령어를 겹쳐서 처리하는 것

이야.

이건 **CPU 파이프라이닝(CPU Pipelining)**, **슈퍼스칼라(superscalar)**라는 기술로 구현할 수 있어.
쉽게 말하자면, 여러 개의 명령어를 물리적으로 동시에, 
**병렬로 처리**하는거야.

예전에는 한 명령어를 실행하는데 3 ~ 10클럭이 필요했어.
하지만 최근에는 1클럭에 2 ~ 4개의 명령어를 실행해.
1개의 명령어를 2 ~ 4클럭에 실행하는게 아니라, 1클럭에 2 ~ 4개의 명령어를 실행하는거야.
이게 어떻게 가능하냐면,
앞서 말한 CPU 파이프라이닝(+슈퍼 스칼라)이 이걸 가능하게해.

그러니까 명령어를 실행할 때는 단계(사이클)가 있어.
가장 보편적으로 표현되는게,

$인출(fetch) \to 해석(decode) \to 실행(execute)$

이런 식인데,
각각의 사이클은 동시에 실행될 수는 없지만,
다른 사이클(단계)이라면 동시에 실행될 수 있어.

그래서, 결국 파이프라이닝이란,
이런 식으로 각 사이클 별로 겹치지 않게 한 단계 씩 실행하는거야.

슈퍼스칼라는 이런 파이프라인을 여러개 두는거라고 이해하면 편해.
정확히는 한 사이클에 여러 명령어를 동시에 인출하고.. 
이런 식으로 병렬적으로 여러 명령어를 다루는거야.
생각보다 개념 자체는 단순해.

근데 파이프라이닝을 쉽게 하려면, 
**모든 명령어의 각각의 사이클의 시간이 비슷**해야 유리하겠지?
파이프라이닝은 명령어를 단계별로 겹쳐 처리하는 구조라,
각 단계(사이클)의 시간이 비슷할수록 병목 없이 매끄럽게 흘러가기 때문이야.

예를 들어,
인출(fetch) 사이클은 빨리 끝나는데
실행(execute) 사이클만 유독 오래 걸린다고 해보자.

그러면 실행 사이클에서 계속 병목이 되고,
앞 단계가 이미 실행된 명령어들이 기다리게 돼.
그리고 뒤에 실행될 명령어들도 밀리겠지.

좀 더 이해하기 쉽게 비유를 해보자면,
세탁 공장 라인 같은 거야.

- 1단계: 세탁
- 2단계: 건조
- 3단계: 포장

이 세 단계가 각각 10분, 10분, 10분이면
10분마다 다음 물건을 자연스럽게 넘길 수 있어.

그런데 만약

- 1단계: 10분
- 2단계: 40분
- 3단계: 10분

이면 어떻게 될까?

2단계가 계속 병목이 돼서
앞 단계는 기다리고, 뒤 단계도 놀게 돼.
즉, 겹쳐서 처리하는 이점이 줄어들어.
