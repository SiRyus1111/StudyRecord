그리고 OS가 추상화를 하는 방법으로,
- 프로세스(Processes) (프로세서(CPU) / 메인 메모리 / 입출력 장치를 추상화한 것)
  - 마치 해당 프로세스가 CPU를 독점하고 있는 것 같이 보이게
  - (Context Switching)
- 가상 메모리(Virtual Memory) (메인 메모리 / 입출력 장치를 추상화한 것)
  - 해당 프로세스가 보고 있는 주소 공간이 본인의 전체 주소 공간인 것 같게
  - (논리(가상) 주소 / paging 등)
- 파일(Files) (입출력 장치를 추상화한 것)
  - 입출력 장치마다 다른 입력방식으로 데이터를 다루지 않고 그저 바이트(비트)열로 데이터를 처리

OS의 컴퓨터의 하드웨어를 보호하는 역할을 달성할 수 있고,

### 2-1. 컴파일 과정과 왜 컴파일 과정을 알아야하는지

나는 이전에,
컴파일 과정이
> 소스 코드를 기계어로 번역하는 과정

이라고만 알고있었어.

하지만 그 과정은 더 세밀했어.

이 과정은 C언어 기준이야.
다른 언어의 컴파일 과정은 다를 수 있어.

```
                 ---소스 코드(텍스트)---> 
--> 전처리기 ---가공된 소스 코드(텍스트)---> 
--> 컴파일러 ---어셈블리 코드(텍스트)---> 
--> 어셈블러 ---재배치가능한 목적 파일(이진수)---> 
--> 링커 ---실행가능한 목적 파일(이진수)---> 
--> 실행
```
이렇게 복잡하게 돌아가더라고.
물론 그 내부에 더 세밀한 과정이 있겠지만 (토큰으로 소스 코드를 나눈다던지..)

대략적인 과정마저도 이렇게 많은 과정을 거친다는게 신기했어.

그리고 컴파일 과정을 알아야 하는 이유도 확실히 체감했어.
내가 체감한 컴파일 과정을 알아야 하는 이유는,

1. 실제로 컴퓨터가 실행하는건 저수준 언어인 기계어(~어셈블리)이므로,
성능이 중요한 부분에서는 내가 짠 코드가 어떻게 기계어(~어셈블리)로 표현될지까지 생각해서
코드를 작성해야한다.
`요약 - 최적화를 위해서`

2. 어디에서 어떤 에러가 발생했는지 더 확실하게 알 수 있고,
그럴수록 에러의 해결이 쉬워진다.
에러가 컴파일 단계에서 발생했는지, 링크 단계에서 발생했는지를 구별할 수 있으면
원인을 훨씬 빠르게 좁힐 수 있고, 생산성도 상당히 오른다.
`요약 - 전체적인 컴파일 과정에서 발생한 에러의 디버깅이 쉬워져서`

3. 실제로 컴퓨터가 실행하는건 기계어(~어셈블리)이므로,
보안상 허점도 이곳에서 발생한다.
컴파일 과정과 어셈블리, 내 코드가 어떻게 어셈블리 / 기계어로 변환되는지 알아야
소스 코드만 봐서는 드러나지 않는 저수준 동작이나 메모리 처리 방식 때문에
실제 실행 단계에서 생길 수 있는 보안상 허점을 막을 수 있다.
`요약 - 보안상 허점을 막기 위해서`

이렇게 세 가지로 요약할 수 있겠네.

### 2-2. 사용자 모드와 커널 모드

나는 그동안 감이 잘 안잡혀있었어.
사용자 모드? 커널 모드? 이게 어떻게 실행이 되는거지?
사용자 모드면 우리가 쓴 코드를 실행?
커널 모드면 커널의 코드를 실행?
두 모드가 동시에 실행될 수는 있나?
커널의 코드를 실행하는 동안에는 우리가 쓴 코드는 실행되지 않나?

이번에 이걸 확실히 이해했어.

솔직히 이건 CS:APP 1장의 주제에서는 어떻게보면 좀 벗어나있긴 한데,
기록해두고싶어서 썼어.

일단 처음부터 생각해보자.
커널 모드란, 애플리케이션 프로세스가 커널이 관리하는 자원을 사용하기 위해
시스템 콜(System Call)이라는 자원 사용 요청을 커널에 날려서
프로세스가 해당 자원을 쉽게 사용할 수 있게 하는거잖아?

내가 헷갈렸던 부분은 이거야.

1. 두 모드가 어떻게 전환이 되는거지?
2. 두 모드일 때 각각 어떤 일이 일어나는거지?

이 두 개는 어떻게 보면 이어져있어.

일단 1번의 답은 시스템 콜을 커널에 날리면, 해당 프로세스는 사용자 모드에서 커널 모드로 전환이 돼.
근데 여기서 2번이랑 연결되는게,
그 사용자 모드 -> 커널 모드 전환을 CPU는 어떻게 알 수 있는거지?
시스템 콜을 날리면 커널은 무슨 일을 하는거지?
애플리케이션 프로세스는 그냥 멈추고 커널 모드 -> 사용자 모드로 전환될 때 다시 시작되는건가?

사용자 모드 -> 커널 모드 전환은 
CPU의 현재 모드를 나타내는 비트의 값이 바뀌는 것으로,
CPU에게 지금 실행하고 있는 프로세스가 사용자 모드로 실행되고 있는지 커널 모드로 실행되고 있는지 알려줘.

시스템 콜을 날린다면,
그저 커널은 무슨 일을 하는게 아니라 커널 영역 내부의 코드가 실행이 될 뿐이야.
이 때, CPU는 방금 설명한 모드 비트의 값으로 
지금 커널 영역 내부의 코드가 실행이 되고있다는걸 아는거지.

애플리케이션 프로세스는 시스템 콜을 날렸을 때 멈추고,
해당 시스템 콜에 해당하는 커널 영역의 메모리 주소의 명령어들을 실행한 후,
다시 애플리케이션 프로세스가 시작되는 식으로 돌아가.
물론 이 때 모드 비트의 값도 바뀌지.

그런데 결국 
- 애플리케이션 프로세스가 시스템 콜을 날리고
- 애플리케이션 프로세스를 잠시 멈추고
- 커널 영역의 코드를 실행하고
- 커널 영역의 코드를 다 실행하면 다시 애플리케이션 프로세스를 실행

이렇게 시스템 콜은 사용자 모드와 커널 모드 사이의 전환을 일으키지.
이 전환 자체에도 오버헤드가 있어.
하지만 이게 다른 프로세스로의 컨텍스트 스위칭을 의미하는 것은 아니야.

어쨌든,
> 자원 사용 = 커널 모드 전환 필요 = 시스템 콜 = 오버헤드 발생

이라,

성능을 극한까지 높이려면 시스템 콜도 어느정도 줄여야할 것 같다고 생각했어.

사실, 이 분량은 책에는 없는데, 어쩌다보니까 쭉 파게 되어서ㅋㅋ

### 2-3. 암달의 법칙(Amdahl's Law)

솔직히 이거 이름만 봐서는 감이 안오는데,
일단 한줄 요약해볼게.

이해하기 쉽게 말하면,    
> 시스템의 일부의 성능을 올린다고 해도,
전체 시스템에 얼마나 영향을 주느냐에 따라 성능 향상률은 달라진다.

이거야.


좀 더 자세하게 말하면,

> 시스템의 한 부분의 성능을 개선했다고 해도,
개선의 효과는 
그 부분이 전체 시스템의 성능에 어느정도 부분을 차지하는지(p),
그리고 어느 정도 성능을 개선했는지(k)에 달려있다.

> 단순히 그 부분의 성능을 얼마나 개선했는지만으로는 
전체적인 시스템의 성능에 끼치는 영향을 추정할 수 없다.

이거야.

책에는 수식들로 표현하는데,
$$
S = \frac{1}{(1-p) + \frac{p}{k}}
$$
이 수식이야. 
- S는 전체 시스템에서 개선된 양
- p는 개선된 부분이 전체 시스템에서 차지하는 비율
- k는 개선된 부분이 얼마나 개선되었는지 나타내는 수

예를 들어, 개선된 부분이 전체 시스템의 병목에서 60%를 차지한다고 하자.
그리고 그 부분을 3배 개선했어.

그러면,
$$
S = \frac{1}{(1-0.6) + \frac{0.6}{3}}
$$
으로,
$$
S = \frac{1}{\frac{2}{5} + \frac{1}{5}}
= \frac{1}{\frac{3}{5}}
= \frac{5}{3}
$$
즉, 전체 시스템을 3/5배 = 1.67배 개선했다는거지.
(1.67×)

근데 해당 부분은 3배나 개선되었는데
전체 시스템은 1.67배 정도밖에 개선이 안 되었지?

이게 암달의 법칙이 말하고자 하는 주제야.

좀 더 극단적으로 가보면,
개선된 부분이 전체 시스템의 병목에서 30%를 차지하고,
그 부분을 무한대로 개선했어. 그냥 병목을 아예 0으로 만들어버린거야.

그러면,
$$
S = \frac{1}{(1-0.3) + \frac{0.3}{\infty}}
$$
$$
\frac{0.3}{\infty} \to 0\;(0에\;수렴)
$$

$$
