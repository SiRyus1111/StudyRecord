## 1. Dividing by Powers of 2 - 2의 제곱에 의한 나눗셈

### 정리

1. 2의 제곱에 의한 나눗셈
    - 곱셈처럼, 나눗셈도 비트 시프트를 이용해서 구현될 수 있다.
    - 하지만, 곱셈처럼 왼쪽 시프트가 아닌, 오른쪽 시프트로 구현된다.
    - 정수의 나눗셈은 항상 0에 가깝게 버림된다.
      - 즉, 소수부를 버리는 것과 같다.
      - C에서 정수 나눗셈은 0을 향해 버림된다.
      - 그래서 나눗셈의 결과가 양수라면 내림과 같고, 결과가 음수이면 올림과 같다.
    - 기호로는, 
    $\lfloor a \rfloor$ = 내림, $\lceil a \rceil$ = 올림
    이렇게 표현된다.
    - 즉, 정수 $x, y$의 나눗셈은 
      - $x \geq 0, y > 0$인 경우,
      즉 결과값이 양수인 경우 $\lfloor x / y \rfloor$라고 할 수 있고,
      - $x < 0, y > 0$인 경우,
      즉 결과값이 음수인 경우 $\lceil x / y \rceil$라고 할 수 있다.
    - 오른쪽 시프트(`x >> k`)의 결과는,
    $x = (\text{상위 w-k비트의 비트열}) * 2^k + (\text{하위 k비트의 비트열})$
    (상위 w-k비트의 비트열에 $2^k$ 곱하는건 가중치 맞추기 용도)
    $x / 2^k = (\text{상위 w-k비트의 비트열}) + (\text{하위 k비트의 비트열}) / 2^{k}$
    $(\text{하위 k비트의 비트열}) / 2^k < 1$이므로 하위 k비트의 비트열은 버려진다.
    즉, `x >> k` = $(\text{상위 w-k비트의 비트열}) = \lfloor x / 2^k \rfloor$이라고 할 수 있다.

2. 부호없는 정수의 2의 제곱에 의한 나눗셈
   - 오른쪽 시프트를 사용한다.
   - $w$비트 부호없는 정수 $x$에 대해 $k$(범위 : $0 \leq k < w$)만큼 오른쪽 시프트 연산을 한다는 것은,
   (`x >> k`)
   하위 $k$비트를 버리면서(이게 수학적으로 내림이 된다.) 
   상위 $w-k$비트를 오른쪽으로 시프트하는 것이기 때문에
   내림이 적용된 $\lfloor x / 2^{k} \rfloor$와 같다.
   - 양수에서 0을 향해 소수부를 버리는 것은 사실상 내림과 다르지 않기 때문에,
   부호없는 정수(unsigned integer)의 `x / 2^k`는
   단순한 논리 오른쪽 시프트만으로 구현할 수 있다.
   
3. 2의보수 정수의 나눗셈
    - 오른쪽 시프트를 사용하지만, 나눈 결과값이 양수일 때와 음수일 때 다른 처리가 필요하다.
      - 양수일 때는 내림을 해야하고($\lfloor x / y \rfloor$)
      - 음수일 때는 올림을 해야한다($\lceil x / y \rceil$)
    - 그리고 부호의 의미를 유지시켜주기 위해 시프트 연산 후 상위 $k$비트를
    부호비트에 맞게 채워주는 산술 시프트 연산을 사용한다.
    - 일단, $w$비트 2의보수 정수 $x$에 대해 $k$만큼 오른쪽 산술 시프트 연산을 한다는 것은,
    (`x >> k`)
    하위 $k$비트를 버리면서 상위 $w-k$비트를 오른쪽으로 시프트하는 것이다.
      - 하지만, $x$가 양수인지 음수인지에 따라 중요한 부분이 있다.
      - 위에서 설명했듯이, `x >> k`는 
      $\lfloor x / 2^{k} \rfloor$, 즉 나눗셈의 결과를 내림하는 것이다.
      - 하지만 시프트하는 값이 음수($x < 0$, 즉 부호비트가 $1$)라면,
      0에 가깝게 소수부를 버려야 하기 때문에,
      내림이 아니라 올림이 필요하다.
      - 기존의 `x >> k`로는 내림밖에 하지 못한다.
      (`x >> k` = $\lfloor x / 2^k \rfloor$)
      - 그래서 일단, $x \geq 0$인 경우에는
      기존과 같이 `x >> k`로, $x / 2^k$로 처리를 한다.
    - $x < 0$인 경우에는, 
    단순한 `x >> k`로는 $x / 2^k$에 올림을 하는 방식을 쓰지 못한다.
      - CS:APP에서는 이런 $x < 0$인 경우의 산술 시프트 연산으로 인한 나눗셈을,
      (`(x + (1 << k) - 1) >> k`)의 형태로 한다고 설명한다.
      이렇게 산술 시프트 연산을 진행하면, $\lceil x / 2^k \rceil$의 효과를 보인다.
      `(1 << k) - 1`로 값의 보정을 해준다고 생각하면 된다.
      - `(x + (1 << k) - 1) >> k`를 수학적으로 표현하면
      $(x + 2^k - 1) / 2^k$라고 할 수 있다.
        - $x$에다가 $2^k - 1$을 더하는 이유는,
        $2^k - 1$은 `x >> k`를 했을 때, 
        즉 $2^k$로 나눌 때의 나머지가 될 수 있는 최댓값이기 때문이다.
        - 즉, 나머지가 있는 음수들만 몫을 0의 방향으로 올려준다.
        - 나머지가 없는 음수는 올림을 해도 올림할 소수점이 없기 때문에 그대로이지만,
        나머지가 있는 음수는 올림을 할 때 0의 방향으로 1이 증가해야한다.
        - 그렇기 때문에 나머지가 될 수 있는 최댓값인 $2^k - 1$을 더함으로써,
        나머지가 없을 때는 내림을 해도 그대로가 되지만,
        나머지가 있을 때는 내림을 하면 0의 방향으로 1을 증가시켜준다.

### 느낀 점 & 배운 점

뭔가 상당히 어려웠던 것 같아..
특히
> 왜 $2^k - 1$을 더해서 보정을 해줬을 때 내림으로 올림이 구현이 되는가?

였던 것 같아.

결국,
- $2^k - 1$이 나머지가 될 수 있는 최댓값이니
- 나머지가 없는 경우에 나머지가 될 수 있는 최댓값을 더한 후 내림하면 해당 수는 변하지 않는다.
  - 나머지가 없는 수에 소수점을 올림한다고 해도 올림할 소수점이 없기 때문이다.
- 나머지가 있는 경우에 해당 수를 더한 후 내림하면 +1이 된다(올림된다).
  - 나머지가 있는 경우에만 올림을 해야한다.

인 것 같아.

그리고, 이 챕터 자체를 요약해보자면,

1. C언어에서 정수의 나눗셈은 0을 향한 소수부의 버림으로 정의되는데,
나누는 값이 2의 제곱일 경우에 `x >> k`의 산술 시프트 연산으로 최적화할 수 있다.
2. 그리고 정수의 나눗셈은 소수부를 절삭하게 된다.
즉, 양수라면 내림하고, 음수라면 올림해야한다.
3. `x >> k`의 산술 시프트 연산은 수학적으로 $\lfloor x / 2^k \rfloor$,
$x / 2^k$의 소수점을 내림하는 것이다.
4. 양수라면 `x >> k`를 그대로 써도 하위 $k$비트가 절삭되어 자동으로 내림되기 때문에
문제가 없다.
5. 하지만 음수라면 `x >> k`의 형태로 나눗셈을 할 때 내림이 아니라 올림을 해야하므로
`(x + (1 << k) - 1) >> k`의, $\lfloor (x + (2^k - 1)) / 2^k \rfloor$의 형태로
보정치를 추가해서 내림하는 식으로 올림을 한다.

라고 할 수 있겠네.

그리고 왜 굳이 상수에 의한 나눗셈도 아닌 $2^k$에 의한 나눗셈만 알려주는지 궁금했었는데,
상수에 의한 나눗셈은 이런 방식이 아닌 곱셈을 이용해서 구현되어서,
이 책에서 설명하기에는 너무 많고 어려워서 일부러 뺀 것 같아.

그건 컴파일러 배울 때 한번 제대로 배워볼 생각.

어쨌든, 딱 세 문장으로 요약하면 이 정도?
> 2의 제곱에 의한 정수의 나눗셈은 0을 향해 소수부를 버리는 것이다.

> 2의 제곱에 의한 정수의 나눗셈은 
`x >> k`로 $\lfloor x / 2^k \rfloor$의 형태로 구현되는데,
`x >> k`는 하위 $k$비트가 절삭되기 때문에 $\lfloor x / 2^k \rfloor$과 같은 효과가 있다.

> 양수일 때는 소수부를 버리는게 내림과 같아서 `x >> k`를 그대로 해도 문제가 없지만,
음수일 때는 올림을 해야하기에 추가적인 보정을 하고 `x >> k`를 수행해야한다.

## 2. Final Thoughts on Integer Arithmetic - 정수 산술연산에 대한 최종 고찰

### 정리

1. 컴퓨터에서 일어나는 정수의 산술 연산은 모듈러 연산($\bmod 2^w$)의 형태에서 일어난다.

2. 컴퓨터는 $w$비트까지만 수를 표현할 수 있으므로,
산술 연산의 결과가 $w$비트를 초과하면 해당 비트들은 절삭된다.
   - 이렇게 오버플로우가 발생한다.
   
3. 부호없는 정수든 2의보수 정수든 동일한 가산기로 덧셈을 한다.
    - 그래서 단 한 종류의 가산기만 있어도 
    **덧셈** / **뺄셈**은 처리할 수 있다.
    **곱셈** / **나눗셈**도 내부적으로
    시프트 / 덧셈 / 뺄셈 등을 활용하는 알고리즘으로 구현될 수 있다.
    - 그래서 결국, 
    컴퓨터는 비트열을 더할 뿐이고,
    따로 signed / unsigned로 해석함에 따라서 해당 비트열의 의미가 부여된다.
    
4. 결국, 개념적으로는 단순해보이지만 
이런 것들로 인해 예상치 못한 버그들이 많이 생길 수 있다.
   - signed가 unsigned와 함께 연산될 때 unsigned로 바뀌는 문제라던지,
   - signed로 해석해야되는걸 최종적으로 unsigned로 해석하게 되어 값의 의미 자체가 바뀌는 문제라던지,
   - 반대로 unsigned로 해석해야되는걸 최종적으로 signed로 해석하게 되어 값의 의미가 바뀌는 문제라던지..

### 느낀 점 & 배운 점

그냥 2.3절의 최종 정리 느낌..

난 여기에다 몇 가지 더 추가해보고 싶어.

1. signed를 unsigned로 바꿔서 덧셈 / 뺄셈하게 되어도 
하위 $w$비트의 결과는 변하지 않는다.
    - unsigned로 바꾼 차이는 오로지 최상위 비트가 1일 때 $2^{w}$의 배수로만 나타난다.
    - 그렇기 때문에 $w$비트 이상의 비트들로만 표현되고, $\bmod 2^w$에 따라 전부 절삭된다.
    - 하지만 최종 비트열을 signed로 해석하느냐 unsigned로 해석하느냐에 따라
    값의 의미는 달라질 수 있다.
2. 곱셈은 비트시프트를 이용한 상수에 의한 곱셈이 가능하지만,
나눗셈은 비트시프트 여러 번으로 상수에 의한 나눗셈이 불가능하다.
   - 나눗셈은 오직 비트시프트 만으로는 $2$의 제곱($2^k$)으로 나누는 것만 가능하고,
   상수에 의한 나눗셈을 하기 위해서는 곱셈이 들어가는 복잡한 로직이 필요하다.
   - 그래서 나눗셈을 할 때의 클럭이 많이 든다.
