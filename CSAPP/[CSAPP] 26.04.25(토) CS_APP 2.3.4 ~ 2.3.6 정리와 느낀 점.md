## 1. Unsigned Multiplication - 부호없는 정수의 곱셈

### 정리

1. 부호없는 정수의 곱셈도 그저 $s \bmod 2^w$,
곱한 값에서 하위 $w$비트 말고 다 절삭하기이다.
- 부호없는 정수 $x$, $y$를 곱할 때,
$x$와 $y$는
$0 \leq x,\;y \leq 2^w-1$
$0 \leq x,\;y \leq UMax$
의 범위를 가지고,
곱한 값($p = x \times y$)은 그에 따라
$0 \leq p \leq (2^w-1)^2$
$0 \leq p \leq UMax^2$
$0 \leq p \leq (2^{2w} - 2^{w+1} + 1)$
의 범위를 가지게 된다.
그리고 계산했을 때 
기존의 $w$비트에서 최대 $2w$비트까지 결과값을 표현하는 비트가 확장될 수 있다.
- 즉, 덧셈은 최대 $w+1$비트가 필요했다면, 
곱셈에서는 최대 $2w$비트가 값을 표현하는데 필요할 수 있다.
- 물론, 결국 덧셈처럼 하위 $w$비트만 남기고, $w$번째 비트 이상의 상위 비트들은 절삭된다.
- 그러므로 $x *_w^u y$는
$x *_w^u y = (x \times y) \bmod 2^w$
으로 정의할 수 있다.

### 느낀 점 & 배운 점

이 파트는 한 페이지 분량도 안 되었는데,
왜 그런지 알 것 같았어.

그냥 곱한 값의 범위랑, $w$비트만 남기고 절삭한다는 것을 설명할 뿐이야.
실제로도 그렇게 돌아가는데다가,
덧셈의 설명과의 차이는
- 값을 표현하는데 최대 $w+1$비트가 아닌 $2w$비트까지 필요할 수 있다.

밖에 없고
- $s \bmod 2^w$
- 하위 $w$비트만 남기고 남은 비트 절삭

이건 덧셈의 설명과 완전히 같아서ㅋㅋ

그래도 곱셈도 덧셈과 상당히 비슷한 것 같아서
아직까진 안심이 되네.
너무 어려울까봐 많이 긴장했었거든.

## 2. Two's-Complement Multiplication - 2의 보수의 곱셈

### 정리

1. 2의 보수의 곱셈도 그저 2의 보수의 덧셈처럼 
두 2의 보수 값을 곱한 뒤,
결과의 하위 $w$비트만 남기고($p \bmod 2^w$),
그 비트열을 다시 2의 보수로 해석하는 것일 뿐이다.
- $w$비트의 2의 보수 두 개($x,\;y$)를 곱하게 된다면,
$x,\;y$의 범위는
$-2^{w-1} \leq x,\;y \leq 2^{w-1} - 1$
$TMin \leq x,\;y \leq TMax$이다.
그리고 두 값을 곱한 값($p$) 범위는 
$-2^{w-1} \times (2^{w-1} - 1) \leq p \leq -2^{w-1} \times -2^{w-1}$
$-2^{2w-2} + 2^{w-1} \leq p \leq 2^{2w-2}$이 된다.
- 그리고 2의 보수의 덧셈과 같이,
그저 $U2T_w((x \times y) \bmod 2^w)$으로 값을 곱한 후
하위 $w$비트만 남기고 절삭 후($p \bmod 2^w$) 2의 보수로 해석한다.
- $x *_w^t y$는 앞에서처럼 이렇게 정의될 수 있다.
$x *_w^t y = U2T_w((x\times y) \bmod 2^w)$
그저 unsigned(부호 없는 정수)와 같은 방식으로 곱한 후,
Two's-Complement(signed, 2의 보수)로 비트 열을 해석한다.

2. 2의 보수를 부호없는 정수(unsigned)처럼 취급해서 곱한 후 상위 비트를 절삭하는 것이
문제가 되지는 않을까?
- 전의 덧셈과는 다르게, 하위 $w$비트의 값에 문제가 생길 수 있지 않을까?
- signed와 unsigned는 같은 비트열이라도 최상위 비트의 의미가 완전히 다르니,
음수의 가중치가 $-2^{w-1}$인 부호 비트를 unsigned의 최상위 비트인 $2^{w-1}$처럼 취급해서
곱해도 괜찮을까?
- 결국 최상위 비트가 1일 경우 $2^w$의 값이 차이나게된다.. 이게 문제가 되지는 않을까?
- 결국, 비트 레벨에서 unsigned의 곱셈과 2의 보수의 곱셈이 같은지(equivalence, 동등성),
절삭되고 남은 하위 $w$비트가 기대되는 값과 같아지는지를 알면 된다.
  - 어떤 $w$비트 2의 보수 정수 $x$, $y$가 있을 때, 
  이들을 unsigned로 해석한 값을 각각 $x'$, $y'$라고 하자.
  - 그러면 $x'$와 $x$는 서로 같은 비트열을 다르게 해석한 값이므로, 
  둘은 $\bmod \;2^w$ 관점에서 같은 값을 가진다.
  - 즉, 비트 레벨에서는,
  $T2B_w(x *_w^t y) = U2B_w(x' *_w^u y')$
  이라고 정의할 수 있다.
- 비트의 관점에서 살펴보자.
  - $w = 3$인 3비트의 정수라고 해보자.
    - $-3 *_w^t 3 = [101] \times [011] = 5 *_w^u 3$
    - $5 *_w^u 3 = 15 = [001111]$
    - $-3 *_w^t 3 = -9 = [110111]$
  - 이렇게 비트 표현이 다르다고 생각할 수 있지만,
  하위 $w$비트만 남기고 절삭하니 
  둘 다 $[111]$이라는 비트열이 된다.
  - 결과의 $2w$길이의 비트열은 다를 수 있지만,
  $w$비트만 남겼을 때의 최종 저장되는 결과 비트열은 같아진다.
- 값의 관점에서 살펴보자.
  - 값의 관점으로는 최상위 비트의 음의 가중치($-2^{w-1}$)가 양의 가중치($2^{w-1}$)로 바뀌니
  최상위 비트가 1이었던 경우 $+2^{w}$만큼의 값이 변한다.
  즉, 
  $x' = x + x_{w-1} 2^w$
  $y' = y +y_{w-1}2^w$
  라고 할 수 있다. 
  원래 값에 최상위 비트의 변한 가중치를 고려해서 더한 것.
  - $(x' \times y') \bmod 2^w$을 한다면,
  $[(x + x_{w-1}2^w) \times (y + y_{w-1}2^w)] \bmod 2^w$
  $= [x \times y + (x_{w-1}y + y_{w-1}x)2^{w} + x_{w-1}y_{w-1}2^{2w}] \bmod 2^w$
  $2^w$의 배수인 항은 $\bmod 2^w$에 의해 0이 되므로,
  $= (x \times y) \bmod 2^w$
  으로 값의 관점으로도, 그러니까 수학적인 관점으로도 
  $p \bmod 2^w$했을 때 기대되는 값과 같아진다.
- 정말 쉽게 풀어보자면,
  - signed 값을 unsigned 값으로 바꿔 해석했을 때 생기는 차이는,
  최상위 비트가 1인 경우에만 $2^{w}$가 더해지는 것이다.
  - 즉, 원래 값에 $2^w$의 배수가 추가된 것이라고 볼 수 있다.
  - 비트의 관점에서는 $2^w$의 배수를 곱셈에 추가한다고 해도,
  그 추가 항들은 전부 $w$번째 비트 이상의 비트들로만 표현이 된다.
  - 하위 $w$비트만 남기는 순간,
  그 추가 항들은 모두 사라진다.
  - 값의 관점에서는 $2^{w}$의 배수를 곱셈에 추가한다고 해도,
  그 추가 항들은 전부 $2^{w}$의 배수이니 
  $(추가\;항들)\bmod 2^w = 0$이므로
  추가 항들의 값은 모두 사라진다.
  - 따라서 signed로 곱하든 unsigned로 곱하든,
  결과의 하위 $w$비트 비트열은 동일해진다.
  
3. 총정리
- signed를 unsigned로 바꾼 영향은 전체 곱셈 값에는 존재한다.
하지만 그 영향은 모두 $2^w$의 배수 형태로만 나타난다.
- 따라서 그 영향은 최종적으로 저장되는 $w$비트의 결과에는 영향을 주지 않는다.
- 따라서 고정 크기 $w$비트 정수 곱셈에서 하위 $w$비트만 남기면,
signed로 곱하든 unsigned로 곱하든 결과 비트열은 동일하다.

## 3. Multiplying by Constants - 상수에 의한 곱셈

### 정리

1. 예전의 많은 컴퓨터에서 곱셈 연산은 덧셈, 뺄셈, 시프트 연산에 비해 많이 느렸다. 
10클럭 이상 잡아먹을 정도로.
- 덧셈, 뺄셈, 시프트, 기타 비트 연산들은 1클럭이면 되었다.
- 그래서 더 빠르게 곱셈에 대한 연산을 최적화해야했다.
- 그 방법이 시프트, 덧셈, 뺄셈 연산의 조합으로
기존의 곱셈 연산을 대체하는 것이었다.

2. 왼쪽 시프트 연산($x << k$)은 시프트하는 비트의 수($k$)에 따라 
값이 $x \times 2^k$로 변한다.
- 다만, 고정 크기 $w$비트 정수에서는 상위 비트가 절삭된다.
$(x \times 2^k) \bmod 2^w$
- (unsigned 기준, $w = 4$)
$[1011] = 11$
`[1011] << 2 == [101100]`
$[101100] = 44 = 11 \times 2^2$
$w = 4$이므로 하위 $4$비트를 제외한 비트 절삭
$[1100] = 12$
값의 관점에서는 $44 \bmod 16 = 12$
오버플로우가 발생할 수야 있지만 어쨌든 이렇게 기존의 값에 $2^k$가 곱해진다.
- 이 연산은 값의 관점에서는 다음과 같다.
$B2U_{w+k} ([x_{w-1}, x_{w-2}, ... x_{0}, 0, ... ,0])$
$= \sum_{i=0}^{w-1}x_{i}2^{i+k}$(전체 비트열이 k만큼 상위 비트로 이동했으니 각 비트의 가중치도 k만큼 증가)
$= [\sum_{i=0}^{w-1}x_{i}2^i] \times 2^k$
$= x \times 2^k$
이렇게 기존의 값에 $2^k$ 가 곱해지게 된다.
- 물론 기존의 최상위 $k$개의 비트는 절삭되지만,
확실히 고정 크기 $w$비트 정수의 곱셈이다.

3. CSAPP의 고정 크기 비트 수준 모델에서의
signed / unsigned 정수의 곱셈은,
일단 `x << k`가 
$x *_w^u 2^k$
$x *_w^t 2^k$(하위 $w$비트의 결과는 같음)
나 다름없다는 점에서부터 시작한다.
- 물론 실제로 이런걸 C언어에서 구현하게 된다면, 
오버플로우나 음수 왼쪽 시프트는 주의해야 한다.
요즘은 컴파일러가 곱셈 연산을 이렇게 해준다.
- 예를 들어, `x * 14`의 연산을 수행해야한다고 하면,
$14 = 2^3 + 2^2 + 2^1$이다.
즉, 산술 연산으로는 $x \times (2^3 + 2^2 + 2^1)$을 수행하는 것과 같다.
- $x \times (2^3 + 2^2 + 2^1)$
$= (x \times 2^3) + (x \times 2^2) + (x \times 2^1)$
으로 분배법칙에 의해 표현될 수 있고,
$x \times 2^3 = x << 3$
$x \times 2^2 = x << 2$
$x \times 2^1 = x << 1$
로 연산될 수 있다.
- 그러므로 결국 `x * 14`의 연산을 수행할 때
$x \times 14 = (x << 3) + (x << 2) + (x << 1)$의 연산을 수행해서
1클럭의 시프트 연산, 덧셈 연산으로,
총 시프트 연산 3번 + 덧셈 연산 2번으로 곱셈 연산을 대체할 수 있다.
- 하지만 여기서 중요한게, 
단순히 $2^k$의 합뿐만 아니라 차로도 상수 곱셈을 표현할 수 있다는 것이다.
- 위에서 한 `x * 14`도
차로 표현한다면 $x \times (2^4 - 2^1)$로 표현할 수 있고,
결국 $x \times 14 = (x << 4) - (x << 1)$처럼
총 시프트 연산 2번 + 뺄셈 연산 1번으로 곱셈 연산을 수행할 수 있다.
- 컴파일러는 이런 표현 중 더 적은 연산으로 계산할 수 있는 형태를 선택할 수 있다.

4. 어떻게 시프트 연산을 해야 상수에 의한 곱셈을 만들 수 있을까?
($x * K$)($K$는 상수)
- 일단 상수의 1과 0을 연속되는 것끼리 부분비트열로 묶어준다. (예시는 $14$)
$14 = [(0...0)(111)(0)]$
- 그리고 1이 연속되는 부분비트열에서 최상위 비트의 위치를 $n$,
최하위 비트의 위치를 $m$이라고 잡는다.
$14 = [(0...0)(111)(0)]$ 
여기서는 연속되는 1의 비트열이 한 개이고, 
최상위 비트 번호($n$)가 3, 최하위 비트 번호($m$)가 1이다.
- 시프트 연산을 사용해서 두 가지 형태로 해당 부분비트열을 표현할 수 있다.
  - $(x << n) + (x << (n-1)) + ... + (x << m)$
  - $(x << (n + 1)) - (x << m)$
  - $x \times 14 = (x << 3) + (x << 2) + (x << 1)$
  $x \times 14 = (x << 4) - (x << 1)$
  그대로 적용된다.
- 컴파일러는 이 두가지 방법 중에 더 연산 횟수가 적은 방법을 사용해서
곱셈 연산을 한다.

### 느낀 점 & 배운 점

이전 파트들이 
> 곱해진 값이 어떻게 표현될까?

에 초점을 맞췄다면,

이번 파트는
> 실제로 어떻게 곱셈 연산을 단순한 연산들의 조합으로 계산할까?

에 초점을 맞춘 내용이었던 것 같아.

진짜 이번 파트는 진짜 신기했어.
단순히 시프트 연산 + 덧셈 연산 + 뺄셈 연산으로
이렇게 간단히 곱셈을 구현할 수 있다는게ㅋㅋ

결국, 곱하는 수가 상수일 경우에, 고정된 값일 경우에는
그 상수를 $2^k$들의 덧셈, 뺄셈으로 표현해서
각각 k만큼 왼쪽 시프트 연산을 수행한 후
그대로 더하거나 빼버려서 곱셈을 할 수 있다는게..

진짜 신세계인 것 같더라..

그리고 `x*K`를 $2^k$들의 덧셈, 뺄셈으로 표현하는 방식을 일반화한 부분이
정말 신기하다고 생각했어.
- $(x << n) + ... + (x << m)$
- $(x << (n+1)) - (x << m)$

이렇게 단순한 공식만으로 연속되는 1을 시프트 연산으로 표현할 수 있다는게ㅋㅋ
