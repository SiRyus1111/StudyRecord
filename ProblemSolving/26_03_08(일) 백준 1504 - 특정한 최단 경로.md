## 2. 문제 요약

### 2-1. 문제 설명

방향성이 없는 그래프가 주어지고,
1번 정점에서 N번 정점으로 최단거리로 이동해야 한다.
또한 임의로 주어진 두 정점을 반드시 통과해야한다.

한번 이동했던 정점 / 간선도 다시 이동할 수 있다.

1번 정점에서 N번 정점으로 이동할 때, 임의의 두 정점을 반드시 거치는 최단 경로를 구하는 문제다.

### 2-2. 입력

첫 번째 줄 : N E
- N = 정점의 개수(2 ≤ N ≤ 800)
- E = 간선의 개수(0 ≤ E ≤ 200,000)

두 번째 줄부터 E개의 줄 : a b c
- a = 양방향 간선으로 연결된 정점 1
- b = 양방향 간선으로 연결된 정점 2
- c = 양방향 간선의 가중치

그 다음 줄 : v1 v2

- v1, v2 = 반드시 거쳐야 하는 정점(v1 ≠ v2, v1 ≠ N, v2 ≠ 1)

### 2-3. 출력

첫째 줄에 두 개의 정점을 지나는 최단 경로의 길이를 출력한다. 
만약 그러한 경로가 없다면 -1을 출력한다.

### 2-4. 예시

입력
```
4 6
1 2 3
2 3 3
3 4 1
1 3 5
2 4 5
1 4 4
2 3
```

출력

```
7
```

설명 

1 -> 2 = 3
2 -> 3 = 3 + 3 = 6
3 -> 4 = 6 + 1 = 7

이 경로가 1 -> N(4)로 가는 두 정점(2, 3)을 지나는 최단 경로이다.
다른 경로는 두 정점(2, 3)을 지나며 1 -> N(4)로 가도 이 경로보다 짧을 수 없다.

// 문제 유형 : 다익스트라
// 권장 시간 복잡도 : 노드(V) = 최대 800개, 간선(E) = 최대 200000개
// 다익스트라 알고리즘의 시간복잡도 = O(E log V)로는 200000 log 800이고,
// 시간 제한이 1초 = 약 1억번의 계산 가능 이므로, 다익스트라로 충분히 풀 수 있다.
// 가중치는 최대 1000, 간선 = 최대 200000개이므로 1000 * 200000 = 200000000 = 2억이므로 int형을 써도 된다. int형은 최대 21억.

// 발상 :
// 그냥 시작 지점이랑 두 정점(v1, v2)에서 다익스트라를 돌려서
// 시작 지점 -> v1 -> v2 -> 도착 지점
// 시작 지점 -> V2 -> V1 -> 도착 지점
// 중에 더 최단경로가 짧은 것을 고르면 된다.
// 그리고 이 두 개의 최단 경로를 구하려면
// 시작지점에서 시작 -> V1,V2
// V1에서 시작 -> V2,도착지점
// V2에서 시작 -> V1,도착지점
// 딱 이렇게 세 정점으로부터 시작하는 다익스트라만으로도 가능하다.

#include <queue> // priority_queue

const int MAX_VALUE = 1 << 30; // INF라고 쓰는게 편했는데 깜빡하고 MAX_VALUE 써버림.. 자바가 어렴풋이 기억이 났나봐.

std::vector<int> dist(int node_num, const std::vector<std::vector<std::pair<int, int>>>& graph){ // graph는 딱 읽기 작업만 하기 때문에 그걸 명시하고자 const를 넣었다.
    
    // 전체 노드 갯수. 나중에 +1할텐데 왜 -1을 했냐면, 전체 노드 갯수라는 것을 의미한다는걸 확실히 할려고 했다.
    // -1을 안하면 값이 전체 노드 갯수 +1이 되므로..
    int n = (int)graph.size() - 1; 

    // 우선순위 큐
    std::priority_queue<
    std::pair<int, int>, // 우선순위 큐에 저장할 값의타입 
    std::vector<std::pair<int, int>>, // 실제로 우선순위 큐 역할을 할, 실제로 값을 담을 컨테이너
    // 처음에는 단순히 std::greater<int>를 사용했지만, 실제로 우선순위 큐에 들어가는 값의 타입은 pair<int, int>이므로 정상적으로 greater<>를 사용하기 위해서는 우선순위 큐에 들어가는 값에 타입에 맞춰야한다.
    // greater<>를 사용하면, compare(A, B)로 값들을 비교하는 식으로 동작하는데, 
    // 여기서는 단순히 std::greater<int>를 사용하면 실제로 비교해야하는 std::pair<int, int>가 아니라 그저 int만 비교하므로 에러가 발생한다.
    std::greater<std::pair<int, int>> // 우선순위 비교 기준 (여기서는 최소 우선) / 왜 최소 우선이 greater이냐고 할 수 있지만, 
    > pq;

    // 시작할 때는 시작 노드로부터 모든 노드로의 최단 경로가 기록되지 않았으므로 
    // 가능할리가 없는 아주 큰 값으로 최단 경로를 저장하는 dist(distance)를 저장하는 vector를 초기화해놓는다.
    // 길이는 각각의 노드의 최단 경로이므로 전체 노드의 갯수 + 1(1 ~ n번 인덱스까지 있어야함)
    // 그냥 그래프의 인덱스 수만큼 만들어주면 된다.
    std::vector<int> dist (n + 1, MAX_VALUE);

    dist[node_num] = 0;
    pq.push({0, node_num});

    // 본격적인 다익스트라
    while (!pq.empty()){ // 종료 조건 = 우선순위 큐가 빌 때

        // 우선순위 큐의 가장 우선순위 높은 원소(top)를 꺼낸다.

        // 그렇게 변수명을 지어준다.
        // 물론 top()으로 가장 우선순위가 높은 원소를 복사한 후
        // pop()으로 꺼내주는 식이다.        
        auto [cur_wei, cur_node] = pq.top(); 
        pq.pop();

        // 만약 이미 기록되어있는 그 노드의 최단경로가 우선순위 큐에서 꺼낸 최단 경로보다 짧다면
        // 우선순위 큐에서 꺼낸 최단 경로는 쓸모가 없다. 이미 더 짧은 경로로 이 노드에 도달할 수 있으니까.

        if (dist[cur_node] < cur_wei) continue;

        // 본격적인 현재 노드와 연결되어있는 노드 탐색
        // graph는 vector라서 for-each문을 돌릴 수 없을 것 같지만,
        // 모던 C++에서는 충분히 가능하다.
        // 컴파일러가 vector라이브러리의 begin()과 end() 함수를 이용해서
        // 시작인덱스부터 끝인덱스까지 순회할 수 있다.
        // 그리고 값은 auto로 받는다. auto는 vector에 저장된 값의 타입을 자동으로 추론해서 저장해준다.
        for (auto [next_wei, next_node] : graph[cur_node]){ 
            
            int w = cur_wei + next_wei; // 현재 노드에서 현재 노드와 연결되어있는 다음 노드로 가는 누적 가중치 계산
            if (w < dist[next_node]){ // 만약 그 값이 지금까지 계산된 그 노드(다음 노드)로 가는 누적 가중치보다 작다면
                dist[next_node] = w; // 갱신 후
                pq.push({w, next_node}); // 우선순위 큐에 push하기
            }
        }
    }

    return dist;
}

### 3-1. 핵심 발상 - 왜 다익스트라를 3번만 돌리면 될까?

발상에 나와있긴 한데,
솔직히 저거 글씨 작아서 잘 안 읽히니까,
이렇게 핵심 발상들은 나눠서 써봤어.

일단 1번 노드에서 v1, v2를 둘 다 거쳐서 N번 노드에 도달하려면,
- `1 -> v1 -> v2 -> N`
- `1 -> v2 -> v1 -> N`

이렇게 두 가지 경우의 수 밖에 없어.

그래서 딱 이 여섯 가지의 정보만 구하면 되겠지.
- `1 -> v1`
- `1 -> v2`
- `v1 -> v2`
- `v1 -> N`
- `v2 -> v1`
- `v2 -> N`

그래서 뭔가 여섯 번의 다익스트라를 돌리면 되겠지 싶어.

물론 이 문제의 그래프는 무방향 그래프라서 
`v1 -> v2`와 `v2 -> v1`의 최단 거리가 같긴 해.

하지만, 다익스트라는 **한 노드에서 다른 모든 노드로의 최단 경로를 구하는 알고리즘**이니까,
같은 시작 지점에서 출발하는 최단 경로의 정보는 한 번의 다익스트라로 구할 수 있어.

위의 여섯가지 정보들은
- 1번 노드에서 다익스트라 1회 : `1 -> v1` / `1 -> v2` 둘 다 구할 수 있음
- v1 노드에서 다익스트라 1회 : `v1 -> v2` / `v1 -> N` 둘 다 구할 수 있음
- v2 노드에서 다익스트라 1회 : `v2 -> v1` / `v2 -> N` 둘 다 구할 수 있음

그렇기 때문에, **단 세 번의 다익스트라**로도 해결할 수 있어.

물론 도착 노드의 값 하나만 반환하지는 않고, 
다익스트라를 완료해서 최단 경로 계산이 다 끝난 vector를 반환해서 모든 노드에 대한 최단 경로를 그대로 반환해.

### 5-1. greater<>

일단 처음에 뭐 이상한 에러가 났는데,~~(사진 못 찍음)~~
바로 `std::greater<>`의 인수를 잘못 써서 에러가 났었어.

처음에는 단순히 `std::greater<int>`를 사용했지만, 
실제로 우선순위 큐에 들어가는 원소의 타입은 `pair<int, int>`야. 
정상적으로 `greater<>`를 사용하기 위해서는 우선순위 큐에 들어가는 값에 타입에 맞춰야해.

왜 그렇냐면,
`greater<>`를 사용하면 `compare(A, B)`로 값들을 비교하는 식으로 동작하는데, 
여기서는 단순히 `std::greater<int>`를 사용하면 
`compare(pair<int, int>, pair<int, int>` 식으로 비교를 해야하는데, 
`compare(int, int)` 식으로 비교하게 되어서
실제로 비교해야하는 우선순위 큐의 원소인`std::pair<int, int>`가 아니라 
그저 `int`만 비교하므로 에러가 발생하게 돼.

실제로 `greater<>`가 비교해야 하는 값의 타입은 `pair<int, int>`인데, 그저 `int`만 비교하게 하면 에러가 날 수밖에 없지.

### 5-2. 간선이 0개일 때 이상한 값이 출력된다?

이게 일단 위의 예시 입력을 넣었을 때는 정상적으로 출력이 됐었어.
그리고 딱히 문제도 없어보였고.
그래서 백준에 채점을 넣었는데,
90% 쯤에서 실패가 뜨더라고.

보통 그 정도에서 실패가 뜨면, 특수 케이스에 대한 예외처리가 문제가 있는거라서
제일 특수한 케이스인 간선이 0개일 때 뭔가를 잘못 처리했다는 느낌이 들었어.

실제로 간선 0개로 설정하고 실행해보니까,
이상한 값이 뜨더라고.
![](https://velog.velcdn.com/images/siryus0907/post/9a6c93a9-700e-4d9b-a103-9f60b488d8a3/image.png)

나는 이게 간선 0개일 때 for-each문에서 잘못된 인덱스(쓰레기 값)에 접근해서
결과로도 쓰레기 값이 나오는 줄 알았어.

하지만 애초에 for-each문은 컨테이너에 원소가 없을 때는 아예 실행조차 안된다고 하더라고.
그래프의 간선이 0개일 때 for-each문에서 잘못된 인덱스에 접근한다는 내 생각이 틀린거지.

어쨌든 그 발상의 결과물이, 간선(e)이 0개일 때 동작하는 이 코드야.
```cpp
if (e == 0){
    std::cout << -1;
    return 0;
}
```
내 의도는 간선이 0개면 1번 노드에서 N번 노드에 절대 도달할 수 없으니
-1을 출력하고 프로그램을 종료하는 코드야.
그래서 간선이 0개일 때는 잘 돌아가던데,
다시 백준에 채점을 넣어보니 실패가 떴어.

하지만 애초에 for-each문은 컨테이너에 원소가 없을 때는 아예 실행조차 안된다고 하더라고.
그래프의 간선이 0개일 때 for-each문에서 잘못된 인덱스에 접근한다는 내 생각이 틀린거지.

이게 왜 그렇지? 하고 계속 생각을 해보다가,
C++이 파이썬이랑 다른 점이 생각이 났어.
파이썬은 수가 너무 커지면 알아서 메모리 공간을 할당해주지만,
C++은 오버플로우가 났었거든.
그런데 내가 기존에 짰던 코드는,
```cpp
int path1 = start_dist[v1] + v1_dist[v2] + v2_dist[v]
```
이런 식이었는데,
셋 다 MAX_VALUE였으면 오버플로우가 나겠더라고.

생각해보면 간선이 0개고 두 정점에서의 dist 값을 더했을 때 
저 세 값이 전부 MAX_VALUE여서 오버플로우가 난 것 같더라고.

그래서 애초에 도달 가능 / 불가 여부를 이렇게 먼저 판단,
그러니까 하나라도 MAX_VALUE 값이 있나 살펴본 후,
```cpp
bool path1_valid = 
start_dist[v1] < MAX_VALUE &&
v1_dist[v2] < MAX_VALUE &&
v2_dist[v] < MAX_VALUE;

bool path2_valid =
start_dist[v2] < MAX_VALUE &&
v2_dist[v1] < MAX_VALUE &&
v1_dist[v] < MAX_VALUE;
```
이렇게 최단 경로를 구했어.
```cpp
int shortest_path = MAX_VALUE;
if (path1_valid) {
	shortest_path = std::min(shortest_path, start_dist[v1] + v1_dist[v2] + v2_dist[v]);
}
if (path2_valid) {
    shortest_path = std::min(shortest_path, start_dist[v2] + v2_dist[v1] + v1_dist[v]);
}
```
그리고 출력은 이렇게.
만약 shortest_path가 그대로라면, 그러니까 처음 초기화 할 때의 값인 MAX_VALUE라면,
shortest_path가 갱신이 안되었다는 거고, 도달할 수 없다는 거니까 -1을 출력했어.
```cpp
// 만약 도착 지점에 도달할 수 없다면 -1 출력
// 노드 두 개일 때 두 노드가 연결이 안되어 MAX_VALUE가 나오는 경우까지 고려
if (shortest_path == MAX_VALUE){     
	std::cout << -1;
}
// 도달할 수 있다면 최단 경로 출력
else{
    std::cout << shortest_path;
}
```

결국, 간선(e)의 개수가 0일 때의 예외처리는 그저 특정 케이스만 막는 땜빵에 불과했고,
진짜 문제는 MAX_VALUE의 값이 더해지면서 오버플로우가 난게 근본적인 원인이었어.

이걸로 느낀건,
> 단순히 특정 케이스만 막는 예외처리와
근본 원인을 해결하는 디버깅은 만족감 자체가 다르다

라는 느낌이었어.

마치 내가 정말 고통받던 질병의 정확한 원인을 찾아내서 치료한 느낌이랄까?
그저 통증을, 증상을 가라앉히려 진통제만 먹는게 아니라,
그 증상의 원인 자체를 치료해버리는 것 같았어.
