## 1. 서론

이번엔 오랜만에 복습이 아닌 새 문제를 풀어봤어.
**백준 2565번 - 전깃줄**문제였는데,
다이나믹 프로그래밍 - LIS(Longest Increasing Subsequence, 가장 긴 증가하는 부분수열)의 기본적인 문제였어.

## 2. 문제 요약

### 2-1. 문제 설명

두 전봇대 A와 B가 서로 마주보고 있고,
여러 개의 전깃줄이 두 전봇대를 연결하고 있다.

각 전깃줄은

- A 전봇대의 어떤 위치
- B 전봇대의 어떤 위치

를 연결한다.

하지만 전깃줄들이 서로 교차하면 문제가 생기기 때문에,
전깃줄들이 **서로 교차하지 않도록 만들려고 한다.**

이를 위해 몇 개의 전깃줄을 제거해야 하는데,
**가능한 한 적은 개수의 전깃줄만 제거**하려고 한다.

즉,

전깃줄들이 **서로 교차하지 않도록 최소한으로 제거해야 하는 전깃줄의 개수**를 구하는 문제다.

---

### 2-2. 입력 형식

첫 번째 줄

- `N`

  - 전깃줄의 개수
  - 범위: `1 ≤ N ≤ 100`

다음 `N`개의 줄

각 줄에는 두 개의 정수 `A`, `B`가 주어진다.

- `A`: A 전봇대의 연결 위치
- `B`: B 전봇대의 연결 위치

범위:
`1 ≤ A, B ≤ 500`

각 전봇대의 위치는 서로 중복되지 않는다.



### 2-3. 출력 형식

* 전깃줄이 서로 교차하지 않도록 하기 위해
  **제거해야 하는 전깃줄의 최소 개수**를 출력한다.

---

### 2-4. 예시

입력

```
8
1 8
3 9
2 2
4 1
6 4
10 10
9 7
7 6
```

출력
```
3
```
설명

전깃줄들이 서로 교차하지 않도록 만들기 위해서는
일부 전깃줄을 제거해야 한다.

가능한 조합 중에서
**최소 3개의 전깃줄을 제거하면**
나머지 전깃줄들이 서로 교차하지 않게 만들 수 있다.

따라서 정답은 **3**이다.

## 3. 발상(주석)과 코드

```cpp
// 백준 2565번 - 전깃줄
// 문제 유형 : 다이나믹 프로그래밍

// 권장 시간 복잡도 : 전깃줄 개수 = 100이하의 자연수, 전봇대 위치의 번호 = 500이하의 자연수. 
// 제한시간 = 1초
// 일일히 전깃줄 하나하나, 전봇대 번호 하나하나 탐색할 때 100 * 500 = 50000번의 연산 필요함.
// = 1초에 1억번 연산이라고 가정하면 O(N^3)까지도 가능.

// 발상 : 
// 이 문제는 정석적인 LIS(Longest Incresing Subsequence) 문제이다.
// 결국 전선이 교차하지 않게 최소 갯수의 전선을 뺀다는 것은 최대한 전선을 많이 남겨놓는 것과 같다.
// 그렇다면, 전선이 교차하지 않는다는 조건을 만족할 때의 가장 긴 부분 수열의 길이는
// 최대한 전선을 많이 남겨놓는 것
// 이나 다름이 없다.

// 그래서 이 문제의 핵심은 이것이다.
// "어떻게 전선을 교체하지 않는다는 조건을 구현해야할까?"

// 그 방법은 일단 전봇대 A의(물론 B를 기준으로 삼아도 된다) 
// 각각의 전깃줄의 위치를 기준으로 오름차순으로 입력받은 전선들을 정렬한 후,
// 계속 현재 전선보다 이전의 전선들을 살펴보면서
// 이전의 전선의 B에 연결되어있는 위치의 값을 보고
// 현재의 전선의 B에 연결되어있는 위치의 값보다 크다면 
// 교차되어있는 것으로 판단하는 식으로 구현을 한다.
// 이미 전봇대 A를 기준으로 전선들이 정렬되어있으므로 
// 전봇대 A는 무조건 A에 연결되있는 위치가 이전의 전선의 값보다 현재의 전선의 값이 클 수밖에 없다.
// 그러므로 교차되어있는 것은 B의 이전의 전선의 B에 연결되어있는 위치의 값이
// 현재의 전선의 B에 연결되어있는 위치의 값보다 큰 경우이다.

// 솔직히 이건 그림을 그리면 이해하기 쉽다.

#include <iostream> // 표준 입출력
#include <vector> // vector
#include <algorithm> // max(), sort()

struct wire{ // 구조체로 각각의 전선 구현
    int A_pole = 0; // 전봇대 A에 연결된 전선
    int B_pole = 0; // 전봇대 B에 연결된 전선
};

bool compare(wire A, wire B){ // 전봇대 A를 기준으로 오름차순 정렬할 때 사용하는 함수
    return A.A_pole < B.A_pole;
}

int main(){
    std::ios::sync_with_stdio(false);
    std::cin.tie(nullptr);

    // 전선의 수 입력받기
    int n;
    std::cin >> n;

    // 전선들을 저장할 poles vector. 전선의 수만큼 vector의 길이 설정.
    std::vector<wire> poles(n);

    // 최대 부분 수열 길이를 저장할 dp vector. 
    // 모든 인덱스를 1로 초기화. 일단 자기자신만으로 최대 부분 수열 길이가 1이기 때문.
    std::vector<int> dp(n, 1);

    // 전선들 입력받기
    for (int i = 0; i < n; i++){
        int a, b;

        std::cin >> a >> b;

        poles[i].A_pole = a;
        poles[i].B_pole = b;
    }

    // 전봇대 A를 기준으로 오름차순 정렬
    // 디버깅 포인트
    // 밑의 dp vector 갱신 문제를 고쳤는데도 그저 dp vector가 갱신만 될 뿐 아예 이상한 값이 나왔다.
    // 애초에 이 로직이 poles vector가 A_pole을 기준으로 오름차순 정렬되어있다는 가정 하에 만들어진 로직이라서,
    // 이렇게 정렬을 해줘야했다.
    std::sort(poles.begin(), poles.end(), compare);


    // i = 지금까지의 가장 긴 증가하는 부분 수열을 구할 인덱스 번호
    for (int i = 1; i < n; i++){
        // j = 이전의 전선들을 탐색할 용도의 인덱스 번호
        for (int j = 0; j < i; j++){

            // 이전의 전선들을 탐색할 때 전선이 교차하지 않는지 확인하는 조건문
            // 최적화 포인트
            // 원래는 이 if문의 조건이 이랬다.
            // !(poles[i].A_pole > poles[j].A_pole && poles[i].B_pole < poles[j].B_pole) || (poles[i].A_pole < poles[j].A_pole && poles[i].B_pole > poles[j].B_pole)
            // 두 전선이 교차하는지 아닌지 판별하기 위해서 만든 조건식이다.
            // 하지만 이미 A_pole에 대해 오름차순 정렬 되어있으므로, 그저 B_pole이 교차하는지만 보면 된다. 이렇게 복잡한 조건식을 쓸 필요가 없다.
            if (poles[i].B_pole > poles[j].B_pole){

                // 지금까지의 가장 긴 증가하는 부분수열인지 확인.
                // 디버깅 포인트
                // 처음에 dp vector가 갱신이 안되어서 여기서 cout로 출력을 해봄으로써 위의 if문이 정상적이라는 것을 확인했다.
                // 하지만 dp vector가 갱신이 안 되었기 때문에 아래의 max() 함수가 비정상적이었다고 생각해서, 결국 문제를 찾아냈다.
                // dp[i] = std::max(dp[i], dp[j]);
                // 이렇게 dp[i] = 현재의 전선 수와 이전의 전선 수 중에 더 큰 것
                // 으로 하고있었다.. 실제로는 dp[i]는 dp[j]의 다음 단계가 되므로 dp[i]와 dp[j] + 1을 비교해야한다.
                // 역시 밤에 집중력 딸리는 상태로 알고리즘 문제를 푸는 것은 위험하다. 이런 초보적인 실수를 하게 된다.
                dp[i] = std::max(dp[i], dp[j] + 1);
            }
        }
    }

    // 가장 긴 증가하는 부분 수열의 길이를 저장할 변수
    int max_amount = 0;

    // 가장 긴 증가하는 부분 수열을 dp vector를 순회하며 찾기
    // 디버깅 포인트
    // 원래는 반복문이 
    // for (int i = 1; i < n; i++)
    // 이었는데 n == 1인 경우를 처리하지 못했다.
    // 사실 원래는 dp[i]랑 dp[i-1]을 비교하게 하려고 했는데,
    // 그 때 잘못된 인덱스에 접근하는걸 막기 위해 i = 1번부터 시작했었다.
    // 하지만 max를 사용하는 방식으로 바꿨는데 그걸 수정하지 않은 탓에
    // n == 1인 경우를 처리하지 못했다.
    // 코드를 다 짠 후 수정이 필요한 곳이 있는지 꼼꼼히 확인해야겠다..
    for (int i = 0; i < n; i++){
        max_amount = std::max(dp[i], max_amount);
    }

    // 출력
    // 문제에서 요구하는게 
    // 최소 몇 개의 전선을 제거해야하는지ㅣ임므로
    // 전체 갯수 - 남길 수 있는 최대 갯수(가장 긴 증가하는 부분수열의 길이)를 출력해야한다.
    std::cout << (n - max_amount);
}

// 결과 : 메모리 = 2024KB, 시간 = 0ms
```

### 3-1. 핵심 발상

이 문제는 결국 **전선이 서로 교차하지 않도록 최대한 많이 남기는 문제**야.
즉, 제거해야 하는 전선의 최소 개수를 구하려면 먼저 **교차하지 않는 전선의 최대 개수**를 구해야 해.

그러면 여기서 핵심은 이거야.

> 어떤 전선들이 서로 교차하지 않는지, 그 조건을 어떻게 단순하게 볼 수 있을까?

전선 두 개를 생각해보자.
예를 들어 `(A1, B1)`과 `(A2, B2)`라는 두 전선이 있을 때,

- `A1 < A2`인데 `B1 > B2`이면 두 전선은 교차
- 반대로 `A1 > A2`인데 `B1 < B2`여도 역시 교차

즉, **A 전봇대에서의 순서와 B 전봇대에서의 순서가 서로 뒤집히면 전선이 교차한다**고 볼 수 있어.

여기서 전봇대 A를 기준으로 전선을 오름차순 정렬하면,
이제 전선들은 **A 전봇대에서는 항상 작은 번호에서 큰 번호 순서로만 나열**돼.

그 말은 곧,
이후에는 A 쪽 순서는 더 이상 신경 쓰지 않아도 되고
**B 전봇대에서의 순서만 보면 된다**는거지.

정렬된 상태에서 어떤 이전 전선 `poles[j]`와 현재 전선 `poles[i]`를 비교할 때,
만약 `poles[j].B_pole < poles[i].B_pole`라면
A에서도 순서가 맞고 B에서도 순서가 맞으므로 두 전선은 교차하지 않아.

반대로 `poles[j].B_pole > poles[i].B_pole`라면
A에서는 앞에 있던 전선이 B에서는 뒤로 가게 되므로 순서가 뒤집혀 교차하게 되겠지.

즉, 전봇대 A를 기준으로 정렬한 뒤에는
**전봇대 B의 위치값이 증가하는 전선들만 고르면 서로 교차하지 않는다는거야.**

결국 이 문제는
전봇대 A 기준 정렬 이후, **전봇대 B에 대한 LIS(Longest Increasing Subsequence)** 를 구하는 문제라는거야.

그리고 이 LIS의 길이가 곧 **서로 교차하지 않게 남길 수 있는 전선의 최대 개수**이므로,
최종 정답은 다음과 같다.

> 전체 전선 개수 - LIS 길이

---

### 3-2. 네 줄 요약

1. 전선이 교차한다는 것은 A와 B에서의 **순서가 뒤집히는 것**이다.
2. 전봇대 A를 기준으로 정렬하면 A에서의 순서는 고정된다.
3. 따라서 이제는 전봇대 B의 값이 **증가하는지**만 보면 된다.
4. 그래서 이 문제는 **B에 대한 LIS 문제**가 된다.

## 4. 최적화 포인트

### 4-1. if문 조건식 최적화

처음에는 내가

- `A1 < A2`인데 `B1 > B2`이면 두 전선은 교차
- 반대로 `A1 > A2`인데 `B1 < B2`여도 역시 교차

이 **두가지 조건을 전부 생각해서** 교차를 판별하는 if문의 조건식을
```cpp
!((poles[i].A_pole > poles[j].A_pole && poles[i].B_pole < poles[j].B_pole) 
|| (poles[i].A_pole < poles[j].A_pole && poles[i].B_pole > poles[j].B_pole))
```
이렇게 길게 잡아놨었는데,

애초에 **이미 전봇대 A를 기준으로 오름차순을 정렬했기 때문**에, 
`(poles[i].A_pole < poles[j].A_pole && poles[i].B_pole > poles[j].B_pole)`
이 부분과
`poles[i].A_pole > poles[j].A_pole`
이 부분이 필요가 없었어. 

어차피 **전봇대 B에서의 전선의 위치**만 보면 되기 때문에..

## 5. 디버깅 포인트

### 5-1. dp vector 갱신 안됨

내가 이거 백준에 나와있는 예시 입력값을 넣어서 실행해봤는데
이상한 값이 뜨는거야.
그래서 가장 긴 부분수열 구하는 곳(사실상 출력 직전) 전에 중단점 찍고 디버거로 디버깅을 해봤어.

근데 `dp vector`에 저장된 값이 전부 초기값이었던거야..
그래서 일단 교차하지 않는 것을 검사하는 조건문(`poles[i].B_pole > poles[j].B_pole`)에 문제가 있나 싶어서
그 조건문 안에 `cout`로 메시지를 출력하는 코드를 추가로 작성한 후 다시 실행해봤어.

그런데 메시지가 출력은 잘 되는데 `dp vector`의 값은 하나도 갱신이 안되더라?

그래서 그 if문 안에 있었던 `max()` 함수를 살펴봤어.

### 5-2. max() 함수의 인수 잘못됨

```cpp
dp[i] = std::max(dp[i], dp[j]);
```
결국 이 `max()` 함수가 문제였는데,
이게 현재 전선 이전까지의 LIS의 길이(`dp[j]`)와 현재 전선까지 저장된 LIS의 길이(`dp[i]`)를 비교해서 
만약 `dp[i]`가 더 크다면 값을 유지하고, `dp[j]`가 더 크다면 `dp[j]`로 값을 갱신하는 코드였어.

여기서 내가 간과했던게, 
결국 이전까지의 LIS의 길이(`dp[j]`)는 현재 전선을 포함하지 않고 계산된 길이였고
현재 전선까지 저장된 LIS의 길이(`dp[i]`)는 현재 전선을 포함해서 계산된 길이였단거야.

그래서 `dp[i]`와 **`dp[j]`에 현재 전선까지 포함한 값**을 비교해야했던거지.
결국 `dp[j]`에 현재 전선까지 포함한 길이는 `dp[j] + 1`이니까,
`max()` 함수를
```cpp
dp[i] = std::max(dp[i], dp[j] + 1);
```
이렇게 고쳤어.

고치니까 `dp vector`의 값이 변경이 되긴 하더라.

그런데..

### 5-3. 전봇대 A를 기준으로 오름차순 정렬 빼먹음

이건 진짜 기초적인 실수인데,
내가 입력이 전봇대 A 기준으로 오름차순으로 들어오는줄 알고 딱히 정렬을 할 생각을 못했어..

그래서 전에 있던 버그들이 고쳐졌어도 `dp vector`의 값만 변할뿐 이상한 값이 나왔었지.

결국 `algorithm` 라이브러리의 `sort()` 함수를 사용해서 정렬했어.
그리고 정렬 기준은 따로 함수를 정의해서 딱 전봇대 A를 기준으로 정렬하도록 했어.


### 5-4. dp vector의 최댓값을 구하는 부분의 초기 i값을 잘못 설정함

처음에는 `dp[i]`와 `dp[i-1]`을 비교해서 최댓값을 계속 갱신하는 식으로 for문을 돌리려 했기 때문에 초기 `i`값이 1이었었어.
하지만 중간에 max() 함수를 쓰는 것으로 로직이 갈아엎어졌었는데, 
딱히 초기 `i`값은 바꾸지 않아서 
`i = 0`부터 탐색해야할 것을
`i = 1`부터 탐색하는 것으로 대참사가 일어났었어.
```cpp
for (int i = 1; i < n; i++){ // 0번 인덱스부터 탐색해야 하는데 1번 인덱스부터 탐색하려고 했던 잔재가 남아있었다.
        max_amount = std::max(dp[i], max_amount);
    }
```
이러면 `n == 1`인 경우를 고려할 수 없었을 뿐만 아니라,
첫 인덱스를 탐색을 못하니 최댓값을 판단할 때 모든 인덱스를 고려하지 못한 것으로 인한 문제가 발생할 수 있어서..
