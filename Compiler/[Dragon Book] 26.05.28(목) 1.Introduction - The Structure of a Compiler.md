## 1. Analysis - 분석

일단 분석(Analysis)은, 앞에서 설명한 것처럼
> 소스 코드를 분석해서 
합성에 사용할 IR(Intermediate Representation, 중간 표현)을 생성하는 작업

이라고 할 수 있어.
일단 소스 코드(문자열 스트림)를 받고,
최종적으로 IR(중간 표현)을 반환하는 과정이지.

결국 한 마디로 요약하면 이거야.
> 소스 코드를 어셈블리 코드가 생성되기 쉬운 형태로 만든다.

이 과정에서
- 토큰
- 구문 트리
- 심볼 테이블(Symbol Table)

같은 개념이 나오는거야.

그리고 이 과정은,
대표적으로
- Lexical Analysis(어휘 분석) : 소스 코드를 토큰으로 분해하기
- Syntax Analysis(구문 분석) : 토큰들로 구문 트리 만들기
- Semantic Analysis(의미 분석) : 구문 트리가 제대로 된 구문 트리인지 검사하기
- Intermediate Code Generation(중간 표현 생성) : 구문 트리 기반으로 중간 표현(IR, 중간 코드) 생성하기

이런 과정을 거치게 돼.

---

### 1-1. Lexical Analysis - 어휘 분석

일단 소스 코드를 받았으면 
컴파일러는 먼저 해당 소스 코드를
토큰(Token)이라는 단위로 나눠.

그리고 그걸 `Lexical Analysis`(어휘 분석)라고 해.

예를 들어,
```cpp
y = x + 10;
```
이라는 코드가 있다면,
```
y -> <IDENTIFIER, 0x01>
= -> <ASSIGN_OP, NULL>
x -> <IDENTIFIER, 0x02>
+ -> <ADD_OP, NULL>
10 -> <NUMBER, 10>
; -> <SEMICOLON, _>
```
이런 식으로 토큰으로 분해하는거지.

토큰은 Introduction 파트에서는 이 정도로 설명이 되어있어.
```text
<token-type, attribute-value>
```
토큰의 종류와, 속성 값.
- token-type = 해당 토큰이 뒤에 나올 Syntax Analysis 과정에서 어떤 의미인지를 나타내는 식별자
  - `KEYWORD`(키워드), `IDENTIFIER`(식별자), `NUMBER`(상수), `SEMICOLON`(세미콜론(`;`)) 등
  - 해당 토큰의 의미라고 생각하면 됨.
- attribute-value = 토큰 종류만으로는 표시할 수 없는 구체적인 데이터나 참조 값.
  - 상수(literal)인 경우 : 실제 값
  - 식별자(identifier)인 경우 : 심볼 테이블에서 해당 식별자가 위치한 인덱스 또는 포인터(주소) 


뭔가 변수의 타입 / 상수의 타입같은건 보이지 않는데,
이런건 따로 심볼 테이블(Symbol Table)이라는 테이블(표)에 적어놔서 참조하는 식으로 사용해.

심볼 테이블에는,
- 식별자 이름(Lexeme/Name) : 변수나 함수의 실제 이름
- 타입(Type) : Data Type(`int`, `float`, `char*`, `void` 등)
- 종류(Kind/Category) : 해당 식별자가 무엇인지(지역 변수, 전역 변수, 함수, 매개변수, 상수 등)
- 스코프(Scope) : 해당 식별자가 유효한 범위(전역, 특정 함수 내부, `{}` 내부)

이런 것들이 들어가.

쉽게 설명하면,
- Token(토큰) = 컴파일러가 해당 토큰의 종류를 식별해서 **구문 트리를 작성하는데 필수적인 정보**를 담음.
- Symbol Table(심볼 테이블) = 소스코드의 등장하는 식별자의 정보를 모아두는 데이터베이스.

이라고 할 수 있어.

그리고 심볼 테이블은 단순히 Lexical Analysis에서만 쓰이는게 아니라,
앞으로의 과정에서 계속 갱신되고, 계속 참조돼.
각 변수 / 함수들의 정보를 모아놓은 데이터베이스 역할을 하다보니까,
구문 트리를 생성할 때도,
구문 트리가 실제로 실행될 수 있는지 검사할 때도,
중간 표현을 생성할 때도,
그 후의 synthesis(합성) 과정 에서도..

그리고 TMI로,
심볼 테이블은 개인적으로
내 블로그 글과 비슷하게 느껴지네.

내가 특정 개념이 기억 안 날 때
예전에 쓴 블로그 글을 찾아보듯,

컴파일러도 토큰 자체에 모든 정보를 저장하지 않고
심볼 테이블을 참조해서

- 타입
- 스코프
- 변수/함수 여부

등을 확인하는 느낌이니..

---

### 1-2. Syntax Analysis - 구문 분석

이제 소스코드를 토큰으로 분리했으니
IR을 생성하기 쉬운 구문 트리(syntax tree)의 형태로 바꿔줘.

단, **해당 구문 트리의 의미가 실제로 실행되었을 때에 문제가 없는지**,
즉 **언어 의미 규칙이 맞는지**는 검사 / 수정 / 보완하지 않아.
그저 **토큰들을 기반으로 구문 트리를 만들 뿐**이야.
그건 다음 단계인 의미 분석에서..

그저 구문 트리(syntax tree)를 만들어줘.

방금 전의 예시로 계속 설명을 해볼게.
```cpp
y = x + 10;
```
이 코드로 생성된
```text
<IDENTIFIER, 0x01> // y
<ASSIGN_OP, NULL>
<IDENTIFIER, 0x02> // x
<ADD_OP, NULL>
<NUMBER, 10>
<SEMICOLON, _>
```
이런 식의 토큰이 있다고 하면,

```text
              <ASSIGN_OP, NULL>
                /          \
  <IDENTIFIER, 0x01>       <ARITH_OP, NULL>
                            /          \
              <IDENTIFIER, 0x02>      <NUMBER, 10>
```
이런 식으로 구문 트리를 생성해줘.

대충 이런 의미임.
```text
    = 
   / \
  y   +
     / \
    x   10
```

구문 트리를 읽어보면,
```cpp
y = x + 10;
```
이라는 코드가 생각이 날 것 같지?

단, 앞에서 말했듯이 
**언어 의미 규칙을 지킨 구문 트리인지**는 검사 / 수정 / 보완하지 않아.
이거 정말 중요함.

그래서 이 단계에서 발생할 수 있는 에러는
`토큰들로 구문 트리가 생성될 수 없는 경우`가 많아.

---

### 1-3. Semantic Analysis - 의미 분석

전에 말했듯이,
만들어진 구문 트리가
**해당 언어의 의미 규칙이 맞는지**를 검사 / 수정 / 보완하는 단계야.

대표적으로 타입 체크가 있지.

```cpp
y = x + 10;
```
만약 방금의 이 코드에서 `y`랑 `x`가 `float`(floating-point) 타입이라면?
그리고 `10`은 평범한 `int`(integer) 타입이고..

`int`와 `float`를 같은 식에서 연산하지?
그리고 최종적으로 `float` 타입의 `y`에 대입되고.

그러면 `int`인 `10`을 `float` 타입으로 바꾼 후 연산을 하는 식으로,
이렇게 구문 트리의 의미를 바꿔주는거지.
```text
              <ASSIGN_OP, NULL>
                /          \
  <IDENTIFIER, 0x01>       <ARITH_OP, NULL>
                            /          \
              <IDENTIFIER, 0x02>      inttofloat
                                          |
                                      <NUMBER, 10>
```

컴파일러는 의미 분석 단계에서 
해당 구문 트리가 **해당 언어의 의미 규칙**을 지키는지를 검사하고 보완해.

물론, 보완 가능한 경우. 방금처럼 inttofloat같은 경우.
이럴 때는 충분히 보완 가능한 경우라서 보완을 하지만,

보완 불가능한 경우. 아예 타입 자체가 안 맞는다거나..
이럴 때는 보완 불가능한 경우로 보고 에러를 발생시켜.

```cpp
// 보완 불가능한 경우 예시
int k = "hello"; // 문자열을 int에 대입(int로 변환 자체가 불가능)

int* ptr = 10; // 정수를 포인터에 직접 대입(이거 의미적으로 말이 안됨)
```

만약 의미가 맞지 않아서 해당 코드를 실행할 수 없다?
그러면 해당 구문 트리는 생성되었지만, 실제로 실행할 수는 없는 상태인거야.

즉, 이 단계에서 발생할 수 있는 에러는
`코드의 문법(syntax)은 맞았지만 의미적으로 보완 자체가 불가능한 경우`가 많아.

### 1-4. Intermediate Code Generation - 중간 코드 생성

이제 구문 트리를 검사해서 **해당 언어의 의미 규칙을 지키는지**가 보장되었으니
해당 구문 트리로 중간 표현(중간 코드, IR)을 만들어야겠지?

즉, Intermediate Code Generation이란 
완성된(유효한) 구문 트리(valid syntax tree)와
`Syntax Analysis`, `Semantic Analysis`를 거치며 채워진 `Symbol Table`을 이용해서,
IR(Intermediate Representation, 중간 표현 / 중간 코드)를 만드는 단계야.

```cpp
y = x + 10;
```
이 코드에서
앞의 세 단계를 거쳐서
```text
              <ASSIGN_OP, NULL>
                /          \
  <IDENTIFIER, 0x01>       <ARITH_OP, NULL>
                            /          \
              <IDENTIFIER, 0x02>      inttofloat
                                          |
                                      <NUMBER, 10>
```
이런 구문 트리가 완성되었지?

```text
t1 = inttofloat(10)
t2 = id2 + t1
id1 = t2
```
그리고 이런 느낌으로 IR로 바꿔줘.
(`id1` = `y를 가리키는 식별자`, `id2` = `x를 가리키는 식별자`)
 
**소스코드를 중간 표현(IR)로 바꾸는 `Analysis` 과정**은 여기가 마지막이야.

그리고 중간 표현(IR)이란,
앞에서 설명했듯이
> 소스 코드로 어셈블리 코드가 생성되기 쉬운 형태

라고 할 수 있어.

심볼 테이블에 대해 좀 짧게 설명하고 넘어갔는데,
사실 이게 상당히 중요해서 한번 더 쓰네.

심볼 테이블은,

계속 사용되고, 계속 신규 정보가 있을 때마다 갱신돼.

이거 정말 중요해.

### 1-6. IR을 왜 쓸까?

그리고 왜 굳이 바로 어셈블리로 바꾸지 않고
중간 표현(IR)으로 바꾸는지 의문이 들 수도 있는데,

만약 IR 없이 
소스코드를 읽자마자 각 CPU 아키텍처에 맞는 어셈블리로 번역한다고 가정해 보자.

만약 우리가 컴파일러를 만들어야 하는 프로그래밍 언어가 3개(C++, Rust, Go)이고, 
타겟 CPU 아키텍처가 3개(x86_64, ARM, RISC-V)라면?

각 언어별로 CPU 아키텍처에 맞는 컴파일러를 전부 따로 만들어야 해서, 
총 **$3 \times 3 = 9$개**의 컴파일러를 밑바닥부터 만들어야 해. 

언어와 CPU가 늘어날 때마다 컴파일러 개수가 제곱으로 폭발하는 지옥이 열리지.

하지만 중간에 **`IR`이라는 공통된 독립 언어**를 딱 끼워 넣으면 판도가 완전히 바뀌어.
- **Front-end(Analysis):** 
소스코드를 분석해서 공통 IR로 만드는 빌더만 언어별로 만들면 OK. 
(3개)
- **Back-end(Synthesis):** 
그 공통 IR을 받아서 각 CPU에 맞는 어셈블리로 조립하는 제너레이터만 아키텍처별로 만들면 OK. 
(3개)

결과적으로 컴파일러를 **$3 + 3 = 6$개**만 만들어도 모든 조합이 완벽하게 작동하게 되는 거지.

이건 분석하는 과정이 실제 어셈블리를 생성하는 과정(합성)과 분리될 수 있기 때문에 가능한거야.

즉,
> 일부러 분석의 결과물을 특정 CPU에 종속되지 않는 고정된 하나의 언어로 뽑아내서,
분석의 결과물을 서로 다른 CPU 아키텍처로 번역만 하면 되게 한다.

라고 할 수 있겠지.

## 2. Synthesis - 합성

이제 Analysis 과정으로 생성된 
IR(중간 표현)을 기반으로 
각 CPU 아키텍처에서 실행 가능한 코드로 변환을 해야겠지.

즉 최종적으로는,
각 CPU 아키텍처에 맞는 어셈블리 코드(기계어와 거의 1 : 1로 대응되는 표현)가 생성되는거야.

근데 이 과정은 어쩔 수 없이 각 CPU 아키텍처마다 꽤 큰 차이를 보여.

당연하지.
IR이 모든 CPU 아키텍처의 어셈블리로 번역될 수 있다고 해도,
번역 과정은 각 CPU 아키텍처마다 다를 수 밖에 없으니까.

그래서, 지금 설명하는 Synthesis(합성) 과정은
> Synthesis 과정에서 구체적으로 무슨 일이 일어나는가.

보다는,
> 결국 Synthesis 과정에서 공통적으로 무슨 일을 하는가.

의 뉘앙스로 설명을 할 생각이야.

그리고, Synthesis 과정에서 공통적으로 하는 일은
이렇게 두 가지 정도로 나눌 수 있어.
- Code Optimization(코드 최적화) : Analysis 과정에서 생성된 IR을 좀 더 최적화하기
- Code Generation(코드 생성) : 최적화된 IR로 실제 어셈블리 코드를 생성하기

그리고 이 과정에서도
**심볼 테이블**이 쓰이지..

심볼 테이블은 Analysis 과정에서만 사용하는게 아니라,
Synthesis 과정에서도

- 변수의 타입
- 저장 위치
- 함수 정보 등

을 참조하기 위해
계속 사용돼.

이래서 심볼 테이블이 중요하다고 한거야.

### 2-1. Code Optimization - 코드 최적화

이제 기존의 소스코드를 분석(Analysis)해서 IR을 생성했으니,
그 생성된 IR을 좀 더 성능을 올리도록 최적화를 해.
물론 CPU 아키텍처에 상관없이 적용될 수 있는 최적화도 많고,
특정 CPU 아키텍처를 고려하는 최적화도 있고.

예를 들어,
- IR을 해당 CPU 아키텍처에 맞게 좀 더 최적화하고..
- IR에 있는 중복, 즉 굳이 필요 없는 코드를 줄이고..
- 어쨌든 IR이 해당 CPU 아키텍처에 맞는 어셈블리로 번역되었을 때
좀 더 빠르고, 짧은 코드가 되도록

최적화하는 과정이야.

기존의
```cpp
y = x + 10;
```

```text
t1 = inttofloat(10)
t2 = id2 + t1
id1 = t2
```
으로 생성한 해당 IR도,
10을 쓸데없이 실수(`float`)로 바꾸는 과정이 있지?
그냥 10.0을 `id2`(`x`)와 더해주면 될텐데..

그래서 이렇게 최적화 될 수 있어.
```text
id1 = id2 + 10.0
```
세 줄의 코드가 한 줄로 줄어서 더 짧아졌지?
그리고 굳이 `inttofloat`연산자를 쓰지 않고 
`10.0`으로 바로 연산을 하는 식으로 성능을 최적화했지?

즉, 굳이 필요 없는 코드를 줄였지?

물론 컴파일러는 이 방법 말고도

- 상수 계산(Constant Folding)
- 죽은 코드 제거(Dead Code Elimination)
- 공통 부분식 제거(Common Subexpression Elimination)

같은 방법으로 IR을 최적화해줘.

Code Optimization이란,
이런 식으로 IR을 말 그대로 최적화하는 과정이야.

물론 다양한 CPU 아키텍처가 있기 때문에,
해당 예시보다 훨씬 다양한 최적화가 적용되겠지.

### 2-2. Code Generation - 코드 생성

이제 IR을 번역해야하는 CPU 아키텍처에 맞게 번역을 해.
그런데 이 과정은 각 CPU 아키텍처마다 다를 수 밖에 없어서,
대표적인 두 가지 과정을 설명해볼게.

1. 레지스터 할당(Register Allocation)
  - IR의 어떤 변수가 실제로 어떤 레지스터에 들어갈지 정하는 작업인
  레지스터 할당(Register Allocation)
  - 이것도 심볼 테이블을 보고 정함.
  - 물론 심볼 테이블 뿐만이 아닌 실시간 정보나 IR 분석 결과도 많이 사용함.
2. 명령어 선택 및 순서 배열(Instruction Selection and Scheduling)
  - IR을 실제 타겟 CPU의 어셈블리로 조합
  
이렇게 크게 두 가지 정도의 과정을 거쳐서
최종적으로 
타겟 CPU의 어셈블리(타겟 머신의 코드 표현)가 만들어지게 되는거야.

### 2-3. Reordering과의 연결점

그리고 Code Optimization과 Code Generation 사이에서 
내가 이 책을 보게 된 원인인,
명령어 재정렬(Instruction Reordering)이 발생해.

메모리에서 데이터를 가져오는 동작은 시간이 많이 걸려.
그래서 컴파일러가 내가 쓴 순서 그대로 어셈블리로 바꾸게 된다면,
CPU가 메모리에서 데이터가 오는걸 기다리느라 아무런 일도 안하고 노는 시간(Stall)이 생겨.

그래서 컴파일러가 성능을 높이려고
> CPU가 노는 시간 없이 계속 실행되게 하기 위해서
연산의 최종 결과만 똑같게 기계어 명령어 순서를 재정렬(Reordering)

해버려.

결국 내가 Reordering에 대해 확실히 알아보기 위해서는
Code Optimization과 Code Generation을 제대로 공부해보면 된단거지.

## 3. 총정리

이제 맨 위쪽의 이미지가 잘 이해될 것 같아.

요약해보자면,
> 소스 코드의 텍스트를 토큰으로 쪼개며 각 토큰들에 의미를 부여하고(**Lexical Analysis**)
> 그 토큰들의 구조를 잡아 구문 트리를 만들고(**Syntax Analysis**)
> 만들어진 구문 트리의 의미가 잘 맞는지 검사 / 보완하고(**Semantic Analysis**)
> 검사 / 보완한 구문 트리를 공통의 중간 표현(IR)으로 변환하고(**Intermediate Code Generation**)
> 생성된 IR을 좀 더 최적화하고(**Code Optimization**)
> 최적화된 IR로 타겟 CPU 아키텍처의 어셈블리를 생성한다(**Code Generation**)

> 심볼 테이블은 어휘 분석(Lexical Analysis) 부터 생성되어
최종적으로 코드 생성(Code Generation)까지 계속 
각 식별자들의 데이터베이스로써 기능한다.

## EX. 간단히 설명하는 어셈블러

어셈블리의 각 명령어(Operator)와 피연산자(Operand)는
실제 해당 CPU의 기계어와 1 : 1로 대응되기 때문에,
컴파일러의 분석(Analysis)과 합성(Synthesis)으로 생성된 어셈블리를
그냥 1 : 1로 기계어와 바꿔주면 기계어가 생성돼.

그리고 그 기계어(목적 파일)를 링커로 다른 기계어(목적 파일), 라이브러리 등과 엮어주면
컴퓨터가 실행할 수 있는 실행 파일이 생성되지.

여기는 메인 주제가 아니라 간단히 설명했어.
실제론 좀 더 복잡한데,
컴파일 과정보단 훨씬 간단함.
