## 2. TCP 서버의 기본 구조와 소켓 API들

**TCP 서버 - 클라이언트의 기본 구조**는 

- ```socket()```
- ```bind()```
- ```listen()```
- ```accept()```
- ```connect()```
- ```recv()```
- ```send()```
- ```closesocket() ```

이렇게 8가지의 소켓 API들로 표현할 수 있어. 
그림과 함께 보자.  
![](https://velog.velcdn.com/images/siryus0907/post/3774f18b-8f60-4d25-b903-1fbdb1ceba7d/image.jpg)
**이미지 오류 있음, listen()을 bind()와 accept() 사이에 추가해야함. 추후 수정 예정**


### 2-1. socket()

일단 처음에 서버와 클라이언트에서 **socket()** 함수로 **소켓을 생성**해. 
저번에 socket() 함수를 이미 했었지만, 
복습하는 차원에서 다시 한번 보자.

일단 socket() 함수는 
```cpp
socket(int 주소_체계, int 소켓_타입, int 프로토콜);
```
이런 인수를 받아.

**주소 체계**에는 
말 그대로 주소 체계(AF_INET, AF_INET6 등)을 넣어.

**소켓 타입**에는 
이 소켓의 타입, 
그러니까 신뢰할 수 있는 통신 / 연결형 통신(TCP)인지, 
신뢰할 수 없는 통신 / 비 연결형 통신(UDP)인지를 넣어주는거야. 
전자면 SOCK_STREAM, 후자면 SOCK_DGRAM.

**프로토콜**에는 
말 그대로 프로토콜 이름(IPPROTO_TCP, IPPROTO_UDP 등)을 넣어줘.

마지막으로 socket() 함수는 **성공하면 소켓(소켓 핸들)**을, 실패하면 **INVALID_SOCKET**을 반환해.
그래서 **SOCKET 자료형으로 반환 값(소켓 핸들)를 받아야 해**.

socket() 예시: 
```cpp
SOCKET sock = socket(AF_INET, SOCK_STREAM, IPPROTO_TCP); // IPv4, TCP로 설정
```

식당으로 비유하면,
> 사업자 등록을 하고 주방 설비를 갖춰라.

라는거야.

그리고 socket() 함수는 윈도우의 경우에는 **성공하면 소켓**을, **실패하면 INVALID_SOCKET**을 반환해.

이번 글에서는 이 단계에서 서버에서 생성한 소켓은 **server_sock**, 클라이언트에서 생성한 소켓은 **c_sock**이라고 할게. 


### 2-2. bind()

그 다음엔 서버에서 **bind()**로 **server_sock(서버 소켓)에 IP 주소와 포트 번호를 할당**해.

bind() 함수는 
```cpp
bind(SOCKET 클라이언트_접속_수용_소켓, const struct sockaddr* 소켓_주소_구조체, int 소켓_주소_구조체_길이);
```
이런 인수를 받아.

**클라이언트 접속 수용 소켓**에는 
말 그대로 **클라이언트의 접속을 수용할 소켓**, 
그러니까 방금 socket() 함수를 호출해서 생성한 server_sock 소켓을 넣으면 돼.

그리고 **소켓 주소 구조체**에는 
**서버의 IP 주소, 서버에서 데이터를 받을 포트 번호, 주소 체계를 담은 
소켓 주소 구조체의 포인터(주소)**를 넣어.

마지막으로 **소켓 주소 구조체 길이**에는 
말 그대로 **소켓 주소 구조체의 길이(바이트 단위)**를 넣으면 돼. 
막 이렇게? 
`sizeof(server_addr)`

bind() 예시: 
```cpp
bind(server_sock, (sockaddr*)&server_addr, sizeof(server_addr)); // server_sock 소켓에 server_addr 소켓 주소 구조체의 IP 주소와 포트 번호를 할당
```

식당으로 비유하면,
> 우리 식당의 주소(IP)를 확정하고 간판(포트 번호)을 달아라.

라는거야.

그런데 이미 다른 가게가 그 주소(포트)에서 장사하고 있는데 내가 거기서 장사하겠다고 하면 싸움나겠지?
그래서 bind()를 호출했을 때 에러가 난다면, 이미 누군가 이 주소와 포트 번호를 선점해서 쓰고있다는 뜻으로 알면 돼.

그리고 전에 포트 설명할 때 말한건데, 
일반적으로 서버는 **```0 ~ 1023```번의 웰 노운 포트(시스템 포트)**나
**```1024 ~ 49151```번의 등록된 포트**를 사용해.
그리고 클라이언트는 일반적으로 **```49152 ~ 65535```번의 동적 포트**를 사용해.
이게 왜 그렇냐면, 나중에 ```connect()``` 설명할 때 설명할거긴 한데,
서버는, **클라이언트가 서버의 IP와 포트를 미리 알고 있어야** ```connect()```(연결)를 시도할 수 있어서
서버의 포트가 ```49152 ~ 65535```번의 동적 포트면 계속 포트 번호가 바뀌어서 클라이언트가 ```connect()```(연결)를 시도할 수 없어.
하지만 클라이언트는 **서버에 잠시 접속했다 나가기 때문에**, 딱히 포트 번호가 고정될 필요가 없어.
그러니까 **딱 서버에 연결하는 순간에만 자신을 식별할 번호가 필요**하니까,
딱히 고정된 포트 번호를 사용하지 않고, OS가 자동으로 할당해준 동적 포트를 써도 되는거야.
그래서 클라이언트는 ```49152 ~ 65535```번의 동적 포트를 사용하는거야.

그리고 만약 식당 주인이 잠깐 자리를 비웠는데(서버 종료),
바로 다시 돌아와서 문을 열려고 해.
그런데 건물주(OS)가 아직 내가 맺었었던 이전 계약 정리(TIME_WAIT)가 안 끝났다며 
지금 이 자리에 못 들어간다고(bind() 못 한다고) 할 수 있거든?
이런 에러도 있어..
이건 **Address already in use**에러라고 해.

그리고 bind() 함수는 윈도우의 경우에는 **성공하면 0**을, **실패하면 SOCKET_ERROR**를 반환해.

### 2-3. listen()

이제 서버는 **```listen()```**으로 소켓의 TCP 상태를 **```LISTENING```**으로 바꿔. 
전에 **3-way handshake** 했을 때 봤었지? 바로 그거야.
서버가 클라이언트 연결 기다리는 상태.
그러니까 클라이언트가 접속 요청(```connect()```)을 보낼 수 있게 소켓을 열어놓는다고 생각하면 돼.

```listen()``` 함수는 
```cpp
listen(SOCKET TCP_상태_바꿀_소켓, int 백_로그);
```
이런 인수를 받아.

일단 **TCP 상태 바꿀 소켓**은 
말 그대로 **LISTENING** 상태로 바꿀 소켓을 넣는거야. 
그러니까 **클라이언트가 연결할 소켓**.

그리고 **백 로그**는 
**서버가 당장 처리하지 않더라도 접속 가능한 클라이언트의 수**야. 
클라이언트의 접속 정보는 연결 큐 형식으로 저장되는데, 
이 연결 큐(백 로그 큐)의 길이를 넣는거야. 
사용 가능한 최댓값을 넣으려면 **```SOMAXCONN```** 값을 넣으면 돼.

그리고 백 로그 큐는 3-way handshake를 진행할 때의 **SYN 세그먼트를 처리하는 큐만 있는게 아니라**,
3-way handshake가 끝나고 **ESTABLISHED** 상태가 되어서 **```accept()```를 기다리고 있을 때의 정보(소켓 핸들 / TCB)를 처리하는 큐**도 있어.
근데 이건 ```accept()``` 볼 때 이해가 될거야 아마..

listen() 예시: 
```cpp
listen(server_sock, SOMAXCONN); // server_sock 소켓을 열고(LISTENING 상태로 바꾸고) 백 로그 큐의 길이를 최대로 설정
```

그런데 왜 굳이 백 로그 큐의 길이를 인수로 넣는걸까?
사실 listen() 함수는 단순히 소켓만 여는게 아니라, OS 내부에 따로 백 로그 큐까지 만드는 과정이기 때문에 그래.
식당으로 비유하면, listen() 함수는 
> 이제부터 손님이 오면 일단 줄(백 로그 큐에)을 세워라. 우리 식당이 처리할 수 있을 때(백 로그 큐가 꽉 찰 때) 까지.

라는거야.

그리고 ```listen()``` 함수는 윈도우의 경우, **성공하면 0**을, **실패하면 SOCKET_ERROR**를 반환해.

### 2-4. connect()

그 다음에 클라이언트가 **```connect()```**로 **서버에 접속 요청**을 해. 
그러니까 좀 더 이론적인 표현으로는, 
**TCP 프로토콜 수준에서 서버와 논리적 연결을 설정한다. **
정도로 할 수 있겠네. 

**근데 왜 클라이언트는 ```bind()``` 함수를 호출하지 않을까? **
사실은 ```bind()``` 함수를 호출하지 않은 상태에서 ```connect()``` 함수를 호출하면, 
그냥 운영체제가 자동으로 IP 주소와 포트 번호를 할당해줘서 그래. 
물론 그런 정책은 운영체제마다 달라지고..

그런데 그렇게 자동으로 할당해주면 굳이 서버도 ```bind()```안 해도 되는거 아닌가 싶겠지만,
이게 클라이언트는 어느 포트로 데이터를 내보내도 상관 없잖아? 결국 서버 IP 주소의 특정 포트에 전송되면 되니까..
하지만 서버는 무조건 특정 IP / 특정 포트로 데이터를 받아야하니까, 저렇게 ```bind()```를 하지 않고 자동으로 할당해버리면
클라이언트가 접속할 **특정 IP / 특정 포트**를 고정할 수 없어.
그래서 딱 서버가 listen()할 소켓에 특정 IP / 특정 포트를 직접 할당해줘야해.

connect() 함수는 
```cpp
connect(SOCKET 서버와_통신할_소켓, const struct sockaddr* 소켓_주소_구조체, int 소켓_주소_구조체_길이);
```
이런 인수를 받아.

**서버와 통신할 소켓**에는 
전에 ```socket()``` 함수로 생성해두었던 ```c_sock``` 있지? 
그걸 넣어주면 돼. 말 그대로 **서버와 통신할 소켓**을 넣는거야.

그리고 **소켓 주소 구조체**에는 
**접속할 서버의 IP 주소, 데이터를 보낼 포트 번호, 주소 체계를 담은 소켓 주소 구조체의 포인터(주소)**를 넣어.

마지막으로 **소켓 주소 구조체 길이**에는 
말 그대로 **소켓 주소 구조체의 길이(바이트 단위)**를 넣으면 돼.

```connect()``` 예시 :
```cpp
connect(c_sock, (sockaddr*) &c_server_addr, sizeof(c_server_addr)); // c_sock 소켓으로 c_server_addr 구조체의 주소에 논리적 연결 설정
```

식당으로 비유하면,
> 손님이 미리 알아둔 주소(IP)와 가게 이름(포트 번호)를 보고 가게를 찾아간다.
그리고 들어가도 되는지 확인한다.

라는거야.

근데 주소(IP)가 틀리면 길을 잃은거고,
주소는 맞는데 그 장소(포트)에 식당이 없으면 여긴 식당이 아니라면서 입장 거절을 당해.

그리고 ```connect()``` 함수는 윈도우의 경우 **성공하면 0**을, **실패하면 SOCKET_ERROR**을 반환해.

추가적으로, ```connect()``` 사실상 3-way handshake에서 **액티브 오픈 호스트가 SYN 세그먼트를 보내는 단계**야.

### 2-5. accept()

이제 클라이언트가 접속 요청을 보냈으니, 서버가 그 요청을 수용해줘야겠지?

서버는 **accept()**로 **클라이언트의 접속을 수용**해. 
그리고 **접속한 클라이언트와 통신할 수 있는 새로운 소켓을 생성하여 반환**해. 

내가 왜 굳이 클라이언트에서 생성한 소켓을 ```c_sock```으로 이름지었는지 알겠지? 
서버에서 **접속한 클라이언트와 통신할 수 있는 새로운 소켓을 따로 생성**하기 때문이야.
그래서 반환값을 ```SOCKET``` 자료형으로 받아야해. 
그리고 이렇게 ```accept()```로 생성한 소켓은 ```client_sock```이라고 할게.

```accept()``` 함수는 
```cpp
accept(SOCKET 클라이언트_접속_수용_소켓, struct sockaddr* 소켓_주소_구조체, int* 소켓_주소_구조체_길이_포인터);
```
이런 인수를 받아.

**클라이언트 접속 수용 소켓**에는 
이전에 ```listen()``` 함수로 LISTENING 상태로 만들어놨던 ```server_sock``` 있지? 
그 소켓을 넣어주면 돼.

**소켓 주소 구조체**에는 **접속한 클라이언트의 주소 정보를 저장할 소켓 주소 구조체**를 넣으면 돼.
그리고 **소켓 주소 구조체 길이 포인터**에는 
말 그대로 **소켓 주소 구조체의 크기를 포인터 형식으로** 넣으면 돼. 
이거 중요해. **꼭 주소(포인터)값을 넣어야함!**

그리고 접속한 클라이언트가 없을 경우 ```accept()``` 함수는 서버를 대기 상태로 만들어. 
그리고 클라이언트가 접속하면 서버는 깨어나고 ```accept()```함수가 소켓을 반환하게 되지.

```accept()``` 예시 :
```cpp
accept(server_sock,(sockaddr*) &client_addr, &addr_len) 
// server_sock 소켓으로 클라이언트의 접속을 수용하고, client_addr 구조체에 클라이언트의 주소 정보를 저장
```

식당으로 비유하면,
> 줄 서서 기다리고 있는 손님을 빈 테이블(새 소켓)로 안내해라.

라는거야.

그런데, ```accept()```는 **"줄 서서 기다리고 있는(백 로그 큐에 있는) 손님"을 빈 테이블(새 소켓)로 안내하는 것**이기 때문에,
**이미 ```connect()```가 되어서 백 로그 큐에 있는 클라이언트와의 연결 정보(소켓 핸들 / TCB)를 기반**으로 새 소켓을 생성해.
그러니까 **```accept()```를 호출하기도 전에 ```connect()```가 성공할 수도 있다**는거야.
왜냐면 이미 OS가 미리 **백 로그 큐에 클라이언트와의 연결 정보를 받아두기 때문**이야.

그런데 만약 아무 클라이언트도 `connect()`를 안해서 백 로그 큐가 비어있다면 어떨까?
사실 `accept()`함수는 백 로그 큐에 연결 정보가 들어올 때 까지 기다리는 
**블로킹(Blocking, 동기)** 방식이라,
클라이언트가 `connect()`할 때까지 기다리다 `connect()`로 백 로그 큐에 연결 정보가 들어온다면,
그 때에 그 연결 정보를 기반으로 소켓 핸들을 반환해.

추가로, 만약 클라이언트 100개가 접속하면, 서버의 포트도 100개가 열릴까 싶은데,
실제로는 포트는 여전히 하나지만, 
```accept()```가 만든 서로 다른 소켓 핸들들의 클라이언트들의 연결 정보(5-Tuple)로 클라이언트를 구분해.

그리고 ```accept()``` 함수는 윈도우의 경우 **성공하면 소켓**을, **실패하면 INVALID_SOCKET**를 반환해.

마지막으로 설명할 게 있는데, 딱 **```connect()```과정**에서 **3-way handshake가 진행**이 되고, 
3-way handshake가 **종료된 후**에, ```accept()```가 **백 로그 큐에서 연결 정보를 꺼내서** **소켓을 반환**해.
그러니까 ```accept()```는 **연결을 만들지는 않고**, 이미 **OS가 3-way handshake로 만들어놓은, 백 로그 큐에 쌓여있는 연결**을
꺼내와서 애플리케이션에 그 **정보(소켓 핸들 / TCB)를 넘겨주는 역할**일 뿐이야. 

### 2-6. 기본적인 소켓 API의 간단한 요약

- ```socket()``` → '소켓 생성'

- ```bind()``` → '서버 주소/포트 고정'

- ```listen()``` → '연결 요청 대기 시작'

- ```connect()``` → '클라이언트가 서버에 연결 요청'

- ```accept()``` → '서버가 연결 수용 + 통신용 소켓 획득'

(여기에 이미지)

### 2-7 accept()까지 완료한 후의 상태

자, 이제 본격적으로 통신하기 전의, 그러니까 **```connect()```와 ```accept()```가 끝났을 때의 상태**를 알아보자.

일단 서버의 ```bind()```와 ```listen()```을 거친 ```server_sock```은 계속해서 **LISTENING** 상태야.
이건 그냥 특정 부서로 안내해주는 역할이라고 생각하면 돼.

그리고 서버의 ```client_sock```은 클라이언트와 1:1로 매칭되어있지.
이건 특정 부서의 고객 전담 가이드라고 생각하면 돼.

이제 ```client_sock```이 실제로 클라이언트와 통신을 주고받겠지.
좀 이따 설명할 ```send()```와 ```recv()```를 수행할거야.
<br>
그리고 클라이언트는 ```connect()```가 성공적으로 반환되어서,
클라이언트의 ```c_sock```은 서버의 **특정 IP / 특정 포트**에 완전히 고정되어있어.

그리고 ```client_sock```과 ```c_sock```은 통신을 할 수 있는 **ESTABLISHED** 상태야.
그러니까 OS가 이 연결을 위해서 **송수신 버퍼를 메모리에 할당**했다는거지.
이 송수신 버퍼에 대해서는 **```send() / recv()```** 볼 때 살펴보자.
어쨌든, 이제 TCP의 바이트 스트림이라는 데이터의 물줄기가 흐를 물길이 준비가 된거야.

이렇게 통신 준비가 다 끝난 상태를 수식하는 중요한 개념이 있는데,
```accept()```에서 클라이언트들의 연결 정보라고 설명했던
**5-Tuple**이야.
소켓 통신이 확립되었다는건, 네트워크에서 이 연결을 식별하는 5가지 정보가 확정되었다는거야.

**1. 프로토콜**
**2. 송신지 IP**
**3. 송신지 포트**
**4. 수신지 IP**
**5. 수신지 포트**

이 5개가 세트로 묶여야 비로소 '세션이 성립되었다'고 해.

## 3. 게임 서버와의 연결

이게 게임 서버는 보통 로그인 서버, 로비 서버, 전투 서버(월드) 등으로 나뉘는데,
```bind()```를 할 때 특정 서버가 어떤 역할을 할지 포트 번호로 결정해.
예를 들어 ```8000번``` 포트는 로그인 서버, ```9000번``` 포트는 1채널 전투 서버인 식으로.

그리고 막 '서버 점검 중'이라는건, 
분명 프로세스는 떠 있어도 ```listen()```을 멈추거나 
백 로그 큐를 비워버려서 ```connect()```요청을 의도적으로 무시하는 상태라고 볼 수도 있어.

그리고 로그인 대기열 알지? 막 신작 게임이 오픈했을 때 "대기 인원 500명"하고 뜨는거 있잖아?
이건 게임 내부의 애플리케이션 레벨 대기열 시스템이기도 하지만, 네트워크 레벨에서는 백 로그 큐가 터져나가는 상황이야.
그래서 백로그 큐의 크기를 너무 작게 잡으면 유저들이 접속 시도조차 못 하고 '서버 연결 실패'창을 보고,
너무 크게 잡으면 3-way handshake만 끝낸 유령 유저들이 메모리를 점유해서 서버가 뻗어버릴 수도 있어.
