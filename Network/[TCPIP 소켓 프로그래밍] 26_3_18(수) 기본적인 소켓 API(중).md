이게 결국 TCP는 **바이트스트림**이라, **데이터의 경계가 없다**는 점이 아마 이번 글의 핵심일 것 같아.
추가로 송신 버퍼 + 수신 윈도우 / 혼잡 윈도우 정도?

솔직히ㅋㅋ 이것만 계속 생각하면서 `send()` / `recv()`를 봐도ㅋㅋ

## 2. TCP 서버의 송수신 과정

TCP 서버는 기본적인 소켓 API로는 연결 소켓을 사용해서 `send()` / `recv()`로 송 / 수신을 해. (listen 소켓 아님)

사실상 기본적인 수준에서는 이 두 함수만 제대로 알아도 TCP의 송수신 과정을 이해할 수 있어.

결국 `send()`로 데이터를 보내고,
`recv()`로 데이터를 받는 과정의 반복이야.

아무리 나중에 다른 소켓 함수를 쓴다고 해도,
TCP로 통신을 하는 이상
이 두 함수의 동작 과정부터 주의할 점,
특히 **이 함수들이 대체 무슨 시점에, 어떤 동작까지 하고 값을 반환하는지** 같은 정보들은
앞으로도 두루두루 쓰일 것 같아.

### 2-1-1. send()의 사용법

```cpp
int send(SOCKET 데이터를_전송할_소켓, const char* 전송할_데이터가_저장된_버퍼, int 전송할_데이터_크기, int 플래그);

int send(SOCKET s, const char* buf, int len, int flags);
```
`데이터를 전송할 소켓(s)`에는 
말 그대로 데이터를 전송할,
**데이터를 받을 수신지와 연결되어있는 소켓**이 들어가.

그리고 `전송할 데이터가 저장된 버퍼(buf)`에는 
**전송할 데이터가 담겨있는 버퍼**가 들어가.
그러니까 "hello world!"라는 데이터를 전송하려면
const char* 자료형의 버퍼에 hello world!를 담아놓은 후
send() 함수를 사용해야해.

그리고 `전송할 데이터 크기(len)`는
말 그대로 전송할 데이터의 크기.
근데 솔직히 len은 전송할 데이터의 전체 크기라기보다는,
**이번 send()를 호출했을 때 총 몇 바이트를 전송하려하겠는지**를 나타내는 값이라고 이해하면 돼.
왜 굳이 **전송하려하겠는지** 인지는 동작 원리 설명할 때 설명할게.

그리고 `플래그(flags)`는 
**추가 옵션**을 넣는 곳이라고 생각하면 돼.
**기본적으로는 0**을 넣는다고 생각하면 편해.

그리고 반환 형식은 `int` 형, 
성공하면 **한번의 send() 호출로 송신 버퍼에 넘겨준(처리한) 바이트 수**를,
실패하면 **SOCKET_ERROR**를 반환해.
윈도우 기준.

### 2-1-2. send()의 동작

이제 **동작**을 알아보자.

이 글에서는 일단 기본적인 **블로킹 소켓 기준**으로 설명할게.
이 글의 모든 함수는 **블로킹 소켓 기준**이야.

일단,

> TCP는 데이터의 경계가 없이 바이트단위로 흐르듯이 전송된다.

이걸 다시 상기해보자.

TCP는 데이터의 경계가 딱히 없어.
그러면 혹시
> `send()` 함수가 보내는 데이터도 경계가 없는거 아닐까?

라는 생각을 해볼 수 있어.

그리고 여기에 이어서,
> 결국 `send()`하려는 데이터의 크기가 MSS(Maximum Segment Size)보다 크다면?
수신 윈도우 / 혼잡 윈도우보다 send()하려는 데이터의 크기가 크다면?
보낼 데이터가 임시로 저장되는 **송신 버퍼의 공간이 부족**하다면?

> 이러면 send() 한 번에 전송되지 않을 수 있는거 아닐까?

라는 생각도 해볼 수 있지.

자, 이제 `send()` 함수가 어떻게 동작하는지를 살펴보자.

`send()` 함수는 운영체제의 송신 버퍼에 송신할 데이터를 넘겨줘.
이게 끝이야.

물론, **보낼 데이터가 많거나 송신 버퍼에 여유 공간이 부족한 경우**에는,
한 번의 `send()`만으로 **모든 바이트를 처리하지 못할 수도** 있어.
그래서 필요한 만큼 여러 번 `send()`를 해야 할 수도 있지.
그러니까 **len만큼 한 번에 다 못 보낼 수도** 있다는거야.
len이 이번 **`send()`를 호출했을 때 총 몇 바이트를 전송하려하겠는지** 라는거야.
다 못 전송할 수도 있으니까.

그래서 결국 len은
> 송신 버퍼에 넘겨줄 수 있는 최대 바이트 수.
이번 send() 호출에서 보내려고 요청한 바이트 수.

그래서 `send()` 한 번에 모든 데이터를 다 못 보낼 수 있으니,
실제로는 이렇게 while문을 이용해.

```cpp
(pseudo code)

총_보낸_바이트_수 = 0
while (총_보낸_바이트_수 < 보내야할_바이트_수){
	현재_보낸_바이트_수 = send(
    	socket, 
    	buf + 총_보낸_바이트_수, 
    	보내야할_바이트_수 - 총_보낸_바이트_수, 
    	0
    );
    
    if (현재_보낸_바이트_수 == SOCKET_ERROR) break; // 오류 처리
    
    총_보낸_바이트_수 += 현재_보낸_바이트_수 
}
```

그리고 반환 값도 딱 이 시점에 나와.
**송신 버퍼에 송신할 데이터를 넘겨준 시점**에.

즉, `send()` 함수가 성공적으로 처리한 데이터의 바이트 수를 반환했다고 해서,
- 데이터가 확실히 송신 버퍼에서 전송이 되었고,
- 전송된 데이터가 확실히 상대 호스트에게 수신되었는지

는 `send()` 함수의 반환 값만으로는 알 수 없어.

송신 버퍼에 올라가있는 데이터는 **운영체제가 전송**해주고,
전송된 데이터의 신뢰성 보장은 **TCP**가 해줘.


신뢰성 보장을 좀 더 엄밀히 말하면,
> 전송, 세그먼트 분할, 재전송, ACK 확인 같은 작업은 운영체제의 TCP/IP 스택이 처리해준다

라고 할 수 있지.

결국 프로그래머가 `send()` 함수를 호출했어도 그저 **송신 버퍼에 전송할 데이터를 넘겨준 것**일 뿐,
실제로 **전송 자체가 됐다는 보장이 있는 것은 아니**고,

- 운영체제가 전송을 해주고,
- TCP가 신뢰성을 보장

해주는거야.

그래서 send() 함수는 송신 버퍼에 넘겨준 바이트 수를 반환하는거야.
딱 송신 버퍼에 전송할 데이터를 넘겨주는 역할을 수행할 뿐, 전송이나 신뢰성 보장은 운영체제 / TCP가 처리하니까.
물론 send() 함수의 반환 값만으로는 확실히 전송이 되었고, 전송된 데이터가 확실히 상대 호스트에게 도착했다는 보장은 없지만.

결국, 
> send() = 보낼 데이터를 운영체제의 송신 버퍼에 올려놓는 함수
반환 값이 정상이라고 해서 확실히 수신 호스트에게 데이터가 도착했다는 보장은 없다.

라는거지.

~~사실 전송과 신뢰성 보장은 운영체제의 TCP 스택이 해줘. 둘이 나뉘어져있지는 않아.~~

**블로킹 소켓 기준**으로는 이 정도면 다 한 것 같네.

이제 다음은 수신지 호스트에서 데이터를 어떻게 받는지 보자.

### 2-2-1. recv()의 사용법

```cpp
int recv(SOCKET 데이터를_받을_소켓, char* 받은_데이터를_저장할_버퍼, int 받을_데이터의_크기, int 플래그);

int recv(SOCKET s, char* buf, int len, int flags);
```

`데이터를 받을 소켓(s)`에는
말 그대로 **송신 호스트와 연결되어있는,
전송된 데이터를 받을 소켓**이 들어가.

받은 데이터를 저장할 버퍼(buf)에는
말 그대로 송신 호스트로부터 받은 데이터를 저장할,
**`char*`형의 버퍼**가 들어가.

그리고 받을 데이터의 크기(len)에는
**딱 이번 `recv()`로 받을** 데이터의 **최대 바이트 수(크기)**가 들어가.
왜 **최대**인지는 원리 설명할 때 같이 설명할게.

그리고 `플래그(flags)`에는
**추가 옵션**들이 들어가.
여기서 기본적으로는 **데이터가 len만큼 다 모일 때까지 기다리도록 요청**하는 플래그인
(더 정확한 표현으로는 **버퍼가 가득 차거나, 연결이 닫히거나, 오류가 날 때까지 기다리도록 요청하는 플래그**)
`MSG_WAITALL`같은 플래그를 설정할 수 있어.

그리고 
반환 형식은 `int` 형이고,
클라이언트가 연결되어있고 성공한다면 **한번의 `recv()` 호출로 수신 버퍼에서 받아온(처리한) 바이트 수**를,
실패하면 **SOCKET_ERROR**를 반환해.

그리고 이게 제일 중요한데,
- 연결이 **정상적으로 종료**되었고, 
- 이미 받아야 할 데이터도 **다 받은 상태**라면 (이거 closesocket() 때 중요함)

`recv()`는 0을 반환해.
윈도우 기준.

### 2-2-2. recv()의 동작 원리

근데 솔직히 이거 앞에서 대부분 설명하긴 했어..
> recv() = 운영체제의 수신 버퍼에서 송신 호스트가 송신한 데이터를 가져오는 함수

이걸로 사실상 거의 끝이긴 해ㅋㅋ

그럼 좀 더 깊게 들어가보자.

일단 `recv()`는 데이터를 어떻게 가져올 것 같아?
이게 `recv()`는 막 데이터를 수신 버퍼에서 **택배마냥 자기가 받을거만 딱딱 잘 보고 들고오지 않아.**
딱 **프로그래머가 지정한 len 범위 안에서, 수신 버퍼의 앞부분부터 읽어오는 거지.**
즉, 프로그래머가 
**"딱 이 메시지 하나만 꺼내와!" **
라고 지정할 수는 **없다**는 거야.

그리고 `send()`랑 비슷하게 `recv()`도 len만큼 **한 번에 다 못 받을 수 있어.**
이런걸 방지하고자 앞에서 나왔던 `MSG_WAITALL`같은 플래그가 있지.
결국 여기서도 len은 그저 
> `recv()` 한 번으로 수신 버퍼에서 가져올 수 있는 최대 바이트 수

라고 보면 돼.

`MSG_WAITALL`같은 플래그를 설정하지 않는다면 
항상 **len만큼 받는다는 보장은 없다**는거야.
그저 **받을 수 있는 최대 바이트 수**.

그래서, `MSG_WAITALL` 플래그를 사용하지 않는다고 하면,
```cpp
(pseudo code)

총_받은_바이트_수 = 0
while (총_받은_바이트_수 < 받아야할_바이트_수){
	현재_받은_바이트_수 = recv(
    	socket, 
    	buf + 총_받은_바이트_수, 
    	받아야할_바이트_수 - 총_받은_바이트_수, 
        0
    );
    
    if (현재_받은_바이트_수 == SOCKET_ERROR) break; // 오류 처리
    
	else if (현재_받은_바이트_수 == 0) break; // 클라이언트에서 연결 종료
    
    총_받은_바이트_수 += 현재_받은_바이트_수 
}
```
처럼 구현할 수 있어.

이렇게 `send()`가 한 번에 모든 데이터를 처리하지 못할 수 있었던 것처럼, 
`recv()`도 한 번에 원하는 길이만큼 다 받아오지 못할 수 있어.
`send()`랑 `recv()`가 꽤 비슷하지?

결국,
> send()든 recv()든 내가 입력한 송 / 수신량만큼 데이터를 송 / 수신한다는 보장이 없으니 
send() / recv() 한 번에 모든 데이터가 송 / 수신될거라고 생각하면 안된다.

라는거야.

### 2-3. send()와 recv() - recv() 한 번이면 send() 한 번으로 보낸 데이터를 다 받을 수 있을까?

여기가 하이라이트야.

내가 계속
> TCP는 데이터의 경계가 없이 바이트단위로 흐르듯이 전송된다.

라고 한 이유야.

일단, 이런 의문을 가져보자.
> 송신 호스트가 `send()`를 한 번 호출했으면,
수신 호스트도 `recv()`를 한 번 호출해서 그 데이터를 그대로 받을 수 있지 않을까?

그런데, TCP에서는 `send()` **한 번으로 데이터를 전송**했다고 해도,
`recv()` 한 번에 `send()`로 **보낸 모든 데이터를 수신한다는 보장이 없어.**

왜냐하면 TCP는 **데이터의 경계가 없는, 바이트스트림**이기 때문이야.

이건 예시로 설명하는게 좋을 것 같네.

만약,
```cpp
send(socket, "ABC", 3, 0);
send(socket, "DEF", 3, 0);
```
이렇게 데이터를 **두 번에 걸쳐서** 보냈다고 하자.
그러면 `recv()`는 어떻게 데이터를 받을 것 같아?
```cpp
ABC
DEF
```
처럼 보낸대로 받는다고 생각할 수 있지?

하지만 아니야.

1.
```cpp
AB
CDEF
```
2.
```cpp
ABCDE
F
```
3.
```cpp
ABCDEF
```
처럼 받을 수도 있어.

이게 왜 그렇냐면, 
TCP는 **데이터의 경계가 없어서**
분명 송신은 두 덩어리로 했더라도,
- MSS에 따라
- 송신 호스트와 네트워크 장비들의 송신 버퍼의 여유 공간에 따라
- 수신 윈도우, 즉 수신 호스트의 수신 버퍼의 여유 공간에 따라
- 혼잡 윈도우, 즉 네트워크의 상태에 따라

중간에 더 잘게 쪼개지기도 하고,
중간에 여러 데이터가 붙기도 하고,
특정 데이터만 붙거나 쪼개지기도 하기 때문이야.

> 몇 번 SEQ부터 몇 번 SEQ까지는 1번 세그먼트, 
몇 번 SEQ부터 몇 번 SEQ까지는 2번 세그먼트 

가 아니라 
> 각각의 바이트(byte)가 물 흐르듯이(stream), 
딱히 세그먼트를 구분짓는 경계 없이, 
어떨때는 여러 바이트 덩어리들이 합쳐지고, 
어떨때는 한 바이트 덩어리가 쪼개진다 

라는거야. 

'어떨때'라는건 앞에서 말한 
**MSS / 송신 버퍼의 여유 공간 / 수신 윈도우 / 혼잡 윈도우 등에 따라 결정**되게 돼. 

혼잡이 발생해서 혼잡 윈도우가 절반이 되어서 수신 윈도우보다 혼잡 윈도우가 작아져버렸다. 
그렇다면 한 번에 보낼 수 있는 데이터의 양이 적어지니 결국 바이트 덩어리가 더 잘게 쪼개져야겠지. 

아니면 혼잡이 계속 발생하지 않아서 혼잡 윈도우가 계속 증가하고있고 수신 윈도우도 상당히 넉넉하다. 
그렇다면 `send()`를 여러번 했다고 해도 한 번에 보낼 수 있는 데이터의 양이 크니 한 번에 뭉쳐져서 보내질 수도 있겠지.

하지만 결국 수신 호스트에서는 SEQ에 따라서 
전송 과정에서 쪼개지고, 뭉쳐진 바이트들이 순서대로 합쳐지며 수신 버퍼에 들어가게 돼. 

그런데 결국 수신 호스트에서의 **순서만 보장**하지, 
**송신 호스트가 보낸 데이터 덩어리대로 도착**하는 것은 보장하지 않잖아? 

중간에 더 잘게 쪼개져서 도착할 수도 있고, 
중간에 다른 데이터와 붙어서 도착할 수도 있고, 
특정 데이터만 붙거나 쪼개져서 도착할 수도 있고.

그리고 이래서 `recv()`는 **수신 버퍼의 맨 앞에서부터 데이터들을, 바이트열을 읽어오는거야.**
딱히 덩어리로 취급하지 않으니까. 

즉 TCP는

> 송신 호스트가 보낸 "데이터 덩어리"의 경계를 보존하는 프로토콜이 아니라,
바이트들의 순서만 보장된 바이트 스트림을 전달하는 프로토콜

이라고 볼 수 있어.

그래서 결국 프로그래머는

> send() 한 번 = recv() 한 번

이라고 생각하면 안 되고,
애플리케이션 레벨에서 직접 데이터의 길이나 경계를 구분할 방법을 설계해야 해.

### 2-4. send()와 recv() - 애플리케이션 레벨에서 데이터의 길이를 어떻게 명시할까?

송신지에서 몇 바이트를 보냈는지
**수신지가 어떻게 알 수있을까?**

이게 결국 문제야.
그래서 우리가 직접 그런걸 해결 해줘야해.
예를 들어,
- 아예 고정된 데이터 크기로 데이터를 주고받거나
- 송신지 호스트에서 맨 앞의 고정된 길이의 헤더로 데이터의 전체 크기 / 페이로드 크기를 수신지 호스트에게 알려주거나
- 데이터 맨 끝에 구분자를 붙여서 데이터의 끝을 알려주거나

하는 방법이 있어.

그런데 여기서
> 송신지 호스트에서 맨 앞의 고정된 길이의 헤더로 데이터의 전체 크기 / 페이로드 크기를 수신지 호스트에게 알려주거나

이건 사실상 프로토콜을 만드는거야ㅋㅋ
내가 만들고있는 에코 서버에도 이 방법을 사용하고 있고.
