## 1. 혼잡 제어

이제 TCP의 신뢰성 보장 기법 세가지.
- 오류 제어
- 흐름 제어
- 혼잡 제어



중에 마지막인 혼잡 제어에 대해서 공부해봤어.

이 **혼잡 제어**라는 키워드는 흐름 제어에 비해 상당히 직관적인 것 같아.
**혼잡 제어**는,
> 네트워크의 혼잡 상태에 따라 한 번에 전송하는 양을 제어

라고 보면 돼.

그러니까 흐름 제어랑 간단히 비교해보면
> 흐름 제어가 수신 호스트가 받을 수 있는 양을 고려하는 것이라면, 
혼잡 제어는 네트워크 전체가 감당할 수 있는 양을 고려하는 것이라고 볼 수 있다.

지금은 이렇게만 생각해도 돼.

### 1-1. 혼잡이 무엇일까?

**`혼잡`**이란, 
> 많은 트래픽으로 인해 패킷의 처리 지연이 커지거나 패킷 유실이 발생할 수 있는 네트워크 상황

이야.

쉽게 사진과 비유로 예시를 들어볼게.

![](https://velog.velcdn.com/images/siryus0907/post/4cc1fb59-e7a3-40d6-a861-160aa1b9d370/image.webp)

짜잔. 그러니까 이거라고 보면 돼.

차들이 한번에 너무 많이 들어와서 거의 통과를 못 하고 있지?

저 차들을 패킷이라고 보면 되고,
<br>
저렇게 차들이 많으면 대기 줄이 길어지고 차들이 통과하는 속도도 느려지겠지?

이건 그러니까 위에서 설명한 **'많은 트래픽으로 인해 패킷 처리 지연이 커지는 것'**이랑 비슷해.

그리고 차가 이렇게 계속 몰리는데 대기 공간마저 부족해지면 
뒤에 새로 들어오는 차량은 더 이상 진입하기 어렵거나 우회해야 하겠지?

이게 위에서 설명한 **'많은 트래픽으로 인해 패킷이 유실될 가능성이 커지는 것'**이랑 비슷해.

### 1-2. 혼잡 제어를 하는 이유와 혼잡 제어의 뜻

그러니까 다시 CS식 설명으로 돌아오면 
라우터에 **한순간에 많은 트래픽이 몰리면** 모든 패킷을 곧바로 처리하지 못할 수 있어.
그니까 라우터에 **과부화**가 발생할 수 있다는거지.

TCP는 **세그먼트 손실이 발생했다고 판단하면 해당 세그먼트를 재전송**해.

근데 이미 라우터가 **혼잡 상태**인데?
근데 또 호스트가 **재전송**을 하네?
그러면 더 혼잡 상태가 **악화**되겠지?

이걸 막기 위해서는 어떻게 해야할까?
호스트에서 **혼잡 상태가 발생하지 않을 정도**로만 세그먼트를 보내야겠지?

그러니까,
> 호스트에서 네트워크 혼잡도를 판단하고 혼잡한 정도에 따라 유동적으로 전송량을 조절하는 것

이게 **혼잡 제어**야.

그런데 **네트워크 혼잡도를 기반으로 판단한 적정 전송량**은 **어떻게 나타나고, 어떻게 결정**되는걸까?

그건 바로 다음의 **혼잡 윈도우**가 표시해주고, **혼잡 제어 알고리즘**이 결정해줘.

### 1-3. 혼잡 윈도우

여기서 익숙한 말 나왔다.
- `윈도우`

이거 흐름 제어 기법으로 나왔었지? 슬라이딩 윈도우.

흐름 제어의 `수신 윈도우`는 
> 수신 호스트가 받을 수 있는 데이터의 양

을 나타내는 개념이었지?

혼잡 제어의 **`혼잡 윈도우`**는 상당히 비슷한 느낌인데,
> 현재 네트워크 상태에서 혼잡 없이 전송할 수 있다고 판단되는 데이터의 양

이야.

그리고 보통 혼잡 윈도우는 Congestion WiNDow에서 **CWND**만 빼서 표시해.

지금까지의 추론으로는 어쨌든 윈도우라는건
> 송신 호스트에서 데이터를 보냈을 때 문제가 생기지 않을 데이터의 양

이라고 생각하면 편해.

하지만 수신 윈도우는 문제가 생기지 않는다는게 
**수신 호스트의 수신 버퍼에서 더 받을 수 있는 양을 초과하지 않는다는**거고,

혼잡 윈도우는 문제가 생기지 않는다는게
**네트워크 상태 상 혼잡이 발생하지 않을 가능성이 높다**는거야.
왜 가능성이 높다고 하냐면 수시로 변하는 네트워크 상태에서 **혼잡이 발생할 가능성이 0이 되지는 않기 때문**이야.
<br>
그리고 수신 윈도우의 크기는 
**수신 호스트가 송신한 TCP 세그먼트 헤더의 윈도우 필드**로 알아낼 수 있었지?

그러면 혼잡 윈도우의 크기는 어떻게 알아낼 수 있을까?

바로 이 다음에 나올 **혼잡 제어 알고리즘**을 통해 알아낼 수 있어.

### 1-EX. 혼잡 윈도우와 수신 윈도우

그런데 여기서 의문이 들 수 있어. 
혼잡 윈도우와 수신 윈도우 둘 다 
> 한번에 송신할 데이터의 양을 조절하기 위해 사용되는 값

들인데,
그게 혼잡 윈도우와 수신 윈도우 두 개면 
대체 어느 쪽에 한번에 송신할 데이터의 양을 맞춰야할까?

그래서 TCP는
> 수신 윈도우와 혼잡 윈도우 중 더 작은 윈도우 크기

에 맞춰서 데이터를 전송해.

## 2. 혼잡 제어 알고리즘

이게 오늘의 메인 디쉬.

내가 공부하고 있는 책에서는 이렇게 네 가지 혼잡 제어 알고리즘이 설명되어있어.

- `AIMD`
- `느린 시작 알고리즘`
- `혼잡 회피 알고리즘`
- `빠른 회복 알고리즘`

앞서 설명했듯 혼잡 제어 알고리즘은 혼잡 없이 전송할 수 있다고 판단되는 데이터의 양(혼잡 윈도우 크기)을 결정하는 알고리즘이야.

### 2-1. AIMD - 계속 올리다가 혼잡이 감지되면 윈도우 크기 절반으로 줄이기

`AIMD`는 `Additive Increase / Multiplicative Decrease`의 약자인데,
말 그대로 합으로 증가, 곱으로 감소야.

그러니까, 일단 RTT(Round Trip Time, 세그먼트를 송신하고 확인 응답을 받기까지의 시간)마다 
혼잡 윈도우를 **1씩 선형적으로 증가**시키고,
만약 혼잡이 감지된다면 **혼잡 윈도우를 절반으로 줄이는** 알고리즘이야.

이건 가장 기본적인 혼잡 제어 알고리즘인데, 이것만으로 혼잡 제어가 이루어지지는 않고,
다른 좀 더 정교하고 복잡한 알고리즘들이 필요해.

### 2-2. 느린 시작 알고리즘 - 지수적으로 윈도우 크기 늘리기

`느린 시작 알고리즘`은 
혼잡 윈도우를 **1부터 시작해** 문제없이 수신된 **확인 응답(ACK 세그먼트) 하나마다** 혼잡 윈도우를 **1씩 증가**시키는 방식이야.

1. 세그먼트 하나를 송신하면 ACK 세그먼트가 하나가 오니까 혼잡 윈도우가 1이 늘음
2. 그리고 혼잡 윈도우가 2가 되었으니 세그먼트 두 개를 송신할 수 있고, 두개를 송신하면 ACK 세그먼트가 두 개가 옴
   - 그러면 또 혼잡 윈도우를 받은 ACK 세그먼트 수만큼 늘이는건데, 
     결국 기존 혼잡 윈도우 2에 ACK 세그먼트 두 개니까 혼잡 윈도우는 4가 됨
3. 계속 그렇게 혼잡 윈도우에 맞춰 세그먼트 4개 송신 -> ACK 세그먼트 4개 옴 -> 혼잡 윈도우 4 늘어서 8됨
4. 반복

개념적으로 보면,
```
다음 혼잡 윈도우 = 현재 혼잡 윈도우 + 받은 ACK 세그먼트의 수
```

```
next_cwnd = cur_cwnd + received_ACK_count
```

이런 방식으로 혼잡 윈도우를 증가시켜.

느린 시작이라는 이름과 달리, 실제로는 혼잡 윈도우가 매우 빠르게 증가하는 알고리즘이지.

쉽게 비유하자면, 가속하는 자동차.

출발 자체는 느린 속도에서 시작하지만, 점점 가속도가 붙게 되는거지.

`AIMD`랑 비슷한 느낌이 없지않아 있는데, 
`느린 시작 알고리즘`은 RTT마다 혼잡 윈도우가 **2배**씩 지수적으로 늘지만,
`AIMD`는 **처음 연결이 수립된 뒤 혼잡 윈도우가 증가하는 속도**가 느린 시작 알고리즘에 비해 **느려**.

그러니까 자동차로 비유하면, 제로백(시속 100km 도달 시간)이 느리다고 할 수 있겠네.

그런데 계속 가속만 할 수는 없잖아?
계속 가속만 하게되면 언젠가 앞에 평범한 속도로 가는 차를 만났을 때 난감하지.

그러니까 계속 윈도우가 지수적으로 증가하게 된다면
결국 언젠가는 혼잡 상황을 마주할 수밖에 없어.

그래서 느린 시작 알고리즘을 사용할 때 **느린 시작 임계치(slow start threshold, ssthresh)**라는 값이 정해져있어서,

혼잡 윈도우 값이 계속 증가하다가 **느린 시작 임계치 이상**이 된다면
이제 빠르게 말고 조심스럽게 증가시키는 식으로
`느린 시작 알고리즘`을 종료하고 `혼잡 회피 알고리즘`을 실행해.

그런데 만약

- 재전송 타이머가 다 되어서 타임아웃이 발생하거나,
- 세 번의 중복된 ACK 세그먼트를 수신

하는 식으로 혼잡이 감지된다면,

각각
- 타임아웃 발생 - 심한 혼잡이 발생했다고 보고 혼잡 윈도우 값을 1로 바꾸고, 
  - 느린 시작 임계치(ssthresh)를 절반으로 설정하고 다시 느린 시작 재개
- 세 번의 중복된 ACK - 세그먼트 유실이 발생했다고 판단하고 (빠른 재전송 후) 빠른 회복 알고리즘 수행

처럼 대처해.

### 2-3. 혼잡 회피 알고리즘 - 조금씩 윈도우 올리면서 간보기

`혼잡 회피 알고리즘`은 한 RTT 동안 **결과적으로** 혼잡 윈도우가 **1 MSS만큼씩 증가**하게 만드는 알고리즘이야.
왜 **결과적**이라고 했냐면, 실제로 `혼잡 회피 알고리즘`은 보통 **ACK마다 혼잡 윈도우가 조금씩 증가**하고,
그 결과를 **RTT 단위로 봤을 때** 총 1 MSS 정도로 선형 증가하는 식으로 돌아가거든.

그래서 혼잡 윈도우가 **지수적으로 증가**하는 `느린 시작 알고리즘`과 다르게,
`혼잡 회피 알고리즘`은 혼잡 윈도우가 **선형적으로 증가**해.

그러니까 `느린 시작 알고리즘`의 **느린 시작 임계치**를 넘은 후에는 **혼잡이 발생할 우려**가 있으니까
**조심해서 혼잡 윈도우를 증가**시키는 방식이라고 보면 돼.

요약하면,
> 일단 느린 시작 알고리즘으로 혼잡 윈도우를 빠르게 느린 시작 임계치까지 올린 후,
느린 시작 임계치에 도달하거나 넘어서면 혼잡 회피 알고리즘으로 조심조심 혼잡 윈도우를 늘린다.

라고 보면 돼.

여기에서도 
중간에 타임 아웃이 발생해서 네트워크 상에서 **심한 혼잡이 발생**했다고 판단될 때는 **혼잡 윈도우 값을 1**로 바꾸고, 
**느린 시작 임계치**를 **절반으로 설정**한 뒤 다시 **`느린 시작 알고리즘`을 수행**해.
이건 사실상 느린 시작 알고리즘이랑 같아.

그리고 혼잡 회피 알고리즘 도중 **세 번의 중복된 ACK**가 발생해서 세그먼트 유실이 발생했다고 판단될 때는
**혼잡 윈도우 값을 절반**으로 줄이고, **빠른 재전송** 후 `빠른 회복 알고리즘`을 수행해.

그리고 물론 타임 아웃이 발생했든 세 번의 중복된 ACK가 발생했든 **문제가 생긴 세그먼트는 재전송**해.

### 2-4. 빠른 회복 알고리즘 - 윈도우 크기 1부터 시작하지 않고 절반부터 시작하기

느린 시작 알고리즘과 혼잡 회피 알고리즘의
**세 번의 중복 ACK** 문제에서 계속 
> 빠른 재전송 후 빠른 회복 알고리즘을 수행한다

고 했지?

빠른 회복 알고리즘은 세 번의 중복 ACK를 수신했을 때 
**느린 시작 알고리즘은 건너뛰어버리고** 바로 **혼잡 회피 알고리즘**을 수행하는 알고리즘인데,
이름처럼 **빠르게 전송률을 회복**하기 위한 알고리즘이야.

왜 느린 시작 알고리즘을 건너뛰는지 설명해보자면,
세 번의 중복 ACK를 수신했다는건 
아예 아무 ACK도 도착하지 못한 타임 아웃에 비해서 **그나마 네트워크의 혼잡이 심하지 않다**는거잖아?
그러니까 **어떤 세그먼트 하나가 중간에 유실**됐는데 **그 뒤의 세그먼트들은 계속 도착**하니까
수신 측이 **그 세그먼트**만 못 받았다며 계속 ACK 세그먼트를 보내고 있는 상황인거지.
그래서 굳이 느린 시작 알고리즘으로 **혼잡 윈도우를 1로 과하게 줄이지 말고**, 
혼잡 회피 알고리즘으로 **혼잡 윈도우를 절반까지만 줄여서**
전송률을 덜 줄인 상태로 시작해서 더 빨리 전송률을 회복하게 하는거야.

그런데 타임아웃은 재전송 타이머(RTT 기반)가 끝날 때까지 필요한 ACK를 받지 못한 상태라서 
**어떤 세그먼트가 제대로 전송되었는지 확인도 못하는 상태**니까
혼잡이 심한 상태라고 판단해서 다시 혼잡 윈도우를 1부터 시작하는거야.

## 3. ECN : 라우터가 보내주는 혼잡 알림

앞서 설명했던 혼잡 제어 알고리즘은
데이터의 전송량을 **송신 호스트가 주로 결정**했었지?
송신 호스트가 타임 아웃이 발생하거나 세 번의 중복 ACK를 받아서 혼잡을 감지하면
송신 호스트가 혼잡 윈도우를 줄였지.
그리고 송신 호스트가 시간(RTT 등)을 기준으로 혼잡 윈도우를 점차 늘렸고.

그러니까 이런 방식은 결국 **혼잡이 발생한 후**에 혼잡 윈도우를 줄이는,
**유실이나 지연이 이미 발생한 뒤**에 눈치채는 방식이잖아.

그런데 솔직히 이거 뭔가 비효율적이지 않아?
그냥 전송하려 하는 **라우터에게 네트워크의 상태를 알려달라고 하고 **
라우터가 본 네트워크 상태가 곧 혼잡이 발생할 것 같으면 
라우터가 **곧 혼잡이 발생할 것 같다는걸 송신 호스트에게 알려줘서** 송신 호스트가 혼잡 윈도우 크기를 조절하는 식으로 하면 편할텐데.

그러니까 앞서 설명했던 혼잡 제어 알고리즘들은 일기 예보를 볼 수 없는 상태에서 밖에 나가는거야.
그러니 밖에 나가서 비가 온다면, 즉석에서 우산이나 우비를 사야하지.

하지만 그냥 미리 일기 예보를 보고 밖에 나간다면?
이미 비가 온다는 예보를 받았고, 우산을 챙겨갈 수 있으니 딱히 우산이나 우비를 살 필요가 없지.

``ECN(명시적 혼잡 알림, Explicit Congestion Notification)``은

이렇게 
> 라우터 자체에서 호스트들에게 네트워크가 혼잡해질 것 같다는 사실을 알려줘서 
호스트들이 스스로 혼잡 윈도우를 줄여 전송량을 조절하게 하는 기법

이야.

그리고 ECN은 **선택적인 기능**이라 ECN을 지원하는 호스트도 있지만, ECN을 지원하지 않는 호스트도 있어.

### 3-1. TCP/IP 헤더와 함께 보는 ECN의 동작

일단 ECN의 동작을 설명하려면 다시 IP 헤더와 TCP 헤더를 살펴봐야해..
왜냐하면 ECN은 선택적인 기능이라서, ECN을 지원하는 호스트가 TCP/IP 프로토콜로 정보를 주고받을 때
IP 헤더와 TCP 헤더에 ECN 관련으로 **기존 비트를 사용**하기 때문이야.
ECN은 **새로운 필드를 추가하는 기능이 아니라**, 
기존 IP 헤더와 TCP 헤더의 일부 비트를 이용해 혼잡 신호를 주고받는 방식이야.

IP 헤더는 대표적으로 송수신지 IP 주소, 식별자, 플래그, 단편화 오프셋, TTL, 프로토콜 정도의 필드가 있었지?
ECN을 나타내는 부분은 이런 것들은 아니고, IP 헤더의 **서비스라는 필드(혹은 DS 필드)**에 있어.
서비스 필드(혹은 DS 필드)의 **맨 오른쪽 두 비트가 ECN으로 사용**돼.

일단 처음에 **ECN을 사용 가능하다고 명시**를 할 필요가 있겠지? ECN은 선택적인 기능이니까.
그래서 송수신 호스트 둘 다 ECN이 가능해서
그 호스트간의 전송에서 ECN이 사용가능하다면 
일단 송신 호스트가 IP 헤더의 **ECN으로 사용되는 두 비트를 10(ECT(0))으로 바꿔서** 라우터에게 ECN을 사용가능하다고 알려.
그리고 이걸 **ECN-Capable Transport**. 줄여서 `ECT`라고 해.
이 패킷은 **ECN을 쓸 준비가 되어있다**는 뜻이야.

~~TMI : 두 비트가 01이면 ECT(1)이라 하는데, 최신 네트워크에서는 ECT(0)과 큰 차이는 없음~~

그리고 라우터로 전송되었을 때, 
라우터가 봤을 때 **네트워크가 혼잡**해질 것 같다고 판단할 경우, **이 두 비트를 라우터가 11(CE)로** 바꿔.
물론 **ECN이 사용 가능할 때**(ECT - 두 비트가 10)만.

TCP 헤더는 대표적으로 송수신지 포트 번호, 순서 번호(SEQ), 확인 응답 번호(ACK), 윈도우, 제어 비트 정도의 필드가 있었지?
ECN을 나타내는 부분은 **제어 비트의 CWR, ECE 플래그**야.
만약 라우터를 거쳐서 온 IP 패킷의 서비스 필드 **맨 오른쪽 두 비트가 11(CE)로** 표시되어있어서
라우터가 혼잡을 감지했다는걸 알았을 때,
그 패킷을 받은 호스트(수신 호스트)는 TCP 헤더의 **ECE 플래그를 1로** 바꿔서 
패킷을 보낸 호스트(송신 호스트)에게 보내.

그리고 송신 호스트는 그 패킷을 받았는데 ECE 플래그가 1로 설정되어있으니 **혼잡 신호가 전달**됐다는걸 알 수 있고, 
**혼잡 제어 알고리즘(느린 시작 : 1, 혼잡 회피 : 절반 등)에 따라** 혼잡 윈도우를 줄인 후에
**CWR 플래그를 1로** 바꿔서 수신 호스트에 **혼잡 제어에 반응했다는걸 표시**를 한 세그먼트를 보내는거야. 

## 4. 게임 서버와의 연결

혼잡 제어는 게임에서 발생하는 **렉**과도 밀접한 관련이 있어.

실제로 온라인 게임에서는 렉의 원인 중 상당수가
**네트워크 지연, 패킷 유실, 혼잡 같은 네트워크 문제**와 연결되어 있어.

특히 실시간성이 중요한 게임에서는
중간에 타임아웃이 발생하거나 일부 패킷이 유실되어 재전송이 일어나는 것 자체가
곧 체감 렉으로 이어질 수 있어.

TCP는 신뢰성을 보장하기 위해
유실된 세그먼트를 재전송하고, 상황에 따라 혼잡 윈도우를 줄이는데,
이 과정에서 **전송률이 낮아지면** 유저 입장에서는 입력 반응이 늦거나
캐릭터가 순간이동하는 것처럼 보이는 현상이 나타날 수 있어.

특히 **타임아웃**이 발생하면 TCP는 혼잡이 심하다고 판단해
혼잡 윈도우를 1로, **정말 크게** 줄이고 다시 천천히 전송률을 회복하려고 해.
이런 동작은 신뢰성 측면에서는 의미가 있지만,
실시간성이 중요한 게임에서는 **체감 성능을 크게 떨어뜨릴 수 있어**.

반면 빠른 회복 알고리즘은
세 번의 중복 ACK가 발생했을 때 혼잡 윈도우를 1까지 떨어뜨리지 않고
혼잡 회피 단계로 더 빠르게 이어지게 해 주기 때문에,
타임아웃이 발생했을 때보다 전송률 회복이 더 빠를 수 있어.
즉, **일부 상황**에서는 렉이 더 심해지는 것을 어느 정도 완화해 줄 수 있는 거야.

하지만 실시간성이 중요한 게임에서는
이것만으로는 근본적인 해결책이 되기 어려워.

예를 들어 **플레이어의 이동 정보**처럼
**이번 패킷 하나를 놓쳐도 다음 패킷을 받으면 곧바로 상태를 갱신**할 수 있는 데이터는
매번 재전송해서 신뢰성을 보장하는 것보다,
일부 손실을 감수하더라도 더 빠르게 보내는 편이 유리할 수 있어.

그래서 이런 정보는 TCP보다 **UDP**를 사용하는 방식이 자주 고려돼.
물론 UDP를 쓴다고 해서 혼잡 자체가 사라지는 것은 아니지만,
TCP처럼 모든 데이터를 재전송하고 순서를 맞추기 위해 기다리지 않아도 되기 때문에
실시간성이 중요한 상황에서는 더 적합할 수 있어.

또한 TCP를 써야 하는 경우라면,
작은 데이터를 한 번에 모아서 보내는 `Nagle 알고리즘`이 오히려 지연을 만들 수 있기 때문에
지연을 줄이기 위해 이를 비활성화하는 방식(`TCP_NODELAY`)도 고려해볼 수 있어.
