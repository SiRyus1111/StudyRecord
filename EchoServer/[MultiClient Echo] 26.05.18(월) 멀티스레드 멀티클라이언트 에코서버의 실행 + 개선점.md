드디어 멀티클라이언트 에코서버를 처음으로 실행시켜봤어.
나 진짜 엄청 고생하며 디버깅할 각오 하고 실행시켰는데,
한 방에 잘 실행되더라..

역시 설계를 잡아놓고 코딩을 한게 신의 한 수 였던 것 같아.

전에 싱글클라이언트 싱글스레드 에코서버 때 에러가 많았던 것도

이렇게 요약할 수 있어.
잘 되는 것 같지.

각 송 / 수신마다 포트 번호가 다르기 때문에
여러 클라이언트에서 접속했음을 알 수 있고.

그리고 quit을 입력해서 1번 클라이언트를 종료하면,
![](https://velog.velcdn.com/images/siryus0907/post/a125e2b2-f743-4bde-8699-b5e5311b8f85/image.png)

이렇게 제대로 종료된 것이 보이지.
서버 로그에도 `클라이언트에서 연결을 종료하였습니다.`라고 출력되고.

어쨌든 이번 테스트로 확인한건,

1. 서버의 `accept()` 루프가 여러 클라이언트 연결을 연속적으로 수락한다.
2. 각 클라이언트마다 서로 다른 포트 번호가 할당됨으로써 여러 클라이언트가 접속할 수 있다.
3. 각 클라이언트가 보낸 메시지를 서버가 정상적으로 수신한다.
4. 서버가 각 클라이언트에게 정상적으로 에코 메시지을 보낸다.
5. `quit` 입력 시 해당 클라이언트의 연결 종료를 서버가 감지한다.

이 정도야.

## 2. 개선점들

일단 현재의 멀티클라 에코서버에서 내가 체감한 개선점은 두 가지야.

1. 서버의 로그가 좀 더 세부적이어야 한다.
    - 현재는 그저 송 / 수신 내역과 메시지를 보낸 클라이언트 + 받은 메시지 정도
    - 하지만 좀 더 세부적인 로그가 필요해보인다.
      - 클라이언트가 접속할 시 접속한 클라이언트의 `IP:Port` + `세션 ID`를 출력한다던지
      - 클라이언트가 접속 종료할 시 접속 종료한 클라이언트의 `IP:Port` + `세션 ID`를 출력한다던지..
2. 클라이언트의 확실한 구분이 필요하다.
    - 현재는 그저 `IP:Port`를 가지고 클라이언트들을 구분짓는다.
    - 하지만 좀 더 세부적인 구분이 필요하다.
      - `세션 ID`라던지, `닉네임`이라던지..
      - 물론 `IP:Port`로도 서로 다른 클라이언트라는건 알 수 있지만,
      서버 내부에서 클라이언트를 추적하거나 로그를 읽기 쉽게 만들기에는
      `세션 ID` / `닉네임`같은 별도의 식별자가 있는 편이 더 좋다고 생각한다.
      
이렇게 두 개.
