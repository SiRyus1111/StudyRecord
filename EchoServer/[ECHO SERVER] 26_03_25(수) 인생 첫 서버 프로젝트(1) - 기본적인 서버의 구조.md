## 1. 기본적인 서버 구조

일단 이게 멀티 클라이언트 에코 서버는 아니고, `싱글 클라이언트 에코 서버`야.
그래서 서버는 **동시에 한 클라이언트만 처리**할 수 있어.

왜 이렇게 했냐면, 
시작부터 멀티 스레드 멀티 클라이언트 에코 서버를 만들게되면,
**중간에 드랍할 가능성**도 높아지고,
멀티 스레딩 쪽에 집중이 되어서
**서버-클라이언트 간의 송수신 구조를 깊게 팔 수가 없기 때문이야.**

## 2. 전체 서버-클라이언트 구조

![](https://velog.velcdn.com/images/siryus0907/post/ddc767f8-d361-4b60-8a33-8ab829dbc16d/image.png)

설명을 좀 해보자면,
일단 이건 연결이 다 되었다는 전제에서
**서버와 클라이언트가 어떤 흐름으로 send / recv를 하는지 나타내는 구조**라고 생각하면 돼.

1. 클라이언트 콘솔 창에 입력이 들어오면
2. 클라이언트가 서버로 `send()` 하고,
3. 서버가 `recv()` 하고,
4. 서버가 서버 콘솔 창에 출력하고,
5. 서버가 클라이언트로 `send()` 하고,
6. 클라이언트가 `recv()` 하고,
7. 클라이언트가 클라이언트 콘솔 창에 출력한다.

이런 과정을 그림으로 그렸어.

근데 이건 솔직히 텍스트보다 그림이 더 이해가 잘될 것 같아..

## 3. 송 / 수신하는 데이터 구조

![](https://velog.velcdn.com/images/siryus0907/post/d6051ac1-9e04-47d7-9807-6f9abe740f03/image.png)
이건 굳이 왜 정하냐면,

TCP는 메시지의 경계가 없기 때문에, 
바이트스트림 방식으로 데이터가 전송되기 때문에
**클라이언트에서 n바이트를 보냈다고 했을 때
서버는 정확히 n바이트를 받아야해.**
그래서 서버는 클라이언트가 몇 바이트를 보냈는지를 알아야하는거지.

저번에 설명한건데,
송신지에서 몇 바이트를 보냈는지
수신지가 확인하기 위해서 이런 방법들을 사용한다고 했지?

- `아예 고정된 데이터 크기로 데이터를 주고받기`
- `송신지 호스트에서 맨 앞의 고정된 길이의 헤더로`
`데이터의 전체 크기 / 페이로드 크기를 수신지 호스트에게 알려주기`
- `데이터 맨 끝에 구분자를 붙여서 데이터의 끝을 알려주기`

나는 이 중에 두 번째 방법인,
**고정 길이 헤더**로 송신할 데이터의 크기,
즉 **페이로드의 크기를 알려주는 방식**을 선택했어.
그래서 앞에 `uint32_t`(**32비트(4바이트)짜리 unsigned int**) 형의
`length` 헤더를 붙였어.
그리고 헤더에는 이름처럼 **데이터의 길이(크기)를 명시한 후 송신**해.

이건 고정 길이라, 수신지에서 첫 4바이트만 다 받을 때까지 `recv()`,
즉 **수신 버퍼에서 4바이트만 꺼내온다면
바로 송신 호스트에서 몇 바이트의 데이터를 보내려는지 알 수 있어.**

그렇기 때문에 이렇게 송 / 수신할 때의 데이터 구조를 만드는거야.
사실상 프로토콜을 설계한거지.
통신 규약.

헤더의 송 / 수신을 좀 더 자세히 설명해보자면,
![](https://velog.velcdn.com/images/siryus0907/post/1b61505c-e101-4ff9-b0ad-4d00e77bd3ae/image.png)


일단 전송할 때는 **네트워크 바이트 정렬**로 변환해서 전송해.

왜 그렇냐면, 네트워크 바이트 정렬의 표준은 **빅 엔디언**이지만,
**상대 호스트의 바이트 정렬은 알 수 없기 때문에**
내가 내 호스트의 바이트 정렬로 상대에게 데이터를 보낸다고 해도,
**상대 호스트의 바이트 정렬이 내 호스트의 바이트 정렬과 같다는 보장이 없기 때문이야.**

그래서 그냥 **네트워크 바이트 정렬로 바꿔서 송신**하고, 
**상대 호스트에서 그 쪽 호스트의 바이트 정렬로 변환**하는 식으로 했어.
<br>

그리고 `recv()` 할 때는 **헤더 길이를 먼저 `recv()` 해서 헤더를 해석 후 
해석한 헤더만큼 데이터(페이로드)를 `recv()` 하는 식**으로 `recv()` 해.

왜 그렇냐면, 앞에서도 말했듯이 수신 호스트에서 `recv()` 할 때 
총 몇 바이트를 `recv()` 해야하는지, 그러니까 수신 버퍼에서 꺼내와야 하는지 알아야 하기 때문에,
그걸 나타내는 헤더를 먼저 `recv()` 해서 받아야하는 바이트 수를 알아낸 후,
그 바이트 수만큼 `recv()` 해주는 식으로 해야해서 그래.

## 4. 서버의 동작 과정
![](https://velog.velcdn.com/images/siryus0907/post/a72841da-f18c-470d-923a-cfc2f56b2561/image.png)
이번에는 **서버에서 어떻게 소켓 API들을 호출**해야하는지 정리했어.

일단 

1. `socket()` 으로 listen용 소켓(`Listen_sock`)을 생성하고,

2. `bind()` 로 `Listen_sock`의 `IP:Port`를 바인딩(해당 소켓에 할당)해.

3. 그리고 `listen()` 으로 `Listen_sock`의 상태를 `LISTENING(LISTEN)`으로 바꿔.

4. 그리고 이제 `accept()` 를 호출해.
- *`accept()` 은 블로킹(동기) 방식이라, 
`accept()` 로 백로그 큐에서 꺼내올 연결 정보가 생길 때까지 대기하기 때문에, 
나는 `listen()` 함수가 아닌 `accept()` 함수에 '클라이언트 접속 기다리기'라는 키워드를 썼어.
실제로는 `listen()` 이 '클라이언트 접속 기다리기'에 가까워. 이거 오해없길 바래.*

- 그리고 `accept()` 의 반환값인 클라이언트와 통신할 소켓(`Client_sock`)을 저장해.

5. 이제 송수신. `recv()` / `send()`로 송수신을 해. 
이건 클라이언트가 접속 종료할 때까지 반복해.

6. 클라이언트가 접속 종료했다면, 그러니까 `recv()`의 반환값이 0이라면,
`accept()` 로 생성한 `Client_sock`을 `closesocket()`으로 닫아.

이 정도.

그리고 `bind()` 할 때에 따로 `Listen_sock`에 할당할 `IP:Port`를 정해야하는데,
이게 `bind()` 용 `IP:Port`야.

IP는 이게 헷갈릴 수 있는데, 
**해당 소켓으로 접근 가능한 IP**를 정하는거야.

그래서 **모든 IP(NIC)에서 접근할 수 있게**
`INADDR_ANY`로 IP 주소를 설정해.

그리고 Port는 솔직히 맘대로 정해도 돼.
그냥 서버의 포트 번호를 고정하는데 의의가 있는거라.

나는 9000번으로 했어.
그리고 웰 노운 포트(0 ~ 1023번 포트)는 사용하려면 복잡한 과정이 필요하니까,
나는 이번에는 복잡하고 번거로운 과정을 거칠 필요가 없는 
9000번 포트, 테스트용으로 주로 사용되는 포트를 골랐어.

그리고 실무에서는, 
웬만하면 서버가 웰 노운 포트나 등록된 포트를 쓴다고 하네.

![](https://velog.velcdn.com/images/siryus0907/post/aa65d86b-6dc7-487f-ae8b-00929cdc585c/image.png)

이거는 서버의 송 / 수신 과정인데,

쉽게 말하면 클라이언트가 접속할 때마다 반복되는 과정이야.

일단

1. `accept()`로 백로그 큐에서 연결 정보를 꺼내와서 
클라이언트와 통신할 수 있는 소켓(`Client_sock`) 생성.

2. 클라이언트에서 보내온 정보를 `recv()`.
- 먼저 4바이트짜리 헤더를 다 받을 때까지 `recv()` 하고, 헤더를 해석한 후 
헤더에 명시된 데이터의 크기만큼 데이터를 받을 때 까지 `recv()`를 계속 호출해.

3. 이제 `recv()` 도중에 오류가 발생하거나, 클라이언트에서 연결을 종료 했을 때. 그런 것들을 체크해.
- 뒤에 나올 상태 관리 구조체를 이용해. 어떤 과정에서 어떤 문제가 발생했는지 기록하는거지.
- 그리고 서버에서도 
해당 클라이언트와 연결을 종료(`closesocket()`)하거나, 에러를 출력(`WSAGetLastError()`)해.

4. 이제 클라이언트에 보낼 정보를 `send()`.
- 먼저 4바이트짜리 헤더를 다 보낼 때까지 `send()` 하고, 
헤더에 명시된 데이터의 크기만큼 데이터를 보낼 때 까지 `send()`를 계속 호출해.

5. 여기서도 오류 체크. 
- 상태 관리 구조체를 이용해서 어떤 과정에서 어떤 문제가 발생했는지 기록!
- 그리고 여기서도 연결을 종료 or 에러를 출력!

6. 이제 연결을 종료해야할 때는 `closesocket()`
- 에러가 발생했다던지, 클라이언트에서 연결을 종료했다던지.
- 여기서 닫는 소켓은 `accept()` 함수로 생성한, 클라이언트와 통신하는 소켓이야.

## 5. 클라이언트 동작 과정

![](https://velog.velcdn.com/images/siryus0907/post/b84fa82f-0bd0-4dd8-b208-eacebafbbeee/image.png)

이번에는 클라이언트야.

1. `socket()` 으로 서버와 통신할 소켓(`Connect_sock`)을 생성하고,

2. `bind()`를 하지 않고, 바로 서버 `IP:Port`에 `connect()` 해.
- 클라이언트의 `IP:Port`는 그저 서버에서는 클라이언트들을 식별하는 수단이기 때문에,
딱히 특정 `IP:Port`를 할당할 필요가 없어.

3. 이제 송수신. `recv()` / `send()` 로 송수신을 해.
- 이건 서버와 마찬가지로 해당 클라이언트가 접속 종료하거나, 
서버의 연결이 끊어질 때까지 반복해.

4. 그리고 해당 클라이언트가 접속 종료하는건 `closesocket()` 으로 수행돼.

이 정도.

그리고 `connect()` 할 때 서버의 `IP:Port`에 `connect()` 를 해야하는데,

일단 포트 번호는 서버의 `Listen_sock`의 포트 번호인 9000번으로 설정해서,
어떻게 보면 해당 포트에 할당되어있는 `Listen_sock`에 연결을 시도하는거야.

그리고 IP는 **로컬 호스트(루프백 주소, 자기 자신을 가리키는 주소)**인 
`127.0.0.1`로 설정했어.

내 테스트 환경이 같은 컴퓨터. 그러니까 같은 IP에서 서버와 클라이언트가 동시에 돌아가다보니까
이런 식으로 **자기 자신을 가리키는 `127.0.0.1`로 서버 주소를 설정**했어.

물론, 
- 같은 LAN(네트워크) 내에서만 사용하는 서버라면 해당 사설 IP 주소를 적으면 될테고,
- 다른 LAN(네트워크)에서의까지 사용하는 서버라면 포트포워딩과 함께 
NAPT(PAT)에 필요한 공인 IP:Port를 적으면 되겠지.

![](https://velog.velcdn.com/images/siryus0907/post/723dc53c-25b4-46be-b8e8-858946f0f3e6/image.png)

이거는 클라이언트의 송 / 수신 과정.

클라이언트가 서버에 접속 후 송 / 수신하는 과정이야.

일단

1. `connect()` 로 서버와 통신할 수 있게 이미 `socket()` 으로 생성해놨던 소켓을 서버와 연결.

2. 사용자에게 서버로 보낼 데이터 입력받기.
- 입력받을 때는 구분자를 오직 **\n(줄바꿈)**으로만 봐. 
그래서 공백도 구분자로 치는 `std::cin`이 아닌
공백도 하나의 문자로 치는 `getline()`을 사용해.

3. 데이터의 길이를 잰 후, **네트워크 바이트 정렬로 변환해서 헤더에 기록**.
- 단순히 길이를 나타내는 변수(`len`)와 
네트워크 바이트 정렬로 변환한, 실제로 헤더로 송신할 변수(`net_order_len`)는 분리.

4. 헤더를 먼저 미리 정의해놓은 헤더 크기(4바이트)만큼 다 보낼 때까지 `send()`.

5. `len`을 보고 그 크기만큼 전부 `send()` 될 때 까지 계속 `send()`.
- 이래서 `len`과 `net_order_len`을 분리해놓은거야.

6. 여기서도 서버에서처럼 오류 체크. 
- 상태 관리 구조체를 이용해서 `send()`의 어떤 과정에서 어떤 문제가 발생했는지 기록!
- 그리고 연결을 종료(`closesocket()`) or 에러를 출력(`WSAGetLastError()`)!

7. 서버에서 보내온 데이터를 `recv()`.
- 먼저 4바이트짜리 헤더를 다 받을 때까지 `recv()` 하고, 헤더를 해석한 후 
헤더에 명시된 데이터의 크기만큼 데이터를 받을 때 까지 계속 `recv()`.

8. 오류 체크. 위에서 지겹도록 설명했어.

9. 연결을 종료할 때는 `closesocket()` 호출.
- 사용자가 연결 종료 의사를 보인다던지, 에러가 발생했다던지, 서버에서 연결을 종료했다던지.
- 여기서 닫는 소켓은 맨 처음에 `socket()`으로 만든 소켓이야.

## 6. 상태 관리 구조체

이게 뭘까?

나는 실제로 구현을 해보면서, 
송 / 수신 과정에서 에러가 발생했거나,
클라이언트가 연결을 종료했을 경우에

헤더만큼 `recv()` / `send()` 하는 while문 + 전체 송 / 수신 while문

이렇게 **while문을 두 번 break**해야했어.

여기서 문제는 `send()` / `recv()`에서 에러가 발생한 것과 
`recv() == 0`으로 클라이언트가 연결을 종료한 것을
두 개의 while문 밖에서 어떻게 알고
각각에 맞는 처리.
그러니까,
- 에러 발생 = 발생한 에러를 출력하고 클라이언트와 연결 종료
- 클라이언트가 연결 종료 = 연결 종료 메시지를 출력하고 다음 클라이언트 접속 대기(accept())

이렇게 두 가지 경우에 따라 다른 처리를 할 수 있을까 싶었어.
<br>
결국, 내가 고민한건
- `비트 플래그`
(u_char 형 변수의 각각의 비트에 상태 기록)
- `상태 관리 구조체`
(클라이언트가 접속하면 구조체 생성 후 각 상태 기록(현재 하고있는 작업, 오류 / 클라이언트 연결 종료 상태))

이 두 개였는데,
결국 첫 프로젝트니까 가독성을 위해서 `상태 관리 구조체`를 선택했어.

비트 플래그는 **하나하나의 비트**에 **true / false**를 기록하는거다보니까,
이런 식으로 구현이 되는데,
```cpp
u_char bitflag = 0b00000000;

// 각 상태 정의
u_char HEADER_RECV = 0b10000000;
u_char PAYLOAD_RECV = 0b01000000;

// 상태 갱신
u_char UNABLE_HEADER_RECV = ~HEADER_RECV; // 0을 1로, 1을 0으로 뒤집기 (해당 비트 비활성화 용)

bitflag |= HEADER_RECV; // bitflag = 0b00000000 | 0b10000000; (비트 OR 연산)
bitflag &= UNABLE_HEADER_RECV; // bitflag = 0b10000000 & 0b01111111; (비트 AND 연산, 해당 비트만 비활성화)

// 상태 검사
if (!(bitflag & HEADER_RECV)){ // 만약 헤더를 RECV한 상태가 아니면..
    // 대충 뭔가를 처리하는 코드
}
```
가독성이.. 어.. 개박살이 나지.

그래서 비트플래그는 나중에 패킷 구조를 설계할 때,
1바이트라도 아껴야할 때 써볼까 싶어.

상태 관리 구조체는 이렇게 서버 / 클라이언트 나눠서 잡았어.

현재 하고있는 작업. 이건 왜 `accept()` 같은거 없이 `recv()` / `send()`만 있냐면,
애초에 상태 관리 구조체를 만든 목적이 송수신 과정에서의 상태 관리라서 그래.

일단 현재 하고있는 작업은 나는
- `헤더 recv`
- `페이로드 recv`
- `헤더 send`
- `페이로드 send`

이렇게 네 가지로 분류했고,

각 작업에서의 상태는 
- `함수가 SOCKET_ERROR를 반환(오류 남)`
- `recv() 함수의 반환 값이 0 (클라이언트 / 서버에서 연결 종료)`

이렇게 두 가지로 분류했어.

왜 정상적인 상태를 나타내는 플래그가 없냐면,
정상적인 상태는 그냥 계속 코드를 실행하면 되기도 하고,
그렇기 때문에 정상적으로 실행되었을 때를 기록할 메리트도 없으며,
정상적인 상태를 기록하는 로직이 추가되면 코드가 더 복잡해질 것 같았기 때문이야.

그래서 이 구조체의 각각의 원소를 보면,
- **상대 호스트가 연결을 종료했을 때 연결 종료 여부를 확실히 기록할 수 있고,**
- **이게 핵심인데, 오류가 발생했을 때 어느 작업에서 어떤 오류가 났는지를 바로 알 수 있어.
즉, 디버깅이 쉬워지겠지.**

난 이런 식으로 구현했어.

```cpp
struct flags {

	bool header_recv = false;
	bool payload_recv = false;
	bool header_send = false;
	bool payload_send = false;

	bool if_error = false;
	bool if_client_exit = false;
};
```
(이건 서버의 상태 관리 구조체)

이렇게 내가 그린 그림처럼

> 서버 / 클라이언트에서 어떤 과정에서 어떤 상태가 발생했는지

를 확실히 알 수 있어.
