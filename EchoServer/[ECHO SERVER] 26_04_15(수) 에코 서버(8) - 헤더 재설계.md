## 0. 서론

이번에는 기존의 uint32_t 형의 기존 헤더를 재설계했어.
불편한 점이 참 많았거든.

그 불편한 점들의 근본적인 원인은 이거야.

- 헤더를 uint32_t 형의 4바이트 헤더에서 변경, 확장하기 어렵다

이 문제로 인해서,
내가 에러 메시지 / 정상 에코 메시지를 구분해서 전송하려고 할 때 정말 힘들었어.

처음에는 단순히 
헤더가 1 이상 4096 이하면(1 <= header <= 4096) 정상 에코 메시지로,
헤더가 0이면 에러 메시지로 판별하려 했는데,

애초에 헤더의 역할이 보내는 페이로드의 길이(바이트 수)라서,
0으로 설정해놓으면 수신지에서 에러 메시지의 길이를 알기 어려웠어.

그러니까, 기존의 헤더는 페이로드의 길이를 표현하는 역할과
메시지의 의미를 표현하는 역할을 동시에 하기에는 어려운 구조였어.
그리고 4바이트에서 확장도 어려웠지. 단순히 변수 하나라서..

그래서, 지금의 단순 4바이트 헤더 구조에 모든걸 때려넣는 헤더 구조는 도저히 안되겠다고 판단,
헤더를 대폭 재설계했어.

[깃허브 링크](https://github.com/SiRyus1111/My-First-Echo-Server-Project)

## 1. 재설계한 헤더 구조

기존에는 단순히 4바이트의 uint32_t 형 변수인 header가 헤더를 나타냈어.
이 구조의 단점은 해당 헤더를 확장하기 어렵다는거야.

내가 이번에 고민한 에러 메시지 판별처럼, 추가적인 기능이 필요할 때 확장성이 상당히 낮은 구조야.

그래서, 나는 아예 헤더 자체를 구조체로 정의했어.

이렇게 한다면, 패킷의 의미(type)와 크기(length)를 분리해서 표현할 수 있어서
헤더의 각 필드의 역할을 명확하게 나눌 수 있어.

```cpp
#pragma pack(push, 1)
struct PacketHeader {
	int32_t type;
	uint32_t length;
};
#pragma pack(pop)
```
이렇게. 
int32_t(int 32비트) 형의 type과,
uint32_t(unsigned int 32비트) 형의 length를 멤버로 가져.

쉽게 표현하면,
```
[8Bytes PacketHeader][Payload]

[4Bytes type][4Bytes Length][Payload]
```
이렇게 볼 수 있겠네.
헤더는 총 8바이트,
4바이트의 타입(패킷의 의미)과 4바이트의 길이(페이로드의 바이트 수)로 이루어져있어.

이 구조는, 
헤더를 더 확장할 때 단순히 구조체에 멤버를 추가하고,
그에 맞게 헤더를 직렬화(Serialization) / 역직렬화(Deserialization)하는 부분에
해당 멤버까지 직렬화 / 역직렬화 하는 과정을 넣으면 돼.

그러니까 여기서의 PacketHeader 구조체는
단순히 의미 단위로 다루는게 아니라,
네트워크로 보낼 때는 바이트열로 변환해서 보내고,
받아서 해석할 때는 다시 구조체 형태로 해석하는거야.

기존의 4바이트 uint32_t 형 변수로 처리하는 방식보다
뼈대인 구조체 위에서 단순히 멤버만 추가하는 식이라서
훨씬 확장성이 좋아.

그리고 
```cpp
#pragma pack(push, 1)
// PacketHeader 구조체
#pragma pack(pop)
```
이 #pragma pack은 뭘 의미하냐면,
구조체 멤버 사이의 컴파일러가 임의로 넣을 수 있는 패딩(Padding)을 의미해.

구조체는 멤버 정렬(alignment) 때문에 멤버 사이에 패딩 바이트가 들어갈 수 있는데,
패킷 헤더는 송 / 수신 시 정확한 바이트 배치를 유지해야돼서

이렇게 사이에 패딩을 없애는
```cpp
#pragma pack(push, 1)
```
로 이 구조체의 멤버들 사이의 패딩을 없애는 1바이트 정렬을 적용하고,
```cpp
#pragma pack(pop)
```
구조체 정의가 끝나면 이렇게 이전 정렬 옵션을 복원했어.

## 2. type 필드

일단 length 필드는 기존 헤더와 다름없어서 스킵할게.
이번 재설계한 헤더에 새로 추가된 type 필드만 설명해볼게.

이건 처음에 말했듯, 정상 에코 메시지와 에러 메시지를 판별하기 위해 사용하는 필드야.
그리고 나중에 다른 메시지 타입이 새로 추가될 때도 그저 type 필드의 규칙만 추가해서 그에 맞는 처리를 하면 돼.

일단 지금 정의된 type 필드의 규칙은
- HEADER_SAFE = -1
- HEADER_ERROR = 0

이렇게 두 가지가 있어.

정상 에코 메시지와 에러 메시지를 확실히 구분하기 위해,
이렇게 단순하게 구분했어.
중요한건 값 자체보다, 이 값을 전역 상수로 정의해서
프로토콜 규칙 자체를 코드 전체에서 일관되게 유지하는 거라서,
딱히 각 타입에 대한 의미는 없어.

type 필드의 값이 -1이면 HEADER_SAFE(정상 에코 메시지),
0이면 HEADER_ERROR(헤더 에러 메시지).

이런 식이야.

그리고 저렇게 type 필드의 규칙들은 싹 다 전역 상수로 선언해서,
규칙이 바뀔 경우에, 빠르게빠르게 딱 전역 상수만 수정하면 바로 규칙이 적용되게 했어.

## 3. 에러 메시지의 처리

기존에는 헤더 값이 0이면 에러 메시지라고 가정하고,
일반 메시지와는 다른 분기 로직으로 페이로드를 수신하려고 했어.

그러니까 기존에는 헤더 값 자체에 메시지의 의미(에러 메시지 == 0)까지 억지로 눌러담으려고 해서,
에러 메시지를 일반 메시지랑 다른 예외 로직으로 처리해야했었어.

기존 헤더의 값은 사실상 재설계한 헤더의 length값이었으니까,
고정 길이 페이로드 느낌으로 했던거야.

하지만 재설계한 헤더는,
그저 헤더 / 페이로드 recv_all() 후에,
헤더와 페이로드 모두 수신한 후에
type 필드의 값에 따라 처리 방식을 바꿔.

정상 에코 메시지는 그냥 계속 송 / 수신을 진행하고,
헤더 에러 메시지는 상태 관리 구조체(flags)의 if_peer_error필드를 true로 바꾸고
송 / 수신을 종료 후 if_peer_error == true 조건에 맞춰서 
송 / 수신 종료 후 해당 조건에 맞는 처리를 실행해.

이 구조의 장점은,
> 에러 상황도 프로토콜 내부에서 일관되게 처리 가능

이라는거야.
에러 메시지라고 특별 처리가 아니라,
일반적인 메시지처럼 페이로드까지 recv() 한 후, 
에러 메시지를 위한 특별 처리를 하는거지.

## 4. PacketHeader 구조체의 직렬화 / 역직렬화 과정

### 4-1. recv_all()과 역직렬화

일단 recv_all()로 받을 PacketHeader 구조체의 각 멤버는
전부 네트워크 바이트 정렬(network byte order) 상태야.

그래서 나는 net_header와 host_header를 따로 두고,

- net_header : 네트워크 바이트 정렬 그대로 받은 헤더
- host_header : ntohl()로 호스트 바이트 정렬로 변환한 헤더

이렇게 역할을 나눴어.

```cpp
	PacketHeader net_header;

	server_state.header_recv = true;
	int header_recv_res = recv_all(client_sock, server_state, (char*) &net_header, HEADER_SIZE);

	if (header_recv_res == SOCKET_ERROR || header_recv_res == 0) break;
	
	server_state.header_recv = false;
```
이 코드에서 조금 헷갈릴 수 있는 부분이
왜 PacketHeader 구조체 변수를 char\*로 형 변환해서 받을 수 있냐는 점이야.

그 이유는 recv() 함수가 타입을 해석해서 받는 함수가 아니라,
그저 지정한 메모리 주소에 바이트열(char*)을 그대로 써 넣는 함수이기 때문이야.

즉, recv() 입장에서는
그 메모리가 char\[\] 버퍼인지, PacketHeader 구조체 변수인지 중요하지 않아.
그저
> 어느 메모리 주소부터 몇 바이트를 쓸지

만 중요해.

그래서 `PacketHeader net_header;`라는 구조체 변수도
메모리 관점에서 보면 결국 바이트열, 연속된 바이트들의 집합이고,
그 시작 주소인 `&net_header`를 `char*` 형으로 변환해서 넘긴다면
recv_all() 함수는 그 구조체가 차지하는 메모리 공간에 헤더의 바이트열을 그대로 채워넣을 수 있어.

특히 나는 #pragma pack(push, 1)을 사용해서 구조체 멤버 사이의 패딩을 없앴기 때문에,
PacketHeader 구조체의 메모리 배치는
```
[4Bytes type][4Bytes length]
```
이렇게 정확히 8바이트 연속 공간이 돼.

그래서 헤더를 받을 때도 이 구조 그대로 바이트열을 받아올 수 있어.

하지만 여기서 중요한건, PacketHeader 구조체 자체를 바이트열처럼 받는 것과
구조체 멤버의 값을 올바르게 해석하는것은 별개의 문제라서,
이렇게 구조체를 바이트열처럼 받았다고 해도
net_header의 type과 length 멤버는
여전히 네트워크 바이트 정렬인 상태라서,
이 값을 그대로 사용하면 안돼.

그래서 이렇게 다시 ntoh()로 변환해서
호스트 바이트 정렬의 host_header를 따로 만드는 과정까지 필요했어.
```cpp
// 헤더 해석
PacketHeader host_header;
host_header.type = ntohl(net_header.type);
host_header.length = ntohl(net_header.length);
```

정리해보자면,
- PacketHeader 구조체는 패딩이 없는 연속된 8바이트 공간이므로
 recv() 함수로 그대로 받아올 수 있음
- 하지만 그 안의 멤버 값들은 네트워크 바이트 정렬 상태라서
실제로 PacketHeader 구조체의 각 멤버를 사용하기 전에 반드시 ntohl()로 변환해야함

이렇게 정리해볼 수 있겠네.

이렇게 구조체 자체는 바이트열처럼 다루고,
구조체 멤버의 의미는 역직렬화 과정에서 다시 해석하는 방식으로 처리했어.

정리하면,

- recv_all()과 역직렬화는 네트워크 바이트열을 구조체 메모리에 채워 넣는 과정

이라고 볼 수 있어.

---

### 4-2. send_all()과 직렬화

짧게 요약하자면,

> recv_all()이 네트워크에서 받은 바이트열을 구조체 변수에 채워 넣는 과정이었다면, 
send_all()은 네트워크 바이트 정렬로 변환된 구조체 변수를 바이트열처럼 취급해 
그대로 송신하는 과정이야.

그러니까, 거의 정반대로 하면 된다는거지.

---

recv_all()을 사용할 때는
네트워크에서 받은 바이트열을 PacketHeader 구조체 변수의 메모리 공간에 그대로 채워 넣었어.

반대로 send_all()을 사용할 때는
호스트에서 만든 PacketHeader 구조체를
네트워크로 보낼 바이트열 형태로 그대로 송신해야 해.

즉 recv_all()이
구조체 변수의 메모리 공간에 바이트열을 받아오는 과정이었다면,

send_all()은
구조체 변수에 들어있는 값을 바이트열처럼 취급해서 그대로 보내는 과정이라고 볼 수 있어.

예를 들어, 내가 실제로 서버에서 클라이언트로 헤더를 송신할 때의 코드는 이래.

```cpp
PacketHeader send_host_header;
send_host_header.type = HEADER_SAFE;
send_host_header.length = payload_recv_res;

PacketHeader send_net_header;
send_net_header.type = htonl(send_host_header.type);
send_net_header.length = htonl(send_host_header.length);

server_state.header_send = true;
int header_send_res = send_all(client_sock, server_state, (char*) &send_net_header, HEADER_SIZE);

if (header_send_res == SOCKET_ERROR) break;

server_state.header_send = false;
```
여기서 중요한 건,
송신할 때도 바로 send_host_header를 보내는 게 아니라
send_net_header를 따로 만들어서
send_host_header의 각 멤버를 네트워크 바이트 정렬로 바꿔서 send_net_header에 넣어준다는 점이야.

즉 송신 과정은

1. 호스트 바이트 정렬의 PacketHeader 생성
2. 각 멤버를 htonl()로 네트워크 바이트 정렬로 변환
3. 변환된 PacketHeader 구조체를 바이트열처럼 취급해서 send_all()로 송신

이 순서로 진행돼.

여기서도 send() 함수는 구조체를 이해해서 보내는 함수가 아니라,
그저 주어진 메모리 공간에 들어있는 바이트열을 네트워크로 내보내는 함수야.

그래서 패딩이 없는 PacketHeader 구조체는
그 시작 주소를 char\*로 형 변환해서 그대로 송신할 수 있어.

즉, recv_all()에서 구조체를 받을 때와 마찬가지로,
send_all()에서도 구조체는 결국 연속된 바이트들의 집합처럼 다뤄지는 거지.

다만 여기서도 중요한 건,
구조체 자체를 바이트열처럼 전송할 수 있다는 것과
구조체 멤버의 값이 올바른 바이트 정렬 상태인지 여부는 별개의 문제라는 점이야.

그래서 송신 전에는 반드시 htonl()을 이용해서
type, length를 네트워크 바이트 정렬로 변환한 뒤 전송해야 해.

정리하면,

- send_all()과 직렬화 구조체 메모리에 있는 값을 바이트열처럼 네트워크로 내보내는 과정

이라고 볼 수 있어.

사실, 이 프로젝트에서의 내가 사용한 방식은 아주 복잡한 직렬화 라이브러리를 쓴게 아니라,
> 고정 크기 구조체를 네트워크 바이트 정렬로 맞춘 뒤 그대로 바이트열로 내보내는 단순 직렬화

에 불과해.
