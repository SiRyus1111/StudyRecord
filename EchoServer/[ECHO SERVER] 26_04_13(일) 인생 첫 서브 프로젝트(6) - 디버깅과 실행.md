## 1. 빌드 후 경고 - strcpy() 함수는 권장되지 않는 함수

일단 나는 이전의 코드에서는 strcpy() 함수를 이용해서
사용자에게 std::getline()을 사용해서 입력받은 std::string 타입의 user_input 변수를
전송할 메시지를 담을 `char*`형의 버퍼인 buf에 넣었어.

하지만, 빌드를 해보니 경고가 있었어.
내가 실수로 캡쳐는 못 했었는데,
대충
> strcpy() 함수는 권장되지 않는 함수다. 
이 함수를 사용하려면 ~~~(아마 #define _CRT_SECURE_NO_WARNINGS) 를 추가해라.

이런 느낌의 경고 메시지였는데,
애초에 뭔가 위험이 있어서 strcpy()를 권장되지 않는 함수라며 막았을 것 같아서,
권장되는 함수를 사용하기로 했어.

알아보니까 MSVC에서는 strcpy()를 안전하지 않을 수 있는 함수로 경고하고, 
대신 strcpy_s() 사용을 권장한다고 하네.

기존에 strcpy() 함수는 딱히 문자열을 복사할 버퍼 크기를 지정할 수 없어서,
버퍼 크기보다 문자열의 크기가 더 크다면 버퍼 이상으로 메모리를 수정해버리는
버퍼 오버플로우가 발생할 수 있었대.

그리고 strcpy_s() 함수는 목적지 버퍼의 크기를 함께 받아서, 
복사할 문자열이 그 버퍼에 들어가는지 검사한 뒤 안전하게 복사한다고 해.

그리고 인자를 넣을 때도 내가 실수를 했는데,
```cpp
strcpy_s(char* 값을_쓸_목적지_버퍼, rsize_t 목적지_버퍼의_크기, const char* 복사할_문자열);
```

이렇게 인자를 넣었어야 했는데,

나는 
```cpp
strcpy_s(char* 값을_쓸_목적지_버퍼, rsize_t 복사할_문자열의_크기(user_input.size()), const char* 복사할_문자열);
```
이렇게 인자를 넣었었어. -

그래서 실행하고 나서 클라이언트에 입력을 줬을 때,
이 함수에서 버퍼 공간이 부족하다는 에러(Debug Assertion Failed)가 발생했었어.

분명 최대 버퍼 크기는 4096 바이트였는데,
뭔가 이상해서 찾아보니까,
strcpy_s() 함수의 두 번째 인자는 복사할 문자열의 크기가 아니라, 
복사한 문자열을 저장할 목적지 버퍼의 크기를 넣어야 하더라고.

그래서 sizeof(buf)를 두 번째 인자로 줘서 해결했어.

## 2. 런타임 에러 - htons()와 ntohs()가 다루는 바이트 크기

실행하다가, connect() 부분에서 에러가 발생했었어.
이건 출력된 에러 메시지도 확실하게 있어.
> 요청한 주소는 해당 컨텍스트에서 유효하지 않습니다(WSAEADDRNOTAVAIL, 10049)

이런 에러 메시지였어.
확인해보니까, 해당 소켓 핸들에 서식 자체가 잘못된 IP:Port가 바인딩 되었을 때에 발생하는 에러라고 했고.

마이크로소프트 공식 문서로도,
> WSAEADDRNOTAVAIL
10049
요청된 주소를 할당할 수 없습니다.
요청한 주소가 해당 컨텍스트에서 잘못되지 않음 이는 일반적으로 로컬 컴퓨터에 유효하지 않은 주소에 바인딩 하려는 시도에서 발생합니다. 원격 주소 또는 포트가 원격 컴퓨터(예: 주소 또는 포트 0)에 유효하지 않은 경우 연결, sendto, WSAConnect, WSAJoinLeaf 또는 WSASendTo 에서 발생할 수도 있습니다.

이렇다고 했고 그래서 나는 connect() 함수 내부의 인자로 들어가는 소켓 주소 구조체(server_addr)를 살펴봤어. 
사실상 IP:Port는 이 소켓 주소 구조체에 달려있어서.

여기서 문제였던게, 
포트 번호는 unsigned short 형, 즉 16비트 값인데
나는 여기에 32비트 값을 변환하는 htonl() 함수를 사용해서 바이트 정렬을 변경했었어.

포트 번호를 나타내는 멤버인 `sin_port`는 16비트 멤버인데, 
여기에 32비트 변환 함수인 htonl() 결과를 넣으면
애초에 잘못된 바이트 순서의 값이 만들어지게 되고, 
그 결과가 16비트 멤버에 들어가면서
의도한 포트 번호가 깨져.

그래서 10049 - WSAEADDRNOTAVAIL 에러가 발생했던거였어.

그래서 4바이트의 호스트 바이트 정렬을 네트워크 바이트 정렬로 변환하는 htonl() 함수를 
2바이트의 호스트 바이트 정렬을 네트워크 바이트 정렬로 변환하는 htons() 함수로 바꿨어.

그리고 혹시 다른 부분에도 이렇게 잘못된 바이트 정렬 변환 함수가 사용되지 않았는지 살펴봤고,
서버쪽에서도 포트 번호를 출력하는 부분에서
```cpp
ntohl(client_addr.sin_port)
```
이렇게 되어있던걸 발견해서
```cpp
ntohs(client_addr.sin_port)
```
이렇게 수정했어.

## 3. TIME_WAIT 상태인 포트 - Address already in use 에러

이게 내가 예전에 소켓 함수들 정리했던 글에서, 

서버가 종료되었고,
다시 서버를 실행시켜서 bind()로 이전에 썼던 IP:Port에 새로 생성한 소켓을 바인딩 할 때,

TCP 상태가 해당 포트를 통한 통신의 뒷정리를 하는
(혹시 모를 FIN 세그먼트 재전송 대기, 세그먼트 겹치지 않게 다음 통신 전에 기다리기)
**TIME_WAIT** 상태라면 

OS차원에서 해당 IP:Port에 bind()를 못하게 하는
**Address already in use** 에러를 발생시킨다고 했었지.

좀 많이 간추리긴 했는데,
이 에러는 TCP 상태가 TIME_WAIT 상태이기 때문에 발생한 에러야.

내가 서버를 한번 켜보고 종료한 후,
다시 서버를 켰을 때,

바로 이 에러가 발생했어.

예전에 한번 글로 써봤기 때문에, 바로 기억이 났고,
바로 해결책을 알아봤어.

일단, 내 프로젝트에서의 해결책으로는
TCP 상태가 TIME_WAIT 상태여도 바로 해당 IP:Port에 바인딩 할 수 있게 해주는 소켓 옵션을 설정했어.

이렇게.
```cpp
int option = 1;
if (setsockopt(server_sock, SOL_SOCKET, SO_REUSEADDR, (const char*)&option, sizeof(option)) == SOCKET_ERROR) {
	err_display("setsockopt()");
	return 1;
}
```
여기서 중요한건 
setsockopt() - set socket option 함수의
세 번째 인자인 `SO_REUSEADDR`인데, 
이 인자를 넣음으로써 방금 내가 설명한 TIME_WAIT 상태여도 
해당 소켓이 바로 해당 IP:Port에 바인딩할 수 있게 해줘.

이 옵션이 정말 좋은 옵션이지만,
확실한 부작용도 있어.

이건 환경에 따라 특히 크게 차이나는데,
한 방에 요약하면 이거야.
> 이 옵션이 활성화되어있을 경우, Linux같은 환경에서는 '같은 포트를 여러 소켓이 동시에 바인딩할 수 있다'는 의미이므로,
포트 충돌이 발생할 수 있다.

이게 어떤 문제들을 야기하냐면,
- 다른 프로세스가 같은 포트를 가로챈다거나
- 포트가 하이재킹(악성 프로세스가 해당 포트 바인딩해버림) 당할 수 있다거나
- 멀티 bind 상태에서 디버깅이 어려워진다거나

이런 일들이 발생할 수 있어.

하지만 윈도우(Windows) 환경에서는 `SO_REUSEADDR`옵션은 TIME_WAIT 상태 포트 재사용 허용정도의 옵션이긴 해.
하지만 결국 실무에서는 리눅스 환경에서 서버를 가동하는 경우가 많으니, 확실히 알아둬야겠지.

그래도 서버를 빠르게 실행하고 종료할 때 마다 `SO_REUSEADDR`을 켜지 않으면
이번에 내가 겪은 것처럼 Address already in use에러가 계속 발생해서 불편하기 때문에
테스트 할 때, 개발 할 때는 켜놓고,
진짜로 서버를 운영할 때는 신중하게 선택하는 식으로 `SO_REUSEADDR`옵션을 사용한다고 해.

내 에코 서버 프로젝터에서는 
- 단일 프로세스
- 단일 소켓
- LAN 내부의 에코 서버
- 개발 단계

이므로
켜도 괜찮고, 

오히려 불편하지 않게 켜는게 맞다고 생각해서 이 옵션을 사용해봤어.
