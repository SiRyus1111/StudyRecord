## 1. 객체를 어떻게 표현하는가.

C++에서 객체를 표현할 때 가장 먼저 보이는 선택지들은 대표적으로
- struct
- class
- enum class

정도가 있는데,
실제로도 세 가지의 선택지가 문법만 약간 다른 비슷한 것처럼 보일 수 있어.

나도 예전에는 
- struct = 구조체
- class = 자바의 그 클래스
- enum = 그냥 상수 묶음

정도로만 막연하게 이해하고 있었어.

하지만 이번에 새로 객체지향에 대해 공부하니까,
이 개념들부터 다시 제대로 잡고가야한다는 생각이 들었어.

이 셋은, 결국 문법만 약간 다른 비슷한 것이라기보다,
'표현하려는 것'에 따라서 선택이 달라지는 도구라는 느낌이야.

---

### 1-1. struct vs class

이 두 개는 내가 봤을 때 정말정말 비슷해서 차이점 위주로 설명해볼게.

struct(구조체)는, 
- 멤버 변수 사용 가능
- 멤버 함수 사용 가능
- 생성자 있음
- 소멸자 있음
- 접근 지정자 사용 가능

이고,

class(클래스)는,
- 멤버 변수 사용 가능
- 멤버 함수 사용 가능
- 생성자 있음
- 소멸자 있음
- 접근 지정자 사용 가능

으로,
struct와 class는 문법상으로 상당히 비슷해.

실제로, 클래스를 쓸 곳에 구조체를 써도 컴파일은 잘 되기도 해.

하지만, 문법적 차이점이 하나 있어.
- 기본 접근 지정자 / 기본 상속 접근 지정자

야.

struct는 기본 접근 지정자와 기본 상속 접근 지정자(어쨌든 접근 지정자)가 public이야.
외부에서도 쉽게 참조, 수정할 수 있지.

class는 기본 접근 지정자와 기본 상속 접근 지정자가 private이야.
외부에서 참조, 수정할 수 없지.

근데 이 정도로는 굳이 두 개를 구분할 필요가 없어보여.
하지만, 문법적이 아닌,
**C++의 관례**와 **struct와 class의 설계 의도** 상으로 보면 명확한 차이점이 있어.

앞에서 말했듯이, 
'표현하려는 것'에 따라서 선택이 달라지는 도구라는 느낌이야.

---

struct(구조체)는 단순한 데이터 묶음이야.
그저 멤버 변수들이 논리적으로 묶여있을 뿐,
**주로 특별한 보호를 할 필요가 없는 공개된 데이터 묶음**으로 사용되는걸 의도하고 설계되었어.
그래서 단순한 데이터 구조에서는 주로 struct를 사용해.

내가 설계해보려 했던 멀티클라 에코서버에서는 
클라이언트의 정보(sock, addr)를 저장하는 ClientInfo같이,
특별한 보호를 할 필요가 없는, 
단순한 데이터들의 묶음이 필요한 경우에는
struct를 사용해야할 것 같아.

실제로 내가 설계한 ClientInfo는 특별히 외부에 숨겨야하거나,
실수로 건드리는 상황이 생기지 않고,
단순히 클라이언트의 소켓 핸들(SOCKET sock)과 주소 구조체(sockaddr_in addr)의
단순한 데이터들의 묶음이라서, struct를 사용했어.

---

class(클래스)는 객체 지향적인 데이터 / 행동(메서드)의 묶음이야.
멤버 변수는 private으로 두고,
메서드를 통해서 데이터를 제어하는, 
객체 지향적인 데이터 / 메서드의 묶음을,
**주로 내부 데이터의 상태를 보호하고 캡슐화가 필요할 때** 사용하는걸 의도하고 설계되었어.
그래서 캡슐화가 필요한, 
내부 데이터의 상태를 보호할, 
복잡한 객체를 정의할 때 주로 class를 사용해.

내가 설계해보려 했던 멀티클라 에코 서버에서는 
클라이언트들의 정보가 들어있는 컨테이너를 관리하는 ClientManager같이,
내부 상태를 감추고, 
메서드로 관리를 해줘야 하는 경우,
그러니까 캡슐화가 필요한 경우에는
class를 사용해야할 것 같아.

실제로 나는 원래 
전역에 공유 컨테이너를 넣어놓고,
그 컨테이너에 접근하는 함수(삽입/삭제/카운트) 만들어넣고,
여러 스레드가 동시에 접근하면 안되니까 해당 컨테이너에 접근하기 전에 락 걸고..
이렇게 기존 로직을 짰어.

하지만, 이렇게 ClientManager같이 class로 묶는다면,
일단 
- 전역 공유 컨테이너
- 그 컨테이너에 접근하는 함수
- 해당 컨테이너에 접근할 때의 락(mutex)

이렇게 공유 컨테이너의 관리에 관련된 것들을 한 군데에 모을 수 있고,

나중에 코드가 커져서 발생할 수 있는 상황을 커버할 수 있어.

예를 들어,
- 실수로 컨테이너를 직접 건드릴 수도 있고
- 락 없이 접근할 수도 있고
  
이런 상황을
> ClientManager 클래스의 공유 컨테이너의 접근 제어자를 private으로 해놓고,
해당 클래스의 함수들로만 해당 컨테이너에 접근 가능 + 클래스의 메서드들 안에 락 구현

```cpp
// pseudo code. 대략적으로 이렇다는거임.
class ClientManager {
private:
  std::vector<ClientInfo> clients;
  std::mutex mtx;
  
public:
  void add_client(ClientInfo client); // 이 안에 락 구현
  void remove_client(ClientInfo client); // 이 안에도 락 구현
  int count(); // 물론 이 안에도
};
```

이렇게 해놓는다면,
- 실수로 공유 컨테이너를 건드리는걸 차단할 수 있고,
- 락 없는 직접 접근 실수를 크게 줄일 수 있어.

### 1-2. enum class

일단 기존의 enum부터 가볍게 설명해볼게.
enum의 목적은,
> 연관된 상수를 하나로 묶는 것

이라고 할 수 있어.

지금 내 프로젝트에서도 PacketHeader의 type 멤버의 값이 될 수 있는 값들을 enum (class)으로 표시했지?
이렇게 연관된 상수들을 하나의 타입으로 묶어보리는거야.

이걸 안 쓰면, 나중에 PacketHeader의 type 값이 0이 무슨 의미고, -1이 무슨 의미었는지
기억이 안 날 때에 정말 큰일날 수 있어.
막 type값에 들어갈 수 없는 값을 넣어도 딱히 경고를 주거나 하지는 않아.

하지만 enum을 쓴다면,
이렇게 하나의 타입으로 묶어버려서,
우리가 코드를 읽을 때에 딱히 기억에 의존하지 않고도
값을 직접 넣을 필요 없이,
단순히 enum으로 묶은 타입만으로도 간단히 알아볼 수 있어.
```cpp
enum PacketType {
  HeaderError = 0,
  Safe = -1
};

PacketType host_type = HeaderError; // == 0
```

하지만 기존 enum은 결정적 한계가 있었는데,

첫 번째로,
만약 enum 내부의 식별자(방금의 예시에서는 HeaderError, Safe)의 이름에서
충돌이 발생할 수 있다는 점이야.
```cpp
enum PacketType {
  HeaderError = 0,
  Safe = -1
};

enum StateType {
  Error = 1,
  Safe = 0
};
```
이렇게 있을 때, 오류가 발생해.
이미 Safe가 PacketType에 있어서, 다른 enum에서 같은 이름을 사용하고 있는데 또 Safe라는 식별자를 사용했기 때문에.

두 번째로, 
enum은 내부적으로 정수(int)로 취급되어서
방금 코드에서 분명 StateType과 PacketType이라는 다른 타입임에도,
```cpp
PacketType p = Safe;
StateType s = Error;

if (p == s) { ... } // PacketType과 StateType이라는 별개의 타입에 관계없이 int로 취급해서 비교
if (p == -1) { ... } // PacketType을 그저 int로 취급해서 -1과 비교
```
이런 상황이 발생할 수 있어.

이게 왜 문제가 되냐면,
만약 PacketType의 Safe가 1, StateType의 Error가 1이라고 해보자.
그런데 만약 두 enum을 비교하게 된다면?
`Safe == Error`가 True를 반환하는 정말 웃긴 상황이 발생할 수 있겠지.

그래서 enum class는 
기존에 enum에다가 아예 독립적인 타입이라는 것을 명시하기 위해 class를 붙여서 
기존의 문제들을 해결했어.

```cpp
enum class PacketType { Safe = -1 };
enum class StateType  { Safe = 0 }; // 가능! 중복 에러 안 남

PacketType p = PacketType::Safe;
StateType s = StateType::Safe;
```
이렇게 범위 지정 연산자 없이 접근할 수 없게 해서 
이름 중복도 가능하고,
<br>
```cpp
if (p == s) { ... }      // 컴파일 에러! (타입이 다름)
if (p == -1) { ... }     // 컴파일 에러! (타입이 다름)
```
이렇게 int로의 암시적 자동 형 변환을 금지해서
기존의 문제들이 해결되었어.

어쨌든 enum class는,
기존 enum의 목적인 
> 연관된 상수를 하나로 묶는 것

의 목적으로 사용되고,
그저 enum의 한계들을 보완한 것일 뿐이야.

그래서 사실상 C++에서는 enum class를 더 권장해.

내 설계에서는 왜 enum class를 쓰냐면,
그저 `type = -1` 대신 `type = PacketType::Safe`라고 씀으로써 
이 코드를 읽었을 때의 가독성을 챙길 수 있고(바로 type이 안전한 상태라고 읽히지)

실수로 PacketType을 넣어야하는 곳에 StateType을 넣었을 때도 
귀찮은 논리 오류가 발생하지 않고 컴파일러가 바로 에러를 표시해줘서
코드의 안전성을 챙길 수 있고

나중에 PacketType에 새로운 타입이 추가될 때도 다른 식별자와 겹칠까봐 걱정할 필요도 없어져.

그러니까

1. 가독성
2. 안전성
3. 유지보수성

때문에 enum class를 쓴다고 해.

그리고 실제 설계 글에
```cpp
enum class PacketType : int32_t {
    HeaderError = 0,
    Safe = -1
};
```
이렇게 `int32_t`라고 적혀있는건
> 이 열거형 변수 하나가 메모리에서 정확히 4바이트를 차지하고, 부호 있는 정수로 읽힐 수 있게 강제

라는 뜻이야.

PacketType이 쓰이고 있는 PacketHeader는 1바이트 단위가 정말정말 중요한 곳이고,
값의 의미도 갑자기 unsigned로 해석된다거나 하면 안되는 부분이라서,
이렇게 기반 타입 지정(Underlying Type Specification)을 해줬어.

그래도 당연히 네트워크로 보낼 때는 여전히 바이트 정렬이랑 직렬화는 별도로 신경써야하고.

## 2. 객체의 생명 주기

일단 앞에서는 C++에서 객체를 어떻게 표현하는지에 대해,
struct, class, enum class가 어떤 상황에서 쓰이는지를 정리해봤어.

이번 파트는 이렇게 만들어진 객체가 언제 생성되고, 언제 사라지는지에 대한 내용이야.

이번 파트는 내 멀티클라 에코 서버 설계에서 특히
클라이언트 스레드에 ClientInfo를 넘기는 부분에 적용이 될 것 같아.

---

### 2-1. 객체는 언제 생성되고 언제 소멸되는가

일단, 단순히 예시를 들어보자면,
```
// client_thread 내부
ClientManager client_manager;
```
지역 객체는 보통 변수가 선언되는 순간 생성이 되고,
그 변수가 속한 범위(scope)를 벗어나는 순간 소멸해.

이렇게 예시처럼 선언한 객체는, 해당 스레드가 끝나면 소멸하겠지.
물론 이건 함수 내부에서 선언한 객체는 해당 함수가 끝나면 소멸한다는 말이랑 같아.

아니면 이렇게 중괄호 블록 안에서 선언한다면,
```cpp
if (뭐시기) {
  ClientManager client_manager;
}
```
중괄호 블록이 끝날 때 소멸해.

하지만 동적 할당으로 선언된 객체는 개발자가 따로 delete를 호출하기 전까지는
소멸하지 않고 계속 남아있게 되지.

사실상 이게 객체가 언제 생성되고 언제 소멸되는지에 대한 설명 끝이야.

하지만 객체의 생성 / 소멸 시점이 프로그래머 입장에서는 상당히 중요해.

---

### 2-2. 객체의 생성과 소멸 시점이 중요한 이유

일단 첫 번째로, 메모리 누수가 발생할 수 있어.
동적 할당으로 선언된 객체는 개발자가 따로 delete를 호출하기 전까지는
소멸하지 않는다고 했잖아?

그런데 만약 개발자가 delete를 호출하는 것을 까먹는다면?
메모리 누수가 나고 만약 그런게 1초에 수십번만 발생해도 얼마못가고 메모리가 터져나가겠지.

두 번째로, 댕글링 포인터(Dangling Pointer)라는건데,
그러니까 이미 소멸된 객체의 주소를 가리키고 있는 포인터를 사용할 때에 문제가 생긴다는거야.

쉽게 설명하자면, 이미 철거된 집의 주소에 택배를 보내는거야.
이미 집이 철거됐는데, 원하는 사람한테 잘 가겠어?

실제로 이런 댕글링 포인터 문제가 발생한다면, 프로그램이 즉사(Crash)할 수도 있어.
그만큼 심각한 문제지.

세 번째로, 이중 해제(Double Free)라는건데, 
이미 소멸한 객체를 또 소멸시키려고 할 때,
이미 해제한 메모리를 또 해제하는거나 다름이 없어서 힙 메모리 꼬이고,
프로그램이 비정상 종료되거나 정의되지 않은 동작이 발생할 수 있어.
요약하자면, 어쨌든 메모리 관리 시스템이 꼬일 수 있다는거.

마치 전셋집에서 이전 세입자가 나갔는데,
집주인이 새로 들어온 세입자를 이전 세입자인 줄 알고 내쫓으면 안되겠지?
이렇게 이해하면 돼.


---

### 2-3. 내 설계에서 객체의 소멸 시점이 중요했던 이유

client_thread 함수
int client_thread(ClientInfo info) - 여기 값 복사로 받아야할까 참조로 받아야할까 포인터로 받아야할까 모르겠음.
먼저 스레드를 관리해주는 list / vector / map 등의 접속 정보 저장하는 컨테이너에 어떻게든 info 구조체 추가.

---

이 부분이 상당히 중요했어.
내가 값 복사로 받아야할까 참조로 받아야할까 포인터로 받아야할까 모르겠다고 적어놨었는데,
결국 이것도 객체의 소멸 시점에 관한 문제야.

일단 info값은 main() 함수에서 accept()가 끝난 소켓과 해당 소켓의 소켓 주소 구조체를 담은 구조체인
ClientInfo 구조체에 담아서 client_thread에 전달해야해.
실제로 함수 호출은 이렇게 되겠지.
```cpp
while (true) {
  sockaddr_in client_addr{};
  int addr_len = sizeof(client_addr);
  
  SOCKET client_sock = accept(listen_sock, (sockaddr*) &client_addr, &addr_len);
  
  if (client_sock == INVALID_SOCKET) {
  	// 예외처리
  }
  
  ClientInfo client{};
  client.sock = client_sock;
  client.addr = client_addr;
  
  std::thread client_th(client_thread, 여기에_ClientInfo_구조체);
  
  client_th.detach();
}
```
이런 식으로.

그런데, 만약 client_th 스레드 내부에서 
ClientInfo의 info(메인 함수에서는 client) 구조체에 접근을 했는데,
그 구조체가 이미 main() 함수에서 소멸시켜서 실제로는 없다면?

확실히 문제가 되겠지..

어떨 때 문제가 되는지 알아보자면,
이게 결국, 전에 객체를 중괄호 블럭 안에서 선언했다면 중괄호 블럭이 끝날 때 소멸한다고 했지?
이게 while문도 다음 반복이 될 때, 그러니까 매 루프가 끝날 때마다 소멸돼.

그런데 detach() 함수까지 끝나서 다음 반복에 접어들었을 때,
ClientInfo 타입의 client 구조체도 소멸되는데,
만약 client 구조체를 레퍼런스나 포인터로 넘겨줬다면?
바로 그냥 댕글링 포인터 문제가 생겨.
큰일나겠지.

결국 이거에 대한 가장 편한 정답은 그냥 client 구조체의 값을 복사해서 스레드에 넘기면 돼.
하지만 결국 이것도 지금은 ClientInfo 타입의 크기가 
SOCKET(8바이트) + sockaddr_in(16바이트) 정도로 작으니 문제 없이 값을 복사할 수 있지만,
client 구조체의 역할을 할 구조체(꼭 ClientInfo 타입이 아니더라도)의 크기가 커지고
접속자의 규모가 커지게 되면 무조건 값을 복사하는데 확실히 자원이 크게 낭비가 되고,
그로 인한 지연 체감이 커질 수 있어.

이거에 대한 해결책은 정말 난감하지.
복사 비용은 아까운데, 포인터는 위험해.. 참 어렵지.

해결책을 간단하게 설명해보자면,

1. 스마트 포인터(std::shared_ptr)를 쓰거나
- 객체를 가리키는 포인터가 몇 개인지(참조 횟수)를 세고,
- 그 횟수가 0이 되지 않으면 객체가 소멸되지 않음.
- main() 함수에서 루프가 끝나서 객체가 사라지려 해도,
스레드가 여전히 그 객체를 붙잡고 있다면 객체를 가리키는 포인터(참조 횟수)는 0개가 아니므로
객체는 소멸되지 않음.

2. 동적 할당을 한 후 move로 소유권 자체를 스레드에게 넘겨서 객체를 스레드 안에 살려두는 방법
- std::unique_ptr을 사용함.
- main() 함수는 객체를 만들자마자 스레드에게 소유권을 던져버리고 그 객체에서 손을 떼는 것.
- 그러니 루프가 끝나도 객체는 이미 main() 함수를 떠났으니 스레드가 끝날 때까지 살아있게 됨.

이 있어.

이건 나중에 스마트 포인터 공부할 때 좀 더 파보기로 하자.

## 3. 생성자와 소멸자의 역할

지금까지 C++에서 객체가 어떻게 표현되고, 생성되고, 소멸되는지 알아봤는데,

C++에서는 생성될 때와 소멸될 때, 자동으로 특수한 함수들이 호출돼.
그게 생성자와 소멸자야.

말 그대로 생성자는 객체가 생성될 때 호출되고,
소멸자는 객체가 소멸될 때 호출되지.

그러니까 객체의 시작과 끝을 담당하는 함수라고 생각하면 돼.

생성자는 보통 객체가 처음 만들어질 때 필요한 초기화를 맡고,
소멸자는 객체가 사라질 때 필요한 정리 작업을 맡게 돼.

중요한 점은, 
이 함수들은 내가 직접 호출하지 않아도, 
내가 직접 객체 내에 정의하지 않아도
객체의 생명 주기에 따라 자동으로 호출된다는 점이야.

물론 직접 생성자와 소멸자를 정의할 수도 있어.

생성자는 그냥 아예 반환 형식이 없고, 해당 클래스 명과 동일한 이름의 함수야.
```cpp
class MyClass {
public:
    int* data;
    // 생성자 정의
    MyClass() {
        data = new int[10]; 
    }
};
```
이렇게. 함수의 반환 형식이 아예 없고 함수 이름이 해당 클래스 명과 동일하지?

소멸자는 생성자와 비슷한데,
```cpp
class MyClass {
public:
    int* data;
    // 소멸자 정의
    ~MyClass() {
        delete[] data; // 리소스 해제
    }
};
```
이렇게 함수의 이름 앞에 `~`를 붙여주면 소멸자라고 할 수 있어.

이런 생성자와 소멸자는
객체의 생성과 소멸과 동시에 실행되기 때문에,
확실한 용도가 있어.

생성자는 객체 생성 시 필요한 초기화를 해서
안정적인 상태를 만들어주는 역할을 해주고,

소멸자는 객체 소멸 시 할당된 자원을 자동으로 해제해서
메모리 누수를 방지하는 역할을 해줘.

그래서, 결국 내가 나중에 구조를 설계할 때,
어떤 객체가 소켓, 파일, 메모리 같은 자원을 관리하게 된다면,
생성자에서는 그 객체가 사용할 초기 상태를 만들고,
소멸자에서는 사용이 끝난 자원을 정리하는 식으로 이 개념을 응용할 수 있겠지.

## 4. 이것들이 왜 RAII로 이어지는가

앞의 모든 것들이 이걸 설명하기 위해서 설명한 것들이야.
정말정말 쉽게, 축약해서 결과만 말하면,
> RAII = 소멸자에 해제 코드 넣기

라고 할 수 있어.

생성자는 객체 생성 시에 필요한 자원을 할당해주는 역할을 하기도 해.

그러면 할당된 자원들을 객체가 소멸될 때 해제를 해줘야겠지..
안 그러면 자원 / 메모리가 확실히 누수가 날테니까.

마치 malloc()으로 메모리를 할당했는데 free()로 해제를 안 했을 때나,
new로 메모리를 할당했는데 delete를 안 했을 때와 비슷해.

그래서, 소멸자가 객체가 소멸될 때에 객체에 할당된 자원들을 책임지고 해제하게 만드는 방식이 필요했어.

그게 모던 C++의 핵심 디자인 패턴이자 자원 관리 철학인
RAII(Resource acquisition is initialization)야.

그런데 단순히 소멸자에 자원 해제 코드를 넣는다기 보다는,
> 객체의 생명 주기에 자원을 묶는다

라고 생각하는게 좋아.

객체에 할당된 자원을 단순히 객체가 소멸될 때 소멸자로 없앤다기보다,
객체에 할당된 자원도 객체의 생명 주기에 따라 해제된다고 보는거야.

요약하면,
> RAII란, 자원의 획득과 해제를 객체의 생명 주기에 묶는 방식

이라고 할 수 있어.
