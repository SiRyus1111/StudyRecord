## 1. RAII란 무엇인가

저번 글 마지막에 잠깐 설명하긴 했었는데,
다시 한번 설명해볼게.

RAII란, Resource acquisition is initialization의 약자로,
정말 간단하게 설명하자면
> 객체의 생명 주기에 자원의 획득과 해제를 묶는 것

이라고 할 수 있어.

예를 들어 어떤 객체가 메모리, 파일, 소켓, 락 같은 자원을 사용한다면,
그 객체가 살아 있는 동안에는 그 자원도 함께 관리되고,
객체가 소멸할 때는 그 자원도 자동으로 정리되게 만드는 거야.

그러니까, 굳이 개발자가 자원을 일일이 수동으로 관리하는 방식이 아니라,
객체가 생성되고 소멸되는 생명 주기에 맞춰서 자원도 함께 관리하겠다는 발상이라고 보면 돼.

특히, 소멸할 때 해당 객체가 할당받은 자원들을 전부 해제(반환)하는게 중요한데,
이건 다음 파트에서 이어서 설명할게.

## 2. 왜 RAII가 필요한가


### 2-1. 자원 관리의 단순화

객체의 생명 주기에 묶어서,
그러니까 객체가 생성되고 소멸되는 시점에 맞춰서 자원을 함께 관리하지 않으면,
개발자가 그 수많은 자원들의 획득 / 해제 시점을 하나하나 전부 관리해줘야해.

그리고 생성된 자원들을 해제할 필요가 있을 때 
개발자가 일일이 자원을 해제해줘야하니 
실수로 특정 자원을 해제해야하는 것을 까먹어서 
해제를 못할 수도 있겠지?

그렇다면 결국 메모리 누수가 발생할 수 있어.
해당 자원이 영원히 해제되지 않을 수도 있고.

지역 객체가 스코프를 벗어날 때, 그러니까 소멸될 때에 
소멸자가 호출되어서
할당된 메모리나 파일, 소켓, 락 등을 자동으로 해제해줌으로써 
개발자가 직접 하나하나 자원을 해제하지 않더라도 객체가 사용하는 자원들을
안전하게, 확실하게 해제할 수 있어.

그리고 개발자가 일일이 new / delete / open / close 등을 
일일이 수동으로 호출할 필요가 없기 때문에,
코드도 한층 간결해지고
실수로 특정 자원을 해제하지 못하는 경우가 확실히 줄겠지.

### 2-2. 예외에 대한 대처

만약 객체의 함수를 실행하는 도중에 예외가 발생한다면,
RAII를 적용하지 않을 경우
자원이 영원히 해제되지 않을 수 있어.

물론 개발자가 어떻게 코딩을 해놨는지에 따라 다르겠지만,
자원이 영원히 해제되지 않을 수 있는 위험은
RAII를 적용하는 것 보다 높을 수 밖에 없지.

RAII를 적용해서 객체의 생명 주기와 자원의 획득과 해제를 묶는다면,
이렇게 예외가 발생했을 때에도 
스코프를 벗어나는 과정에서 객체의 소멸자가 자동으로 호출되니까
자원이 영원히 해제되지 않을 수 있는 위험을 막을 수 있어.

### 2-3. 소유권 명확화

RAII의 또 다른 장점은,
어떤 자원을 어떤 객체가 책임지고 있는지가 비교적 명확해진다는 점이야.

예를 들어, 메모리 / 파일 / 소켓 / 락 같은 자원을
그냥 여기저기서 원시 포인터(raw pointer, 우리가 쓰는 일반적인 포인터)나
핸들 형태로만 들고 다닌다면,
해당 자원을 누가 해제해야하는지 점점 헷갈리기 쉬워.

하지만 어떤 객체가 그 자원을 자신의 멤버로 가지고 있고,
그 객체가 생성될 때 자원을 획득하고,
소멸될 때 자원을 해제하도록 만들어둔다면,
적어도 그 자원의 책임은 해당 객체에게 있다고 볼 수 있어.

즉, 자원의 소유권이 객체 단위로 묶이기 때문에
코드의 구조도 더 명확해지고
어떤 객체가 자원을 정리해야 하는지 헷갈리는 상황도 확실히 줄일 수 있겠지.

## 3. RAII는 왜 생성자와 소멸자와 연결되는가

솔직히 앞에서 답이 다 나왔어.

객체의 생성과 동시에 자원이 획득되고,
객체의 소멸과 동시에 자원이 해제된다면,

객체의 생성과 동시에 실행되는 생성자와
객체의 소멸과 동시에 실행되는 소멸자가

연관이 있을 수 밖에 없지.

객체의 생성과 동시에 실행되는 생성자로 자원을 획득하고,
객체의 소멸과 동시에 실행되는 소멸자로 자원을 해제하는거야.

예시로 의사 코드(pseudo-code) 수준의 C++코드 하나를 보여주자면,
```cpp
class Resource {
public:
    Resource() { 
        // 여기서 자원 획득(생성자) 
    }
    ~Resource() { 
        // 여기서 자원 해제(소멸자) 
    }
};

int main() {
    {
        Resource resource; // 객체 생성, 생성자 호출로 자원 획득
    }
    // 스코프를 벗어나서 소멸자가 호출되어 resource 객체의 자원이 자동 해제됨

    return 0;
}
```
이런 식으로 객체가 생성될 때 생성자가 호출되어 필요한 자원을 획득하고,
스코프를 벗어나 객체가 소멸할 때 소멸자가 호출되어 그 자원을 자동으로 해제하게 돼.

## 4. RAII의 간단한 예시

소켓을 RAII로 관리하는 간단한 예시를 들어볼게.
```cpp
class SocketGuard {
private:
    SOCKET _sock; // 해당 클래스(객체) 내부에서만 접근할 수 있게 private 접근지정자.
public:

    // 객체 생성과 함께 실행되는 생성자로 _sock 멤버를 획득한 소켓인 s로 초기화!
    SocketGuard(SOCKET s) : _sock(s) { } 

    ~SocketGuard() {
        if (_sock != INVALID_SOCKET) {
            closesocket(_sock); // 소멸자에서 무조건 획득한 소켓을 닫아줌!
        }
    }
    SOCKET get() const { return _sock; } // 객체 외부에서 해당 소켓을 사용할 수 있게 하기 위한 메서드
};
```
대략 이런 식인데,
이 코드도 완벽한 코드는 아니야.
그냥 직관적으로 RAII가 사용되는 예시 정도.

일단 이 클래스는 생성자에서 이미 획득한 소켓의 소유권을 받아서 내부에 저장하고,
객체가 소멸할 때 소멸자에서 closesocket()을 호출해서 획득한 소켓을 닫아주는 구조야.

즉, 소켓이라는 자원의 획득과 해제가
SocketGuard 객체의 생명 주기에 묶여있는거야.

이렇게 하면 개발자가 함수의, 스레드의 모든 종료 지점마다 
closesocket()을 직접 써주지 않아도 되기 때문에,
소켓을 닫는 것을 깜빡해서 발생하는 자원 누수를 줄일 수 있어.

다만 이 객체는 소켓의 소유권을 가지는 객체라서,
실수로 이 객체가 복사된다면 같은 소켓을 두 번 닫으려는 문제가 생길 수 있어서
복사는 막아둬야해.
그래서 실제로 사용되는 코드라면 복사 자체를 막거나, 
소켓이라는 자원의 이동만 허용하는 식으로 설계해야해.

이 코드에서는 복사는 막아두지 않고 단순히 생성자와 소멸자,
그리고 획득한 소켓을 객체 외부에서 사용할 수 있게 해주는 함수로만 구성했어.
이게 이해는 더 쉬우니까.

그리고 내 서버 코드에서도, 
각각의 클라이언트 스레드에서
이 방식으로 SocketGuard 클래스로 소켓을 RAII로 관리하는 객체를 생성해서
소켓을 관리한다면,
해당 클라이언트와의 연결을 종료할 때에, 
해당 스레드 함수 안에서 SocketGuard를 지역 객체로 두었다면,
일일이 closesocket()으로 소켓을 닫지 않고도
함수가 return되어 스코프를 벗어날 때 소멸자가 자동으로 호출되어 소켓도 함께 닫히게 돼.

SocketGuard 객체가 클라이언트 스레드가 종료되어서 스코프에서 벗어났을 때 
자동으로 해당 객체의 소멸자가 호출되어서,
클라이언트와의 연결을 담당하던 소켓이 자동으로 closesocket() 될테니까.

그리고 다음 글의 주제인 스마트 포인터(std::shared_ptr / std::unique_ptr), 
마지막 글의 주제인 동기화 라이브러리의 std::lock_guard 등이
RAII의 예시라고 볼 수 있어.

이거는 해당 글에서 추가로 설명해볼게.

## 5. 내 서버 설계와의 연관점

앞에서 설명한 것처럼, RAII로 소켓을 관리하면 
스레드 종료 시점에서 일일이 closesocket()을 호출하지 않아도 된다는 장점이 있어.

그리고 추가적으로 내 서버 로직 / 코드에 RAII가 적용될만한 부분들이 있는데,
일단 대표적으로 윈속 초기화 / 윈속 종료 부분이야.

서버 시작 시, WSAStartup()으로 Winsock을 초기화해야하고,
서버 종료 시, WSACleanup()으로 Winsock을 종료해야해.

윈속 초기화를 해줘야 모든 소켓 함수를 사용할 수 있어.
그리고 main() 함수가 종료될 때,
윈속 초기화로 얻은 자원을 정리해야해.

그래서 이렇게,
```cpp
class WinsockGuard {
public:
    WinsockGuard() {
        WSADATA wsa;
        int WSAStartupRes = WSAStartup(MAKEWORD(2, 2), &wsa);
        if (WSAStartupRes != 0) {
           throw std::runtime_error("윈속 초기화 실패");
        }
    }
    
    ~WinsockGuard() {
        WSACleanup();
    }
};

int main() {
    try {
        WinsockGuard winsock; // 객체 생성 시 자동으로 생성자가 호출되어 윈속 초기화.
    
        // 서버 로직
    }
    catch(const std::exception& e) {
        std::cerr << e.what() << '\n';
        return 1;
    }
    // 스코프를 벗어나게 되어 자동으로 소멸자가 호출되어 윈속 종료.
    
    return 0; 
}
```
WinsockGuard 클래스의 winsock 객체를 생성하는 방식으로
RAII를 적용해서 윈속 초기화 / 종료를 해주면
혹시나 예외 상황에서 깜빡하고 WSACleanup()을 호출하지 않는 상황을 예방할 수 있어.

그리고 클라이언트들의 정보를 관리할 공유 컨테이너에는 락이 필요하지?
그런데 임계 구역(Critical Section)에 들어가기 전에 락을 걸고,
Critical Section에서 나온 후에는
공유 컨테이너에 클라이언트의 정보를 다 추가했거나, 
다 지웠거나, 클라이언트의 갯수를 다 셌다는 뜻이겠지?
그러니까 공유 자원을 다 사용했으니 락을 해제해야하겠지?

그리고 이것도 까먹을 수 있어. 
Critical Section에서 나왔는데, 
락을 해제하지 않는다면 
다른 스레드들이 공유 컨테이너에 대한 작업을 진행할 수 없어.
이미 락이 걸려있는데 해제를 안 했기 때문에.

그래서 딱 공유 컨테이너를 건드리는 함수가 종료되면,
스코프를 벗어나면 락을 해제하는
std::lock_guard를 사용할 수 있어.
```cpp
std::mutex mtx;

void func() {
    std::lock_guard<std::mutex> lock(mtx);
    // 여기서는 자동으로 lock
} 
// 스코프 끝나면 자동 unlock
```
이렇게.

이게 객체의 생명 주기에 맞춰서 자원을 함께 관리하는
RAII와는 크게 관련이 없어보일 수 있지만, 

std::lock_guard는 생성될 때 mutex에 락을 걸고,
스코프를 벗어나서 소멸될 때 자동으로 락을 해제하는
RAII 객체라고 할 수 있어.

락을 객체 생명 주기에 묶은, 
C++ 표준 라이브러리의 대표적인 RAII 예시라고 할 수 있어.
