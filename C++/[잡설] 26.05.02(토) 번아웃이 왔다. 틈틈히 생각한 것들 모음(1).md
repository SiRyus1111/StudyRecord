## 1. 스마트포인터


shared_ptr은 단순히 참조 횟수를 관리한다는게 아니야. 

해당 객체를 소유하고 있는 shared_ptr들이 해당 객체를 더이상 소유하지 않게 된다면 
해당 객체를 소멸시키는 객체 lifetime 관리 도구라고 난 생각해. 

아무도 소유하지 않는 객체는 의미가 없으니까. 

만약 계속 객체가 메모리에 남아있다면 메모리 누수 / 자원 누수가 발생하겠지. 
그리고 애초에 해당 자원을 소유한다는 것 자체가, 
ref count를 올린다는 것 자체가 
shared_ptr에서는 객체 lifetime에 영향을 주는 동작이야. 

객체 lifetime에 영향을 주면 안되는 경우지만, 
해당 객체를 참조를 해야할 때는 weak_ptr을 사용하지. 

사실 그냥 단순 참조만 할거면 raw pointer를 써도 되겠지만, 
해당 객체의 소유권을 가진 shared_ptr가 
전부 소멸되어서 
해당 객체가 소멸되어있다면 
댕글링 포인터 문제가 발생할 수 있기 때문에 
weak_ptr을 써서 control block을 보는 식으로 
해당 객체가 남아있다면 접근 할 수 있게 하는거지. 

물론 해당 객체가 소멸되면 
weak_ptr이 무효화되어 nullptr을 반환해서 
객체가 소멸했을 때의 처리를 할 수 있게 하고.

unique_ptr은 해당 객체의 소유권을 명확히 하고, 
그러니까 어떤 unique_ptr가 해당 객체를 소유할 수 있는지,
더 쉽게 말하면 어떤 포인터가 해당 객체의 소멸에 책임을 지는지..

해당 객체를 소유하고 있는 unique_ptr가 소멸된다면 해당 객체도 소멸되는 식이야. 

그래서 복사 불가능 이동 가능인거고.
결국 다 ownership과 객체 lifetime을 보장하기 위해 나온거지.
shared_ptr이든 weak_ptr이든 결국 객체의 소유권을 가짐 + 소유하지 않을 때 객체를 소멸.
더 쉽게 말하면, 스마트 포인터와 객체의 생명 주기를 일치시키는 것이라고 할 수 있지.
