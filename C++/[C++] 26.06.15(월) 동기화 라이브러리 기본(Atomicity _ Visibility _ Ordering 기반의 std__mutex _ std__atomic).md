### 1-1. 만악의 근원, 레이스 컨디션

레이스 컨디션이 뭘까?

그냥 여러 실행 흐름에서 동시에 공유하는 데이터(공유 자원이라고 하지)에 접근해서,
그 데이터의 의미가 망가지는거라고 할 수 있어.

예를 들어,

```c
// 전역 변수 (공유 자원)
남은_좌석 = 1

function 수강신청(학생_이름): // 어떤 학생이 수강 신청을 했을 때
    // 1. Time-of-Check (조건 검사)
    if 남은_좌석 > 0: // 좌석이 남아있으면
        print(학생_이름 + ": 빈자리 확인! 등록 진행합니다.")
        
        // [컨텍스트 스위칭 포인트] 
        // 검사는 통과했지만, 아직 남은_좌석을 깎지 않은 '찰나의 틈'이 발생!!
        
        // 2. Time-of-Use (실제 데이터 사용 및 변경)
        남은_좌석 = 남은_좌석 - 1
        print(학생_이름 + ": 수강 신청 성공!")
    else:
        print(학생_이름 + ": 남은 자리가 없습니다.")
```

수강 신청 사이트의 코드가 이렇다고 생각해보자.

`남은_좌석`은 공유 자원이야.

그래서 수강 신청을 하려는 여러 학생들이 공유하지.

자, 이제 
학생 A(스레드 1에서 수강 신청이 처리됨)와 
학생 B(스레드 2에서 수강 신청이 처리됨)가 있다고 해보자.

1. 일단 학생 A가 수강 신청을 눌렀고, 
스레드 1이 실행됐을 때 `남은_좌석`이 하나가 남아서
해당 강의에 남은 좌석이 있는지 확인하는 if문을 통과했어.

2. 근데 `남은_좌석`을 1 감소시키기 전에
다른 스레드와 전환이 되는 컨텍스트 스위칭이 발생해버렸고,

3. 학생 B의 수강 신청이 진행중인
스레드 2가 실행됐어.

4. 근데 아직 스레드 2가 보는 `남은_좌석`은 1이라서, 
스레드 2는
남은 좌석이 있는지 확인하는 if문을 통과해버려.

5. 근데 여기서 또 `남은_좌석`을 갱신하기 전에
컨텍스트 스위칭이 발생해서
다시 학생 A의 수강 신청이 진행중인 스레드 1이 실행됐어.

6. 그래서 스레드 1은 `남은_좌석`을 1 감소시키고,
수강 신청 성공 메시지를 띄웠어.

7. 그리고 다시 스레드 2로 전환되어서,
스레드 2도 `남은_좌석`을 1 감소시키고,
수강 신청 성공 메시지를 띄웠어.

> 자리는 한 자리밖에 안 남았었는데,
두 명이 수강 신청을 성공해버렸지?

> 게다가, 이게 unsigned 자료형으로 `남은_좌석`을 관리했으면 더 대참사인게,
단순히 signed 자료형이면 -1이 되어서 다음 학생의 수강 신청은 막히지만,
unsigned는 비트 관점에서는 (signed로 해석했을 때) -1이 되지만, 
값 관점에서는 해당 자료형이 표시 가능한 최댓값이 되어버려서,
이론상으로는 (uint32 기준) 42억명까지 수강신청을 할 수 있게 되는 대참사가 벌어져.

시간 순으로 보면 이런 느낌이지.

|시간|학생 A (스레드 1)|학생 B (스레드 2)|`남은_좌석` 상태|
|---|---|---|---|
|T1|if 남은_좌석 > 0 검사 (true)||1|
|T2|[컨텍스트 스위칭으로 대기]||1 (아직 안 깎임)|
|T3||if 남은_좌석 > 0 검사 (true)|1|
|T4|남은_좌석 = 1 - 1 (성공!)||0|
|T5||남은_좌석 = (컨텍스트 스위칭 전에 읽었던)1 - 1 (성공!)|0(남은 좌석 하나에 두 명이 신청된 대참사)|

어쨌든 이렇게,

> 여러 실행 흐름(스레드)에서 동시에 공유하는 데이터(공유 자원)에 동시에 접근하게 되어서,
해당 데이터의 의미가 망가지는 것

> 여러 실행 흐름이 동일한 공유 자원에 동기화 없이 접근하여, 
실행 순서에 따라 결과가 달라질 수 있는 상태

을 레이스 컨디션이라고 해.

이 수강 신청 예시는 그 중에서도
TOCTOU(Time-of-Check to Time-of-Use) 형태의 대표적인 레이스 컨디션이야.

딱 봐도 여러 스레드가 동시에 공유 자원에 접근해서
값의 의미가 망가지는게 보이지?

좀 더 내려가면
어셈블리 레벨까지 가긴 하는데,
이 예시도 충분히 레이스 컨디션이 뭔지를 나타내니까
어려운 어셈블리 이야기는 하지 않을게.
근데 결국 위에서 말한 근본적 원인이 어셈블리에도 적용이 돼.

결국, 이걸 해결하려면
방금의 수강 신청 코드는,

```c
// 전역 변수 (공유 자원)
남은_좌석 = 1
방어벽 = mutex() // 공유 자원에 접근하기 위한 자물쇠 준비

function 수강신청(학생_이름): // 어떤 학생이 수강 신청을 했을 때

    방어벽.lock() // 자물쇠 채우기. 여기서부터 한 번에 한 스레드만 진입 가능 (임계 구역 시작)
    
    // Check와 Use가 하나의 덩어리로 보호됨
    if 남은_좌석 > 0: // 좌석이 남아있으면
        print(학생_이름 + ": 빈자리 확인! 등록 진행합니다.")
        
        // 2. Time-of-Use (실제 데이터 사용 및 변경)
        남은_좌석 = 남은_좌석 - 1
        print(학생_이름 + ": 수강 신청 성공!")
    else:
        print(학생_이름 + ": 남은 자리가 없습니다.")
        
    방어벽.unlock() // 공유 자원에 대한 접근이 끝났으니 자물쇠 해제
```

이렇게 수정할 수 있을거야.

그리고 저기에 방어벽이라고 한 거 있지?
저게 동기화 도구 중 한 종류인 mutex(뮤텍스, mutual exclusive(상호 배제))야.

이렇게 레이스 컨디션을 방지하기 위해,
다양한 도구들을 사용해.

### 1-2. 레이스 컨디션을 막는데 필요한 성질들

그렇다면 레이스 컨디션을 막으려면,
즉 동기화를 하려면
어떤 성질들이 필요할까?

사실 이거 학부에서 잘 안 가르쳐 주는 파트긴 한데,
동시성 프로그래밍 제대로 파보려면
필수 수준으로 중요한 개념들이야.

동기화에는,

- 원자성(Atomicity) 
: 공유 자원에 접근했을 때 중간 상태를 다른 코어 / 스레드에서 관측할 수 없어야 한다.
  - 만약 원자성이 지켜지지 않으면,
  중간에 컨텍스트 스위칭이 발생했을 때,
  위의 사례처럼 중간 상태에 다른 코어 / 스레드가 간섭해서
  값의 의미가 망가질 수 있다.
- 가시성(Visibility) 
: 공유 자원에 대한 접근이 끝났을 때의 변경 사항이 
바로바로 다른 코어 / 스레드에서 관측할 수 있어야 한다.
  - 각각의 다른 코어에서 실행되는 다른 스레드들은
  원자성에 의해서 다른 스레드가 접근했던 공유 자원의 값을 참조할 필요가 있을 때,
  항상 최신 값을 사용해야 한다.
  - 그렇지 않으면 각 코어가 서로 다른 값을 보고 연산하게 되어,
  값의 의미가 망가질 수 있다.
  - 각 코어가 접근하는 메모리(RAM)는 모든 코어가 공유하지만,
  L1 / L2 캐시는 각 코어에 별도로 존재한다.
  - 그렇기 때문에 각 코어의 L1 / L2 캐시에 최신 값이 반영이 되지 않는다면,
  각 코어가 서로 다른 값을 보고 해당 데이터를 수정할 수 있게 된다.
  - 그렇기 때문에 공유 자원에 대한 접근이 끝났을 때의 변경 사항이
  바로바로 다른 코어 / 스레드에서 관측할 수 있어야 한다.
- 순서성(Ordering)
: 의도한 실행 순서가 유지되어야 한다.
  - 컴파일러나 CPU는 서로 다른 실행 흐름(스레드) 사이의 의존성을 신경쓰지 않고,
  싱글 스레드 관점에서 코드의 (예상) 실행 결과가 똑같이 나오게 하는 것을 목적으로 
  어셈블리 / 기계어를 재정렬하는 식으로 코드를 최적화한다.
  - 하지만, 그렇게 됐을 때에 서로 다른 실행 흐름 사이의 의존성이 고려되지 않았기 때문에
  값의 의미가 망가질 수 있다.
  - 이게 왜, 어떻게 문제가 되는지는 `std::atomic`에서 좀 더 자세히 알아보자.
  - 예시 없으면 설명 못하겠다 진짜.

이렇게 세 가지의 성질들이 필요해.

참고로 전문적인 동기화 / 동시성 프로그래밍 교재같은거 보면,
이 세 개념들이 밥먹듯이 나옴..

그만큼 중요하다는거야.

그리고, 동기화 도구에서는
이 성질들을 만족시키기 위해
다양한 방법을 사용하지.

결국, 동기화의 본질이란
> 원자성 / 가시성 / 순서성을
레이스 컨디션이 발생할 수 있는 곳(임계 구역)에 만족시켜서,
레이스 컨디션을 막는 것

이라고 할 수 있어.

즉,
> 임계 구역에 대한 원자성 / 가시성 / 순서성이 만족된다면,
레이스 컨디션으로 인해 공유 자원의 의미가 망가지는 문제를 방지할 수 있다.

라는거지.

원자성 / 가시성 / 순서성을 만족시키기 위한 방법들은
대표적으로,

- 원자성 : 캐시 일관성 프로토콜(MESI)을 이용한 캐시 락(Cache lock)을 이용해 보장
- 가시성 : 캐시 일관성 프로토콜(MESI)을 이용한 타 캐시로 변경 사항 전파를 해서 보장
- 순서성 : 메모리 배리어(메모리 펜스)를 이용한 코드 재정렬(Reordering) 방지를 이용해 보장

이딴 것들이 있어.

이건 지금은 몰라도 됨..

왜냐하면 이것들 전부 뒤에 나올거임ㅋㅋ
근데 적어놓고 보니까
싹 다 `std::atomic`쪽 원자성 / 가시성 / 순서성 보장 방식이네ㅋㅋ

`std::atomic` 파트에서 진짜 죽어나갈거임ㅋㅋ

### 1-3. 레이스 컨디션을 잡는 동기화 도구들

크게 두 가지야.
- `std::mutex` - Lock-based 
: 그냥 자물쇠 걸어서 한 번에 한 스레드밖에 못 들어오게 해버려서 레이스 컨디션 막기
- `std::atomic` - Lock-Free
: 자물쇠 필요 없이 일단 들이박아보고 레이스 컨디션 발생할 것 같으면 무른 후 다시 들이박아보기

이 정도..
그냥 뮤텍스 / 아토믹이 아니라
Lock-Based / Lock-Free로 이해하는게 더 좋을거야..

## 2. Lock-Based 동기화 도구 : `std::mutex` 계열

### 2-1. 뮤텍스란 무엇인가

뮤텍스란, 용어적으로는

> Mutual Exclusive, 상호 배제의 약자로,
말 그대로 여러 실행 흐름이 동시에 임계 구역에 들어가는 것을 막는
동기화 도구

라고 할 수 있어.

### 2-1-1. High-Level 수준의 이해

솔직히, 이것만 보고는 감이 잘 안올텐데,
딱 화장실 비유로 하면 감이 와.

이게, 각 칸마다 변기가 있고,
문이 잠겨있으면 누가 있다는 뜻이니까 못 들어가잖아?

그러니까, 

- 사람 = 스레드
- 화장실 칸 = 임계 구역
- 변기 = 공유 자원
- 문 잠금쇠 = 락(뮤텍스)
- 들어가서 문을 잠그는 행위 = 임계 구역에 들어가며 락을 거는 것(뮤텍스를 얻는 것)
- 나오면서 문의 잠금을 풀고 나오는 행위 = 임계 구역에서 나오며 락을 해제하는 것(뮤텍스를 해제하는 것)

어쨌든 그렇게 문 잠금쇠를 걸고 들어가면,
이미 사람이 들어가 있는 칸은 문이 잠겨있으니까,
다른 사람은 그 칸에 들어갈 수 없게 되겠지?

즉, 한 번에 한 사람만 해당 변기를 사용할 수 있게 되는거지.

이게 뮤텍스에서 락을 거는 행위랑 같아.
한 스레드가 임계 구역 안에 들어가서 락(뮤텍스)을 획득하면,
다른 스레드들은 해당 임계 구역 안에 못 들어온다 이말이지.

근데 내가 화장실 칸에 들어갈 때 문 잠금쇠를 걸고 들어가야
다른 사람이 안에 사람 없는 줄 알고 내가 볼일보고있는데 들어오는 불상사를 막을 수 있잖아?

설마 현실에서 그런 일은 없겠지만,
코딩 / CS적으로 보면 문을 열고 들어와서 내가 쓰고 있는 변기를 같이 쓰는..
어우..
더럽..

어쨌든 그러면 안되겠지?

이건 임계 구역에 들어갈 때 락을 안 얻고 들어가는 상황이라고 볼 수 있어.
락을 안 걸면, 다른 스레드가 동시에 내가 접근한 것과 같은 공유 자원에 접근하게 되어서,
레이스 컨디션이 발생할 수 있겠지.

그리고 문 잠금쇠를 안 풀고 그냥 안에서 벽을 타고 나오면
뒤에 기다리는 사람들은 그 변기를 계속 사용할 수 없잖아?

그러면 사실상 그 변기는 못 쓰게 되는거지.
화장실 칸 안에 사람이 없는데 문 잠금쇠가 안 열려있으니..

이건 락(뮤텍스)을 획득하고 깜빡하고 락을 해제하지 않은 상황이라고 볼 수 있지.
그래서 항상 락을 얻고 임계 구역에 들어갔으면
임계 구역에서 나올 때는 무조건 락을 해제해줘야해.

즉, 임계 구역에 들어갈 때 락을 걸고,
임계 구역에서 나올 때 락을 해제해야하는거지.

의사 코드로 보면 이런 느낌.

```cpp
// 전역 변수 (공유 자원)
static int a = 1;


a_mutex = mutex()


// 스레드 1

a_mutex.lock()
// 공유 자원 a에 접근하는 임계 구역
a_mutex.unlock()



// 스레드 2

a_mutex.lock()
// 공유 자원 a에 접근하는 임계 구역
a_mutex.unlock()
```

실행 흐름을 보자면,

| 시간 | 스레드 1                 | 스레드 2                       |
| -- | --------------------- | --------------------------- |
| T1 | `a_mutex.lock()` 성공   |                             |
| T2 | 공유 자원 `a` 접근 시작       | `a_mutex.lock()` 시도         |
| T3 | 공유 자원 `a` 접근 중        | 뮤텍스가 이미 잠겨 있어서 대기(Blocked)  |
| T4 | `a_mutex.unlock()` 호출 |                             |
| T5 |                       | 대기 해제 후 `a_mutex.lock()` 성공 |
| T6 |                       | 공유 자원 `a` 접근                |
| T7 |                       | `a_mutex.unlock()` 호출       |

그리고, 락을 이미 다른 스레드가 얻어서,
해당 스레드가 락을 얻을 수 없다면,
해당 스레드는 대기 상태(blocking)가 돼.
즉, sleep 상태라는거지.

그리고, 스레드가 락(뮤텍스)을 얻는다는건
결국 뮤텍스라는 복사불가능한 공유 자원을 해당 스레드가 소유한다는 것,
소유권을 얻는다는 것과 같아.

이렇게 보면 사실상 뮤텍스 요약은 끝이야.

### 2-1-2. Low-Level 수준의 이해

방금까지가 일단 뮤텍스의 하이 레벨 동작이라고 할 수 있어.
그니까 초초초 압축 요약 정리본 느낌(기능 위주).

이제 너무 깊게 파진 않고 적당히(적당히 안하면 이 글 쓰는데 한 달 이상 걸림),
뮤텍스가 대체 뭔지,
어떻게 만들어져있는건지 알아보자.

일단 방금까지의 설명을 보면 이런 의문이 들 수 있어.

> 두 스레드가 동시에 `lock()` API을 실행하면 어떻게 되지?

사실, `lock()` / `unlock()` 함수를 호출할 때,
락을 얻고, 락을 해제하는 과정 자체가 원자적으로 일어나.

위에서 원자성 나왔지?

> 다른 코어 / 스레드에서 연산의 중간 과정을 관측할 수 없다.

라고..

나중에 원자적 연산(CAS 연산)에서 나올텐데,
간단하게 말하면 이거야.

- 일단 별도의 `lock()` 함수를 호출하는데 사용하는 락 같은 거는 얻지 않고 
`lock()` 함수를 호출해 뮤텍스를 얻으려고 해본다.
- 하지만 그 사이에, 다른 실행 흐름이 락을 얻었다면, 지금 당장 락을 얻는 것을 포기하고,
대기 상태(blocking)에서 기다린다.
- (그리고 이게 원자적 연산이라 필연적으로 false sharing같은 문제가 발생할 수 있는데,
이건 추후에 atomic 파트에서 더 자세히 알아보자.)

그러니까, 

> 락을 얻는 행동의 시작과 끝만 중간 과정을 생략한 채 외부에서 관측되고,
끝은 오로지 성공 / 실패로 귀결된다는거야.
락을 '얻었다' / '못 얻었다'로.

내가 락을 얻으려고 시도하던 중에 다른 스레드가 락을 얻었다면,
그냥 락을 얻는 것을 포기하고 `sleep()`(blocking 상태로 만드는 함수임) 하는거지.
(사실 이런 동작이 나중에 `std::atomic`에서 설명할 CAS 연산의 본질이야.)
(지금은 간단하게만.)


그런데, 다른 스레드가 락을 얻었는지 안 얻었는지는 어떻게 알 수 있을까?

뮤텍스는 간단하게,
기본적인 느낌으로는 이렇게 구성되어있어.
(실제 구현 아님, 의사 코드에 매우 가까움)

```cpp
struct mutex {
    locked;
    owner;
    wait_queue;
}
```

쉽게 설명하면

- `locked` : 해당 락을 누군가가 점유하고 있는지 (여부)
  - 즉, 해당 락이 일단 누구라도 점유하고 있어서 다른 스레드가 해당 `lock()`함수를 실행하면
  `sleep()`이 호출되어 blocking 상태로 바꾸는거.
- `owner` : 해당 락을 어떤 실행 흐름이 점유하고 있는지 (락의 주체)
  - 누가 뮤텍스라는 자원을 소유하고 있는지.
  - 즉, 누가 `unlock()`을 호출해서 락(뮤텍스)을 해제할 수 있는지
- `wait_queue` : 어떤 스레드들이 해당 락을 얻으려고 기다리고 있는지 (대기 명단)
  - 즉, 락을 얻는데 실패해서 blocking 상태에 들어간 실행 흐름들을 모아놓은 큐.
  
이 정도야.

그리고, 계속 blocking과 `sleep()`이 나오는데,
이거는 뮤텍스를 얻지 못한 실행 흐름이
계속 CPU를 점유하며 자원을 낭비하지 못하게
운영체제가 해당 스레드를 잠재운다는 느낌이야.

(여담으로, 락을 얻을 때 CPU를 점유하며 락을 기다리는 것을 스핀락이라고 해.)

그렇게 잠재우는 함수가 `sleep()`이고,
그렇게 잠에 들어있는 상태가 blocking 상태(대기 상태)인거지.

근데, 이렇게 잠에 들었다 깨는데는 비용이 많이 들어.
흔히 컨텍스트 스위칭 오버헤드(Context Switching Overhead)라고 하지.

이게 캐시 상태도 깨질 수 있고,
명령어 파이프라인과 TLB도 꽤 영향을 받기 때문에
이렇게 실행 흐름이 잠에 들었다 깨어나는데는
비용이 상당히 많이 들어.

이렇게 단순히 컨텍스트 스위칭을 하는데만 5000 ~ 10000 클럭이 들지.

그런데, 이렇게 컨텍스트 스위칭을 하는데 많은 클럭이 든다면,
딱 락을 얻으려는 스레드가 락을 못 얻어서 
blocking 상태가 되자마자 락이 해제되면,
컨텍스트 스위칭 오버헤드가 쓸데없이 발생하겠지?

그냥 조금만 기다렸으면 락이 나왔는데,
5000 ~ 10000 클럭의 무거운 컨텍스트 스위칭을 하게 되니까..

그래서 락을 얻으려는 실행 흐름이 락을 못 얻었을 때 잠깐동안 기다리며,
그러니까 잠깐동안 CPU를 점유하며 락을 기다리는 뮤텍스를
어댑티브 뮤텍스(Adaptive Mutex)라고 해.

뭐 '이런 것도 있다~' 그런 느낌이지.

그리고 마지막으로, 
되게 중요한 개념인 락 경합(Lock Contention)에 대해 설명을 해볼게.
여러 실행 흐름이 동시에 한 뮤텍스(락)를 얻으려고 경쟁하는 상황을
락 경합(Lock Contention)이라고 해.

이건 그러니까,

```text
Thread 1 ─ lock 획득

Thread 2 ─ 대기

Thread 3 ─ 대기

Thread 4 ─ 대기
```

이렇게 여러 스레드가 한 뮤텍스를 얻으려고 몰려서,
락을 차지하지 못한 스레드들이 대기 상태(blocking)가 되는..

이전 챕터의 화장실 비유로 풀어보자면,
화장실 칸이 꽉 차서 뒤의 사람들이 계속 화장실 자리가 나는걸 기다리고 있는 상황이지.

근데 문제는 이렇게 락 경합이 발생한다면,
blocking과 컨텍스트 스위칭이 자주 발생하게 되어서,
결국 컨텍스트 스위칭 오버헤드로 인해 성능이 많이 떨어져.

그래서 락 경합이 큰 구조는 별로 좋지 않아.

락 경합을 최소화하려면
각 실행 흐름이 해당 락(뮤텍스)을 최대한 적은 시간동안 가지고 있는 구조가 필요하겠지.

그러면 뒤에서 대기하고 있는 실행 흐름들이
빠르게빠르게 뮤텍스를 얻고 빠르게빠르게 다음 실행 흐름이 뮤텍스를 얻을 수 있게 될테니까.

### 2-2. 뮤텍스는 어떻게 원자성 / 가시성 / 순서성을 보장할까?

자 드가자..
솔직히 이게 처음으로 머리 아픈 파트일거야.

근데 뒤의 아토믹에 비하면 의외로 별게 없어.

- 원자성 보장 : 해당 뮤텍스를 얻지 못한 스레드는 해당 공유 자원에 접근할 수 없게 해서 보장
  - 즉, 락을 사용해 임계 구역에 대한 접근을 통제(상호 배제, Mutual Exclusion)
  - 락도 앞에서 설명했듯이 원자적으로 락을 얻고 해제하기 때문에 
  락의 획득과 해제에 대한 레이스 컨디션도 발생하지 않음.
  - 즉, 락 중복 획득도 방지되어
  임계 구역에는 락을 획득한 하나의 스레드만 들어갈 수 있는 것이 보장됨.
- 가시성 보장 : 메모리 배리어를 사용해서 
락을 획득할 때와 해제할 때 강제로 해당 코어와 타 코어의 캐시를
최신 값으로 교체함.(결과론적으로는 이게 맞음.)
  - 그렇게 함으로써 스레드가 임계 구역에서 들어올 때와 나갈 때,
  가시성을 보장할 수 있음
  - 결과론적으로는 이 설명이 맞지만,
  실제로는 락을 획득할 때(`lock()`)와 해제할 때(`unlock()`) 
  캐시를 강제로 최신 값으로 교체한다는 느낌보다는,
  `unlock()`이 메모리 배리어의 Release처럼 임계 구역 안의 메모리 연산이 임계 구역 밖으로 밀리지 않게 하고
  `lock()`이 메모리 배리어의 Acquire처럼 락을 얻은 뒤의 메모리 연산이 임계 구역 안으로 당겨지지 않게 해서
  이전 스레드가 임계 구역에서 한 변경을 이후 스레드가 관측할 수 있게 하는것임.
  - 근데 결과론적으로는 어쨌든 메모리 배리어로 해당 코어와 타 코어의 캐시를
  최신 값으로 교체하는 것임.
  - 어쨌든 임계 구역 안의 메모리 연산이 임계 구역밖에서 무조건 보여지게 하는거니까.
  - 이것에 대한 원리(캐시 일관성 / MESI 프로토콜) / 메모리 배리어는 아토믹에서 다룸.
  - (솔직히 초반부터 이런거 깊게 다루면 안 될 것 같음ㅋㅋ)
- 순서성 보장 : 메모리 배리어를 사용해서
임계 구역 안의 코드와 임계 구역 밖의 코드를 뒤섞지 못하게 함.
  - 메모리 배리어란, 간단하게 코드의 실행 순서를 뒤바꾸지 못하게 막아주는 배리어.
  - 메모리 배리어를 기점으로 앞과 뒤의 메모리 연산(직관적으로는 코드 느낌)를 뒤섞지 못하게 된다.
  - 메모리 배리어를 뮤텍스가 임계구역 앞과 뒤,
  즉 `lock()` / `unlock()`에 전개해서,
  임계 구역 안의 코드와 임계 구역 밖의 코드가 뒤섞이지 않게 해줌.
  - 이것에 대한 원리(Acquire / Release)는 아토믹에서 다룸
  - (이런거 초반에 넣기엔 더럽게 매움ㅋㅋ)
  
일단 이 정도면 될 듯.
이게 뮤텍스랑 아토믹에 둘 다 나오거든?

근데 뮤텍스는 다뤄야 할 라이브러리들이 좀 많아..

이것들을 한번에 다 설명하는건 솔직히 좀 에바임..ㅋㅋ

그래서 여기서는 

- 캐시 일관성 / MESI 프로토콜
- 메모리 배리어

는 깊게 설명하지 않고

> 이게 어떻게 이런 것들을 보장하고, 대충 이런 느낌이다.

정도로만 설명했어.

깊은 이야기는 아토믹 파트에서 이어서 할게.

### 2-3. 뮤텍스의 장단점(= atomic 밑밥깔기)

장점부터 소개해보자면,

1. 임계 구역 전체에 대해 원자성을 보장할 수 있는 유일한 방법이기 때문에,
임계 구역이 넓을 때, 로직이 복잡할 때 안전하게 사용하기 좋다.
   - 물론 안전성만큼 병렬성은 낮아질 수 있지만, 안전성은 확실하게 챙길 수 있다.
2. 단순히 임계구역에 들어갈 때 `lock()`
임계구역에서 나올 때 `unlock()`을 해주면 돼서
사용 난이도가 쉽다.
   - 직관적인 사용법으로 사용할 때의 안전성이 높아진다.
3. 올바르게 사용한다는 전제에서는,
`std::atomic`같은 복잡한 Lock-Free 설계보다 안전성을 확보하기 쉽다.
    
장점을 하나로 모아보자면,

> 사용이 직관적이고, 안전성이 좋다.

이거 하나로 요약할 수 있겠지.

사용하기 쉬우니,
잘못된 코드를 입력할 가능성도 (Lock-Free에 비해선) 크게 줄어들고,
사고가 터져도, 상대적으로 디버깅도 쉬운 편이야.

단점도 소개해보자면,

1. 락 경합이 발생하는 부분에서는
스레드가 계속 재우고 깨워져서 발생하는
컨텍스트 스위칭 오버헤드가 성능을 크게 깎아먹는다.
    - 앞에서 설명했듯이, 5000 ~ 10000클럭이 낭비된다.
2. 데드락 위험이 꽤 있다.
    - 뮤텍스는 락을 얻은 스레드가 직접 락을 해제하기 때문에
    여러 뮤텍스를 사용하는 복잡한 로직에서 락 획득 순서가 꼬이면
    서로가 서로가 필요한 락을 잡고 
    서로의 락을 기다리는 코드가 진행이 안되는
    데드락이 발생할 위험이 꽤 있다.
3. 값 하나를 보호할 때는 뮤텍스를 쓰기에는 무겁다.
    - 단순 카운터 / 플래그를 보호하기 위해
    스레드를 재우고 깨우는건 비효율적이다.
4. `std::lock_guard` 등의 RAII 객체를 쓰면 상당히 해결되지만,
`lock()` / `unlock()`을 사람이 직접 하는 경우가 있다보니
휴먼 에러로 인한 `lock()` / `unlock()`의 누락이 발생할 위험이 있다.
    
이 정도가 있겠네..

근데, 
결국 단점을 하나로 모아보자면,

> 컨텍스트 스위칭 오버헤드로 인한 성능 저하가 심하다.

> 변수 하나에 대한 연산에 뮤텍스를 사용하기에는 아깝다.

이정도가 가장 크게 보이잖아?

사실상 이게 어찌보면 뮤텍스의 최대 단점이야.

결국, 
성능이 중요하지 않은 곳에서는 닥치고 안전성 좋고 편리한 뮤텍스를 쓸텐데,

컨텍스트 스위칭 오버헤드로 인한 성능 저하를 
감당할 수 없을 정도로
성능이 중요한 곳이나,
~~(물론 뮤텍스를 쓰는 경우도 있기야 한데, 진짜 락 경합을 최대한 줄이는 방식으로 쓰지.)~~
뮤텍스의 컨텍스트 스위칭 오버헤드를 감당하기에는 
변수 하나만 보호하면 돼서 뮤텍스를 쓰기에는 아까운 경우에는,
다른 동기화 도구를 써.
바로 뒤 챕터에 있네.

## 3. Lock-Free 동기화 도구 : `std::atomic` 계열

### 3-1. 아토믹이란 무엇인가

아토믹, 즉 atomic operation(원자적 연산)이란,
1-2에서 설명한 원자성이 보장되는 연산이야.

그러니까 연산의 중간 과정이 외부 코어 / 스레드에 관측되지 않는 연산이라고 할 수 있지.

좀 더 이해하기 쉽게 말해보자면,
해당 스레드 / 코어 내부에서는 연산이 정상적으로 여러 중간 과정으로 실행되지만,
외부에서 관측했을 때는 오직 연산의 중간 과정이 보이지 시작과 끝만 보이는 연산이야.

실제 어셈블리상으로도

(x86_64 asm 기준)

- `lock xadd`(Fetch and Add)
- `lock cmpxchg`(Compare and Swap)

처럼 하나의 명령어로 해당 코어 외부에서는
시작과 끝만 보이는 형태로 실행돼.

그리고 연산의 끝에서는 오직 연산의 결과가 실패 / 성공으로만 나타나지.
물론 실패하면 재시도하거나 하고.

그리고 `C++`에서 `std::atomic`같은 클래스는,
원자적 연산이 가능한 변수를 만들고,
원자적 연산을 실행해주는 역할을 맡아.

역할을 맡는다고 하면 좀 이상하긴 한데,
나름 적절한 표현이라고 생각해.

왜 하나의 메모리 위치에만 원자적 연산을 수행할 수 있는지 의문이 들 수도 있는데,
그건 추후에 원자적 연산의 기본적인 하드웨어 구현인

- 캐시 라인
- 캐시 일관성 프로토콜(MESI)

같은걸 설명할 때 추가로 설명해볼게.
일단 지금은 

- 하나의 메모리 위치에만 연산을 원자적으로 만들 수 있다.

라고 기억해두고 있자. 일단은 그게 편할 듯.

그리고 원자적 연산 자체로는
원자성(atomicity)만 확실히 보장할 수 있을 뿐,
가시성(visibility)과 순서성(ordering)까지는 만족시킬 수가 없기 때문에,

단순히 

> 그냥 std::atomic 썼으니까 끝!
레이스 컨디션으로부터 안전함!

같은 느낌이 아니라,

> 가시성과 순서성이 확실히 보장돼서
레이스 컨디션을 확실히 막을 수 있을지 확실히 확인해보자.

라는 느낌으로 사용해야해.

물론, `C++`의 `std::atomic`에서는 
기본적 옵션으로는 가시성과 순서성까지 확실히 보장되는 옵션으로 `std::atomic`이 작동되어서
성능이 크게 필요하기 전까지는 그냥 순정 `std::atomic`을 사용해도 괜찮지만,

성능 최적화를 하기 시작하면
기본 옵션을 너머
가시성과 순서성을 좀 넉넉하게 푸는 옵션을 설정하게 돼.

거기서부터는 진짜 논리 싸움이고.

계속 가시성과 순서성이 만족되는지,
레이스 컨디션이 발생하지 않는지 생각해봐야 해.

그리고,
이런 원자적 연산이 어떤 곳에서 수행되어서
레이스 컨디션을 막는지에 대한 예시로,

- CAS(Compare And Swap)
- FAA(Fetch And Add)

이렇게 두 가지를 준비했어.
이거 실제로 `std::atomic`에서 메서드로 쓰임.

근데 이런 실제 `C++`의 메서드는 좀 나중에 보자.

- `std::compare_exchange_*()`
- `std::fetch_add()`

### 3-1-1. CAS(Compare And Swap)

> Compare And Swap : 비교 후 교체

> 바꾸기 전의 값과 비교해서 바꿀 곳의 값이 변하지 않았다면 바꾸는 것을
하나의 원자적 연산으로 처리하기

앞에 인용구로 설명한거 그대로야.

비교하고, 교체한다.

사실상 어떻게 보면 Lock-Free 동시성 프로그래밍의 근본같은 느낌이지.

일단 매커니즘을 대략 설명해볼게.

1. 수정할 메모리 주소에 저장되어있는 값을 연산 전에 저장해놓는다.
2. 연산을 수행한다.
3. 원자적 연산으로 이전에 저장해놓았던 값(Expected)과 현재의 값을 대입해보고,

    3-1. 만약 이전에 저장해놓았던 값과 현재 메모리에 있는 값이 같다면,
    해당 메모리 주소의 값이 이전과 같으니 연산을 진행하는 동안
    다른 실행 흐름이 해당 메모리의 값을 교체하지 않았다는 의미이므로
    레이스 컨디션이 발생하지 않기 때문에,
    해당 메모리 공간을 연산한 값으로 교체한다.   
    
    3-2. 만약 이전에 저장해놓았던 값과 현재 메모리에 있는 값이 다르다면
    다른 실행 흐름이 해당 메모리에 쓰기를 실행해서 값이 바뀌었다는 것이므로
    연산한 값을 그대로 교체하면 레이스 컨디션이 발생하기 때문에
    해당 CAS 연산을 실패 처리한다.    
    
    3-ex. 이 대입과 교체가 원자적으로,
    중간 과정을 다른 스레드가 관측할 수 없게 일어난다.
    
이 정도야.

결국,

- 연산 전에 저장해놓은 값과 일치하면 아무도 해당 메모리를 안 건드렸다는 뜻이니까 
현재 연산한 값이 유효한 값이라는 뜻이므로 값 교체.
- 연산 전에 저장해놓은 값과 다르면 누가 해당 메모리를 건드렸다는 뜻이니까
현재 연산한 값은 유효하지 않은 값이라는 뜻이므로 실패 처리.

이거야.

그리고 저렇게 

- 일치하는지 확인하는 과정
- 실제로 메모리의 값을 교체하는 과정

이 원자적으로 일어나는거야.

그리고 이 원자적 연산이
x86_64 어셈블리 레벨에서는

- `lock cmpxchg` (**C**o**MP**are and e**XCH**an**G**e)
    
라는 하나의 명령어로 나타내져.

말 그대로 CAS(Compare and Exchange)라고 할 수 있지.

그리고 이 CAS 연산은,
아주 치명적인 문제를 하나 가지고있어.

바로 ABA 문제라는건데,
이건 좀 이따 따로 챕터로 빼서 설명해볼게.

### 3-1-2. FAA(Fetch And Add)

> Fetch And Add : 인출 후 더하기

> 값 읽기 + 연산하기 + 쓰기를 한 번의 원자적 연산으로 처리하기

이건 정말 간단해.

앞에 쓴 것 대로

메모리에서 값 읽기(load) / 연산하기 / 메모리에 값 쓰기(store)를 
한 번의 원자적 연산으로 처리하지.

근데 이거 단점이,
단순 증감밖에 못한다는거?

더하기 / 빼기밖에 못함.
연산하는 시간이 길면 스핀락땜에 오버헤드가 커져서.
그래서 간단한 더하기 / 빼기 연산(증감 연산)밖에 못해.

- 논리 연산
- 비트 연산을 섞어서 하기
- 복잡한 로직의 연산(조건문을 포함한 연산 등)

등등은..
그냥 못해.

그런 것들은 CAS가 맡지.

단순 증감밖에 못하기 때문에,
카운터같은 느낌의, 공유 자원에 대한 연산을 할 때
주로 사용돼.

그리고 스마트 포인터(`std::shared_ptr`)의
ref count도 카운터니까 이 FAA 연산이 사용된다고 해.

어쨌든 이건 단순 증감밖에 필요없는 카운터에 되게 잘 맞아.

그리고 이렇게 load / 연산 / store 을 원자적으로 처리하는 원자적 연산을
x86_64 어셈블리 기준으로는

- `lock xadd`

라는 하나의 명령어로 나타내져.

### 3-2. 아토믹이 어떻게 원자성 / 가시성 / 순서성을 보장할까?

이건 하드웨어 ~ 소프트웨어의 여러가지가 
전부 얽혀서 돌아가는거라, 더럽게 어려워.

진짜 준비 단단히 하는게 좋음.

쉽게 말하면,

- 원자성 보장 : 캐시 일관성 프로토콜(MESI)을 이용한 캐시 락
- 가시성 보장 : 캐시 일관성 프로토콜(MESI)을 이용한 타 코어의 캐시로의 전파
- 순서성 보장 : CPU의 비순차적(Out-of-Order) 명령어 처리와
컴파일러의 코드 최적화가
코드의 순서를 뒤바꾸지(Reordering) 못하도록
메모리 배리어를 CPU(하드웨어) / 컴파일러(소프트웨어)에 전개

뮤텍스에 비해 짧아보이지?
근데 이거 더럽게 길다? 밑 챕터들을 보면 바로 체감이 될거야..
뮤텍스에서 설명 안하고 넘어간

- Cache Coherence(캐시 일관성) : 공유 변수 하나가 갑자기 분산 시스템이 됨
- MESI 프로토콜 : 각 코어의 L1 / L2 캐시들의 정치질을 4글자로 압축해서 돌아가는 프로토콜
- 메모리 배리어 : 인간의 기본적인 직관이 박살나는 지옥

다 설명하는거라..

지옥에 온걸 환영해!

이게 매운 정도는

```text
Cache Coherence(열라면 - 7500SHU) 
       ↓
     MESI(틈새라면 - 9413SHU)
       ↓
 Memory Barrier(염라대왕라면 - 21000SHU)
```

이런 느낌임.
메모리배리어에서 진짜 ㅈㄴ 어려워짐.
난 진짜 어렵게 했음..

### 3-2-1. Cache coherence(캐시 일관성)와 MESI 프로토콜

일단, 캐시 일관성 프로토콜이 어떻게 원자성과 가시성을 보장해주는지에 앞서,

캐시 일관성과 MESI에 대해 기초적인 부분부터 시작해보자.

왜 캐시 일관성 프로토콜이 필요할까?
왜 굳이 그렇게 MESI같은 규칙으로 관리를 해줘야 하는것일까?

일단 여러 코어는 같은 주기억장치(메모리)를 가지지만,
같은 캐시 메모리를 가지진 않아.

L1 / L2캐시는 코어별로 분리되어있지.

근데 내가 특정 값을 수정했는데,
다른 코어가 가지고 있는 캐시에는 반영이 안 되어있다면?

그러면 수정하기 이전 값을 가지고 다른 코어가 연산을 하게 되는 대참사가 벌어지겠지.

이게 가시성이 보장이 안되어 생기는 문제라고 볼 수 있어.

그래서, 그 캐시들의 상태를 일관성있게 맞춰주는게
캐시 일관성 프로토콜(MESI 프로토콜)이야.

내가 수정한 값이 다른 코어의 캐시와도 일관성있게 보이게 하기 위해서
이런 캐시 일관성 프로토콜을 사용해.

근데 솔직히 여기까지 보면 이게 왜 원자성 / 가시성과 관련되어있는지 모르겠지?

조금만 더 들어봐.

- 캐시 메모리에는 단순히 1바이트 단위로 메모리의 데이터가 캐싱되지 않고
64바이트 단위의 캐시 라인(Cache Line)이라는 단위로 캐시 메모리에 캐싱돼.
- 그리고 MESI 프로토콜도 그 캐시 라인 단위로 
해당 캐시 라인의 상태를 결정하는 것으로 동작해.

MESI 프로토콜은

- Modified : 수정한 데이터, 다른 코어에서는 갱신 필요, 메모리에는 안 올라왔을 수 있음(다른 코어에서는 Invalid 상태가 됨)
- Exclusive : 해당 코어의 캐시에만 반영되어있는 데이터, 메인 메모리의 값과 같음, 해당 코어만 가지고 있을 수 있음, Modified 상태로 전환 가능(다른 코어에서는 Invalid 상태가 됨)
- Shared : 모든 코어의 캐시가 같은 값을 가지고 있는 데이터, 다른 캐시가 그 데이터를 읽을 수 있음
- Invalid : 유효하지 않은 데이터, 다시 메모리에서 데이터를 가져와야 함

이렇게 네 상태를 기반으로 캐시의 일관성을 지켜줘.
그러니까, 하드웨어적인 캐시의 통신 규약이라고 할 수 있겠지.

캐시 락을 설명하기 위해 우리가 주목할 것은,
Exclusive와 Invalid, Modified야.

Exclusive는 해당 코어의 캐시에만 반영되어있고,
메인 메모리의 값과 같으며,
다른 코어가 건드릴 수 없어.
즉, 다른 코어의 해당 캐시 라인(데이터)은 Invalid 상태가 된다는거지.

이게 뭐다?
딱 해당 코어 / 스레드 외부에서 관측할 수 없는 
원자성을 보장하는데 쓰인다.

> 추가 : 사실 Exclusive로 해당 캐시 라인을 독점하고 
다른 코어의 캐시 라인을 Invalid 상태로 만든다고 해서
원자성이 보장되지는 않아. 
다른 코어에서는 여전히 메인 메모리 등에서 최신 값을 읽어오려 할 수 있거든.
만약 atomic operation 사이에 해당 값을 읽고 그걸 가지고 연산하게 된다면
원자성이 지켜지지 않을테고..
여기에는 추가로 atomic operation 중에 
해당 캐시 라인을 타 코어에서 읽을 수 없게 하는
캐시 릭을 따로 걸어.
결국 이게 atomic operation도 하나의 연산으로만 수행되지도 않다보니,
여러 단계에 거친 atomic operation(사실상 모든 atomic operation)은 
MESI 프로토콜만으로 원자성을 보장할 수 없다는게 핵심이야.
이게 내가 기존에 착각했던 부분인데,
MESI 프로토콜 = 가시성
캐시 락(MESI 프로토콜과 별도임) = 원자성
이렇게 보면 될 것 같아.
그러니까 MESI 프로토콜은 원자성을 완벽히 보장 못한다는거야.
결국 이 파트는 가시성 중심으로 읽어주면 좋을 것 같아.
이 파트는 수정하기 너무 힘들 것 같아서,
이렇게 주석 느낌으로 남겨놓았어.

결국 코드가 실행되는건 코어야.
그래서 해당 코어를 제외한 다른 코어의 캐시에
해당 데이터가 올라갈 수 없게 한다면,
원자성을 보장할 수 있게 되지.

그리고 연산을 시작할 때 해당 코어가 연산할 해당 메모리 위치의 캐시 라인을 
다른 코어에서는 Invalid로 전환하고,
해당 코어에서는 Exclusive 상태(물론 CPU 구현에 따라 다를 수 있음)로 전환해서
다른 코어가 캐시에 해당 캐시 라인을 캐시 메모리에 올려놓지(캐싱) 못하게 해.
그리고 연산이 끝나면 Modified로 전환해서 수정이 끝났다는걸 알리지.

Exclusive는 CPU 구현에 따라 다를 수 있지만,
어쨌든 어떤 CPU 구현에서도
```text
해당 메모리 위치(캐시 라인)에 대한 독점적 수정 권한 확보 -> 해당 메모리 위치에 쓰기 실행 -> Modified로 전환
```
이 순서로 실행된다고 보면 돼.

이러면 원자성을 보장할 수 있겠지?
다른 코어 / 스레드가 연산 중간을 관측할 수 없고,
연산의 시작과 끝만 관측할 수 있겠지.
물론 원자적 연산 중간에도 해당 값을 수정할 수 없을테고.
이게 원자적 연산(atomic operation)의 핵심 원리인 캐시 락이야.

(여담으로 예전에는 해당 메모리 버스를 잠가버리는 버스 락이라는 것도 쓰였는데,
버스를 잠가버리는건 너무 병목이 커서, 요즘에는 사실상 캐시 락을 사용한다고 보면 돼.)

그리고 연산이 끝났을 때 해당 코어의 해당 캐시 라인을 Modified 상태로 전환해서,
다른 코어가 해당 캐시 라인에 대한 캐싱(어떻게 보면 접근으로 봐도 됨)이 필요할 때,
해당 캐시 라인이 Invalid 상태이니,
무조건 최신 값을 다시 받아오게 되겠지.

그러면 이렇게 MESI 프로토콜로 가시성까지 보장할 수 있어.

### 3-2-1-1. false sharing 문제

일단 이 문제를 이해하기 위해서는,

> MESI의 각 상태는 1바이트 단위가 아닌 64바이트 단위의 캐시 라인 단위로 정해진다.

라는걸 잘 알고 있어야 해.

여기서부터 문제가 터지거든?

그러면 원자적 연산을 수행할 때,
해당 캐시 라인의 수정 권한을 독점하고(Exclusive) 연산을 수행하는건데,
그러면 다른 코어의 해당 캐시 라인이 Invalid 상태가 돼.

근데, 한 캐시 라인에 원자적 연산이 가능한 `std::atomic` 변수가
두 개 있다면 어떻게될까?

```cpp
struct A {
    std::atomic<bool> a;
    // 사이에 패딩이 없어서 같은 캐시 라인에 두 std::atomic 변수가 존재할 수 있음
    std::atomic<int> b;
};
```

한 번에 두 코어에서 그 캐시 라인을 독점할 수는 없잖아?

만약 방금 내가 말한 것처럼
한 캐시 라인에 원자적 연산이 가능한 `std::atomic` 변수가
두 개 있어서,
그 두 개의 변수가 계속 원자적 연산을 하게 된다면,

계속 각자의 캐시 라인을 독점하고,
계속 상대의 캐시 라인을 무효화하고..

이걸 계속 반복하게 되어버려서,
성능이 떡락해.

CPU는 a와 b가 다른 변수라고 생각하지 않고,
그저 같은 캐시 라인에 있는 데이터라고만 생각하거든.

마치 컨텍스트 스위칭 오버헤드와 비슷하지.

이게 결국 해당 캐시 라인을 독점해서 
다른 코어의 해당 캐시 라인에 대한 접근을 막는 것이 원자적 연산의 원리인데,
같은 캐시 라인의 `std::atomic` 변수 두 개가
같은 캐시 라인에 있어서
서로의 캐시 라인을 계속 무효화하게 되어서
양쪽 다 제대로 된 원자적 연산을 수행하기 어려워지는거지.

그리고 이게 진짜 성능을 심각하게 갉아먹어.

해결법은,
두 개의 `std::atomic` 변수의 사이를
`alignas()` 지정자를 사용해서 
같은 캐시 라인에 다른 변수가 올 수 없도록 하는거야.
캐시 라인은 보통 64바이트니,
`alignas(64)`이렇게 해주면 되겠지.

```cpp
struct A {
    alignas(64) std::atomic<bool> a;
    // 64바이트 단위로 정렬되어서 두 std::atomic 변수는 64바이트 떨어져 있게 됨. 
    // 즉, 다른 캐시 라인에 있음
    alignas(64) std::atomic<int> b;
};
```

물론 이게 절대적으로 캐시라인이 64바이트인건 아니고,
하드웨어에 따라 다를 수 있어.

~~(..근데 이거 프로젝트에서 `#pragma pack` 안 쓰고 이거 써도 되는거 아님?ㅋㅋ)~~
~~(아 오버 정렬만 되는구나.. `#pragma pack`은 기본 정렬보다 더 작게도 가능하고..)~~
~~(근데 전처리 지시문이랑 단순 키워드의 차이 때문에 솔직히 어쩔 수 없을 것 같긴 했어.)~~

그리고 이건 뮤텍스에도 적용돼.
뮤텍스도 결국 `lock()` / `unlock()`은 원자적 연산으로 진행되기 때문에,
여러 뮤텍스가 동시에 같은 캐시 라인에 있게 된다면,
똑같이 false sharing이 발생할 수 있어.

### 3-2-2. Memory Barrier(Memory Fence) & `std::memory_order`

이게 사실상 이번 글의 최종보스 아닐까?
이해하기나 설명하기나 정말 난해하거든..

일단 처음부터 차근차근 봐보자.

우리가 작성한 코드는,
사실 CPU에서 우리가 작성한 순서대로 하나하나씩 실행된다는 보장이 (일반적으로는) 없어.

우리가 작성한 코드는,
컴파일러와 CPU에게는 비효율적인 코드로 보일 수가 있거든.

그래서, 컴파일러와 CPU는

- 컴파일러 : 컴파일 과정에서(Analysis / Synthesis 둘 다 해당) 코드의 흐름을 재정렬
- CPU : 비순차적 명령어 처리(Out-of-Order Excution)로 비순차적으로 명령어를 실행

이렇게 우리가 쓴 코드를 최적화해줘.

뭐 깊게 파면

- Store Buffer(CPU) : 대충 CPU가 메모리의 쓸 데이터들 넣어놓는 버퍼
- Load-Store Queue(CPU) : 대충 CPU가 메모리에 접근할 순서를 관리하는 큐
- Store Forwarding(CPU) : 대충 파이프라인 멈춤(stall) 막아주는 기술
- Code Optimization(컴파일러) : 대충 컴파일러가 Analysis를 거친 IR을 가지고 해당 IR을 최적화하는 과정

이딴것들이 나오는데,
이런것들 하면 이게 컴퓨터 구조 / 컴파일러 글이 될 것 같아서,
딱 위의 저 두 가지만 알고 가자.

어쨌든 계속 이어가자면, 
컴파일러나 CPU는 우리가 쓴 코드의 멀티 스레드 관점에서의 스레드 간의 의존성은 몰라.
좀 더 정확히는, **동기화 수단으로 표현해놓지 않은** 스레드 간의 의존성은
컴파일러나 CPU가 보존해주지 않는다고 할 수 있어.
그저, 딱히 명시하지 않는다면 싱글 스레드 관점에서만 최적화를 해줘.

멀티 스레드가 아닌 싱글 스레드 관점에서
서로 연관이 없는 독립된 변수들에 대해서는
지멋대로 최적화를 하는거야.
멀티스레드 환경에서는 의존성이 있는 변수들이더라도.

그냥

> 어차피 내 입장에서는 싱글 스레드에서 결과만 똑같이 나오면 되니까,
좀 더 효율적이게 바꿔봐야겠다.

하고 지멋대로 실행 순서를 뒤바꿔..

근데 문제는?

- MESI 프로토콜을 이용한 원자적 연산은 오로지 
하나의 메모리 공간(하나의 변수)에 대한 연산만 원자적으로 만들 수 있음
  - 하나의 변수에 대해서는 무조건 원자성 / 가시성을 보장할 수 있지만,
  **여러 변수에 대한 연산의 순서성**은 
  **MESI 프로토콜을 이용한 원자적 연산만으로는 보장할 수 없음.**
  - 즉, 여러 변수가 얽힌 연산의 실행 순서성은 
  MESI 프로토콜만으로는 보장할 수 없음.
  - 컴파일러와 CPU의 명령어 재정렬은 
  **MESI 프로토콜보다 더 앞의 과정(컴파일러)과 뒤의 과정(CPU)**에서
  일어나기 때문임.
- 여러 변수에 대한 연산을 진행할 때, 특정 순서로 코드가 실행되어야만
레이스 컨디션이 발생하지 않는 경우가 있음(이게 순서성을 보장해야하는 이유)

이렇다는거야.

예시를 한번 들어볼게.
`std::atomic`에는 기본적으로 가장 강력한 메모리 배리어인
`std::memory_order_seq_cst`를 사용해서
Reordering이 발생하지 않지만,
순서성(Ordering)이 지켜지지 않으면 어떤 일이 발생하는지 예시를 들기 위해서
`std::memory_order_relaxed`를 사용해서
메모리 배리어를 없앴어.

```cpp
#include <iostream>
#include <atomic>

int data = 0;
std::atomic<bool> ready = false;

int thread_1(){
    data = 42;
    ready.store(true, std::memory_order_relaxed); // 예시를 들기 위해 기본 메모리 배리어 해제
}

int thread_2(){
    if (ready.load(std::memory_order_relaxed) == true) { // 예시를 들기 위해 기본 메모리 배리어 해제
        std::cout << data << '\n';
    }
}
```

스레드 1의 의도한 실행 순서는 

```text
data = 42 -> ready = true
```

이겠지만,
CPU나 컴파일러는 멀티 스레드의 실행에서의 의존성을 신경쓰지 않고,
어차피 `data = 42` / `ready = true`이라는 싱글 스레드에서의 결과만 똑같으면 되니,
두 대입 연산의 실제 실행 순서를

```text
ready = true -> data = 42
```

으로 뒤틀어버릴 수 있어.
두 변수 사이에는 의존성이 없으니
순서가 달라져도 된다고 생각하고 재정렬(Reordering)하는거야.

그런데, 
스레드 2에서는 `ready == true`을 확인 후
`data`를 출력해.

코드의 논리적으로는 스레드 1에서 `data -> ready` 순서로 바꿨으니 문제가 없어 보이지만,
앞서 말한 재정렬로 `ready -> data` 순서로 뒤틀려서 
우리의 머릿속으로는 "`ready`가 `true`로 변경되었으니 `data`도 변경됐겠지."
라고 생각이 들겠지만,
실제로는 `data`가 변경되기 전에 재정렬로 `ready`가 먼저 변경되어서
출력한 `data`값이 42가 아닌 0이 나올 수도 있는거야.

그리고, 이렇게 재정렬이 사람의 논리 / 직관을 거스르다보니
`std::atomic`을 사용한 동시성 프로그래밍이 어려운거고.

일단 지금까지는 이 정도 직관만 잡고 가도 충분해.

> 우리가 작성한 고급 언어의 소스 코드와
실제 CPU에서 실행되는 기계어는
실행되는 순서가 다를 수 있다.

> 그리고 그 원인은 
컴파일러의 코드 최적화와
CPU의 비순차적 명령어 처리(OoO Excution)에서 발생한다.
즉, 한 가지가 아닌 컴파일러와 CPU라는 두 가지 계층으로 인해 발생한다.

결국, 
위의 예시(생산자-소비자)같은 느낌의
우리가 의도한 대로 실행되어야 레이스 컨디션을 막을 수 있는 경우,
컴파일러 / CPU의 코드 재정렬을 각각 막아줘야해.

재정렬을 막아주는 기술이, 메모리 배리어(Memory Barrier / Memory Fence)야.

앞에서 말했듯이,
컴파일러 / CPU의 코드 재정렬을 각각 막아줘야 하기 때문에,
재정렬을 막는 메모리 배리어는 컴파일러와 CPU 두 계층에 나뉘어서 쳐져.

컴파일러에는 말 그대로 메모리 배리어를 기점으로
코드를 최적화할 때 재정렬을 하지 못하게 하고,

CPU에서도 비순차적 명령어 처리를 실행할 때 메모리 배리어를 기점으로
무조건 다 store해야지만 메모리 배리어 다음 코드를 실행할 수 있게 해.

그리고, 메모리 배리어에도 두 종류가 있어.
이게 어떤 방향으로 재정렬을 막을지에 대한 차이인데,
이 두 종류를 구별하는게 상당히 중요하다고 볼 수도 있어서.

물론 양방향 메모리 배리어도 있는데..ㅋㅋ
이건 생각보다 잘 안써.
특히 고성능 Lock-Free 자료구조같은거 구현할 때는..

실무에서는 이 두 가지를 구분해서 메모리 배리어를 쓰면서 코딩해.
이게 양방향 메모리 배리어보다 오버헤드가 상대적으로 낮아서..
그리고 아토믹을 쓰는 이유도 결국 성능 챙기려고 쓰는 경우가 많긴 해서.
오버헤드가 낮으니 단방향 메모리 배리어가 좋지.

이미 뮤텍스의 원자성 / 가시성 / 순서성 파트에서 대충 설명하긴 했는데,
이번에는 좀 자세히 설명해볼게.

- Acquire : 해당 메모리 배리어를 가지고 load(메모리 읽기) 한 이후의 메모리 연산들이
해당 load 이전으로 당겨지지 않게 막아준다.
- Release : 해당 메모리 배리어를 가지고 store(메모리 쓰기) 한 이후의 메모리 연산들이
해당 store 이후로 밀려나지 않게 막아준다.

근데 솔직히 이건 정말 비직관적이잖아?

좀 더 쉽게 설명해볼게.

- Acquire : 해당 메모리 배리어를 기점으로 뒷쪽이 앞쪽으로 당겨지지 않게 막기
  - 해당 메모리 배리어를 기점으로 **뒷쪽의 메모리 연산**들이 앞쪽에서 실행될 수 없음
  - 즉, '앞쪽의 메모리 연산이 다 끝나기 전의 뒤의 작업을 하지 마라.'
- Release : 해당 메모리 배리어를 기점으로 앞쪽이 뒷쪽으로 밀리지 않게 막기
  - 해당 메모리 배리어를 기점으로 **앞쪽의 메모리 연산**들이 뒷쪽에서 실행될 수 없음
  - 즉, '뒷쪽의 작업을 실행하기 전의 앞쪽의 메모리 연산들은 다 끝내놓아라.'

이렇다는거야.

예시를 보여주면,

(Acquire)
```text
          │  
          │  절대 아래쪽의 메모리 연산들이
          ▼  이 장벽 위로 기어올라갈 수 없음!
   ============================================= [WALL]
     if (ready.load(std::memory_order_acquire));
   ============================================= [WALL]
          │  장벽이 열렸으니 이제 안심하고
          │  최신 데이터를 읽어옴
          ▼  
   - std::cout << data;
```

(Release)
```text
          │
          │  하드웨어가 순서를 뒤바꾸다가도...
          ▼
   ============================================= [WALL]
     ready.store(true, std::memory_order_release);
   ============================================= [WALL]
          │
          │  절대 위쪽의 메모리 연산들이 
          ▼  이 장벽 밑으로 내려올 수 없음!
```

이게 결국 순서성을 보장할 부분이,
메모리 배리어로 보호할 부분이

- 해당 메모리 배리어 뒷쪽이 주체인가(Acquire)
  - 뒷쪽의 메모리 연산이 앞쪽으로 당겨지면 안되나
- 해당 메모리 배리어 앞쪽이 주체인가(Release)
  - 앞쪽의 메모리 연산이 뒷쪽으로 밀리면 안되나

에 따라 Acquire / Release가 갈린다고 보면 돼.

앞쪽의 코드를 수정해보면

```cpp
#include <iostream>
#include <atomic>

int data = 0;
std::atomic<bool> ready = false;

int thread_1(){
    data = 42;
    
    // release 배리어 : 위쪽의 data = 42가 해당 메모리 연산 아래로 밀리는 것을 방지
    ready.store(true, std::memory_order_release);
}

int thread_2(){

    // acquire 배리어 : 아래쪽의 data 읽기(출력하기 위해선 읽어야 하니까) 이 위로 당겨지는 것을 방지
    if (ready.load(std::memory_order_acquire) == true) { 
        std::cout << data << '\n';
    }
}
```

이렇게 되겠지.

### 3-2-2-1. happens-before 관계

추가적으로,
이렇게 acquire과 release로 만들 수 있는
happens-before 관계라는
멀티스레드에서 순서성이 보장되는 관계에 대해 설명해볼게.

이건 바로 위에 있는 수정한 코드의
acquire / release가 하는 **역할**에다가
이름을 붙인 거라고 할 수 있어.
(두 연산을 모아놓은 이름같은거 아님)

```text
(두 스레드는 시간순 아님, 그저 코드의 흐름)
   ---------------스레드 A----------------------------
          │
          │  절대 위쪽의 메모리 연산들이 
          ▼  이 장벽 밑으로 내려갈 수 없음!
   ============================================= [WALL]
     ready.store(true, std::memory_order_release);
   ============================================= [WALL]
          │  
          │  이 사이에는 store 전 연산들 / load 후 연산들이 들어올 수 없다.
          ▼  (synchronizes-with)
   ---------------스레드 B----------------------------
          │  (synchronizes-with)
          │  이 사이에는 store 전 연산들 / load 후 연산들이 들어올 수 없다.
          ▼  
   ============================================= [WALL]
     if (ready.load(std::memory_order_acquire));
   ============================================= [WALL]
          │  절대 아래쪽의 메모리 연산들이
          │  이 장벽 위로 기어올라갈 수 없음!
          ▼  
```

이렇게,
같은 `std::atomic` 변수에 대해
release-acquire로
서로의 연산 순서를 보장해주는거야.

방금 코드의 흐름이 딱 이거고.

설명을 위해 스레드 A / 스레드 B라고 적어놨는데,
실제로는 두 스레드가 어떤 스레드이든 상관이 없이,
같은 `std::atomic` 변수에 대해서 release로 store -> acquire로 load의 관계만 있으면 돼.

그러니까,
정말 간단하게(?) 말하면,

- 같은 `std::atomic` 변수에 대해서,
- 이전에 `release` 배리어를 이용해 store한 결과는
다른 스레드에서 해당 `std::atomic` 변수에 대해서 `acquire` 배리어로 load했을 때
`release` store 이전의 메모리 연산들은 전부 다 끝났다는게 보장되고(synchronizes-with)
- `acquire` 배리어로 load한 스레드에서는
load 이후에 코드에서는 load한 결과가 무조건 반영된다는게 보장된다.
  - 하지만 store 이전의 메모리 연산들이
  load 이후의 코드에 무조건 반영된다는 것이 보장되는 것은
  **acquire 배리어로 load한 결과가
  release 배리어로 store한 결과일 때만 성립**한다.

라는건데,

즉 앞의 순서도를 정리하면

```text
store하기 전의 메모리 연산들 → ready.store(release) → ready.load(acquire) → load 이후의 작업(메모리 연산)들
```

이런 순서가 무조건 보장된다는거야.

- store하기 전의 메모리 연산들이 release해도 반영이 안된다거나,
- load이후의 작업(메모리 연산)들이 load하기 전에 실행된다거나

하지 않는다는거지..

이런 순서가 보장되니,
acquire로 load할 때는 release 이전의 메모리 연산들이 적용되었다는걸 보장할 수 있겠지.

이렇게 멀티스레드 환경에서도 실질적으로 순서성이 보장되는거지.

**같은** `std::atomic` 변수에 대한 release와 acquire로.

그리고 `acquire`로 load를 했을 때 happens-before 관계가 보장되는 경우는
`release`로 store했을 때 뿐이고,
`release`의 특성이 없는 메모리 배리어(예 : relaxed)로 store 하면
store 하기 전의 메모리 연산들이 무조건 store 전에 모두 실행된다는 보장이 없기 때문에
happens-before 관계가 성립되지 않아.
release-acquire가 짝이라는거지.

그러니까, release로 store하고
acquire로 load해야만 happens-before 관계가 성립한다는거야.

어쨌든 이런걸 happens-before 관계라고 하고,
이건 같은 `std::atomic` 변수에만 적용되기 때문에,
다른 `std::atomic` 변수에 대한 release-acquire로 이어지는 관계는
멀티스레드 환경에서의 순서성이 무조건 보장되지는 않아.
이거 중요함. 이것 때문에 버그가 꽤 터져..
(물론 중간에 다른 `std::atomic` 변수를 거쳐서 순서성이 보장되는 경우도 있음)
(이걸 happens-before chain이라고 하는데, 이건 좀 더 난이도가 높은 개념인지라 이 글에서는 안다룸..)

조금 더 happens-before 관계에 대해 중요한 점을 나열해보자면,

- 무조건 release와 acquire가 짝으로 실행되어야 happens-before 관계가 성립된다.
  - release가 해당 `std::atomic` 변수에 store한 값을 acquire로 load해야만 성립된다.
- 이 happens-before 관계는 국지적으로만 적용될뿐, 전역적으로 적용되지 않는다.
  - 딱 해당 `std::atomic` 변수에 대한 store / load를 기점으로만 happens-before 관계가 성립된다.
  - 딱 해당 `std::atomic` 변수만 happens-before 관계가 성립하는 것은 아니고,
  store 전의 메모리 연산 / load 후의 메모리 연산들도 happens-before 관계가 성립한다.
- 어떤 스레드가 store / load를 했는지는 상관 없이 같은 `std::atomic` 변수에 대한 store / load라면 happens-before 관계가 성립한다.
  - 이 조건만 만족하면 `store 전 연산들 → store(release) → load(acquire) → load 후 연산들`의 실행 순서가 보장된다.

이 정도가 있겠네.

좀 중구난방이라 딱 핵심만 정리해보자면

> 같은 `std::atomic` 변수에 대해, release로 쓴 값을 acquire로 읽었다는 그 사실 하나만 있으면,
> 어떤 스레드가 썼는지(store)/어떤 스레드가 읽는지(load)와 무관하게
> `store 전 연산들 → store(release) → load(acquire) → load 후 연산들` 순서가 보장된다

이라고 할 수 있어.

### 3-2-2-2. `std::atomic`에서 사용가능한 메모리 배리어 옵션

마지막으로,
이렇게 `std::atomic`에 쓸 수 있는 메모리 배리어 옵션들을 나열해볼게.

- `std::memory_order_relaxed` : 메모리 배리어 없음
  - 진짜로 없음. 그냥 메모리 배리어가 없음.
- `std::memory_order_acquire` : Acquire 메모리 배리어
- `std::memory_order_release` : Release 메모리 배리어
- `std::memory_order_seq_cst` : 가장 강력한 메모리 배리어
  - `std::atomic`의 메서드에 아무 옵션도 안 넣었을 때의 기본값
  - 모든 스레드에서 전역적인 단일 실행 순서를 보장.
  - 즉, 존나 세다.
  - 그 대신 비용도 정말 높다.
  - happens-before 관계로 보장할 수 없는 전역적인 실행 순서를 보장할 때 사용한다.
- `std::memory_order_acq_rel` : Acquire과 Release를 합친 메모리 배리어
  - 말 그대로임. 
  - 하지만 전역적인 단일 실행 순서까지는 보장되지 않음
  
이런 옵션들도 당연히 비용이 있어.

높은 순서대로 오름차순 나열해보자면,

1. `std::memory_order_relaxed` : 배리어가 아예 없으니 사실상 비용이 없음
2. `std::memory_order_acquire` / `std::memory_order_release` : 단방향 배리어니 낮은 편
    - (일부 CPU 아키텍처(x86_64 등)는 공짜 수준)
    - (Reordering을 잘 안하는 CPU 아키텍처인 경우)
3. `std::memory_order_acq_rel` : acquire / release를 동시에 사용하니 조금 높은 편
4. `std::memory_order_seq_cst` : 양방향 메모리 배리어가 전역에 적용되기 때문에 진짜 ㅈㄴ 높은편
    - (앞의 것들과 비교할 수 없는 수준으로 높음)
    - 하지만 이게 디폴트 메모리 배리어라서,
    성능을 위해서는 다른 메모리 배리어 옵션들을 사용해야함

이게 기본 옵션인 `std::memory_order_seq_cst`는 정말 비용이 높아서,
성능이 중요한 곳에서는 웬만해선 순정 `std::atomic`을 사용하지 않아.

### 3-3. ABA 문제

원자적 연산하면 절대 빼놓을 수 없고,
빼놓아서도 안되는 문제야.

이 문제는 특히 CAS에서 문제가 돼.

ABA 문제란,
> CAS 연산을 수행할 때 값이 바뀐 적이 있음에도 불구하고
저장해놓았던 기존의 값과 새로 바뀐 값이 똑같으니
정상적으로 CAS 연산이 처리되는 문제

라고 할 수 있어.

예시를 들어보자면,

```text
A(여기서 연산할 값을 저장함) -> B(값이 다른 스레드에 의해 바뀜) -> A(다른 스레드에 의해 다시 원래의 값으로 돌아옴)
(하지만 저장한 값과 현재 값이 같으니 정상적으로 Swap됨)
```

이렇게 발생하는 문제야.

해당 메모리 주소의 값은 바뀌었지만,
CAS 연산을 수행하는 스레드는 바뀌었다는걸 알아채지 못하는거지.

말 그대로 A -> B -> A 문제야.
한 번 B로 바뀌었는데 Swap될 시점에는 다시 A로 바뀌어서
B로 바뀌었다는걸 모르고 연산한 결과를 해당 메모리 주소에 대입해버리는 문제.

이게 왜 문젠가 싶을 수도 있어.

사실, 이 문제는 단순히 해당 변수의 값만 저장하는 일반 변수보다
**포인터 변수**에서 크게 눈에 띄어.

일반 변수는 솔직히 바뀌고 원래에 값으로 돌아와도
크게 문제되지는 않아. 값의 의미는 깨지진 않으니까.

하지만 포인터 변수라면?

트리, 연결 리스트같은 것을 포인터를 사용해서 구현한다고 했을 때,
특정 노드가 삭제된 후 새 노드가 그 메모리 주소에 할당되었다면?

그 노드를 가리키는 포인터 변수의 값은 같은거잖아?

그래서 정상적으로 CAS 연산이 수행되는데,
문제는 원래 의도했던 이전에 삭제된 노드가 아닌,
새로 메모리에 할당된 노드에 대해 접근하게돼.

그 노드는 기존에 접근하려던 노드와 포인터 주소 값은 일치하지만
아예 다른 노드잖아?

아예 의미가 다른 노드에 접근하는거니까,
결국 그 연산의 의미는 망가지겠지.

이게 바로 ABA 문제의 위험성이야.

뒤의 두 챕터에서는 ABA 문제를 막기 위한 두 가지 방법인

- 태그드 포인터(Tagged Pointer)
- 해저드 포인터(Hazard Pointer)

에 대해 설명해볼게.

### 3-3-1. 해결책 - 태그드 포인터

> 포인터의 사용하지 않는 비트에 버전 기록하기

ABA 문제를 막기 위한 방법 중 하나인 태그드 포인터는,
바로

> 메모리 주소를 저장하는 포인터 변수는 (64비트 환경에서는) 64비트이므로 
실제 주소 공간 전체를 사용하지 않거나,
객체 정렬(64비트 기준 8바이트)에 따라 포인터의 하위 몇 비트가 항상 0이 되는 경우가 있다.

을 이용해서,

저렇게 주소를 표시하는 비트 이외에 남는 다른 잉여 비트들(예 : 하위 3비트)에다가
해당 `std::atomic` 변수를 이용한
CAS 연산을 할 때마다
버전을 하나씩 올리는거야.

물론 포인터 변수를 사용할 때는 비트 마스킹을 통해 
주소를 나타내는 부분만 잘라내서 사용하고.

그러면, CAS 연산을 하기 전에 저장하는 값에다가
버전까지 함께 기록해서,

```text
[A, 버전 : 1](CAS 전 저장) -> [B, 버전 : 2] -> [A, 버전 : 3](값이 같지만 버전이 다르므로 swap되지 않음)
```

이렇게 ABA 문제를 예방할 수 있지.

장점으로는,

- 성능적 이점 : 포인터랑 버전을 함께 저장해서 처리하니까
메모리 접근 횟수가 확연히 줄어듬
- 메모리 절약 : 하나의 포인터 변수에 속성값까지 넣어버리니
별도의 관리 객체같은걸 만들 필요가 없음

이 정도가 있겠고,

단점으로는

- 태그의 크기가 제한됨 : 포인터의 남는 잉여 비트들에 정보를 넣어야 하기 때문에
태그를 넣을 수 있는 크기가 제한됨
- 제한된 태그 크기로 인한 버전 오버플로우 : 태그에 쓸 수 있는 비트 수가 작으면
버전 값이 오버플로우되어 다시 돌아올 수 있고,
포인터 값도 어긋나는 상황이 발생할 수 있기 때문에,
수많은 변경 후에는 다시 ABA 문제가 발생할 수 있다.
- 구현이 복잡해짐 : 해당 포인터 변수를 읽고 쓸 때마다 속성이 저장된 비트까지 
읽고 쓰는 코드(비트 마스킹 등)를 추가해야함.

이 정도가 있겠네.

### 3-3-2. 해결책 - 해저드 포인터

> 해당 객체를 참조하고 있음을 다른 실행 흐름에 알려서 해당 객체의 소멸을 막기

- 일단, 이건 뭔가 단순한 ABA 문제가 아니라
ABA 문제로 인한 포인터와 객체의 소멸과 관련되어있는 내용이야.
- 그러니까, 포인터 변수와 객체 중심으로 전개되니까,
관점 조금만 바꿔서 봐줘.
- (변수 -> 포인터 변수)
- (변수의 값을 A에서 B로 변경 -> 객체 A를 메모리 해제 후 객체 B를 해당 객체의 주소에 할당)
- 이런 느낌으로.
- 어차피 같은 말이야.

앞서 설명한 태그드 포인터가 포인터 자체에 
버전이라는 값 변경 흔적을 남겨서 ABA문제를 해결한다면,
해저드 포인터는 그냥 아예 전역적으로
해당 메모리를 참조하고 있다는 것을 알려서
다른 스레드들의 해당 메모리 주소에 할당되어있는 객체의 소멸을 막아줘.

어떻게 보면 살짝 스마트 포인터(`std::shared_ptr`) 느낌도 나지?

근데 그건 control block의 ref count를 보고 마지막 `std::shared_ptr`이 소멸되었을 때에만
해당 객체를 소멸시키는 느낌이지만,
해저드 포인터는 좀 달라.

해저드 포인터는 전역적인 해저드 포인터 슬롯에
해당 객체를 참조하고 있다는걸 표시하고(해당 객체의 메모리 주소를 남겨놓음),

다른 객체가 해당 객체를 소멸시킬 때 그 해저드 포인터 슬롯을 보고
소멸시킬 객체의 주소가 해저드 포인터 슬롯에 있다면
다른 실행 흐름이 해당 객체를 참조하고 있다는 것을 파악해서
해당 객체의 소멸을 유예시켜.

ABA문제로 인한 
잘못된 객체(의도했던 이전의 객체가 아닌 새로 할당된 객체)를 참조하는 상황을
막을 수 있는거야.

그러니까, 정말 쉽게 설명하자면

> 해당 객체의 참조 여부를 나타내는 모든 스레드가 볼 수 있는 게시판을 두고,
해당 객체가 소멸될 때는 참조하고 있는 스레드가 있는지
확인을 한 후 소멸시킨다.

라고 볼 수 있어.

근데 이게 딱봐도 무거워보이지?
진짜로 무거우니까 함부로 쓰진 말자.

그리고 이건 실제로는 훨씬 복잡하고 어려워서
'이런 것도 있다'는 느낌으로 딱 여기까지만 쓸게..

실제로는 훨씬 복잡해.
이건 진짜 '이런 것도 있다'는 느낌의 파트야.

### 3-4. 아토믹의 단점 (= 뮤텍스 vs 아토믹 밑밥깔기)

이거 뭔가 단점이 없을 것 같지?

사실 이게 어찌보면 뮤텍스보다 훨씬 단점이 많은 것 같아.
하지만, 이걸 왜 쓰는지는 죄다 장점을 얻으려고 쓰는거야.

장점으로는,

1. 컨텍스트 스위칭을 발생시키는 뮤텍스에 비해
훨씬 가벼운 캐시 락을 사용하기 때문에 
뮤텍스에 비해 
사용할 때의 성능이 압도적으로 좋다.
2. 변수 하나의 연산에 대해
동기화가 필요할 때는
사실상 최강 수준의 안전성과 성능을 보인다.

이게 끝이야..
근데 솔직히 이게 상당히 압도적이야.

뮤텍스의 컨텍스트 스위칭 오버헤드 5000 ~ 10000 클럭에 비해,
캐시 락을 이용한 원자적 연산의 비용은 
적으면 20 ~ 40클럭,
캐시 미스 / 경합이 발생해도 50 ~ 150 클럭 정도야.
(물론 대략적인 수치일 뿐, 절대적인 수치는 아님. 하지만 뮤텍스와 비교해보면..)

그리고 하나의 변수만 동기화가 필요할 때는
성능과 비용 둘 다 챙길 수 있는 수준이고.

하지만..
단점이 어마무시하지.

1. 오직 한 변수만 무적 수준으로 보호할 뿐,
여러 아토믹 변수가 얽힌 로직의 경우에는 
해당 로직의 원자성 / 순서성이 보호되지 않을 수 있다.
    - 각 변수에 대한 가시성은,
    즉 타 코어의 캐시에 즉시 반영되는지에 대한 여부는
    MESI 프로토콜로 인해 보장된다.
    - 하지만 각 변수가 메인 메모리에 기록되는 순서와
    각 변수의 연산의 순서(순서성)는 보장되지 않는다.
2. 사용 난이도가 극단적으로 어렵다.
    - 변수 하나가 아닌 변수 여러개에 대해 동기화가 필요할 때만 해당되지만,
    변수 여러개에 대해 동기화가 필요할 때는 분명히 있다.
    - 그럴 때, 직접 메모리 배리어 옵션을 하나하나
    전부 논리적으로 설정해서,
    직접 논리적으로 해당 로직을 검증해야한다.
    - 게다가 멀티 스레드 환경이니 정말 검증하기 힘들다.
    - 역설적이게도, 뮤텍스보다 훨씬 좋은 성능을 얻은 대신에,
    사용 난이도가 극단적으로 어렵다.
    - Lock-Free 자료구조는 일반적인 자료구조에 뮤텍스를 사용하는 것보다
    압도적으로 구현이 어렵다.
3. 작은 실수가 큰 영향을 미치고, 디버깅이 극도로 어렵다.
    - 메모리 배리어를 하나 잘못 설정하면
    바로 로직이 전부 터져나간다.
    - 하지만 그게 항상 그러는게 아니고,
    몇 개월에 한번 발생하는 수준으로 터지기도 한다.
    - 게다가, 아토믹을 많이 사용한 코드는
    디버깅이 불가능한 수준으로 디버깅이 어렵다.
    - 디버깅을 하려고 관측을 하는 순간
    버그가 발생하지 않는 경우가 상당히 많다.
    (Heisenbug)(이게 매우 끔찍하다.)
    - 결국, 코드만 보고 디버깅을 하는게 아닌
    명제 / 술어 / 한정기호 등등을 사용해서
    논리적으로 증명을 하고, 반례를 찾아야 하는 수준이다.
4. 여러 변수에 대해서는 안전성이 극도로 낮다.
    - 뮤텍스처럼 단순히 아토믹만 쓴다고 해결되는게 아니라,
    직접 논리적으로 해당 구조의 무결성을 증명해야한다.
5. CAS 연산을 수행할 때의 라이브락의 위험성이 꽤 있다.
    - 특히 경합이 심할 때,
    한 번에 한 실행 흐름만 해당 메모리 주소에 swap하니
    많은 스레드에서 동시에 해당 메모리 주소에 CAS 연산을 수행할 때
    라이브락이 발생할 수 있다.
    - 이렇게 경합이 극도로 심한 변수는,
    MESI 프로토콜에 의해
    해당 캐시 라인 수정 권한이
    미친듯이 이동하고,
    그것이 오버헤드를 극도로 발생시켜서
    뮤텍스보다도 성능이 저하될 수 있다.
    (캐시 바운싱)
    
    
단점들이 어마무시하지..

쉽게 요약하면,

> 변수 여러개를 아토믹으로 동기화를 할 경우
원자성 / 순서성을 추가로 신경써야하고,
디버깅도 매우 어려워진다.

> 경합이 심할 때 오히려 뮤텍스보다 성능이 저하될 수 있다.

근데

- 한 변수에 대한 동기화에서는 안정성 / 성능 / 비용 전부 최고로 좋다.
- 여러 변수에 대한 동기화에서는 안전성이 바닥을 치지만,
성능 / 비용이 극도로 중요한 곳에서는 오직 성능 / 비용만 봤을 때는 뮤텍스보다 훨씬 낫다.

이런 특성 때문에,

아토믹은

- 플래그 / 카운터 등 **한 변수에 대한 동기화**만 필요한 경우
- 저런 리스크들을 감수할 필요가 있을 정도로
**성능이 무엇보다 압도적으로 중요한 경우**

에 주로 사용돼.

주로라고 했지만,
사실상 저 두개 제외하고는
보통 현업에서는
아토믹의 논리 구조를 짜는데 쓰는 시간이 너무나 아깝기 때문에,
아토믹을 거의 사용하지 않고, 뮤텍스를 사용해.

솔직히 생산성도 중요하니까 어쩔수 없지.

## 4. `std::mutex` vs `std::atomic`

이제 마지막으로 뮤텍스(`std::mutex`)와 아토믹(`std::atomic`)에 대한 비교를 해보자.

|특성|뮤텍스(`std::mutex`)|아토믹(`std::atomic`)|
|---|---|---|
|기본적인 동기화의 범위|임계 구역 전체|단일 메모리 공간(변수 하나 느낌)|
|실패 시 제어 방식|해당 실행 흐름을 blocking 상태로 전환(sleep-wait)|성공할 때까지 재시도(spin lock)|
|실제 동작이 일어나는 레벨|OS커널 / 스케줄러|CPU / 캐시 메모리(MESI 프로토콜) / 컴파일러(재정렬 방지)|
|주요 오버헤드의 원인|컨텍스트 스위칭 비용(유저 / 커널 모드 전환, 캐시 오염 등)|MESI 프로토콜(캐시 바운싱 등)|
|오버헤드|약 5000 ~ 10000클럭(락 경합 발생 시)|일반적으로는 약 20 ~ 150클럭(경합 심화시 폭등 가능)|
|최대 위험 요인|데드락(Deadlock)|라이브락(Livelock) / ABA 문제 / 하이젠버그(뮤텍스보다 훨씬 높음)|
|구현 및 사용 난이도|낮음|매우 높음|
|디버깅 난이도|중간|매우 높음|
|안전성|매우 높음|매우 낮음|
|특히 잘 사용되는 곳|I/O 작업이 포함된 임계구역, 긴 임계구역, 복잡한 로직 등|단순 플래그 / 카운터 변수, 극강의 성능 핫스팟 구간 등|

조금만 간단하게 마지막 설명을 해보자면,

뮤텍스는 락을 잡지 못하면 스레드를 그냥 잠재워버려서
물론 컨텍스트 스위칭 오버헤드가 발생하긴 하지만,
락을 못 잡고 잠들어있는 도중에는 오버헤드가 0에 가까워.

하지만,
아토믹은 swap을 못하면 성공할 때까지 루프를 계속 돌며
그냥 계속 연산을 갈기기 때문에
경합이 있다면 사실상 스핀락처럼 계속 CPU 자원만 먹어치워.(라이브락)
경합이 없을 때는 20 ~ 150 클럭 이내로 끝나지만,
경합이 미친듯이 심해지면 아토믹의 단점 파트에서도 말했듯이
MESI 프로토콜이 미친듯이 돌아가서(Cache Bouncing)
오히려 뮤텍스를 쓰는 것보다 성능이 더 안 좋아질 수도 있어.
(물론 false sharing 문제가 발생했을 때도 그렇고.)

그리고 디버깅 할 때,
아토믹은 뮤텍스에 비해 압도적으로 하이젠버그가 발생하는 비율이 높아.
이게 직접 메모리 배리어로 순서성을 논리적으로 지켜줘야하다보니까,

자동으로 순서성을 맞춰주는 뮤텍스에 비해
주로 순서성이 지켜지지 않아 발생하는 하이젠 버그가 많이 발생하는거지.

게다가 그런 버그가 진짜 몇 달에 한 번씩 터질 정도로
랜덤성도 심해..

하이젠버그(heisenbug)란,

### 4-1. 최종 결론

솔직히, 
이런 이론이 아닌 실제 개발에서
가장 비싼 자원은 CPU의 클럭 / 오버헤드같은게 아니라
사실상 개발자의 시간(생산성)이라고 생각해.

아무리 아토믹이 극강의 성능을 보여준다고 해도,
그 로직을 한 달 내내 짜면서 버그랑 씨름하면서 수학적 증명을 하는건
솔직히 생산성이 바닥이야..

그래서 솔직히 나는,
순수히 내 의견은,

성능이 중요한 곳에서도 진짜 아토믹 아니면 안되는 수준 아니면,
락 범위를 최대한 줄여도 해결이 안되는 수준 아니면,
최대한 생산성이 높은 뮤텍스(`std::mutex` / `std::lock_guard` 등)를 사용하는거야..

아토믹은 최대한 단일 변수에 대한 동기화 도구로 사용하고..

이게 내 결론이야.
