## 1. 정리와 증명

### 정리

1. 정리란?
    - 참이라고 증명된 명제
    
2. 공리란?
    - 참이라고 증명하지 않아도 참으로 간주하는 명제
    - 증명의 하한선이 되어준다.
    - 공리는 따로 약속되어있다.
    - 예시 )
      - 물론 공리의 예시는 공리계마다 달라질 수 있다.
      - 평행선 공리(평행선 공준): 직선 밖에 있는 한 점을 지나 그 직선과 평행한 직선은 단 하나뿐이다.
      - 동일성의 원리: 어떤 것과 같은 것들은 서로 같다.
      - 전체와 부분: 전체는 부분보다 항상 크다. 
      - 최초의 수: $1$은 자연수이다.
      - 후속자(계승자): 모든 자연수 $n$에 대해, 그 다음 자연수 $n^{\prime }$이 존재한다. 
      - 시작점의 유일성: $1$은 어떤 자연수의 후속자도 될 수 없다. 
      - 등등..
    - 정리는 밑바닥에서는 공리들로 인해 만들어진다.
    
3. 논증의 전제와 정리의 관계
    - 정리는 논증의 전제로 쓸 수 있다.
    
4. 논증과 정리의 관계
    - 논증은 공리 / 정리 등의 전제들로부터 결론을 도출하는 **과정**이다.
    - 정리는 공리 / 정리 등을 사용한 논증으로 도출된 **결론**이다.
    
5. 증명이란?
    - 증명은 어떤 명제가 참이란 것을 논리적으로 확실히 보여주는 과정이다.
    - 즉, 증명은 정리를 만들어내는 수단이다.
    - 즉, 논증과 증명의 차이는
      - 논증 : 전제들로부터 결론을 도출하는 과정
      - 증명 : 어떤 명제가 참인지를 논증을 통해 보여주는 과정 (논증보다 더 엄밀한 느낌)
    - 라고 할 수 있다.
    - 반대로 어떤 명제가 거짓임을 보이는 것은 보통 반례나 반증이라고 볼 수 있다.

### 느낀 점 & 배운 점

이번 파트는 확실히 증명 튜토리얼의 느낌이 강했어.

- 정리의 뜻 : 참이라고 `증명`된 명제
- 공리의 뜻 : 참이라고 `증명하지 않아도` `참으로 간주`하는 명제
- `정리`는 **`논증`의 `전제`**로 쓸 수 있다.
- `논증`은 `전제`로부터 `결론`을 `도출`하는 **`과정`**이다.
- 증명의 뜻 : 어떤 명제가 참임을 논증을 통해 논리적으로 **확실히 보여주는 `과정`**
- 논증과의 차이
  - 논증 : 그저 전제들로부터 결론을 도출
  - 증명 : 어떤 명제가 참이라는 것을 **논증을 통해 보여줌**

이렇게 단어들의 뜻을 좀 더 명확히 하는 느낌.

결국 이렇게 두 문장으로 요약할 수 있겠네.

> 증명(논증을 엄밀하게 쌓은 것)을 통해 정리(참인 명제)를 만들어낸다.

> 그렇게 만들어진 정리는 새로운 논증 / 증명의 전제가 될 수 있다.

## 2. 직접 증명 & 간접 증명

### 정리

1. 직접 증명
    - 직접 증명이란 전제($P$)가 참임을 바탕으로 논리적 단계를 거쳐 
    결론($Q$)이 참이라는 것을 직접적으로 도출하는 것이다.
    - $p \to q$에서 
    $p$(전제)가 참이라고 가정하고 공리 / 정의 / 정리 / 추론 규칙을 사용해서 
    $p$가 참일 경우에 항상 $q$(결론)가 참이라는걸 증명하는 것이다.
    - 즉, 전제들로부터 평범하게 결론을 도출해내는 것이다.
    - 간단해보이지만, 상당히 어렵다.
    - 예 :
      - '모든 정수 $x$에 대해서 $x$가 짝수이면 $x^{2}$도 짝수이다'라는 명제를 직접 증명한다.
      - $P(x)$ = $x$가 짝수이다. / $Q(x)$ = $x^{2}$가 짝수이다.
      - $\forall x (P(x) \to Q(x))$ ($x$는 정수)
      - 짝수의 정의 : $n = 2k$가 되는 정수 $k$가 존재한다면 $n$은 짝수이다.
      - $x = 2k$로 놓고 양변을 제곱하면 $x^{2} = (2k)^{2}$이다.
      - 즉, $x^{2} = 4k^{2} = 2(2k^2)$가 된다.
      - 여기서 $2k^{2}$는 정수이므로, $x^{2}$는 2의 배수이다. 즉, 짝수이다.
      - $\therefore$ 짝수의 정의에 의해서 $x$가 짝수이면 $x^{2}$도 짝수이다.
    - 이렇게 전제들에 의해서 결론이 참임을 직접적으로 도출하는 것을 직접 증명이라고 한다.
    
2. 간접 증명
   - 간접 증명이란 증명할 명제가 참임을 직접 증명하기 어려울 때,
   증명하기 쉬운 형태로 변형하거나 
   다른 성질을 이용해 
   해당 명제가 참이라는 것을 간접적으로 도출하는 것이다.
   - 간접 증명은 다양한 방법으로 시도될 수 있다.
   
3. 간접 증명 - 대우에 의한 증명
    - $p \to q$와 그 대우인 $\neg q \to \neg p$가 논리적 동치임을 이용한 간접 증명 방식이다.
    - 짧은 수식으로 표현하면, $(p \to q) \equiv (\neg q \to \neg p)$임을 이용한 증명 방식이다.
    - 명제 $p \to q$가 있을 때,
    그 대우인 $\neg q \to \neg p$를 증명한다면
    $(p \to q) \equiv (\neg q \to \neg p)$이기 때문에 $p \to q$도 참이라는 것이다.
    - 예 :
      - '$a$와 $b$가 양의 정수일 때 $n = ab$면 $a \leq \sqrt n$ 또는 $b \leq \sqrt n$이다.' 라는 명제를 간접 증명한다.
      - ($a$와 $b$는 양의 정수)
      - $P(a, b, n)$ = $n = ab$이다. 
      - $Q(a, b, n)$ = $a\leq \sqrt {n}$이거나 $b \leq \sqrt {n}$이다.($(a\leq \sqrt {n}) \lor (b \leq \sqrt {n})$)
      - $P(a, b, n) \to Q(a, b, n)$
      - 전제($P$)로 결론($Q$)을 도출하는 방법이 분명하지 않으므로 대우에 의한 증명을 시도한다.
      - $\neg Q(a, b, n) \to \neg P(a, b, n)$
      - $\neg Q(a, b, n)$ = $(\neg(a \leq \sqrt {n}) \land \neg (b \leq \sqrt {n}))$ = $(a > \sqrt {n}) \land (b > \sqrt {n})$
      - $\neg Q(a, b, n)$ = $a$와 $b$ 모두 $a > \sqrt {n}$이고 $b > \sqrt {n}$이다.
      - $\neg P(a, b, n)$ = $ab \neq n$이다.
      - 즉, $\neg Q(a, b, n) \to \neg P(a, b, n)$란 '$(a > \sqrt {n}) \land (b > \sqrt {n})$일 때, $ab \neq n$이다.'라는 의미이다.
      - $ab > \sqrt {n} \times \sqrt {n}$
      - $ab > n$
      - 즉, $\neg Q(a, b, n) \to \neg P(a, b, n)$가 참이므로 $P(a, b, n) \to Q(a, b, n)$도 참이다.
      - $\therefore$ $a$와 $b$가 양의 정수일 때 $n = ab$라면 $a \leq \sqrt n$ 또는 $b \leq \sqrt n$이다.
    - 이렇게 대우를 사용해서 결론($Q$)이 거짓이라면 전제($P$)도 거짓이라는 것을 증명하는 것을
    대우에 의한 증명이라고 한다.
      
4. 간접 증명 - 모순에 의한 증명
    - 어떤 명제 $r$을 증명해야할 때, $\neg r$을 가정하면 모순이 생긴다는 것을 증명해서
    $r$이 참이라는 것을 증명하는 간접 증명 방식이다.
      - 어떤 명제 $r$을 증명해야할 때,
      일단 $r$이 거짓이라고 가정한다.
      - 그리고 $r$이 거짓일 때 모순이 발생한다면,
      $r$은 거짓일 수 없다는 사실이 증명된다.
      - 그러므로 $r$은 참이 된다.
    - 예 : 
      - '$\sqrt {2}$는 무리수이다.'($p$)라는 명제를 간접 증명한다.
      - 모순에 의한 증명을 위해서 $\neg p$, 
      즉, '$\sqrt {2}$는 무리수가 아니다.'라고 가정해보자.
      - 즉, $\neg p$는 '$\sqrt {2}$는 유리수이다.'라고 할 수 있다.
      - $\sqrt {2}$가 유리수라면, 서로소인 두 정수 $a$, $b$가 존재한다는 의미이다.
      - (단, $b \neq 0$이라고 가정한다.)
      - $\sqrt {2} = \frac{a}{b}$
      - 양변을 제곱하면,
      $2 = \frac{a^{2}}{b^{2}}$
      - 양변에 $b^2$를 곱하면,
      $2b^{2} = a^{2}$
      - 즉, $a^{2}$는 $2$의 배수이므로 짝수이다.
      - $a^{2}$가 짝수라면 $a$도 짝수이다.
      - 즉, $a = 2k$라고 쓸 수 있다.
      - 이걸 $2b^{2} = a^{2}$에 대입하면,
      $2b^{2} = (2k)^{2}$
      $2b^{2} = 4k^{2}$
      - 양변을 $2$로 나눈다면,
      $b^{2} = 2k^{2}$
      - 즉, $b^2$도 $2$의 배수이므로 짝수이다.
      - 그리고 $b^2$가 짝수라면 $b$도 짝수이다.
      - 즉, $a$와 $b$는 둘 다 짝수이다.
      - 하지만, 처음에 $a$와 $b$는 서로소라고 가정했는데,
      서로소라는 것은 두 수가 $1$ 이외의 공약수를 가지지 않는다는 뜻이다.
      - 하지만 $a$와 $b$는 둘 다 짝수이기 때문에 공약수 $2$를 가진다.
      - 즉, $a, b \text{는 서로소이다.}$
      $a, b \text{는 둘 다 짝수이다.}$
      가 동시에 참이 되어버린다.
      - 이는 모순이다.
      - 따라서 처음 가정인 '$\sqrt{2}$는 유리수이다.'는 거짓이다.
      - $\therefore$ $\sqrt {2}$는 무리수이다.
    - 이렇게 원래 명제의 부정이 모순이라는 것을 증명함으로써
    원래 명제가 참이라는 것을 간접적으로 증명하는 것을 모순에 의한 증명이라고 한다.
      
### 느낀 점 & 배운 점

생각했던 것보다는 훨씬 어려웠어..

그리고 수학적 사고가 얼마나 어려운건지 확 체감이 되네..

사실 정리에서 쓴 증명들 책에 있던거 내 나름대로 해석해서 써본건데,
도저히 나 혼자서는 못하겠어서 지피티의 도움을 받았거든..

진짜 분량은 적었는데 난이도는 정말 미치겠더라..

하지만 뭔가 증명이 뭔지 확실히 감이 잡힌 것 같아.
결국, 
> 엄밀하게 논증하기

가 증명인 것 같아..

예를 들어 직접 증명으로 $p \to q$라는 명제를 증명해야할 때,
$p$가 참이라고 가정 후 정의 / 공리 / 정리 / 추론 규칙을 활용해서
$p$를 사용해 $q$가 참이라는 것을 도출한다던지..

간접 증명은 전제로 바로 결론을 증명하기 어려울 때,

(대우에 의한 증명)
$p \to q$를 증명하는 대신 동치인 $\neg q \to \neg p$를 증명해서
$p \to q$가 참이라는 것을 도출한다던지,

(모순에 의한 증명)
$r$을 증명하는 대신 부정인 $\neg r$이 모순이라는 것을 증명해서
$r$이 참이라는 것을 도출한다던지..

이런 식으로 $전제 \to 결론$으로 바로 직행하기 어려울 때 쓸 수 있다는 것..

프로젝트에 한번 응용해본다면,
> 내 코드가 특정 조건에서 항상 안전하게 동작한다는 것을 설명하는 방식

으로 쓸 수 있지 않을까..

만약 실제 프로젝트라면,
'정리' 자체가 '설계 불변식'

즉
> 특정 상태 조건이 만족된다면, 코드가 항상 안전한 동작을 한다.

> 이 위험 조건이라면, 이 코드 동작은 실행되지 않는다.

라는 명제가 돼.
$p \to q$.



예를 들어,
- Session이 closing 상태라면 SendPacket()을 호출하지 않는다.
- 페이로드 길이가 PAYLOAD_SIZE를 초과하면 정상 패킷으로 처리하지 않는다.
- 모든 SessionID는 고유해야한다.
- broadcast 중, 자신의 SessionID를 가지고있는 Session에는 SendPacket()을 호출하지 않는다.

이런 것들.
결국 이런 시스템 명세들도 다 명제니까..

해당 명제들이 코드에서 항상 참이라면,
내 코드는 정상적으로 동작한다는거지.

직접 증명은,
코드 흐름대로 안전성을 보일 때 쓰이겠지.

예를 들어,
> 패킷 헤더의 length 필드가 PAYLOAD_SIZE 이하이고, type이 정의된 PacketType이라면,
RecvPacket()은 해당 패킷의 페이로드를 수신할 수 있다.

라는 시스템 명세가 있다고 해보자.

그러면,
1. length <= PAYLOAD_SIZE라고 가정한다. (전제)
2. type이 정의된 PacketType이라고 가정한다. (전제)
3. RecvPacket()은 먼저 header를 읽는다. (코드 흐름)
4. length 검사를 통과한다. (코드 흐름)
5. type 검사를 통과한다. (코드 흐름)
6. 따라서 해당 패킷의 페이로드는 수신될 수 있다. (결론)

이렇게 코드 흐름을 따라 해당 코드가 시스템 명세대로 동작하는지 확인할 수 있어.

대우에 의한 증명은,
특정 조건이 성립하지 않을 때의 시스템 명세의 안전성을 보일 때에 쓰이겠지.
> 모든 세션에 대해 SendPacket() 이 정상적으로 실행되었다면, 해당 세션은 송신 가능한 상태이다.

라는 명제,
즉 $\forall session (SendPacketSuceeded(session) \to Sendable(session))$를

> 모든 세션에 대해 해당 세션이 송신 가능한 상태가 아니라면, SendPacket()이 정상적으로 실행되지 않는다.

라는 대우,
$\forall session (\neg Sendable(session) \to \neg SendPacketSuceeded(session))$로 바꿔서,
해당 시스템 명세를 더 쉽게 실제 코드와 연결해서 
특정 조건이 성립하지 않을 때의 시스템 명세의 안전성을 보일 수 있어.

모순에 의한 증명은,
불가능한 상태를 보일 때 쓰이겠지.
> 서로 다른 두 세션은 같은 세션ID를 가질 수 없다.

라는 명제를 증명해야할 때,
(전제)
> 서로 다른 두 세션 A, B가 같은 세션ID를 가진다고 가정한다.

(추론)
> 하지만 세션ID는 오직 accept() 직후에만 발급되고,
main 스레드를 제외한 다른 스레드에서 accept()하지 않으므로,
세션 ID를 발급하는 곳은 main 스레드의 accept() 한 곳 뿐이다.

> 세션ID를 지정하는 next_session_id는 한 세션에 세션ID가 발급된 후에 무조건 1 증가한다.

> 그런데 A와 B가 같은 세션ID를 가지려면, 
같은 next_session_id 값이 두 번 발급되어야 한다.

> 하지만 SessionID는 main thread의 accept() 흐름에서만 발급되고, 
발급 직후 next_session_id는 1 증가하기 때문에 
같은 next_session_id 값이 두 번 발급될 수 없다.

(결론)
> 따라서 서로 다른 두 세션 A, B는 같은 세션ID를 가질 수 없다.

이렇게 `불가능한 상태 -> 가능한 상태라고 가정`을 통해서,
가능한 상태가 발생할 수 없다는 것을 증명해서,
불가능한 상태임을 증명할 수 있는거야.
