## 0. 서론

저번에는 
- 증명의 정의 
: 특정 명제가 참임을 보이는 엄밀한 논증이자 정리를 생성하는 수단
- 직접 증명 
: 전제가 참임을 바탕으로 결론을 추론
- 간접 증명 
: 증명하기 쉬운 형태로 명제를 변환 / 성질을 이용해서 원래 명제를 간접적으로 증명
  - 대우에 의한 증명 
  : 원래 명제와 논리적 동치인 **대우를 증명**함으로써 원래 명제를 간접 증명
  - 모순에 의한 증명 
  : 원래 명제를 부정하고 **부정한 명제가 모순**임을 보임으로써 원래 명제가 참임을 간접 증명
  
즉, 증명이 대체 뭔지와
**기본적인 증명 방법**들에 대해 배웠었지.

**1.8 - 증명 방법과 전략**에서는
전에 배웠던 기본적인 증명 방법들에 이어서,
> 좀 더 심화적인 증명 방법들

이라고 할 수 있는

- 전수 증명 & 경우에 의한 증명
- 존재 증명 & 유일성 증명
- 증명 전략
- 반례

에 대해서 공부했어.

## 1. 전수 증명 & 경우에 의한 증명

### 정리

1. 전수 증명(exhaustive proof)
    - 모든 경우에 대해서 성립하는 것을 증명해야할 때,
    즉 $\forall$ 한정기호가 사용된 명제를 증명해야 할 때,
    증명해야할 경우, 즉 정의역의 원소의 개수가 적다면,
    정의역의 모든 원소에 대해서 증명함으로써 해당 명제가 참임을 보일 수 있다.
    - 경우의 수가 유한하고, 적을 때 사용한다.
    - 예 :
      - '$n$이 $n \leq 4$인 양의 정수일 때, $(n + 1)^{3} \geq 3n$이다.'라는 명제를 증명한다.
      - $n$은 $n \leq 4$인 양의 정수이다.
      - $P(n)$ : $(n + 1)^{3} \geq 3n$이다.
      - $n$의 범위는 $n \leq 4$인 양의 정수이므로, $1 \leq n \leq 4$의 정수이다.
      즉, 명제 함수 $P(n)$의 정의역은
      그저 $\{1, 2, 3, 4\}$로, 원소가 4개이다.
      - 즉, 모든 경우를 증명함으로써 해당 명제가 참임을 보일 수 있는 수준의 원소 개수이다.
      - 즉, 전수 증명이 충분히 가능하다.
        - $n = 1$인 경우, $8 \geq 3$이므로 참이다.
        - $n = 2$인 경우, $27 \geq 6$이므로 참이다.
        - $n = 3$인 경우, $64 \geq 9$이므로 참이다.
        - $n = 4$인 경우, $125 \geq 12$이므로 참이다.
      - 즉, 모든 정의역의 원소(모든 $n$)에서 $P(n)$는 참이다.
      - $\therefore$ $n$이 $n \leq 4$인 양의 정수일 때, $(n + 1)^{3} \geq 3n$이다.
    - 이렇게, 모든 경우에 대해서 명제가 성립하는 것을 증명해야할 때,
    정의역의 원소의 개수가 적다면
    모든 정의역의 원소에 대해서 해당 명제를 증명해서 
    해당 명제가 참임을 보이는 것을 전수 증명(exhaustive proof)이라고 한다.
    
2. 경우에 의한 증명(proof of cases)
    - 모든 경우에 대해서 성립하는 것을 증명해야할 때,
    정의역의 원소를 특정 성질에 따라 
    **전체 경우를 모두 고려할 수 있게** 나눈 후,
    각각의 경우마다 참임을 증명함으로써 해당 명제가 참임을 보일 수 있다.
    - 주의할 점은 나눈 경우들이 해당 명제의 모든 가능한 경우를 전부 고려해야 한다는 점이다.
    - 전부 고려하지 못한다면, 반례가 있을 수 있게 된다.
    - 예 :
      - '모든 정수 $n$에 대해 $n^{2} + n$은 짝수이다.'라는 명제를 증명한다.
      - 모든 정수 $n$에 대해, $n$은 짝수이거나 홀수이다.
      - $n$이 짝수인 경우  
        - $n$이 짝수라면, 어떤 정수 $k$에 대해서 $n = 2k$라고 쓸 수 있다.
        그렇다면, 
        $n^{2} + n = (2k)^{2} + 2k$
        $4k^{2} + 2k$
        $2(2k^{2} + k)$
        이고,
        여기서 $2k^{2} + k$는 정수이므로 $n^{2} + n$은 짝수이다.
      - $n$이 홀수인 경우
        - $n$이 홀수라면, 어떤 정수 $k$에 대해서 $n = 2k + 1$이라고 쓸 수 있다.
        그렇다면,
        $n^{2} + n = (2k + 1)^{2} + (2k + 1)$
        $4k^{2} + 6k + 2$
        $2(2k^{2} + 3k + 1)$
        이고,
        여기서 $2k^{2} + 3k + 1$은 정수이므로 $n^{2} + n$은 짝수이다.
      - 정수 $n$은 짝수이거나 홀수이고, 두 경우 모두에서 $n^{2} + n$이 짝수임을 보였다.
      - $\therefore$ 모든 정수 $n$에 대해 $n^{2} + n$은 짝수이다.
    - 이렇게, 모든 경우에 대해서 명제가 성립하는걸 증명해야 할 때,
    정의역을 몇 개의 경우로 모든 가능성을 고려할 수 있게 나눠서,
    모든 경우에 대한 명제를 증명함으로써 
    해당 명제가 참임을 보이는 것을 경우에 의한 증명(proof of cases)이라고 한다.
    
3. 전수 증명과 경우에 의한 증명의 차이
    - 전수 증명 : 가능한 값에 대해 전부 증명해서
    명제가 참임을 보인다.
    - 경우에 의한 증명 : 가능한 값을 몇 개의 묶음으로,
    모든 경우를 고려할 수 있도록 나눈다.
    각 묶음에서 명제가 참임을 보인다.

### 느낀 점 & 배운 점

이번에 배운 전수 증명과 경우에 의한 증명은
결국 
> 모든 경우에서 명제가 참이라는 것을 보이는 방법

이라는 점이 공통점인 것 같아.

하지만,
전수 증명은
- 정의역의 원소의 개수가 적을 때

사용하는게 좋을 것 같고,

경우에 의한 증명은
- 정의역의 원소들을 하나도 빠짐없이 몇 개의 묶음으로 나눌 수 있을 때

사용하는게 좋을 것 같은 느낌.

특히 경우에 의한 증명을 할 때,
꼭 모든 경우를 빠짐없이 다루는게 정말 중요할 것 같아.
하나의 경우라도 빼먹으면,
그렇게 빠진 경우에서 반례가 튀어나올 수 있을 것 같거든.

정리하자면,
> 전수 증명 / 경우에 의한 증명은
모든 경우에 대해서 참임을 증명해야 할 때
사용할 수 있는 증명법으로,
정의역을 어떻게 나눌지에 따라
맞는 증명법을 사용해야한다.

라고 요약할 수 있겠네.

프로젝트 관점에서는,
경우에 의한 증명을
- 코드의 상태 분기
- 에러 처리 흐름

등등을 검증할 때 쓸 수 있을 것 같아.

예를 들어, 
`RecvPacket()`이 패킷을 처리할 때 계속 통신을 할 수 없는 경우를 나눠본다면,

- 패킷 헤더의 length가 PAYLOAD_SIZE를 초과하거나 0 이하인 경우.
- 정의되지 않은 PacketType을 받은 경우.
- recv() 과정에서 Transport Error가 발생한 경우.
- 상대가 연결을 종료한 경우.

이런 식으로 나눠서,
각각의 경우마다 코드가 의도한 흐름대로 동작하는지 검증함으로써
예외 처리가 제대로 작동하는지 확인할 수 있고,
해당 조건들이 만족되지 않는 경우에 
계속 통신을 하는지 확인할 수도 있어.

어쨌든 핵심은
> 모든 경우에 대해서 명제 함수가 참임을 보여야할 때,
명제 함수의 정의역이나 가능한 상태를 여러 경우로 나눠서 증명한다.

인 것 같아.

## 2. 존재 증명 & 유일성 증명

### 정리

1. 존재 증명(existence proof)
    - 해당 명제가 어떤 한 경우라도 성립한다는 것을 증명해야할 때,
    즉 $\exists$ 한정기호가 사용된 명제를 증명할 때,
    성립하는 경우가 적어도 하나 존재한다는 것을 증명하는 것을 존재 증명이라고 한다.
    - $\exists x P(x)$의 존재 증명은
    $P(a)$가 참인 정의역의 원소인 $a$를 찾아내는 것으로 완료할 수 있다.
    이걸 구성적(constructive) 증명이라고 한다.
    - $\exists x P(x)$의 존재 증명은
    $P(a)$가 참인 정의역의 원소인 $a$를 찾아내지 않고,
    다른 방법으로, 간접적으로 $a$가 존재할 수 있음을 보이는 것으로도 완료할 수 있다.
    이걸 비구성적(nonconstructive) 증명이라고 한다.
    - 구성적 존재 증명 예 :
      - '양의 정수의 세제곱들의 합으로, 두 가지 다른 형태로 나타낼 수 있는 정수가 존재한다.'라는 
      명제를 구성적 존재 증명해보자.
      - $1729 = 10^{3} + 9^{3} = 12^{3} + 1^{3}$으로
      양의 정수의 세제곱들의 합으로, 두 가지 다른 형태로 특정 정수를 나타냈다.
      즉, 명제의 한 경우라도 성립한다는 것을 나타냈다.
      - $\therefore$ 양의 정수의 세제곱들의 합으로, 두 가지 다른 형태로 나타낼 수 있는 정수가 존재한다.
    - 비구성적 존재 증명 예 :
      - '$a^{b}$가 유리수가 되는 무리수 $a, b$가 존재한다.'라는 명제를
      비구성적 존재 증명해보자.
      - $a = \sqrt {2}$, $b = \sqrt {2}$인 경우를 생각해보자.
      - 즉, $\sqrt {2}^{\sqrt{2}}$를 생각해보자.
      - $\sqrt {2}^{\sqrt{2}}$가 유리수인지 무리수인지 알 수 없기 때문에,
      두 가지 경우로 나누어 생각할 수 있다.
        - 만약 $\sqrt {2}^{\sqrt{2}}$가 유리수인 경우 (가정)
          - 그냥 $a = \sqrt {2}$, $b = \sqrt {2}$로 둘 수 있다.
          - 즉, $a^{b}$가 유리수가 되는 무리수 $a, b$가 존재한다고 할 수 있다.
        - 만약 $\sqrt {2}^{\sqrt{2}}$가 무리수인 경우 (가정)
          - 새로운 $a = \sqrt {2}^{\sqrt{2}}$(무리수라고 가정했기 때문에), $b = \sqrt {2}$라고 두고,
          $a^{b}$를 계산하면 $(\sqrt {2}^{\sqrt{2}})^{\sqrt {2}} = \sqrt{2}^{\sqrt{2} \times \sqrt{2}} = \sqrt{2}^{2} = 2$가 된다.
          - 즉, $\sqrt {2}^{\sqrt {2}}$가 무리수라고 가정해도 
          결국 유리수가 되는 무리수 $a, b$가 존재한다는 것이 증명된다.
      - 결국 $\sqrt {2}^{\sqrt {2}}$가 유리수인 경우에도 무리수인 경우에도
      $a^{b}$가 유리수가 되는 무리수 $a, b$가 존재한다.
      - 즉, 두 경우 중 어떤 경우인지는 모르지만,
      두 경우 중 한 경우는 무조건 해당되기 때문에,
      적어도 $a^{b}$가 유리수가 되는 무리수 $a, b$는 존재한다.
      - $\therefore$ $a^{b}$가 유리수가 되는 무리수 $a, b$가 존재한다.

2. 유일성 증명
    - 해당 명제에서 딱 정의역의 원소 하나에 대해서만 성립한다는 것을 증명한다고 할 때,
    즉 $\exists !$ 한정기호가 사용된 명제를 증명할 때,
    그 한 경우를 찾아내고 해당 경우가 정말로 유일한지를 증명하는 것을
    유일성 증명이라고 한다.
    - 유일성 증명은 두 단계로 나뉠 수 있다.
    - 명제가 $\exists ! x P(x)$라고 했을 때,
      - 1. 해당 명제에서 어떤 한 경우라도 성립한다는 것을 증명한다.
        - $\exists x P(x)$에서 $P(a) = T$를 만족하는 $a$를 하나 찾는다.
      - 2. 해당 경우가 유일하다는 것을 증명한다.
        - 유일하다는 것을 증명하기 위해서는,
        해당 명제 함수($P(x)$)를 만족한다고 가정하는 임의의 원소($x$)를 잡고,
        해당 원소도 $a$와 같음을 도출하면 된다.
    - 즉, $P(x)$를 만족하는 두 원소 $x$, $y$가 있다고 가정한 후,
    조건을 이용해 $x = y$일 수 밖에 없음을 보여서
    $P(x)$를 만족하는 원소는 둘 이상일 수 없다는 것을 보이는 것이다.
    - 유일성 증명 예 : 
      - '$a$, $b$가 정수이고 $a \neq 0$일 때 일차방정식 $ax + b = 0$의 실수 해는 유일하다.'
      라는 명제를 증명해보자.
      - 일단, $a(-\frac{b}{a}) + b = - b + b = 0$이므로 $ax + b = 0$을 만족하는 실수 $x$가,
      즉 해가 존재한다.
      - 만약 일차방정식 $ax + b = 0$의 해가 $x_{1}$, $x_{2}$로 두 개 존재한다고 가정해보자.
      - $ax_{1} + b = 0$
      $ax_{2} + b = 0$
      이렇게 두 가지의 식이 나온다.
      - 양변을 빼면
      $(ax_{1} + b) - (ax_{2} + b) = 0 - 0$
      $a(x_{1} - x_{2}) = 0$
      $a \neq 0$이므로 $x_{1} - x_{2} = 0$이다.
      - 즉, $x_{1} = x_{2}$이므로,
      일차방정식 $ax + b = 0$의 해는 하나밖에 존재할 수 없다.
      - $\therefore$ $a$, $b$가 정수이고 $a \neq 0$일 때 일차방정식 $ax + b = 0$의 실수 해는 유일하다. 
    - 결국 $\exists ! x P(x)$를 증명한다는 것은,
    $\exists x (P(x) \land \forall y (P(y) \to (x = y)))$
    를 증명하는 것과 같다.
    - 명제가 참일 때의 정의역의 값과 비교해서 
    다른 경우를 찾아봐도 
    전부 기존의 명제가 참일 때의 정의역의 값과 같다면,
    결국 명제가 참이 되는 것은 오로지 해당 경우뿐이라는 논리이다.

### 느낀 점 & 배운 점

솔직히 이게 전수 증명 & 경우에 의한 증명보다 훨씬 어려운 듯..

전수 증명 & 경우에 의한 증명은 그나마 
> 정의역을 여러개로 나눠서 각각의 경우를 증명한다

라는 쉬운 원리라도 있었지,

존재 증명 & 유일성 증명은
그냥 처음부터 끝까지
생각만 해야하는.. 그런ㅋㅋ

(존재 증명)
그냥 $\exists x P(x)$를 증명할 때
$P(x) = T$인 $x$가 존재한다는 것을 알아내는 방법이
- 구성적 존재 증명 : $P(a) = T$인 정의역의 원소 $a$를 찾는다. (노가다 느낌)
- 비구성적 존재 증명 : 한 경우라도 $P(x) = T$일 수 밖에 없음을 증명한다. (하다가 머리 깨지는 느낌)

이렇게 된다는건데..

하.. 이게 무슨ㅋㅋ $P(a) = T$가 되는 정의역의 원소 $a$를 찾는 것도,
한 경우라도 $P(x) = T$일 수 밖에 없다는 것을 논리적으로 보이는 것도..

그냥 죽겠다 죽겠어 아주..

뭔가 직접 증명 vs 간접 증명을 보는 것 같아.

그냥 직접적으로 $P(x) = T$가 되는 경우를 보이는 
구성적 존재 증명과

바로 $P(x) = T$가 되는 경우를 보이지 않고 
한 경우라도 $P(x) = T$가 될 수밖에 없음을 간접적으로 보이는
비구성적 존재 증명..

직접 증명 / 간접 증명과 상당히 비슷하게 느껴졌어.

솔직히 내가 머리아팠던건 존재 증명보다 유일성 증명 쪽이었는데,

결국 
> $P(a) = T$인 $a$가 존재하고,
$P(x) = T$를 만족시키는 다른 경우들도 $a$밖에 없다는 것을 증명

해서, 딱 $a$만이 해당 명제 함수를 성립시킬 수 있다고 증명하는거였어.

정리 파트에서 쓴 것처럼,

1. 일단 $P(a) = T$를 만족시키는 $a$를 찾는다.
2. $P(x) = T$를 만족시키는 정의역의 다른 원소($x$)들이 
$a$와 같음을 보인다.

이렇게 증명을 할 수 있고,

논리식으로 쓰면

$\exists x (P(x) \land \forall y(P(y) \to (x = y)))$

이걸 증명을 하는 것 같아.

이 챕터도 솔직히 프로젝트에 충분히 연결해볼 수 있어.

예를 들어 
- 한 `ClientSession`의 송신이 진행될 때 
송신 임계구역에는 동시에 최대 한 스레드만 진입할 수 있어야 한다.
  - 해당 명세가 실제 코드 / 다른 명세와의 관계에서 유효한지 증명할 때는 유일성 증명을 사용할 수 있다.
- 현재 접속 중인 클라이언트 중 송신자를 제외한 세션이 하나 이상 있다면, 
broadcast 대상 세션이 존재한다.
  - 해당 명세가 실제 코드 / 다른 명세와의 관계에서 유효한지 증명할 때는 존재 증명을 사용할 수 있다.
  
이런 예시들이 있겠네.

## 3. 증명 전략

### 정리

1. 전향 추론(forward reasoning) 
    - 전향 추론이란, 전제로부터 결론을 도출하는 것이다. 
    - 증명의 시작점을 전제로 정한다.
    - 직접 증명에 매우 가깝다.
    - 즉, 전제에서 출발해서 결론으로 나아가는 방식이다.

2. 후향 추론(backward reasoning)
    - 후향 추론이란, 결론을 증명하려면 무엇이 필요할지 거꾸로 생각해서
    전제를 도출하는 것이다.
    - 증명의 시작점을 결론으로 정한다.
    - 어떻게 보면 결론의 부정으로부터 전제의 부정을 도출하는 대우에 의한 증명과 비슷하다.
    - 즉, 결론에서 출발해서 결론을 만족할 전제를 찾아가는 방식이다.
 
3. 증명법 선택 기준
    - $p \to q$ 꼴(조건문)이면 직접 증명부터 생각한다.
      - 직접 증명이 어려우면 우선적으로 대우에 의한 증명을 고려한다.
      - 명제의 부정을 가정했을 때 모순이 발생할 것 같다면 모순에 의한 증명을 고려한다.
    - $\forall x P(x)$꼴(전체 한정기호)이라면 경우에 의한 증명 / 전수 증명을 고려한다.
      - 정의역의 원소의 개수가 적으면 전수 증명을 고려한다.
      - 가능한 경우가 나뉠 수 있으면 경우에 의한 증명을 고려한다.
    - $\exists x P(x)$꼴(존재 한정기호)이라면 존재 증명을 고려한다.
      - 직관적으로 $P(x) = T$인 경우를 찾을 수 있을 것 같다면 구성적 존재 증명을 고려한다.
      - 직관적으로 $P(x) = T$인 경우를 찾을 수 없을 것 같다면 비구성적 존재 증명을 고려한다.
    - $\exists ! x P(x)$꼴(유일 한정기호)이라면 유일성 증명을 고려한다.

### 느낀 점 & 배운 점

이번 파트는 솔직히 막 새로운 증명법을 알려준다는 느낌보다는,
> 어떤 상황에 어떤 증명을 우선적으로 시도해보는 것이 좋은가

에 대한 내용인 것 같아.

증명해야하는 명제를 봤을 때,
그냥 무작정 식을 전개하는게 아니라

> 이건 $p \to q$ 형태니까 우선적으로 직접 증명을 시도해보자.

> 이건 모든 경우에 참이 되어야 하는데 정의역의 원소의 개수가 너무 많으니
경우에 의한 증명을 시도해보자.

> 이건 한 경우라도 참인 경우가 존재해야하니 존재 증명을 시도해보자.
하지만 직관적으로 봤을 때 
직접적으로 참인 경우를 찾을 수 없을 것 같네. 비구성적 존재 증명을 시도해보자.

이런 느낌으로 해당 명제의 조건에 맞춰서 알맞은 증명법을 선택하는 방법을 알려주는 것 같았어.

프로젝트와 연결해보자면,
버그를 고치거나 설계를 검증할 때, 
무작정 코드를 보고 설계대로 작동하고 있는지 무작정 위에서 아래로 읽는게 아니라,
- 일반적인 경로를 따라갈지 (직접 증명)
- 에러 케이스를 나눠서 생각해볼지 (경우에 의한 증명)
- 불가능한 상태를 먼저 가정해볼지 (모순에 의한 증명)

이렇게 각각의 필요에 맞게 선택을 하는거지.

## 4. 반례

### 정리

1. 반례
    - 어떤 명제가 항상 참이 아니라는 것을 보여주는 예시이다.
    
2. 반례가 중요한 경우
    - 전체 한정기호($\forall$)가 붙은 명제
    - 즉, "모든 ~에 대해 성립한다."는 조건이 있는 시스템 명세 (즉, $forall$이 붙은 명제)
    
3. 반례가 $\forall$이 붙은 명제에 끼치는 영향
    - 수많은 경우에서 해당 명제함수가 참이 될 수 있더라도,
    단 하나의 반례가 있다면 해당 명제는 거짓이 된다.
    - 즉, 전칭 명제($\forall$)에서의 반례는
    해당 명제의 참 / 거짓 여부를 결정짓는다.
    - 몇 개의 예시가 참이라고 해서 전체 명제가 참이 되지는 않는다.
    - 하지만 하나의 반례를 찾게 된다면, 해당 명제가 거짓임을 보일 수 있기 때문에
    $\forall$이 붙은 명제에서는 반례의 존재 여부가 정말 중요하다.
    
4. 예시 
    - '모든 정수 $n$에 대해 $n^{2} + n + 41$은 소수이다.'라는 명제의 반례를 찾아보자.
    - 반례 : $n = 41$이라면,
      - $41^{2} + 41 + 41 = 41(41 + 1 + 1) = 41 \times 43$이므로 
      소수가 아니게 된다.
      - 따라서 해당 명제는 거짓이다.

### 느낀 점 & 배운 점

반례는 뭔가 지금까지 배운 증명법이랑은 좀 다른 것 같고,
> 어떤 명제가 항상 참이 아니라는 것, 즉 거짓인 경우가 있다는 것을 증명하는 방법

에 가까운 것 같아.
특히 
'어떤 명제에 거짓인 경우가 있을 때' 아예 참 / 거짓이 달라지는
전체 한정기호($\forall$)가 포함된 명제의 증명을 할 때 정말 중요하다고 느꼈어.

전체 한정기호가 포함된 명제는,
아무리 많은 경우에서 해당 명제가 참이 되더라도,
단 하나의 경우에서 명제가 거짓이 되면
'모든 경우에서 참이다'라는 전체 한정기호의 의미에 따라
해당 명제가 거짓이 되니까..

그러니까,
> 여러개의 예시가 참이라고 해서,
해당 전칭 명제가 참이라고는 단정할 수는 없지만,
하나의 반례만으로는 전칭 명제가 거짓임을 증명할 수 있다.

라는거지.

프로젝트 관점에서는
> 내가 놓친 예외 상황(edge case)

을 찾는 것과 반례를 찾는 것이 정말 비슷하다고 느꼈어.

예를 들어, 
> 이 함수는 항상($\forall time$, 모든 시간에서) 예외를 발생시키지 않는다.

라는 시스템 명세가 있을 때,
해당 명세를 깨는 입력이나 상태가 단 하나라도 존재한다면 
해당 명세가 참이라고 볼 수 없어.

그래서, 결국 if문 / 예외 처리 / 뮤텍스 등등에 의존하지 않는
**논리적인 무결성**을 가지게 하려면,
항상 실제 코드 흐름과 실제 설계에서 해당 시스템 명세에 대한 반례를 꼭 찾아봐야 할 것 같아.

## 챕터를 마치며

이걸로 논리와 증명 챕터는 끝이네.

처음에 비하면 정말 많이 실력은 늘은 것 같아.

처음엔 대략적인 명제의 개념과 기초적인 논리 연산자정도밖에 몰랐는데,
- 역 / 대우 / 이 
: 대우라면 원래 명제와 동치이기에 바꿔 적어도 문제가 없지만, 
역 / 이는 원래 명제와 동치가 아니므로 바꿔 적을 시 논리적으로 문제가 발생할 수 있다.
- 논리적 동치
: 논리적으로 동치인 명제라면 바꿔 적어도 문제 없다.
- 만족 가능성
: 해당 명제가 참인 경우가 단 하나라도 있는지
- 술어
: 변수에 따라 참 / 거짓이 나뉘는 명제
- 한정 기호
: 술어(명제 함수)의 각 변수에 따른 결과를 하나의 진리값으로 묶어주는 기호
- 중첩된 한정기호
: 순서에 따라 의미가 달라진다.
$\forall x \exists y$ : 모든 $x$에 대해서 $P(x, y)$가 참인 $y$가 적어도 하나 존재한다.
$\exists x \forall y$ : $x$ 중에 하나라도 모든 $y$에서 $P(x, y)$가 참인 경우가 적어도 하나 존재한다.
- 논증
: 전제로 결론을 추론하는 것
- 추론 규칙
: 추론 과정에서 쓰일 수 있는 규칙
- 증명 / 정리 / 공리
: 정리는 증명으로 만들어진다
- 직접 증명 / 간접 증명
: 전제로 결론을 증명 / 해당 명제를 변형하거나 성질을 이용해서 결론을 증명
- 한정기호가 포함된 명제의 증명
$\forall$ : 전수 증명 / 경우에 의한 증명
$\exists$ : 구성적 존재 증명 / 비구성적 존재 증명
$\exists !$ : 유일성 증명(존재 증명 + 유일성 보장)

까지..

뭔가 별거 안한 것 같아 보였는데 적어보니까 많은 것 같아.

특히
> 내 설계 명세를 명제로 옮기고, 
해당 설계 명세가 
다른 명세와 충돌하지는 않는지,
코드에서도 옳은지 수학적으로 확실히 증명하는 감각

은 확실히 얻은 것 같아.
